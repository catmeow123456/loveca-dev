# 3.7.2 -> 3.7.3 发布迁移说明

> 文档类型：版本迁移说明
> 适用范围：从 `v3.7.2` 或仍使用 v1 原始 bcrypt 凭据的生产环境发布 `v3.7.3`
> 权威通用流程：`docs/production-release-runbook.md`

## 1. 变更与生产影响

- 本次没有新增 Drizzle schema SQL；仍应在停机窗口执行 `pnpm db:migrate`，确认生产 migration 状态与发布提交一致。
- 认证凭据切换改为无损兼容路径：可识别的原始 bcrypt 摘要会被包裹为 `$loveca-bcrypt-raw$...`，用户首次使用原密码成功登录后再升级为当前 `$loveca-bcrypt-sha256$...` 格式。
- 认证数据脚本会删除全部 refresh token、邮箱验证 token 和密码重置 token。发布后所有设备都需要重新登录，旧验证和重置链接失效。
- `sync-cards-loveca-excel.ts` 现在以 Excel `カードタイプ` 或 CloudBase `loveca.type` 为卡牌类型权威来源，并支持将 `double` 展开为两枚 `GRAY` 判心。
- `sync-cards-cloudbase-new.ts` 同样支持 `double`，且只接受 CloudBase `type` 作为卡牌类型来源。
- 本次发布不要求重跑卡牌同步脚本，也不修改对象存储。只有计划把新来源数据或 BP7 `double` 数据写入生产卡牌库时，才按第 4 节另开数据同步窗口。

如果生产环境已经执行过原 `v3.7.2` 的 reset-only 认证脚本，原始 bcrypt 摘要已经被覆盖，`v3.7.3` 无法从 reset-required 标记恢复旧密码。此时必须选择从执行前 Postgres dump 恢复后重新迁移，或明确执行逐账号密码恢复；不得直接绕过 `resetRequiredPasswordCount` 阻断。

## 2. 发布前置与停机顺序

1. 确认 API 镜像、前端产物、迁移脚本和本说明来自同一个 `v3.7.3` 提交。
2. 确认生产环境变量已准备，尤其是 `DATABASE_URL`、两个不同的 32 字节以上 JWT secret、对象存储配置和 `FRONTEND_URL`。
3. 停止旧 API、队列及所有会写入用户、认证 token 或卡牌数据的任务。
4. 完成 Postgres dump，记录并验证恢复命令；如本次另行执行卡牌同步，也要确认对象存储备份或快照。
5. 在具备源码、devDependencies、Node、pnpm 和生产 `DATABASE_URL` 的发布环境执行迁移。API runtime 镜像只含运行产物，不能执行 Drizzle CLI 或 TypeScript 数据脚本。
6. 执行 `pnpm db:migrate`，再按第 3 节完成认证 dry-run、apply 和 SQL 验证。
7. 认证 apply 成功后立即部署同提交的 `v3.7.3` API 与前端，不得重新启动 `v3.7.2` API。

## 3. 认证凭据数据迁移

先执行只读检查并保存报告：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx \
  drizzle/data-migrations/auth-v1-to-v2-credential-cutover.ts \
  --dry-run \
  --report=tmp/release-3.7.3/auth-v2-dry-run.json
```

只有报告满足 `resetRequiredPasswordCount = 0` 且 `unsupportedPasswordCount = 0` 时才能继续。`legacyBcryptPasswordCount` 表示将被兼容包裹的凭据数量，不是需要重置密码的人数。

确认停机、备份、dry-run 数量和当前发布提交后执行：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx \
  drizzle/data-migrations/auth-v1-to-v2-credential-cutover.ts \
  --apply \
  --yes \
  --report=tmp/release-3.7.3/auth-v2-apply.json
```

保存两份报告并核对 before / after 数量。脚本会在一个事务中包裹旧 bcrypt 摘要、删除三类 token，并在提交前验证 postcondition。不得增加绕过 reset-required 或未知凭据的参数。

迁移后执行等价 SQL 验证：

```sql
SELECT count(*) FILTER (WHERE password_hash LIKE '$loveca-bcrypt-sha256$%') AS current_v2,
       count(*) FILTER (WHERE password_hash LIKE '$loveca-bcrypt-raw$%') AS compatible_legacy,
       count(*) FILTER (WHERE password_hash ~ '^\\$2[aby]\\$') AS unwrapped_legacy,
       count(*) FILTER (WHERE password_hash = '$loveca-password-reset-required$v1') AS reset_required,
       count(*) FILTER (
         WHERE password_hash NOT LIKE '$loveca-bcrypt-sha256$%'
           AND password_hash NOT LIKE '$loveca-bcrypt-raw$%'
           AND password_hash <> '$loveca-password-reset-required$v1'
           AND password_hash !~ '^\\$2[aby]\\$'
       ) AS unsupported
FROM users;

SELECT (SELECT count(*) FROM refresh_tokens) AS refresh_tokens,
       (SELECT count(*) FROM email_verification_tokens) AS verification_tokens,
       (SELECT count(*) FROM password_reset_tokens) AS reset_tokens;
```

预期 `unwrapped_legacy = 0`、`reset_required = 0`、`unsupported = 0`，三类 token 均为 0。完整认证切换语义以 `drizzle/migration-notes/auth-v1-to-v2-credential-cutover.md` 为准。

## 4. 卡牌同步口径与可选操作

本次升级本身无需重跑卡牌同步。生产现有卡牌记录会继续保留；只有发布者明确要同步新来源数据时才执行以下步骤。

当前发布准备使用的候选源文件为
`docs/card-data-sync/sources/loveca_20260722221351.xlsx`，SHA-256 为
`f7cd5a774400eaa59e1715a50b5e41b803420692d873bfabf45bb733350812e5`。该目录不进入仓库；
生产执行前必须重新提供并核对同一文件，不要依赖未知或临时来源。

先使用经人工确认的源文件做 dry-run：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx \
  src/scripts/sync-cards-loveca-excel.ts \
  --xlsx=docs/card-data-sync/sources/loveca_20260722221351.xlsx \
  --dry-run
```

人工审核至少包括：

- `カードタイプ / DB card_type corrections` 清单是否符合预期。
- 缺失或非法卡牌类型导致的 skipped 行是否为 0，或已有明确处理结论。
- `double` 是否只在预期卡牌上展开为两个 `GRAY` Heart。
- source-only、DB-only、冲突字段和 transform warning 是否均有处理结论。

审核通过后才能在维护窗口正式执行：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx \
  src/scripts/sync-cards-loveca-excel.ts \
  --xlsx=docs/card-data-sync/sources/loveca_20260722221351.xlsx \
  --yes
```

`--yes` 会写入生产卡牌数据，必须保留 dry-run 输出并再次确认源文件路径。CloudBase-only 新卡导入、图片上传和对象存储变更不属于本次默认发布动作；若另行执行，按 `docs/card-data-sync/cloudbase-new-card-sync.md` 独立 dry-run、人工审核并选择明确的图片策略。

## 5. 同版部署与发布后 smoke

认证迁移完成后，API 与前端必须同版部署；旧 API 不能读取兼容前缀凭据，不允许连接迁移后的生产数据继续写入。

除生产 runbook 的容器、日志、health、首页、manifest、图片代理和联机房间检查外，至少验证：

- 迁移前的原始 bcrypt 测试账号可用原密码登录，并获得当前 v2 refresh Cookie。
- 该账号首次成功登录后，`password_hash` 从 `$loveca-bcrypt-raw$...` 升级为 `$loveca-bcrypt-sha256$...`；错误密码仍被拒绝。
- 当前 v2 测试账号可登录、刷新和登出。
- 旧 refresh Cookie、邮箱验证链接和密码重置链接均失效。
- 卡组创建、复制版本、导入与默认能量处理正常。
- 桌面对局阶段快捷操作、休息室判心统计和窄屏布局正常。
- 如另行执行卡牌同步，抽查 `GRAY` 只补必要 Heart 总数、不替代指定颜色，`double` 显示并结算为两个无色 Heart。

## 6. 回滚边界

- 认证 apply 前发现问题：保持写入停止，可修复后继续；若取消发布，未执行数据写入时无需恢复数据库。
- 认证 apply 后：不能只回滚 API。旧 API 无法读取带兼容前缀或升级后的密码，并且三类 token 已删除；必须停止所有写入，从发布前 Postgres dump 恢复，同时回滚 API 与前端到上一稳定版本。
- 只有前端异常且新版 API、数据库和认证验证正常时，可以单独回滚前端静态产物，并处理 Service Worker / CDN 缓存。
- 本次默认不执行卡牌同步或对象存储更新，因此默认回滚无需恢复卡牌数据或图片。若实际追加这些动作，必须使用对应发布前备份，并在发布记录中写明源文件、命令和影响数量。

## 7. 本次不包含

- 不自动执行生产 `pnpm db:migrate` 或认证数据迁移。
- 不自动重跑 Excel / CloudBase 卡牌同步，不导入 source-only 新卡。
- 不上传或替换生产卡图及其他对象存储内容。
- 不构建或分发 Android APK/AAB。
