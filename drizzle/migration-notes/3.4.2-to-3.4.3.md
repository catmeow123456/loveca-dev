# 3.4.2 -> 3.4.3 生产迁移说明

> 文档类型：版本迁移说明
> 适用范围：生产环境从 `v3.4.2` 升级到 `v3.4.3`
> 当前状态：发布前人工清单；权威部署流程仍以 `docs/production-release-runbook.md` 为准

本次升级包含数据库结构变更、卡牌数据同步脚本字段口径变化，以及前后端展示字段切换。发布时不要只替换 API 或前端产物，必须把数据库迁移、卡牌数据同步和前后端部署作为同一发布窗口处理。

## 1. 本次迁移改变什么

可执行迁移文件：

- `drizzle/0005_add_card_multilingual_excel_fields.sql`

主要结构变化：

- `cards.name` 重命名为 `cards.name_cn`。
- `cards.card_text` 重命名为 `cards.card_text_cn`。
- `cards.name_cn` 取消 `NOT NULL`。
- 删除旧索引 `idx_cards_group_name` / `idx_cards_name`。
- 新增 `name_jp`、`work_names`、`group_names`、`unit_name_raw`、`card_text_jp`、`image_source_uri`、`product_code`、`source_external_id`、`source_flags`。
- 将旧 `group_name` 按换行拆入 `work_names`，然后删除 `group_name`。
- 新增约束：`name_jp` / `name_cn` 至少一个非空。

影响面：

- 新版 API、服务端 card registry、前端 `cardService` 和管理页都读取新字段。
- 旧版 API/前端仍按旧字段访问 `name`、`card_text`、`group_name`，不能和迁移后的数据库长期混跑。
- 运行时仍派生 `card.data.name` / `card.data.cardText` / `card.data.groupName`，但数据库不再存这些旧列。

2. 已确认发布机具备源码、devDependencies、pnpm 和生产 `DATABASE_URL`；不要假设 API runtime 镜像内能跑 Drizzle CLI。

## 3. 推荐执行顺序

### 3.1 发布前 dry-run

先在不会写入生产库的模式下检查两个同步源能被解析：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx src/scripts/sync-cards-llocg.ts --dry-run
pnpm exec tsx src/scripts/sync-cards-loveca-excel.ts --dry-run
```

说明：

- `sync-cards-llocg.ts --dry-run` 仍需要 `DATABASE_URL`，因为它要读取现有卡牌并统计 insert/update 差异。
- `sync-cards-loveca-excel.ts --dry-run` 在没有 `DATABASE_URL` 时只做 parse-only；生产发布前更建议带生产 `DATABASE_URL` 跑 dry-run，以便看到实际 DB 匹配、Excel-only、DB-only 和待更新字段。
- 如果 dry-run 出现大量非预期规则字段更新，先停下审查，不要继续正式写入。

### 3.2 停止写入或进入维护窗口

本次迁移会重命名/删除 `cards` 表列。为了避免旧 API 在迁移中继续读写旧字段，建议进入短维护窗口：

1. 暂停 API 写流量，或至少停止管理端卡牌写入和同步脚本。
2. 保持数据库备份可恢复。
3. 执行数据库迁移。

### 3.3 执行 Drizzle 迁移

```bash
DATABASE_URL='postgres://...' pnpm db:migrate
```

不要使用 `pnpm db:push` 代替生产迁移。

### 3.4 验证迁移结构

迁移后可以执行以下只读 SQL：

```sql
SELECT column_name
FROM information_schema.columns
WHERE table_name = 'cards'
  AND column_name IN (
    'name',
    'name_cn',
    'name_jp',
    'card_text',
    'card_text_cn',
    'card_text_jp',
    'group_name',
    'work_names',
    'group_names',
    'unit_name_raw',
    'image_source_uri',
    'product_code',
    'source_external_id',
    'source_flags'
  )
ORDER BY column_name;
```

期望：

- 不再有 `name`、`card_text`、`group_name`。
- 应有 `name_cn`、`name_jp`、`card_text_cn`、`card_text_jp`、`work_names`、`group_names`、`unit_name_raw`、`image_source_uri`、`product_code`、`source_external_id`、`source_flags`。

继续检查名称约束和旧 group 迁移结果：

```sql
SELECT count(*) AS cards_without_any_name
FROM cards
WHERE (name_jp IS NULL OR btrim(name_jp) = '')
  AND (name_cn IS NULL OR btrim(name_cn) = '');

SELECT count(*) AS cards_with_work_names
FROM cards
WHERE work_names IS NOT NULL;
```

期望：

- `cards_without_any_name = 0`。
- `cards_with_work_names` 不是硬性固定值，但如果迁移前生产库已有 `group_name`，这里不应异常为 0。

### 3.5 同步卡牌数据

迁移后推荐重跑两个同步脚本，顺序固定为先 `llocg_db`，后 Loveca Excel：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx src/scripts/sync-cards-llocg.ts
DATABASE_URL='postgres://...' pnpm exec tsx src/scripts/sync-cards-loveca-excel.ts
```

原因：

- `sync-cards-llocg.ts` 是主数据/规则字段同步，负责卡牌类型、费用、Heart、BLADE、LIVE 分数、必要 Heart、图片文件名、稀有度、收录商品、作品数组和基础中日文本。
- `sync-cards-loveca-excel.ts` 只补强已有卡牌的中日名称、中日效果文本、真实团体、真实小队、商品编号、图片来源 URI 和外部来源标识；不插入 Excel-only 新卡，也不覆盖费用、Heart、BLADE、LIVE 分数或必要 Heart。
- 如果先跑 Excel，后跑 llocg，可能把 Excel 修正过的展示/来源字段又退回到 llocg 口径。

生产注意事项：

- `sync-cards-llocg.ts` 遇到已有卡差异会进入交互审核；非 TTY 环境发现待更新卡会终止，避免无人值守覆盖。
- `sync-cards-loveca-excel.ts` 支持 `--yes`，但生产环境不建议无审查使用。优先交互确认；只有 dry-run 输出已被保存并人工确认后，才考虑 `--yes`。
- 两个脚本都可能更新 `PUBLISHED` 卡牌；执行前确认这是本次发布期望。
- Excel 文件路径可用 `--xlsx=/abs/path/file.xlsx` 显式指定，避免误读旧文件。

### 3.6 部署新版 API 和前端

数据库迁移完成后，部署 `v3.4.3` API 和前端产物。不要让旧版 API 长时间连接迁移后的数据库；旧版 API 仍可能查询旧列并失败。

## 4. 发布后验证

除 runbook 的健康检查外，本次额外建议：

```sql
SELECT card_code, name_jp, name_cn, card_text_jp IS NOT NULL AS has_text_jp, card_text_cn IS NOT NULL AS has_text_cn
FROM cards
ORDER BY card_code
LIMIT 20;
```

人工 smoke：

- 登录管理端，打开卡牌列表，确认卡牌名称正常显示。
- 打开任意卡牌编辑弹窗，确认日文名、中文名、日文效果、中文效果字段能显示和保存。
- 打开卡组编辑器，确认搜索、筛选、卡牌详情抽屉正常。
- 打开一局对战，确认卡牌详情浮窗、判定面板、对局记录中的卡名/效果不为空。
- 抽查新卡效卡组，确认 active effect 弹窗标题和正文仍显示正确卡名/效果。

## 5. 回滚边界

本次数据库迁移不是低风险可逆迁移：

- 迁移会删除旧列 `group_name`，虽然数据会先拆入 `work_names`，但列级回滚需要手写恢复 SQL。
- 迁移会把 `name` / `card_text` 改名为 `name_cn` / `card_text_cn`；回滚旧 API 需要恢复旧列名或恢复发布前备份。
- 如果已经运行两个同步脚本，卡牌文本/来源字段也会被新脚本更新。

建议回滚策略：

1. 如果仅前端或 API 产物异常、数据库迁移和同步结果可用，优先修复并重部署新版，不要直接回滚旧 API 连接新 schema。
2. 如果必须回滚到 `v3.4.2`，停止 API 写入，使用发布前 Postgres dump 恢复数据库，再部署旧 API 和旧前端。
3. 如果只是同步脚本写入了不期望的卡牌展示字段，但 schema 本身正确，优先用重新同步或管理端修正；不要为展示字段问题直接恢复整库，除非影响范围不可控。

## 6. 本次不处理

- 不迁移或复制生产图片对象；图片仍由对象存储/图片代理负责。
- 不构建 Android TWA 包；Android 发布材料按 `android/twa/README.md` 另行处理。
- 不在 API runtime 镜像内执行 Drizzle CLI；迁移必须在具备源码和 devDependencies 的发布环境执行。
