# 3.7.3 -> 3.8.0 发布迁移说明

> 文档类型：版本迁移说明
> 适用范围：从 `v3.7.3` 发布 `v3.8.0`
> 权威通用流程：`docs/production-release-runbook.md`

## 1. 变更与生产影响

- 本次没有新增 Drizzle schema SQL，也没有生产数据转换脚本、卡牌同步口径变化、对象存储变更或新增生产环境变量。
- 权威对局状态新增必填的 `manualOperationMode`，当前值为 `RULES` 或 `FREE`。在线对局、对墙打、调试对局、玩家视图和前端操作入口必须使用同一版协议。
- 业务运行时不接受缺少 `manualOperationMode` 的新状态。仅有两个已审计的复水边界会处理旧记录：历史回放和服务端可记录的对墙打恢复会把早期缺字段 checkpoint 规范化为 `FREE`，避免把历史自由操作记录误解释为当前默认规则模式。该例外不适用于 live session 或普通 command 路径。
- 规则模式新增统一命令防火墙，限制自由拖拽、手动区域移动、任意横置、直接调整分数等调试操作；自由模式继续保留这些能力。既有卡效登场、状态变化、换手与费用路径已改为显式卡效来源，不依赖自由操作入口。
- 前端修复多处异步竞态、轮询重入、交互草稿残留、弹层焦点和 reduced-motion 边界。本次前端静态产物与 API 应按同一提交协调发布。

## 2. 发布前置与停机顺序

1. 确认 API 镜像、前端产物和本说明来自同一个 `v3.8.0` 提交。
2. 确认生产环境变量沿用现有有效配置，尤其是 `DATABASE_URL`、两个不同的 32 字节以上 JWT secret、对象存储配置与 `FRONTEND_URL`；本版没有新增变量。
3. 停止旧 API 及其他会写入对局记录的任务，避免旧版在新版部署窗口继续写入不含 `manualOperationMode` 的运行态数据。
4. 完成 Postgres dump，并记录、验证恢复命令。对象存储本版无写入变化，仍应确认常规备份可用。
5. 在具备源码、devDependencies 和生产 `DATABASE_URL` 的发布环境执行：

   ```bash
   DATABASE_URL='postgres://...' pnpm db:migrate
   ```

6. 核对迁移状态后，部署同提交的 `v3.8.0` API 与前端；不要让旧前端长期连接新 API，也不要让新前端连接旧 API。

本次不需要重跑 `v3.7.3` 的认证凭据迁移，也不需要执行 Excel / CloudBase 卡牌同步脚本。

## 3. 发布前后验证

### 3.1 数据库与配置

本版没有新增表、列或数据迁移。`pnpm db:migrate` 前后应确认 Drizzle 迁移命令正常结束，并使用等价检查确认核心表仍可访问：

```sql
SELECT count(*) AS users_count FROM users;
SELECT count(*) AS match_records_count FROM match_records;
SELECT count(*) AS timeline_entries_count FROM match_timeline_entries;
```

同时确认 API 启动日志没有缺失环境变量、数据库连接或对象存储配置错误。

### 3.2 发布后 smoke

除生产 runbook 的容器、日志、health、首页、manifest、图片代理和联机房间检查外，至少验证：

- 新建规则模式对墙打与联机对局，确认玩家只能执行当前规则允许的命令。
- 新建自由模式调试对局，确认自由拖拽、手动区域移动、横置和分数调整仍可用。
- 在安全操作点切换规则/自由模式，确认双方视图及时同步；存在待处理效果或其他不安全状态时切换被拒绝。
- 验证普通登场、换手、费用支付、卡效登场、成员状态变化和 LIVE 结算各一条基本路径。
- 打开一条 `v3.7.3` 或更早的历史回放，确认缺少 `manualOperationMode` 的旧 authority checkpoint 仅在历史复水边界按 `FREE` 打开。
- 检查联机房间、管理员页面与观战页面的轮询不会重入，切换房间或视角后旧响应不会覆盖新状态。
- 检查弹层键盘焦点、关闭后焦点恢复和系统 reduced-motion 设置下的主要页面。

## 4. 同版部署与兼容边界

- API、前端和新建对局状态必须使用 `v3.8.0` 同版协议。
- 不为 live session、command payload 或普通玩家视图提供缺字段 fallback。发布时应结束或明确放弃旧进程中的活动对局，再启动新版。
- 历史记录读取保留 `/api/battle/match-records...` 规范路径；旧 `/api/online/match-records...` 仅为已公开协议的临时 alias，本版没有扩展其兼容范围。
- 历史回放和可记录对墙打恢复的缺字段规范化不代表新旧 API 可以混跑。

## 5. 回滚边界

- 本次没有数据库 schema 或生产数据写入迁移。若仅 API 或前端出现问题，可以在停止新写入后把 API 与前端一起回滚到 `v3.7.3`，通常不需要恢复数据库。
- 已由 `v3.8.0` 创建的运行态对局状态包含 `manualOperationMode`；回滚前应结束新版活动对局，不要尝试让旧进程继续承接新版内存会话。
- 若发布窗口中除本说明外另行执行了卡牌同步、对象存储更新或人工 SQL，则这些动作不属于本版默认范围，必须使用各自的发布前备份和记录单独判断回滚。
- 回滚前端后应处理 Service Worker、CDN 和 Nginx 缓存，避免客户端继续加载 `v3.8.0` 静态资源。

## 6. 本次不包含

- 不自动执行生产 `pnpm db:migrate`。
- 不重跑认证凭据、Excel 或 CloudBase 卡牌同步脚本。
- 不上传或替换生产卡图及其他对象存储内容。
- 不构建或分发 Android APK/AAB。
