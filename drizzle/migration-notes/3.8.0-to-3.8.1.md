# 3.8.0 至 3.8.1 发布迁移说明

适用发布：`v3.8.1`  
适用 SQL：`drizzle/0008_add_public_table_beta.sql`

## 变更范围

- 新增公共牌桌候场票据、配对预留与跨模式玩家占用三张运行表，以及相关索引、外键和检查约束；
- 将 `match_records.origin_kind` 的允许值扩展为 `ONLINE_ROOM`、`PUBLIC_TABLE`、`SOLITAIRE`；
- API 新增公共牌桌候场、确认与房间引导流程，前后端必须同版部署；
- 不需要重跑卡牌数据或对象存储同步脚本，也不新增生产环境变量。

本次迁移不转换已有对局记录。旧 API 不识别 `PUBLIC_TABLE`，不允许旧 API 连接完成迁移后的 schema；停机窗口内须停止旧版本和所有新对局写入，再迁移并部署 `v3.8.1`。

## 发布前置与执行顺序

1. 确认生产 `DATABASE_URL` 指向目标库，完成 Postgres dump 并记录恢复命令和备份位置；确认对象存储已有独立备份或快照。
2. 在具备源码、devDependencies 与生产 `DATABASE_URL` 的发布环境执行：

   ```bash
   pnpm db:migrate
   ```

3. 确认迁移成功后，先部署 `v3.8.1` API，再部署同版 `client/dist`；不得将生产机用于重新构建 API 镜像。
4. 本次无需运行数据同步脚本，也不得以 `pnpm db:push` 代替迁移。

## 迁移后验证

```sql
SELECT conname
FROM pg_constraint
WHERE conname IN (
  'gameplay_participations_kind_check',
  'public_table_reservations_state_check',
  'match_records_origin_kind_check'
)
ORDER BY conname;

SELECT indexname
FROM pg_indexes
WHERE indexname IN (
  'uq_public_table_tickets_active_user',
  'uq_public_table_tickets_requeued_from',
  'uq_public_table_reservation_ticket_pair'
)
ORDER BY indexname;
```

两条查询均应各返回 3 行。随后按生产 runbook 检查 API health、前端 manifest、图片代理，并以两个测试账号完成公共牌桌候场、双方确认、房间引导和首个对局快照的 smoke。

## 回滚边界

若尚未产生 `PUBLIC_TABLE` 对局记录，可停止 API 写入、删除新增表和相关对象，并将 `match_records_origin_kind_check` 恢复为只允许 `ONLINE_ROOM / SOLITAIRE`，再回滚 API 与前端。若已产生 `PUBLIC_TABLE` 记录，禁止直接回滚旧约束或旧 API；必须停止写入并从发布前 Postgres dump 恢复，或先制定这些记录的保留/失效方案。
