# 3.7.1 -> 3.7.2 生产迁移说明

> 文档类型：版本迁移说明
> 适用范围：生产环境从 `v3.7.1` 升级到 `v3.7.2`
> 当前状态：发布前人工清单；权威部署流程仍以 `docs/production-release-runbook.md` 为准

本次升级新增平台公告与维护状态表，并将认证凭据运行时强制切换到 v2 格式。认证切换会让旧密码凭据进入必须重置状态，并撤销全部刷新令牌、邮箱验证 token 与密码重置 token；因此必须安排停机窗口、先备份并完成 dry-run，再执行数据库迁移和认证数据切换，最后同版部署 API 与前端。不得让旧 API 在认证数据切换后重新接收流量。

## 1. 本次迁移改变什么

可执行 Drizzle 迁移：

- `drizzle/0006_add_site_announcements.sql`：新增 `site_announcements`，保存维护、更新和动态公告，包含发布状态、展示时间、优先级、影响范围及创建/更新账号引用。
- `drizzle/0007_add_site_status_config.sql`：新增单行 `site_status_config`，保存维护生命周期、限制新对局时间、影响范围、限制说明和行动提示。
- 两个迁移均为新增表和索引，不修改既有业务表字段；仍必须通过 `pnpm db:migrate` 正式应用，不要使用 `pnpm db:push`。

一次性认证数据切换：

- 新运行时只接受带版本前缀的 v2 密码凭据、刷新 Cookie 和一次性 token，不保留 v1 fallback 或 dual-read。
- `drizzle/data-migrations/auth-v1-to-v2-credential-cutover.ts` 会把旧 bcrypt 或未知密码凭据标记为必须重置，并清空全部刷新令牌、邮箱验证 token 与密码重置 token。
- 旧密码哈希不能无损转换；受影响用户必须重新登录并通过邮件找回或人工恢复账号。
- 使用 `@placeholder.loveca.local` 邮箱且需要重置的账号默认阻断正式迁移。必须先处理这些账号；只有明确接受无法自助恢复时才允许使用 `--allow-unrecoverable-accounts`。
- 详细语义与参数以 `drizzle/migration-notes/auth-v1-to-v2-credential-cutover.md` 为准。

服务、接口与配置变化：

- 管理员平台新增公告和维护状态管理；维护状态可限制新建、加入和开始对局，但不会中断已经开始的对局。
- 首页公开配置会读取数据库中的维护状态和已发布公告，并在可见页面后台刷新；数据库不可用或迁移未完成时的环境变量路径仅为兜底，不是常规生产模式。
- 房间号观战、视角切换、轮询退避、跨重开等待和自动续看协议均有变化，API 与前端必须同版部署。
- 生产 compose 新增 `LOVECA_API_IMAGE`，部署时应固定到 `ghcr.io/catmeow123456/loveca-api:v3.7.2` 或发布记录中的 digest，并使用 `docker compose pull api` 与 `docker compose up -d --no-build api`。
- `EMAIL_ENABLED=true` 时，注册邮箱和登录前验证成为强制门禁；服务启动前必须准备完整 SMTP 配置。`JWT_SECRET` 与 `JWT_REFRESH_SECRET` 必须是不同的、至少 32 字节的正式随机值。

卡牌数据、图片与对象存储：

- 本次不要求重跑 `src/scripts/sync-cards-llocg.ts`、`src/scripts/sync-cards-loveca-excel.ts` 或 CloudBase 导入脚本。
- 本次不改变生产对象存储 key、图片同步口径或 `/images/*` 代理路径。
- `assets/card` 与 `assets/images` 的本地补图不得进入发布提交或生产静态产物。

## 2. 发布前必须确认

1. 已进入维护窗口并停止旧 API，以及所有会写入用户、刷新令牌、邮箱验证 token、密码重置 token、公告或站点状态的任务。
2. 已完成 Postgres 发布前 dump，记录备份文件、校验值和恢复命令，并在隔离环境验证可恢复。
3. 已确认发布机的 `DATABASE_URL` 指向正确生产库，且发布机具备源码、devDependencies、Node 与 pnpm；不要在 API runtime 镜像内执行 Drizzle CLI 或数据迁移脚本。
4. 已确认新版 API、前端、Drizzle migration 和认证切换脚本来自同一提交。
5. 已确认生产 `POSTGRES_*`、MinIO、`FRONTEND_URL`、JWT secret 与可选 SMTP 配置均为正式值，不含示例或占位密钥。
6. 已为旧密码用户准备密码重置通知；已单独处理 dry-run 报告中的占位邮箱账号。
7. 已确认 API 与前端将在同一维护窗口部署，且认证切换后不会重新启动 `v3.7.1` API。
8. 已确认 `assets/card` 与 `assets/images` 无发布差异；本次不执行卡牌数据同步或对象存储更新。

## 3. 推荐执行顺序

### 3.1 发布前检查

在发布提交和 `v3.7.2` tag 上执行：

```bash
pnpm install --frozen-lockfile
pnpm --dir client install --frozen-lockfile
pnpm version:check
pnpm typecheck:all
pnpm test:run
pnpm build:server
pnpm --dir client build
```

### 3.2 认证切换 dry-run

正式停机前可以先做一次只读预演；停止旧 API 写入并完成备份后必须再次执行，以停机后的报告作为正式基线：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx \
  drizzle/data-migrations/auth-v1-to-v2-credential-cutover.ts \
  --dry-run \
  --report=tmp/release-3.7.2/auth-v2-dry-run.json
```

人工审核报告中的旧密码、未知格式、已标记重置、占位邮箱影响和三类 token 数量。`placeholderResetRequiredCount` 不为 0 时应先处理账号恢复路径，不要直接执行 apply。

### 3.3 应用 Drizzle migration

```bash
DATABASE_URL='postgres://...' pnpm db:migrate
```

迁移后验证新增表及 migration 记录：

```sql
SELECT to_regclass('public.site_announcements') AS site_announcements,
       to_regclass('public.site_status_config') AS site_status_config;

SELECT id, hash, created_at
FROM drizzle.__drizzle_migrations
ORDER BY created_at DESC
LIMIT 5;

SELECT lifecycle, count(*)
FROM site_status_config
GROUP BY lifecycle;

SELECT type, status, count(*)
FROM site_announcements
GROUP BY type, status
ORDER BY type, status;
```

如果生产使用不同的 Drizzle migration schema 或 table，以实际配置为准。两个新业务表可以暂时为空；管理员首次保存配置后才会产生相应记录。

### 3.4 执行认证凭据切换

确认停机后的 dry-run、备份和恢复安排无误后执行：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx \
  drizzle/data-migrations/auth-v1-to-v2-credential-cutover.ts \
  --apply \
  --yes \
  --invalidate-legacy-passwords \
  --report=tmp/release-3.7.2/auth-v2-apply.json
```

只有明确接受占位邮箱账号无法自助恢复时，才额外增加 `--allow-unrecoverable-accounts`。保存 dry-run 和 apply 两份报告并核对受影响数量；脚本会在单个事务中执行，并在提交前验证旧/未知密码凭据和三类 token 已清零。

可用以下等价 SQL 做人工复核：

```sql
SELECT count(*) FILTER (
         WHERE password_hash LIKE '$loveca-bcrypt-sha256$%'
       ) AS current_v2,
       count(*) FILTER (
         WHERE password_hash = '$loveca-password-reset-required$v1'
       ) AS reset_required,
       count(*) FILTER (
         WHERE password_hash NOT LIKE '$loveca-bcrypt-sha256$%'
           AND password_hash <> '$loveca-password-reset-required$v1'
       ) AS unsupported
FROM users;

SELECT (SELECT count(*) FROM refresh_tokens) AS refresh_tokens,
       (SELECT count(*) FROM email_verification_tokens) AS verification_tokens,
       (SELECT count(*) FROM password_reset_tokens) AS reset_tokens;
```

预期 `unsupported = 0`，三类 token 数量均为 0。不要在验证失败时启动新版或旧版 API。

### 3.5 部署同版 API 与前端

认证切换成功后立即按 `docs/production-release-runbook.md` 部署 `v3.7.2` API 和同一提交生成的 `client/dist`。生产机只拉取已验证的版本镜像或 digest，不重新构建 API；前端必须同时发布 `manifest.webmanifest`、`sw.js`、`version.json` 与 PWA 图标。

本次不执行卡牌同步脚本，不上传本地卡图，也不构建或分发 Android TWA 包。

## 4. 发布后 smoke

除生产 runbook 的容器、日志、health、首页、manifest 与图片代理检查外，至少验证：

- 使用现有 v2 测试账号登录、刷新会话、登出并再次登录；确认刷新 Cookie 为 v2 格式且旧 Cookie 无法继续使用。
- 对一个已被标记为必须重置且邮箱可达的测试账号申请新密码重置链接，完成重置并登录；旧验证/重置链接必须失效。
- 管理员打开平台配置，保存并发布一条测试公告，确认首页可见；开启维护后确认新建/加入/开始对局返回维护提示，已有对局仍可收尾，然后恢复 `NORMAL`。
- 创建联机房间并用房间号进入观战，验证只读、隐藏信息、视角切换、轮询恢复和重开后的自动续看。
- 创建一局对墙打并完成基础阶段推进；抽查公开日志、回放读取和关键卡效窗口。
- 检查 API 日志中没有认证格式错误、migration table 缺失、公告表查询失败或持续观战 429。

## 5. 回滚边界

- 在认证 apply 之前发现问题：保持旧 API 停止写入；如只应用了新增表 migration，可优先修复并继续发布。若决定取消发布，恢复旧 API 前确认没有新版写入；新增空表通常不影响旧版，但仍应记录实际数据库状态。
- 认证 apply 之后不能只回滚 API。旧密码哈希已经被覆盖，三类 token 已被删除；完整回滚必须停止所有写入，恢复发布前 Postgres dump，并同时回滚 API 与前端到 `v3.7.1`。
- 如果只有前端静态产物异常，但新版 API、schema 和认证验证均正常，可以先修复或回滚前端；不要因此恢复旧 API。
- 如果公告或维护配置写入异常但认证正常，先关闭维护限制并保留数据库现场；只有数据损坏且无法前向修复时才恢复发布前 dump。
- 本次没有卡牌数据同步或对象存储更新，因此正常回滚不需要恢复卡牌数据或图片对象；若实际发布时临时增加了这些动作，必须单独记录并采用对应备份恢复。

- 不构建或分发 Android APK/AAB；Android 发布材料按 `android/twa/README.md` 另行准备。
- 不重跑卡牌数据同步，不执行 CloudBase 新卡导入。
- 不改变生产对象存储内容或图片代理链路。
- 不执行旧回放 checkpoint 压缩迁移；该历史迁移仍按其原版本说明单独处理。
