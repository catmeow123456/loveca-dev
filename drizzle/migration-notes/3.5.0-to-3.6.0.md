# 3.5.0 -> 3.6.0 生产迁移说明

> 文档类型：版本迁移说明
> 适用范围：生产环境从 `v3.5.0` 升级到 `v3.6.0`
> 当前状态：发布前人工清单；权威部署流程仍以 `docs/production-release-runbook.md` 为准

本次升级不新增 Drizzle SQL migration，但包含对局回放 checkpoint payload 存储格式变更、旧 checkpoint 一次性压缩迁移脚本，以及卡牌同步脚本字段来源扩展。发布时必须把 API、前端、停机数据迁移和必要的数据同步审查作为同一个发布窗口处理。

## 1. 本次迁移改变什么

可执行 Drizzle 迁移文件：

- 无新增 SQL migration；沿用 `v3.5.0` 已有 schema。

数据格式变化：

- 新写入的 `match_checkpoints` authority checkpoint payload 统一写为 `GZIP/BASE64_JSON` envelope。
- `payload_hash` 继续基于未压缩 stable JSON，`payload_compression` 必须与 envelope `compression` 保持一致。
- 正式 replay 读取、管理员读取和管理员导出路径只接受 `GZIP` authority checkpoint。
- 旧 `NONE/JSON_VALUE` checkpoint 只能由 `src/scripts/compress-match-replay-checkpoints.ts` 的迁移 helper 读取并转换。

同步脚本变化：

- `src/scripts/sync-cards-loveca-excel.ts` 增强 Heart / BLADE Heart / 必要 Heart 结构字段读取。
- 新增 `src/scripts/sync-cards-cloudbase-new.ts`，只从 CloudBase 卡牌集合导入数据库中不存在的新卡；正式写入必须明确选择 `--upload-images` 或 `--skip-images`。
- 新增 `src/scripts/loveca-excel-source.ts`，默认按 `docs/card-data-sync/sources/loveca_*.xlsx` 文件名时间戳选择最新 Excel。

## 2. 发布前必须确认

1. 已按 runbook 完成 Postgres 备份，并记录恢复命令和备份文件位置。
2. 已确认生产 `DATABASE_URL` 指向正确数据库，且发布机具备源码、devDependencies 和 pnpm；不要在 API runtime 镜像内执行 Drizzle CLI 或迁移脚本。
3. 已进入维护窗口，或已阻断新对局写入。旧 checkpoint 压缩脚本会在单事务内更新历史 authority checkpoint，不应和新对局 checkpoint 写入并发执行。
4. 已先部署或准备部署 `v3.6.0` API，使正式读取路径和写入路径都使用 `GZIP` checkpoint；不要让旧 API 长期连接已迁移数据。
5. 已确认 `docs/card-data-sync/sources/` 中的 Loveca Excel 是本次期望版本；该目录被 `.gitignore` 忽略，不随仓库发布。
6. 已确认如需 CloudBase 新卡同步，CloudBase collection、图片字段、MinIO 环境变量和图片上传策略已在 dry-run report 中人工审查。
7. 已确认 `assets/card` / `assets/images` 没有临时补图 diff；生产图片仍走对象存储/图片代理。

## 3. 推荐执行顺序

### 3.1 发布前 dry-run

先在维护窗口前或 staging 库上生成 checkpoint 压缩迁移报告：

```bash
DATABASE_URL='postgres://...' \
pnpm exec tsx src/scripts/compress-match-replay-checkpoints.ts \
  --dry-run \
  --report=tmp/release-3.6.0/replay-compress-dry-run.json
```

审查 report：

- `blockingErrors` 必须为空。
- `legacyCheckpointCount` 应等于待迁移旧 authority checkpoint 数。
- `remainingInvalidCheckpointCount` 在 dry-run 前通常等于旧格式数量。
- 若出现 hash、schema version、payload compression 或 authority state mismatch，停止发布并先定位数据问题。

如本次要同步卡牌数据，先做只读预演：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx src/scripts/sync-cards-loveca-excel.ts --dry-run --report=tmp/card-sync/loveca-excel-dry-run.json
DATABASE_URL='postgres://...' pnpm exec tsx src/scripts/sync-cards-cloudbase-new.ts --dry-run --skip-images --report=tmp/card-sync/cloudbase-new-dry-run.json
```

CloudBase 新卡脚本正式运行前必须根据本次目标选择 `--upload-images` 或 `--skip-images`；生产不建议未经 report 审查直接加 `--yes`。

### 3.2 停止写入或进入维护窗口

1. 暂停 API 写流量，或至少阻断新联机/对墙打对局写入。
2. 确认发布前 Postgres dump 已完成且可恢复。
3. 确认当前执行环境使用本次 `v3.6.0` 代码。

### 3.3 执行 Drizzle 迁移

本次无新增 Drizzle SQL，但仍建议执行一次标准迁移命令，确认发布库 schema 与迁移表状态一致：

```bash
DATABASE_URL='postgres://...' pnpm db:migrate
```

不要使用 `pnpm db:push` 代替生产迁移。

### 3.4 压缩旧 replay checkpoint

正式迁移：

```bash
DATABASE_URL='postgres://...' \
pnpm exec tsx src/scripts/compress-match-replay-checkpoints.ts \
  --apply \
  --yes \
  --report=tmp/release-3.6.0/replay-compress-apply.json
```

迁移后只读验证：

```sql
SELECT checkpoint_type,
       payload_compression,
       payload->>'compression' AS envelope_compression,
       payload->>'encoding' AS encoding,
       count(*) AS count
FROM match_checkpoints
WHERE checkpoint_type = 'AUTHORITY'
GROUP BY 1, 2, 3, 4
ORDER BY 1, 2, 3, 4;

SELECT count(*) AS remaining_invalid_authority_checkpoints
FROM match_checkpoints
WHERE checkpoint_type = 'AUTHORITY'
  AND (
    payload_compression <> 'GZIP'
    OR payload->>'compression' <> 'GZIP'
    OR payload->>'encoding' <> 'BASE64_JSON'
    OR payload_hash <> payload->>'payloadHash'
  );
```

期望：

- authority checkpoint 只剩 `GZIP / GZIP / BASE64_JSON`。
- `remaining_invalid_authority_checkpoints = 0`。

可以再跑一次 dry-run 验证幂等：

```bash
DATABASE_URL='postgres://...' \
pnpm exec tsx src/scripts/compress-match-replay-checkpoints.ts \
  --dry-run \
  --report=tmp/release-3.6.0/replay-compress-post-apply-dry-run.json
```

期望 `legacyCheckpointCount = 0`、`updatePlannedCount = 0`、`remainingInvalidCheckpointCount = 0`。

注意：迁移会更新 JSONB/TOAST 行。执行后 `pg_total_relation_size('match_checkpoints')` 可能短期变大，这是 PostgreSQL MVCC 旧 tuple 尚未回收导致；发布后观察 autovacuum，必要时在单独维护窗口评估 `VACUUM (VERBOSE, ANALYZE) match_checkpoints` 或更强空间回收方案。

### 3.5 可选同步卡牌数据

如本次需要同步 Loveca Excel Heart 字段或 CloudBase 新卡，建议在 checkpoint 迁移完成后按已审查 report 执行。CloudBase 新卡脚本只插入 DB 不存在的新卡，不更新已有卡，不删除卡；图片上传失败默认不插入该卡，除非显式 `--allow-missing-images`。

示例：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx src/scripts/sync-cards-loveca-excel.ts

DATABASE_URL='postgres://...' \
MINIO_ENDPOINT='...' \
MINIO_ACCESS_KEY='...' \
MINIO_SECRET_KEY='...' \
pnpm exec tsx src/scripts/sync-cards-cloudbase-new.ts \
  --cloudbase-collection=loveca \
  --upload-images
```

是否执行 CloudBase 新卡同步不是本次 API 部署的硬性前置；若不执行，应在发布清单中明确跳过。

### 3.6 部署新版 API 和前端

部署 `v3.6.0` API 与前端产物。为了 replay 格式一致性，不建议旧 API 与已迁移 checkpoint 数据长期混跑。

## 4. 本地真实数据验证记录

发布准备时已用真实 ONLINE fixture 验证迁移脚本：

- 数据来源：`data/20260627-cst_20260627T183253Z/loveca-match-replay-2026-06-27-cst-online-only.normalized.sql.gz`。
- 导入临时 PostgreSQL 16 库，包含 6 场 ONLINE 对局、736 条 authority checkpoint。
- 迁移前 SQL：736 条均为 `NONE/JSON_VALUE`。
- dry-run：扫描 736 条，计划迁移 736 条，blocking errors 为 0，estimated envelope JSON savings 为 91.69%。
- apply：更新 736 条，remaining invalid/non-GZIP checkpoints 为 0。
- 迁移后 SQL：736 条均为 `GZIP/BASE64_JSON`，mismatch 为 0。
- 二次 dry-run：legacy 0，already migrated 736，planned updates 0，remaining invalid/non-GZIP checkpoints 0。
- payload byte 口径：未压缩 281,074,897 bytes，gzip 17,344,991 bytes，压缩节省约 93.83%。

## 5. 发布后验证

除 runbook 的健康检查外，本次额外建议：

- 打开一条迁移前已存在的历史对局，确认玩家视角回放和管理员 checkpoint 读取正常。
- 执行上方 SQL，确认 authority checkpoint 没有非 `GZIP` 或 hash mismatch。
- 管理员导出一条历史 replay bundle，确认导出流程会复水校验 checkpoint，但 bundle 内 payload envelope 仍为压缩形态。
- 新开一局对局并推进数步，确认新写入 checkpoint 的 `payload_compression` 为 `GZIP`。
- 如执行了卡牌同步，抽查卡牌详情、卡组编辑、对局卡牌浮窗中的名称、效果、Heart、必要 Heart 与图片显示。

## 6. 回滚边界

- 如果只部署前端异常，且数据库 checkpoint 已迁移成功，可只回滚前端。
- 如果 API 部署异常但数据库已压缩 checkpoint，不要回滚到只能读取 `NONE/JSON_VALUE` 的旧 API。优先修复并重部署 `v3.6.0` API。
- 如果必须回滚到 `v3.5.0` API，需停止写入并从发布前 Postgres dump 恢复数据库；本次脚本没有提供 `GZIP -> NONE` 反向迁移。
- 如果 CloudBase 或 Excel 同步写入了不期望的卡牌数据，优先用同步源修正或管理端修正；影响范围不可控时从发布前 DB dump 恢复。

## 7. 本次不处理

- 不迁移 checkpoint 到 `bytea` blob；本次仍复用 `payload jsonb` 中的 base64 gzip envelope。
- 不改变 checkpoint 保存频率。
- 不删除 timeline/public/private/decision 记录。
- 不构建 Android TWA 包；Android 发布材料按 `android/twa/README.md` 另行处理。
