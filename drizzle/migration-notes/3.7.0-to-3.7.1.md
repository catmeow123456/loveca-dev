# 3.7.0 -> 3.7.1 生产迁移说明

> 文档类型：版本迁移说明
> 适用范围：生产环境从 `v3.7.0` 升级到 `v3.7.1`
> 当前状态：发布前人工清单；权威部署流程仍以 `docs/production-release-runbook.md` 为准

本次升级不新增 Drizzle SQL migration，不改变 `src/server/db/schema.ts`，也不要求重跑卡牌同步脚本。主要变化集中在 API 运行态治理、对墙打运行态恢复、public events 截断保护、历史回放读取上限，以及前端对恢复快照和移动端对局 viewport 的兼容处理。建议 API 与前端同一发布窗口部署，避免新前端/旧 API 或旧前端/新 API 对恢复载荷、截断标记和运行态错误反馈理解不一致。

## 1. 本次迁移改变什么

可执行 Drizzle 迁移文件：

- 无新增 SQL migration；沿用 `v3.7.0` schema。
- `pnpm db:migrate` 仍应按 runbook 在发布环境执行一次，确认迁移表状态一致。

数据与脚本变化：

- 本次不改变卡牌同步字段口径，不要求执行 `src/scripts/sync-cards-llocg.ts`、`src/scripts/sync-cards-loveca-excel.ts` 或 CloudBase 新卡导入脚本。
- 对墙打运行态恢复会从既有 `match_records`、`match_participants`、`match_deck_snapshots`、`match_checkpoints` 与 `match_record_public_events` 读取最近 AUTHORITY checkpoint 和 public event 尾部；不写入新的 schema 字段。
- 历史回放读取新增行数保护，超限时返回 413，后续分页 UI / 离线导出仍属下一阶段能力。

服务与前端接口变化：

- API 进程定期执行联机房间与对墙打运行态 cleanup，并输出 `api-runtime-cleanup` / `api-runtime-stats` 结构化日志。
- public-events 响应可能带 `truncated` / `droppedEventCount`，前端会在截断或恢复快照时重置本地公开日志，避免显示不连续历史。
- 对墙打运行态缺失时，服务端可从最近 AUTHORITY checkpoint 恢复；如恢复发生回退，后续旧写操作会被拒绝并附带恢复快照。
- 前端在同一标签页保存对墙打 `matchId`，刷新后会尝试重新拉取 snapshot 并接回远程对局；用户明确离开对墙打时会清理该记录。
- 移动端对局 viewport 统一读 `BattleViewportShell` 变量，拖拽/长按过程中 viewport 变化会取消当前交互并提示重试。

新增或需确认的生产环境变量：

- `API_RUNTIME_CLEANUP_INTERVAL_MS`：API 运行态清理周期，默认 `60000`。
- `API_RUNTIME_STATS_LOG_INTERVAL_MS`：API 运行态统计日志周期，默认 `60000`。
- `ONLINE_PUBLIC_EVENTS_MAX_BATCH`：public events 单次响应上限，默认 `500`。
- `MATCH_REPLAY_TIMELINE_ROW_LIMIT`：历史时间线读取上限，默认 `10000`。
- `MATCH_REPLAY_VISIBLE_ROW_LIMIT`：单个回放节点可见读取上限，默认 `10000`。
- `MATCH_REPLAY_EXPORT_ROW_LIMIT`：管理员导出估算行数上限，默认 `25000`。

上述变量均有默认值；生产可先沿用默认值，若线上仍有 API 内存压力，再按监控结果调小响应上限或缩短 cleanup 周期。

## 2. 发布前必须确认

1. 已按 runbook 完成 Postgres 备份，并记录恢复命令和备份文件位置。
2. 已确认生产 `DATABASE_URL` 指向正确数据库，且发布机具备源码、devDependencies 和 pnpm；不要在 API runtime 镜像内执行 Drizzle CLI。
3. 已确认生产环境变量使用正式值，尤其是 `DATABASE_URL`、JWT secret、MinIO 配置与 `FRONTEND_URL`。
4. 已确认是否需要调整上方新增运行态/回放保护环境变量；如不调整，应在发布记录中明确使用默认值。
5. 已确认 `assets/card` / `assets/images` 没有临时补图 diff；生产图片仍走对象存储/图片代理。
6. 已确认 API 与前端会同版部署；不建议旧前端长期连接 `v3.7.1` API，也不建议新前端长期连接 `v3.7.0` API。

## 3. 推荐执行顺序

### 3.1 发布前检查

按 runbook 执行发布前构建与验证。发布机或 CI 至少应通过：

```bash
pnpm install --frozen-lockfile
pnpm version:check
pnpm typecheck:all
pnpm test:run
pnpm build:server
pnpm --dir client build
```

### 3.2 执行 Drizzle 迁移状态校验

本次没有新增 schema SQL，但仍按标准流程执行：

```bash
DATABASE_URL='postgres://...' pnpm db:migrate
```

不要使用 `pnpm db:push` 代替生产迁移。

迁移前后可用等价检查确认当前库迁移表没有 pending 项；若生产环境使用 Drizzle 默认迁移表，也可抽查：

```sql
SELECT id, hash, created_at
FROM drizzle.__drizzle_migrations
ORDER BY created_at DESC
LIMIT 5;
```

运行态恢复依赖既有对局记录与 checkpoint，可抽查近期对墙打记录是否保留 AUTHORITY checkpoint：

```sql
SELECT record.match_id, record.status, MAX(checkpoint.checkpoint_seq) AS latest_authority_checkpoint
FROM match_records record
LEFT JOIN match_checkpoints checkpoint
  ON checkpoint.match_id = record.match_id
  AND checkpoint.checkpoint_type = 'AUTHORITY'
WHERE record.match_mode = 'SOLITAIRE'
GROUP BY record.match_id, record.status
ORDER BY MAX(record.updated_at) DESC
LIMIT 10;
```

### 3.3 卡牌数据同步

本次发布不要求重跑卡牌同步脚本。若运营临时决定同步卡牌数据，应另行按 `docs/card-data-sync/` 和对应脚本 dry-run report 流程执行，不要把卡牌数据同步混入本次运行态修复发布窗口。

### 3.4 部署新版 API 和前端

部署 `v3.7.1` API 与前端产物。由于恢复快照、public-events 截断字段、离开确认和移动端 viewport 交互在前后端同时变化，建议同一发布窗口完成 API 与前端部署。

## 4. 发布后验证

除 runbook 的健康检查外，本次额外建议：

- 打开正式联机房间，确认创建、加入、锁定卡组、开始对局和普通阶段推进正常。
- 创建一局对墙打，刷新当前标签页，确认可回到同一 matchId 桌面；明确离开后再次刷新不应恢复旧对局。
- 抽查 public battle log，确认长局或旧 cursor 返回截断时前端不会显示不连续历史。
- 触发或等待 `api-runtime-cleanup` / `api-runtime-stats` 日志，确认 match 数量、stale match 数量与内存摘要可观测。
- 打开历史回放与管理员导出入口，确认普通规模记录可读；超大记录如触发 413，应记录 matchId、估算行数与当前阈值。
- 移动端 390x844 视口打开对局桌面，确认 bottom sheet、判定面板和拖拽 viewport 变化提示正常。

## 5. 回滚边界

- 如果只部署前端异常且 API 未暴露新恢复/截断路径，可先回滚前端静态产物。
- 如果 API 运行态恢复、cleanup 或 public-events 截断异常，应同时回滚 API 与前端到 `v3.7.0`，避免前端继续依赖新版恢复字段。
- 本次没有 schema migration；未执行额外数据同步时，数据库通常无需恢复。
- 如果发布后发现恢复逻辑写入了不期望的历史记录帧或对局记录异常，先停止 API 写入并保留日志，再评估是否从发布前 Postgres dump 恢复相关表。
- 回滚后重新执行 runbook 的健康检查，并抽查对墙打创建、刷新、离开流程。

## 6. 本次不处理

- 不新增 Android TWA 包；Android 发布材料按 `android/twa/README.md` 另行处理。
- 不执行 CloudBase 新卡导入。
- 不重跑卡牌同步脚本。
- 不改变生产图片链路或对象存储配置。
- 不落地 public-events 真分页协议或历史回放分页 UI；这些仍为后续 P1 follow-up。
