# 3.6.0 -> 3.7.0 生产迁移说明

> 文档类型：版本迁移说明
> 适用范围：生产环境从 `v3.6.0` 升级到 `v3.7.0`
> 当前状态：发布前人工清单；权威部署流程仍以 `docs/production-release-runbook.md` 为准

本次升级不新增 Drizzle SQL migration，不改变 `src/server/db/schema.ts`，但包含正式联机观战/公开日志/在线服务接口变更、回放 checkpoint 数据迁移脚本整理，以及卡牌同步脚本字段边界修正。发布时建议 API 与前端同版部署；卡牌同步脚本不要求随本次发布强制重跑，除非运营确认要更新生产卡牌数据。

## 1. 本次迁移改变什么

可执行 Drizzle 迁移文件：

- 无新增 SQL migration；沿用 `v3.6.0` schema。
- `pnpm db:migrate` 仍应按 runbook 在发布环境执行一次，确认迁移表状态一致。

数据与脚本变化：

- `drizzle/data-migrations/3.5.0-to-3.6.0-compress-match-replay-checkpoints.ts` 继续作为历史回放 checkpoint 压缩迁移入口，并从旧 `src/scripts/` 位置收口到 `drizzle/data-migrations/`。
- `src/scripts/sync-cards-llocg.ts`、`src/scripts/normalize-group.ts`、`src/scripts/validate-group.ts` 修正卡牌同步字段边界与团体识别口径。
- `docs/card-data-sync/` 已同步说明新的字段来源和同步职责；这些文档是执行同步前的人工依据。

服务与前端接口变化：

- 正式联机新增玩家视角观战、管理员玩家视角观战入口、公开对局日志和相关在线轮询/同步字段。
- `src/server/routes/online.ts`、`src/server/services/online-room-service.ts`、`src/server/services/online-match-service.ts` 与 `client/src/lib/onlineClient.ts`、`remoteMatchClient.ts` 同步演进，旧前端不应长期连接新版 API，旧 API 也不应长期服务新版前端。

## 2. 发布前必须确认

1. 已按 runbook 完成 Postgres 备份，并记录恢复命令和备份文件位置。
2. 已确认生产 `DATABASE_URL` 指向正确数据库，且发布机具备源码、devDependencies 和 pnpm；不要在 API runtime 镜像内执行 Drizzle CLI。
3. 已确认生产环境变量使用正式值，尤其是 `DATABASE_URL`、JWT secret、MinIO 配置与 `FRONTEND_URL`。
4. 已确认本次是否要重跑卡牌同步脚本；如不重跑，应在发布记录中明确跳过。
5. 若要同步卡牌数据，先确认 `llocg_db`、Loveca Excel 源文件与生产图片对象存储策略均为本次期望版本。
6. 已确认 `assets/card` / `assets/images` 没有临时补图 diff；生产图片仍走对象存储/图片代理。

## 3. 推荐执行顺序

### 3.1 发布前检查

按 runbook 执行发布前构建与验证。发布机或 CI 至少应通过：

```bash
pnpm install --frozen-lockfile
pnpm version:check
pnpm typecheck:all
pnpm test:run
pnpm build:server
pnpm --dir client build
```

### 3.2 执行 Drizzle 迁移状态校验

本次没有新增 schema SQL，但仍按标准流程执行：

```bash
DATABASE_URL='postgres://...' pnpm db:migrate
```

不要使用 `pnpm db:push` 代替生产迁移。

迁移前后可用等价检查确认当前库迁移表没有 pending 项；若生产环境使用 Drizzle 默认迁移表，也可抽查：

```sql
SELECT id, hash, created_at
FROM drizzle.__drizzle_migrations
ORDER BY created_at DESC
LIMIT 5;
```

如生产库使用不同 migration schema/table，以实际 Drizzle 配置为准。

### 3.3 可选同步卡牌数据

本次发布不要求强制重跑卡牌同步。若运营确认要应用新的同步字段口径，先 dry-run 并人工审查 report：

```bash
DATABASE_URL='postgres://...' pnpm exec tsx src/scripts/sync-cards-llocg.ts --dry-run --report=tmp/release-3.7.0/llocg-dry-run.json
DATABASE_URL='postgres://...' pnpm exec tsx src/scripts/sync-cards-loveca-excel.ts --dry-run --report=tmp/release-3.7.0/loveca-excel-dry-run.json
```

正式同步前确认 report 中的新增、更新、跳过与 warning 都符合预期，再按对应脚本文档执行正式命令。不要未经 report 审查直接带 `--yes` 执行会写入生产数据的脚本。

### 3.4 部署新版 API 和前端

部署 `v3.7.0` API 与前端产物。由于观战、公开日志和在线同步字段在前后端同时变化，建议同一发布窗口完成 API 与前端部署，不建议新旧版本长期混跑。

## 4. 发布后验证

除 runbook 的健康检查外，本次额外建议：

- 打开正式联机房间，确认玩家创建/加入/开局流程正常。
- 在已开始房间中生成玩家视角观战链接，确认观战页只读、隐藏信息按玩家视角脱敏、观战人数统计不把管理员链接计入公开观战人数。
- 打开公开对局日志，确认移动、卡效摘要和同步游标显示正常。
- 打开休息室浮窗，确认判心统计基于当前可见休息室卡牌显示。
- 抽查卡组编辑和卡牌详情，确认团体/小队筛选与卡牌同步字段显示未回退。
- 如执行了卡牌同步，抽查更新过的卡牌名称、效果、Heart、必要 Heart 与图片显示。

## 5. 回滚边界

- 如果只部署前端异常且未执行卡牌数据同步，可只回滚前端静态产物。
- 如果 API 异常且未执行卡牌数据同步，可回滚 API 与前端到 `v3.6.0`，数据库通常无需恢复。
- 如果已执行卡牌同步并写入了不期望的数据，优先用同步源修正或管理端修正；影响范围不可控时，从发布前 Postgres dump 恢复。
- 如果观战/公开日志相关数据出现线上兼容问题，应同时回滚 API 与前端，避免新前端继续调用旧 API 不支持的在线接口。

## 6. 本次不处理

- 不新增 Android TWA 包；Android 发布材料按 `android/twa/README.md` 另行处理。
- 不执行 CloudBase 新卡导入。
- 不改变生产图片链路或对象存储配置。
