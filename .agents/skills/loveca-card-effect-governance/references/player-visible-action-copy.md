# 玩家可见操作文案规范

本 reference 适用于 `activeEffect` 的 `stepText`、`selectionLabel`、`confirmSelectionLabel`、`skipSelectionLabel`、`selectableOptions[].label` 和数字输入按钮。

## 核心原则

- 按钮写点击后立即执行的游戏动作，不写界面机制。
- 优先复用同类既有 workflow 的稳定字面量；新增前先用 `rg` 搜索相同目的区域和动作。
- 使用玩家语言，不出现 `source`、`pending`、`stale`、`payload`、`eventId`、`trigger` 等内部术语。
- `确认` 只用于没有更具体游戏动作的纯确认步骤。能写 `加入手牌`、`登场`、`放置于卡组顶` 时，不写 `确定`、`确认选择`、`确定分配`、`选择` 或 `继续处理`。
- 同一 shared family 和同批同型卡使用一致文案；差异必须来自真实规则动作。

## 字段职责

| 字段 | 应表达的内容 | 示例 |
| --- | --- | --- |
| `stepText` | 当前要做什么以及必要的规则结果 | `请选择至多1张放置于卡组顶，其余放置入休息室。` |
| `selectionLabel` | 候选区选择目标 | `选择要放置于卡组顶的卡` |
| `confirmSelectionLabel` | 提交选择后执行的动作 | `放置于卡组顶` |
| `skipSelectionLabel` | 不提交选择时的真实规则分支 | `不发动`、`不加入`、`全部放置入休息室` |
| `selectableOptions[].label` | 玩家可选的规则动作或结果 | `发动`、`获得[BLADE]` |

## 常用动作标准词

| 效果形状 | `selectionLabel` 示例 | 主按钮 |
| --- | --- | --- |
| 卡加入手牌 | `选择要加入手牌的卡` | `加入手牌` |
| 公开后加入手牌 | `选择要公开并加入手牌的卡` | `公开并加入手牌` |
| 卡放置入休息室 | `选择要放置入休息室的卡` | `放置入休息室` |
| 卡放置于卡组顶 | `选择要放置于卡组顶的卡` | `放置于卡组顶` |
| 卡放置于卡组底 | `选择要放置于卡组底的卡` | `放置于卡组底` |
| 多张卡按选择顺序放置 | `按放置顺序选择卡片` | `按此顺序放置于卡组顶` / `按此顺序放置于卡组底` |
| 成员从区域登场 | `选择要登场的成员` | `登场` |
| 选择登场槽位 | `选择登场区域` | `登场` |
| 成员变为待机 | `选择要变为待机状态的成员` | `变为待机状态` |
| 成员变为活跃 | `选择要变为活跃状态的成员` | `变为活跃状态` |
| 成员移动槽位 | `选择移动后的区域` | `站位变换` |
| 选择卡支付费用 | `选择要用于支付费用的卡` | `支付费用` |
| 选择 Heart/BLADE 等结果 | `选择要获得的效果` | `获得[赤ハート]`、`获得[BLADE]` |
| 已公开卡的纯确认 | `公开的卡片` | `确认公开结果` |

卡组顶/底统一使用“放置于”，不要在新增文案中混用“放到”“放回”作为按钮动词。`stepText` 可按卡文说明“放回”，但提交按钮仍描述最终动作，例如 `放置于卡组顶`。

## 可选与跳过

- 整个效果可不发动：正向 option 使用 `发动`，跳过按钮使用 `不发动`。
- 固定来源自身作为成本或动作时，不展示唯一来源卡供玩家选择；使用 `发动 / 不发动`，确认发动后重新校验并直接处理来源。
- 可选择不把卡加入手牌：使用 `不加入`。
- 可选择不放置卡：使用 `不放置`；如果不选会让所有已检视卡进入休息室，应写真实结果 `全部放置入休息室`。
- 不要把负向选项同时放进 `selectableOptions` 和 `skipSelectionLabel`，避免双重跳过按钮。
- 不要使用 `取消` 表示规则上的“不发动”或“不处理”；`取消` 只适合真正撤回尚未提交的界面操作。

## 多步骤窗口

- 每一步只保留本步骤需要的选择字段。进入下一步时清理旧的 `selectableCardIds`、选择数量、确认按钮和跳过按钮。
- 第一步按钮描述第一步动作，不提前概括整个效果。例如先弃手再抽牌时，第一步使用 `放置入休息室`，不要使用 `发动效果`。
- 单纯公开、展示结算结果或 confirm-only 的窗口可使用 `确认`；只要存在更具体动作，就使用具体动作。

## 审查与测试

- focused test 至少断言关键 `selectionLabel`、主按钮和跳过按钮的精确文本。
- 检查按钮是否描述实际规则动作，尤其拒绝 `确定分配`、无上下文的 `确定`、无动作结果的 `选择`。
- 检查可选效果是否出现两个负向按钮，固定来源是否被错误渲染成单卡选择。
- 触及这些字段后，运行 focused workflow test、`card-effect-tokens.test.ts` 和 `card-effect-text-governance.test.ts`。
