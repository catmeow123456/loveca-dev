# Loveca 联机模式重点测试场景

> 文档类型：联机手工测试场景
> 适用范围：联机调试页、`GameSession` 命令链路、`PlayerViewState` 投影、公共事件语义、联机恢复与时机控制
> 编写依据：`docs/online-mode-*.md`、`detail_rules.md`、当前 `src/application/game-session.ts`、`src/online/*.ts`、`client/src/store/gameStore.ts`、`client/src/components/pages/OnlineDebugPage.tsx`
> 最后更新：2026-04-10

---

## 1. 文档目的

这份文档不是穷举所有功能点，而是把联机模式里最值得人工重点回归的场景先抽出来。测试工程师阅读后，可以按场景逐个执行，优先验证三个问题：

- 公开事实有没有被双方一致看到。
- 私密信息有没有因为视图投影或调试兼容态而泄露。
- 当前实现里的命令、窗口、阶段和恢复链路，是否真的符合联机语义，而不是只在本地模式下“看起来能跑”。

当前工程还处于“联机语义已落地，实时会话层仍在推进”的阶段，因此建议测试时优先使用联机调试页，从 `FIRST` / `SECOND` 两个 seat 同时观察同一对局。

---

## 2. 使用建议

- 尽量双端同时操作和观察。很多问题不是某一端单看能发现的，而是两端看到的公开事实不一致。
- 每个场景都同时关注四件事：牌桌实际变化、另一端看到的变化、当前可操作按钮或拖拽权限、刷新后的恢复结果。
- 一旦遇到“看起来结果对了，但过程不对”的情况，也应记为问题。联机首版特别强调过程事件，而不只是最终状态。

---

## 3. 重点场景

### 场景 1：双方进入同一联机调试房间，并在开局后看到同一条公共时间轴

让两个 seat 分别锁定卡组并进入同一场对局，确认对局确实只启动一次，而不是两边各自起了本地局。进入 Mulligan 后，双方都应能看到一致的阶段和子阶段，只是 viewer seat 不同。

重点观察：

- 两边进入的是同一个 `matchId`，不是两个独立房间。
- 初始阶段、子阶段、等待状态一致，且双方的可操作 seat 判断正确。
- 一边操作后，另一边的公共流程也会同步推进，而不是只本地刷新。

### 场景 2：隐藏区张数始终公开，但对手不能拿到隐藏区对象详情

开局后立刻对比双方视角里的手牌、主卡组、能量卡组。对手应当知道这些区域有几张牌，但不应该能看到具体对象、具体顺序，更不应该从详情弹窗、hover、日志或调试浮层里读到真实实例信息。

重点观察：

- 对手能看到张数变化，但看不到具体卡名、卡图、卡类型。
- 对手视角里隐藏区不应表现成可逐张追踪的对象列表。
- 任何 hover、长按、详情面板都不应绕过 `PlayerViewState` 直接读出隐藏卡信息。

### 场景 3：同一张盖放 Live，自己看到 FRONT，对手只看到 BACK

在 Live 设置阶段，从手牌盖放一张 Live。操作者应继续知道自己放了什么，对手只应该知道那里出现了一个可追踪的背面对象。

重点观察：

- 操作者端能继续读到这张 Live 的正面信息。
- 对手端只能看到背面占位，不能看到卡名、卡种、分数、需求心。
- 这张牌在双方界面中应仍是同一个连续对象，而不是一边有对象、一边完全没有对象。

### 场景 4：明置 Live 与盖放 Live 的公开语义必须分开

分别测试明置放入 Live 区和盖放放入 Live 区。明置时，对手应直接拿到正面公开信息；盖放时，对手只能拿到背面移动，不应被事件或 UI 额外泄露牌面。

重点观察：

- 明置 Live 是“公开并移动”。
- 盖放 Live 只是“背面对象移动”。
- 盖放场景下，日志、动画、操作记录、调试输出都不应把卡面信息带出来。

### 场景 5：公开对象进入手牌时，对手能看到它进入手牌，但之后不能继续追踪

把一张已经公开的牌从成员区、Live 区或成功区收回手牌。联机文档要求“进入手牌”这一步仍然是公共事实，但进入手牌之后，对手不应继续持有该对象的持续投影。

重点观察：

- 对手能看到这张公开牌确实被收回手牌。
- 这个动作发生后，对手不再能继续在手牌区域里追踪这张牌的对象。
- 如果这张牌之后重新回到公开区，系统应把它当作同一个连续对象回到公共链路，而不是无故变成完全陌生的新对象。

### 场景 6：检视开始时，牌应真实进入检视区，而不是只弹出一个本地草稿窗口

从主卡组或能量卡组发起检视。根据当前设计，检视区是正式区域，不是临时 UI。检视者应看到正面，对手应看到背面，并且双方都要感知到对象进入了检视区。

重点观察：

- 检视区的数量变化对双方都可见。
- 检视者看到 FRONT，对手看到 BACK。
- 检视区窗口打开后，流程确实进入检视状态，而不是主流程继续自由操作。

### 场景 7：检视区内的每一步操作都应即时生效，而不是最后一次性结算

打开检视后，不要只测“最终结果对不对”，而要测试区内逐步操作。把一张牌放回顶部、一张放到底部、一张移入手牌或休息室，确认每一步都立刻改变当前牌桌，而不是等“完成检视”才统一落账。

重点观察：

- 每次移动后，双方界面都立刻看到检视区数量和顺序变化。
- 检视结束前，中间状态就已经是真实状态。
- “完成检视”只负责关闭流程，不负责补做之前没发生的牌桌变化。

### 场景 8：检视区重排必须对对手可感知

一次检视两张以上牌，在检视区内交换顺序或拖动重排。当前设计里，检视区顺序是真值，对手虽然看不到正面，但应该能感知顺序变化过程。

重点观察：

- 检视区同区重排后，对手看到的是“背面对象顺序调整”，不是完全静默。
- 重排后再放回顶部或底部时，结果要和重排后的顺序一致。
- 若检视区里有多张牌，顺序变化不应在刷新后丢失。

### 场景 9：检视区公开一张牌后，对手应立即从 BACK 变为 FRONT

在检视区中主动公开其中一张牌。当前实现允许检视区内直接公开，不要求必须先移出检视区。这个场景最容易暴露投影和 UI 更新不同步的问题。

重点观察：

- 对手端这张牌会从 BACK 立即切成 FRONT。
- 对手可以开始读取这张牌的正面信息。
- 检视区里其他未公开牌仍应保持 BACK，不应被顺带泄露。

### 场景 10：进行中的检视流程必须锁住时机，只允许检视拥有者继续操作

当一方正在检视时，尝试让自己去结束阶段、让对手去执行别的命令、或者让双方继续做普通主阶段操作。当前代码明确把检视视为独立窗口，其他命令应被拒绝。

重点观察：

- 非检视拥有者不能插入操作。
- 检视拥有者在没处理完检视牌前，不能强行结束检视或切阶段。
- UI 上的可操作区和按钮禁用状态，要和实际命令校验结果一致。

### 场景 11：同一轮检视只能从同一来源区追加，不能混入另一副隐藏堆

先从主卡组检视，再尝试在同一流程中追加能量卡组；或者反过来先检视能量卡组，再尝试追加主卡组。当前命令层已经收口了“同一来源区追加”的规则，这一点非常关键，因为它直接关系到审计和重放能否成立。

重点观察：

- 同来源区追加应成功，并且是加到当前检视流程里。
- 不同来源区追加应明确失败，而不是悄悄开第二个检视上下文。
- 失败后当前检视区与窗口状态不能被破坏。

### 场景 12：应援翻牌进入解决区时，应先是可追踪背面，再翻为正面

在 Live 判定阶段执行应援翻牌。当前联机模型要求解决区是正式共享区域，未翻开时对手看 BACK，翻开后看 FRONT。

重点观察：

- 牌进入解决区这一刻，对手看到的是背面对象，而不是直接知道牌面。
- 翻开后，对手才看到正面信息。
- 解决区中的对象在翻面前后应保持同一个连续对象，不应因为翻开而换对象。

### 场景 13：进入演出阶段时，系统自动翻开 Live 必须作为系统公开事件出现

把流程推进到演出开始，验证系统自动翻开 Live。这个动作不是玩家手动声明，而是规则文本要求自动发生的系统处理，因此它必须以系统来源的公开事件表现出来。

重点观察：

- 双方都看到 Live 从背面转为正面。
- 这个过程不能依赖玩家再点一次“翻开”按钮。
- 若刷新页面，翻开后的状态和日志锚点仍然成立。

### 场景 14：系统自动补能量到公开能量区时，双方都应感知完整的公开移动

推进到会自动补能量的流程，或者使用主阶段把能量卡组中的牌拖到能量区。该动作本质上是隐藏区到公开区的正式过程，因此既要验证玩家真的能做，也要验证公开结果是否正确。

重点观察：

- 牌应从能量卡组顶进入能量区，而不是从能量卡组任意位置出现。
- 对手在它进入公开区前不应知道牌面，进入公开区后才看到 FRONT。
- 若当前边界允许通过自由拖拽跨越隐藏区和公开区，则应重点验证拖拽是否真的打通，以及结果是否符合公开语义。

### 场景 15：公开区之间的自由移动和跨公开/隐藏边界的拖拽，都要和当前边界保持一致

从休息室移去成功区、成员槽位之间换位、能量区附着到成员下方，这些都是公开世界里的桌面过程。若当前边界已经放开手牌、牌库、能量卡组跨公开/隐藏边界的自由拖拽，则也要一并验证这些路径在联机里确实可用，而不是仍被旧限制挡住。

重点观察：

- 公开区到公开区的移动过程要对双方可见，并带正确落点信息。
- 成员槽位移动要正确带出 `slot` 语义，成员下方能量要正确带出 overlay 语义。
- 若当前设计允许把隐藏区卡牌直接拖入公开区，则要重点验证这条路径是否真能做、以及公开方式是否正确。

### 场景 16：成员换位、附着能量、成员登场替换旧成员，这三类场景要一起验证

这三类动作都发生在成员区，但规则含义完全不同，也最容易在前端看起来差不多、实际上事件语义完全错位。建议把它们作为一组连续场景测试。

重点观察：

- 成员换位只是在公开区内部移动成员对象。
- 能量附着到成员下方后，应落到正确槽位和正确叠放位置。
- 新成员登场替换旧成员时，旧成员应进休息室，新成员应以公开方式进入成员区，而不是把两者混成一条普通移动。

### 场景 17：Live 设置阶段和表演阶段成功效果窗口中，允许的“回手/调整”是新的正式动作，不是撤销

当前联机文档明确区分“本地预览”和“正式事实”。因此在 Live 设置阶段把刚放上的 Live 拿回手牌，或者在表演阶段成功效果窗口里把公开对象回手，语义上都应是新动作，不是把上一条公开历史抹掉。

这里的“成功效果窗口”指 `PERFORMANCE_PHASE` 内的正式流程窗口，不是 `LIVE_RESULT_PHASE` 的正式子阶段。若测试时看到 `LIVE_RESULT_PHASE + RESULT_FIRST_SUCCESS_EFFECTS` 或 `RESULT_SECOND_SUCCESS_EFFECTS`，应记为当前实现仍残留旧状态机命名，不应反向修改联机规范。

重点观察：

- 回手后，对手应看到一次新的公开移动。
- 历史日志中应同时保留“放上去”和“收回来”两个过程。
- UI 不能把这种行为表现成“撤销成功”，否则测试应记为语义错误。

### 场景 18：结果阶段应从分数确认开始，并体现“双方都能确认分数”的并行提交语义

推进到 `LIVE_RESULT_PHASE` 的分数确认步骤，让双方分别确认分数。这个阶段不是单边优先权，而是双边都可以提交自己的确认。

重点观察：

- 双方都应拥有提交分数的权限。
- 一边确认后，另一边仍应保留自己的确认入口。
- 双边都确认完后，流程才应继续推进。

### 场景 19：错误 seat、错误阶段、错误窗口的操作都必须被明确拒绝

故意在不属于自己的时机点按钮，或者在非主要阶段尝试做主要阶段动作。联机首版强调服务端硬校验，不能依赖客户端自己不让点。

重点观察：

- 错误 seat 提交命令时，要得到明确拒绝，而不是静默失败。
- 非当前阶段的动作要被拒绝，不应被系统偷偷修正到“最近合法”的行为。
- 客户端若误放开了按钮，也不能绕过服务端校验。

### 场景 20：中途刷新页面或重新进入调试页后，当前 seat 应恢复到自己的安全视图

联机调试页当前通过拉取调试快照重建状态。测试时可以在检视中、Live 判定中、公开对象回手后等中间态刷新页面，确认回来后看到的仍然是同一局、同一 seat 的安全视图。

重点观察：

- 刷新后不应串 seat，也不应掉回权威态全量视图。
- 对手隐藏区、检视区背面信息、盖放 Live 等脱敏结果，刷新后必须仍然正确。
- 正在进行中的窗口、等待对象和当前牌桌顺序，刷新后不能丢。

## 4. 附加高风险工程场景

下面这些场景不一定每轮都做，但一旦联机层、调试接口层、状态同步层有改动，就很值得补测。

### 场景 21：快速连续点击同一命令，确认不会造成重复结算

在网络较慢或界面有延迟时，连续点击一次“检视”、一次“确认”、一次“回手”或一次“放到能量区”，看是否会把同一动作执行两遍。这个场景主要用来捕捉命令重复提交、前端未禁用按钮、后端去重缺失等问题。

### 场景 22：检视或判定中刷新其中一端，另一端继续操作，确认恢复后不会读到越权信息

这个场景能同时覆盖恢复、时机锁、私密视图和过程事件四条线。尤其要看刷新回来后，对手是否会因为状态重建顺序错误而读到本来只属于当前操作者的正面信息。

### 场景 23：一张牌多次进出公开链路后，公共追踪对象要保持连续

把一张公开牌回手，再重新打出；或把一张检视牌公开、移出、再回到公开区。重点不是最终位置，而是整条公共观察链是否能让人认出“还是那一张对象”，而不是每次重进公开区都像凭空生成了一张新牌。

## 5. 建议执行顺序

如果测试时间有限，建议至少先做以下优先级：

1. 场景 2、3、5、6、9，先把“可见性和泄露边界”测稳。
2. 场景 7、8、10、11，确认检视区流程确实符合联机即时生效模型。
3. 场景 12、13、14、17、18，覆盖系统动作、窗口和公开事件语义。
4. 场景 19、20，再补时机校验和刷新恢复。

如果这些场景中任意一个失败，都不建议把联机模式视为“只差一点 UI 小问题”。这些场景覆盖的是联机模型本身的边界，而不是简单的交互瑕疵。
