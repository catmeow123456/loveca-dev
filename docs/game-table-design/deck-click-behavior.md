# 卡组点击行为定稿

> 目的：固定“点击主卡组 / 点击能量卡组”在牌桌上的默认语义，避免继续沿用当前代码里偏调试工具化的理解。
>
> 本文是 `docs/game-table-design/` 下的子文档。
> 若与 `game-table-ui-current-state.md` 的旧描述冲突，以本文为准。

## 1. 设计目标

这一轮定稿的目标不是让卡组区域变成一个“多功能操作台”，而是让它更接近实体 Loveca 的桌面手感：

- 牌库点击应表达“翻顶 1 张”的实体动作
- 检视应表现为正式牌桌过程，而不是本地草稿面板
- 点击行为只负责很窄的一件事，不顺带承担完整整理牌库的职责
- 主卡组与能量卡组的点击语义应明确分开，不能继续混用

## 2. 术语对齐

本文中的“隐藏区”直接等同于当前联机模型中的正式 `INSPECTION_ZONE` / 检视区。

也就是说：

- 点击主卡组后，牌不是进入一个新的临时区域
- 牌会真实进入现有的检视区
- 检视区内的后续操作继续沿用现有联机文档中的即时生效语义

相关上位文档：

- `docs/online-mode-preparation.md`
- `docs/online-mode-visibility-matrix.md`
- `docs/online-mode-free-drag-checklist.md`

## 3. 默认点击行为

### 3.1 点击己方主卡组

点击己方主卡组的默认行为固定为：

- 从己方主卡组顶翻出 1 张牌
- 将该牌真实移动到检视区
- 该动作本身就是正式牌桌过程，不是“先开面板再决定是否翻牌”

进一步约束：

- 每次点击只翻 1 张，不提供“一次翻多张”的点击快捷语义
- 不因点击主卡组而打开旧式“卡组检视面板”
- 不因点击主卡组而自动附带排序、回手、弃置、回顶、回底等后续处理

若当前没有进行中的同源检视流程：

- 本次点击等价于开启一次正式检视流程
- 并执行一次 `OpenInspection(count=1)` 语义

若当前已经存在由当前玩家发起、来源仍为主卡组的检视流程：

- 本次点击等价于向当前检视流程继续追加 1 张
- 追加的牌来自主卡组顶
- 新牌加入检视区末尾

### 3.2 点击己方能量卡组

点击己方能量卡组的默认行为固定为：

- 不触发任何动作

这条约束是显式设计，而不是“暂时还没做”：

- 点击能量卡组不等价于“放 1 张能量到能量区”
- 点击能量卡组不等价于“打开能量卡组检视面板”
- 点击能量卡组不等价于“从能量卡组翻 1 张到检视区”

能量卡组后续若需要支持操作，应通过独立交互表达，例如拖拽或明确按钮，而不是复用默认点击。

### 3.3 点击对手的卡组

点击对手主卡组或对手能量卡组：

- 不应触发任何私有操作
- 不应代替对手发起检视
- 不应打开带有私有语义的面板

## 4. 点击与后续处理的边界

点击主卡组只负责一件事：

- 把主卡组顶 1 张翻进检视区

它不负责：

- 在同一次点击里决定这些牌如何结算
- 自动把牌放回顶 / 放回底 / 放进手牌 / 放进休息室
- 自动关闭检视流程
- 打开一个把上述操作全部包进去的“控制台式”弹窗

检视区中的后续处理，仍按现有检视区正式语义执行：

- 检视区是正式区域
- 区内原子操作即时生效
- `FinishInspection` 负责结束流程，而不是补做草稿结算

## 5. 与拖拽交互的分工

为了避免“点击”和“拖拽”语义互相污染，本文固定如下分工：

- 点击主卡组：翻顶 1 张到检视区
- 点击能量卡组：无默认动作
- 其它跨区移动：由拖拽或其它显式专用交互承担

因此后续实现和文案都不应再把下列动作混进“点击主卡组”里：

- 检视区牌 -> 手牌
- 检视区牌 -> 休息室
- 检视区牌 -> 主卡组顶 / 底
- 能量卡组顶牌 -> 能量区
- 能量区 -> 能量卡组

这些动作即使最终仍由拖拽触发，命令层也应继续走各自的专用语义，不把“点击卡组”扩展成万能入口。

### 5.1 当前牌桌实现补充

当前牌桌中的检视区 UI 可提供以下显式入口，作为对原子命令的前端包装：

- 检视区上方提供四个单独的拖拽目标：加入手牌、放入休息室、回卡组顶、放卡组底。
- 单张检视牌下方仅保留“公开”按钮；其它处理主要由拖拽完成。
- 检视区顶部可提供“全部放入休息区”快捷按钮。
- 检视区顶部可提供“关闭（检视区的牌全部回到卡组顶）”快捷按钮；该按钮的实现应按当前检视顺序逐张回顶，再结束检视流程。

这些都属于检视区内后续处理，不改变“点击主卡组只负责翻顶 1 张”的边界。

## 6. 对当前设计文档的纠偏

以下旧说法现在应视为过时：

- “点击己方主卡组可打开卡组检视面板”
- “点击主卡组：打开卡组检视”
- “`DeckPeekModal` 是点击主卡组后的标准入口”（`DeckPeekModal` 已删除）

新的固定理解是：

- 点击主卡组是一个桌面动作，不是一个面板入口
- 检视区是真实区域，不是本地草稿容器
- 当前的隐藏区浏览器（`HiddenDeckBrowserModal`）通过卡组右上角的独立按钮打开，不是点击卡组的默认结果

## 7. 验收标准

后续实现或重构应至少满足以下验收场景：

- 点击己方主卡组一次，检视区新增 1 张牌
- 连续点击己方主卡组三次，检视区累计新增 3 张，且顺序与翻出顺序一致
- 当前已有来自主卡组的检视流程时，再次点击主卡组，应追加到现有检视区，而不是悄悄新开第二个流程
- 点击己方能量卡组，不发生任何默认状态变化
- 点击对手的主卡组或能量卡组，不发生私有操作
- 文案、注释、交互说明中，不再把“点击主卡组”描述成“打开卡组检视面板”

## 8. 本文不处理的内容

本文只冻结“卡组点击”的默认行为，不额外扩展以下议题：

- 能量卡组未来是否需要单独的检视按钮
- `HiddenDeckBrowserModal` 是否需要与检视流程进一步整合
- 具体代码迁移顺序与重构方案

这些问题如需继续讨论，应在本文约束之上展开，而不是重新改写点击卡组的基础语义。
