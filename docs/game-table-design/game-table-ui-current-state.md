## 1. 设计目标与当前实现取向

从 `game_system_design.md` 和当前实现看，桌面 UI 的核心目标不是严格“只暴露合法动作”，而是：

- 让前端作为一个高可视化的对局操作台
- 将规则判断尽量留给 `GameSession` / `GameService`
- 前端采用“信任玩家”原则，允许较自由的拖拽和手动修正
- 用子阶段、浮层和局部操作面板去补足流程提示

当前桌面因此呈现出一种“规则驱动状态机 + 调试型交互桌面”的风格，而不是完全收敛的正式线上对战桌面。

## 2. 整体桌面结构

整体由 `GameBoard` 组织，采用纵向三段式结构：

- 顶部：对手区域
- 中部：`VS` 分隔条
- 底部：己方区域

外层还有几层全局视觉和功能层：

- 整体背景使用卡背图铺底
- 其上叠加若干全屏遮罩和光效渐变
- 右上角固定主题切换按钮
- 顶部中央固定调试控制条
- 左侧固定日志抽屉
- 左侧固定解决区 / 判定面板及其唤起按钮
- 右下角固定阶段指示器
- 右侧中部固定卡牌详情浮窗
- 中央根据阶段弹出换牌框、分数确认框、阶段横幅、拖拽覆盖层等

这意味着当前桌面不是“单一棋盘式平面布局”，而是“游戏桌 + 多个固定悬浮控制层”的复合界面。

## 3. 对局区域布局现状

### 3.1 上下两侧采用镜像布局

`PlayerArea` 明确采用“中心对称”设计：

- 己方在底部，按自己视角展示 LEFT / CENTER / RIGHT 成员槽
- 对手在顶部，成员槽顺序镜像为 RIGHT / CENTER / LEFT
- 己方资源区在画面右侧，对手资源区镜像后在画面左侧
- 己方成功 Live 区在画面左侧，对手成功 Live 区镜像后在画面右侧

布局目标是让双方“面对面坐在桌子两侧”的感受成立。

### 3.2 己方区域布局

己方区域从上到下分为三层：

- 主战区顶部
- 玩家信息条
- 手牌/能量区底部

其中主战区内部又是三列绝对定位：

- 左列：成功 Live 区
- 中列：Live 区 + 成员槽
- 右列：资源区（主卡组、休息室、能量卡组）

中列是主要视觉焦点：

- 上方是 3 个固定 Live 槽位
- 下方是 3 个成员槽位

成员槽下方还可能叠有附加能量牌，形成“成员卡在上，能量卡向左下偏移叠放”的局部堆叠结构。

底部区域：

- 左下角是能量区
- 中间是扇形展开的手牌
- 手牌最右侧外挂两个小按钮：抽一张、放回一张

### 3.3 对手区域布局

对手区域与己方相反，从上到下分为：

- 手牌/能量区顶部
- 玩家信息条
- 主战区底部

主战区内部仍然是三列，但左右互换：

- 左列：资源区
- 中列：成员槽 + Live 区
- 右列：成功 Live 区

中列内部也镜像：

- 上方是成员槽
- 下方是 Live 区
- 成员顺序为 RIGHT / CENTER / LEFT

这种设计保证了双方的“主战区”都靠近中央 VS 分隔条。

## 4. 各游戏元素的具体布局现状

### 4.1 成员区

成员区固定 3 槽：

- LEFT
- CENTER
- RIGHT

单槽表现：

- 空槽为虚线框
- 有卡时显示标准成员卡
- 可双击切换活跃/等待状态
- 被选中卡会进入 store 的 `selectedCardId`

能量附着表现：

- 附加能量卡与成员卡同尺寸
- 以绝对定位叠放在成员卡下方
- 每张能量卡向左下偏移 10%
- 槽位底部用小胶囊显示“附加能量 xN”

### 4.2 Live 区

Live 区固定 3 槽，与成员槽横向对齐。

表现特点：

- Live 卡横置显示
- 未翻开时按背面/暗面状态处理
- 已翻开的 Live 卡底部会显示所需 Hearts
- 判定结果会在卡片右上角显示绿色 `✓` 或红色 `✗`

己方 Live 区是可拖放区，对手 Live 区只是展示区。

### 4.3 成功 Live 区

成功区采用纵向 3 槽堆叠：

- 每张卡横置展示
- 槽位固定高度
- 上下有遮挡覆盖感
- 空槽显示虚线框占位

当前没有成功区专属展开浮层，信息密度较低，但展示直接。

### 4.4 资源区

资源区由三项横向排布组成：

- 主卡组
- 休息室
- 能量卡组

己方顺序：

- 主卡组 -> 休息室 -> 能量卡组

对手顺序镜像：

- 能量卡组 -> 休息室 -> 主卡组

卡组表现：

- 主卡组显示背面缩略牌堆和数量徽标
- 点击己方主卡组：将主卡组顶 1 张翻入检视区
- 点击己方能量卡组：默认无动作
- 能量卡组顶牌可直接拖拽出来

休息室表现：

- 缩略叠堆显示前几张卡
- 点击后打开休息室弹窗
- 弹窗内使用网格展示所有卡牌
- 弹窗内卡牌仍可拖拽

### 4.5 能量区

能量区在玩家头像行附近的小角落里：

- 己方在左下角
- 对手在右上角

表现形式为横向小缩略图列表：

- 最多先显示 12 张
- 活跃/等待状态通过透明度和灰度表现
- 标题同时显示 `(活跃数/总数)`
- 己方能量区中的能量可随时直接拖回能量卡组顶

### 4.6 手牌区

对手手牌：

- 只显示背面
- 扇形散开
- 超过 10 张后用 `+N` 提示

己方手牌：

- 正面扇形散开
- 相邻卡牌负 margin 重叠
- hover 向上抬起并放大
- 可点击选中
- 可拖拽到其它区域

己方手牌右侧有两个快捷按钮：

- `←` 从主卡组顶抽一张到手牌
- `→` 将手牌最右一张放回主卡组顶

这两个操作明显偏调试/辅助操作性质。

## 5. 提示框与浮层布局现状

当前提示不是集中在一个系统内，而是按用途分散为多种悬浮层。

### 5.1 阶段提示横幅

`PhaseBanner` 为全屏居中提示：

- 阶段切换时短暂出现
- 使用深色背景 + 亮色描边
- 仅负责“阶段名提示”
- 不承载交互

位置：

- 全屏居中
- `pointer-events: none`

### 5.2 阶段指示器

`PhaseIndicator` 固定在右下角，是当前最核心的“流程操作面板”。

展示内容：

- 当前回合数
- 当前主阶段名
- 当前子阶段名、图标、是否“需要操作”
- 当前是否轮到自己

操作内容：

- 主按钮：进入下一阶段 / 确认完成 / Live 判定
- 次按钮：部分子阶段可撤销

已知语义重复：右下角在 Live 结算阶段（`LIVE_RESULT_PHASE`）显示的"进入下一回合"按钮应只出现在分数确认、成功 Live 结算等后续流程完成后。表演成功后的效果处理现在属于 `LIVE_RESULT_PHASE` 的前置正式子阶段，按先攻成功效果、后攻成功效果依次处理。

它既是状态提示，也是主要流程推进入口。

### 5.3 解决区 / 判定面板

`JudgmentPanel` 是当前查看和操作解决区的左侧固定面板：

- 默认可由左侧窄按钮唤起
- 打开后占据左侧最大约 420px
- 收起按钮挂在面板右边缘

面板内部结构：

- 上半部：解决区中的应援牌区域
- 下半部：Live 判定区
- 底部：根据阶段变化的操作按钮

它当前兼具两层职责：

- 作为解决区的主要查看与处理入口
- 作为 Live 判定阶段的流程辅助面板

因此它是当前最复杂的交互提示容器，兼具：

- 阶段说明
- 解决区查看
- 应援翻牌与处理
- 成功/失败确认
- 辅助说明文案

补充说明：

- 正式区域是 `RESOLUTION_ZONE`
- `JudgmentPanel` 只是这个区域当前的 UI 载体，不是独立规则区域
- 表演判定成功后不会立刻进入成功效果窗口；双方表演完成后才进入先攻成功效果、后攻成功效果
- 若当前实现仍把这类窗口挂到 `LIVE_RESULT_PHASE` 的 `RESULT_*_SUCCESS_EFFECTS` 子阶段，应视为状态机尚未按规范收敛
- 相关定稿见 `docs/game-table-design/resolution-zone-behavior.md`

### 5.4 分数确认弹窗

`ScoreConfirmModal` 为中央模态框：

- 按规则目标模型，仅在 `LIVE_RESULT_PHASE` 的分数确认步骤中出现
- 展示双方分数和确认状态
- 允许用户调整并确认己方分数

位置：

- 全屏蒙层中央

在调试模式下：

- 蒙层弱化
- 不完全阻断背景视觉

### 5.5 换牌弹窗

`MulliganPanel` 为中央大型模态框：

- 仅在换牌阶段出现
- 展示当前手牌
- 点击卡牌切换是否换掉
- 底部有“不换牌”和“确认换牌”

该面板是一个很明确的单阶段专用任务弹窗。

### 5.6 隐藏区浏览器

`HiddenDeckBrowserModal` 是当前用于浏览主卡组和能量卡组内容的弹窗组件：

- 通过卡组右上角的浏览按钮打开，不是点击卡组的默认行为
- 以网格形式展示卡组内所有卡牌，顶部在最左侧
- 当前窗口允许时，支持直接从弹窗内拖拽任意卡牌到主桌提交正式移动
- 当前窗口不允许时，仅可查看顺序，不能拖拽提交

它的定位是隐藏区的辅助浏览入口，不是检视流程的一部分，也不是点击卡组后的默认结果。

### 5.7 休息室弹窗

休息室弹窗从 `PlayerArea` 内创建 portal：

- 点击休息室牌堆打开
- 居中显示
- 网格列出所有休息室卡牌
- 支持从弹窗内继续拖牌

### 5.8 卡牌详情浮窗

`CardDetailOverlay` 固定在右侧中部：

- 鼠标悬停任意卡牌时触发
- 展示大图、卡名、编号、效果、成员/Live 专属属性
- 不可交互，只用于阅读

位置固定意味着它不是贴近卡牌的 tooltip，而是独立详情侧栏。

### 5.9 游戏日志

`GameLog` 是左侧可展开日志抽屉：

- 仅在 DEBUG 模式下显示
- 默认收起，左边留一条窄按钮
- 展开后宽 280px
- 自动滚到底部

日志类型：

- info
- action
- phase
- error

## 6. 用户交互现状

### 6.1 核心交互模式

当前桌面的核心交互模式有 4 类：

- 拖拽
- 点击/双击
- 固定面板按钮推进流程
- 模态框内局部任务操作

### 6.2 拖拽交互

拖拽是主要桌面交互方式，基于 `dnd-kit`。当前实现采用“UI 可尝试、服务端定真值”的模型：前端可能允许己方卡牌拖起，但是否成为正式移动由命令窗口、卡种、区域和服务端校验决定。

特点：

- PointerSensor 需要移动 5px 才启动，避免误触双击
- 拖拽时显示 `DragOverlay`
- `DroppableZone` 支持三种视觉状态：
  - 推荐目标高亮
  - 当前悬停高亮
  - 其他区域轻度变暗

当前推荐目标只做提示，不做强限制。

当前应视为已承诺的典型拖拽行为：

- 主阶段、表演自由交互窗口、结果成功效果窗口：手牌成员登场、成员换位、能量附着、公开卡移动、手牌/休息室等己方桌面移动。
- Live 设置阶段：手牌放置到 Live 区、放置能量、公开卡回手/回能量卡组、当前命令覆盖的己方卡移动。
- Live 卡桌面豁免：Live 卡从 Live 区回手/进休息室，或在 Live 区/成功区相关公开桌面目标之间移动。
- 检视流程：检视区排序、检视区 -> 手牌 / 休息室 / 除外区 / 卡组顶底，以及己方手牌/休息室 -> 当前检视区。
- 判定/解决窗口：解决区 -> 手牌 / 休息室 / 主卡组顶。
- 能量卡组顶牌 -> 能量区。
- 能量区 -> 能量卡组。
- 休息室弹窗内卡 -> 手牌 / 其它当前允许区域。

补充约束：

- `手牌 -> 休息室` 由拖拽触发，命令层走 `MOVE_OWNED_CARD_TO_ZONE`。
- `Live 区 -> 手牌` 在当前实现中走公开卡回手路径；Live 卡从 Live 区回手有阶段豁免，非 Live 卡不应按该路径成立。
- `Live 区 -> 成功区` 在 `RESULT_SETTLEMENT` 走 `SELECT_SUCCESS_LIVE`；其它非检视流程中如果是 Live 卡相关桌面整理，则走 Live 卡桌面移动豁免并继续由服务端校验目标是否合法。
- `检视区 -> 手牌 / 休息室 / 卡组顶底` 虽然可以由拖拽触发，但命令层应走检视区专用命令。
- `手牌 / 休息室 -> 检视区` 只有在自己的检视流程中成立，命令层走 `MOVE_CARD_TO_INSPECTION`。
- `解决区 -> 手牌 / 休息室 / 卡组` 虽然可以由拖拽触发，但命令层应走解决区专用命令。
- `休息室 -> 手牌` 虽然可以由拖拽触发，但命令层应走公开区进入私有区的专用回手命令。
- `能量卡组顶牌 -> 能量区` 虽然由拖拽触发，但命令层应走专用”放置能量”命令，而不是通用 `MOVE_TABLE_CARD`。
- `能量区 -> 能量卡组` 虽然由拖拽触发，但命令层应走公开区进入隐藏区的专用回收命令；它只在当前命令窗口允许时成立，不视为撤销。

### 6.3 点击交互

主要点击行为包括：

- 点击己方手牌：选中卡牌
- 点击主卡组：将主卡组顶 1 张翻入检视区
- 点击能量卡组：默认无动作
- 点击休息室：打开休息室弹窗
- 点击左侧解决区按钮：打开解决区 / 判定面板
- 点击阶段指示器按钮：推进流程
- 点击中央模态按钮：换牌/确认分数

### 6.4 双击交互

当前明确存在的双击交互：

- 双击成员卡：切换活跃/等待状态
- 联机语义补充：在`主要阶段`和`表演阶段`，不区分当前是不是自己的回合，玩家都可以双击自己的成员卡让它进入待机状态；当前实现仍保留活跃/等待互切，因此再次双击会切回活跃

### 6.5 Hover 交互

hover 在当前桌面中占比很高：

- 卡牌 hover 时在右侧显示详情浮窗
- 己方手牌 hover 时上抬
- 缩略牌 hover 时放大
- 检视区/解决区卡牌 hover 时显示快捷操作条

因此当前桌面是明显偏 PC 鼠标环境的设计。

## 7. 典型流程下的交互路径

### 7.1 换牌阶段

- 中央出现 `MulliganPanel`
- 当前轮到的玩家点击要换的牌
- 点击“确认换牌”或“不换牌”
- 面板关闭，进入下阶段

### 7.2 主要阶段

- 用户主要在桌面上进行拖拽操作
- 可将手牌拖到成员槽等区域
- 可双击成员切换状态
- 联机补充：即使当前是对手的通常阶段，仍可双击自己的成员卡让它待机
- 完成后在右下角点击 `Live Start!`

主要阶段是普通桌面移动的主窗口；活跃、能量、抽卡等自动阶段不应被理解为同样开放普通自由拖拽。

### 7.3 Live 设置阶段

- 从手牌拖牌到 Live 区
- 系统在该阶段把放入 Live 区视为“里侧放置”
- 右下角按钮用于确认进入下一流程

该阶段还允许当前命令覆盖的放置能量、公开卡回手/回能量卡组和己方卡移动，但不等于开放所有普通桌面移动。非 Live 卡拖入 Live 区只在“手牌放置 Live”路径中成立。

### 7.4 Performance 判定阶段

- 右下角按钮变为 `Live 判定`
- 点击后打开左侧 `JudgmentPanel`
- 用户在解决区 / 判定面板中翻开应援牌、查看 Hearts 汇总
- 联机补充：表演阶段内，双方都可以双击自己的成员卡让它待机，不受当前表演玩家限制
- 选择 `LIVE失败` 或 `LIVE成功`
- 若成功，继续点击“成功效果发动完毕”

表演自由交互窗口包括 `PERFORMANCE_LIVE_START_EFFECTS` 和 `PERFORMANCE_JUDGMENT`。其中 `PERFORMANCE_JUDGMENT` 额外开放应援、解决区移动、判定提交和成功 Live 选择。

### 7.5 结果结算阶段

- 中央 `ScoreConfirmModal` 打开
- 用户查看双方分数
- 可人工修改己方分数
- 点击确认

`RESULT_FIRST_SUCCESS_EFFECTS` / `RESULT_SECOND_SUCCESS_EFFECTS` 是结果阶段中保留普通桌面操作的窗口。`RESULT_SCORE_CONFIRM` 和 `RESULT_SETTLEMENT` 不开放全部普通桌面拖拽；结算阶段的成功 Live 选择走专用命令，Live 卡相关桌面整理走阶段豁免。
- 双方都确认后进入下一回合

## 8. 当前设计的明显特征

### 8.1 优点

- 桌面信息量高，几乎所有重要区域都在主视图中可见
- 上下镜像结构清楚，主战区始终聚焦中央
- 流程按钮集中在右下角，不需要四处找“下一步”
- 左侧解决区 / 判定面板把 Live 相关复杂交互集中管理
- hover 详情、日志、检视面板等辅助阅读能力比较强

### 8.2 局限与设计债务

- 当前交互偏“调试桌面”，不是严格收敛的正式对战体验
- 部分窗口仍允许较自由地尝试拖拽，规则约束更多依赖后端报错，用户心智负担较大
- 提示系统分散，横幅、抽屉、模态、固定面板各自承担部分提示责任
- 左侧日志、左侧解决区面板、右侧详情浮窗、右下角阶段面板同时存在时，界面边缘负担较重
- 许多操作依赖 hover，移动端和触屏适配会比较困难
- 资源区、能量区、成功区尺寸较小，某些状态需要靠 hover 才能读清
- 卡组检视、手牌快捷抽放、自由移动等功能带有明显 debug 属性，正式规则体验边界不够清楚

## 9. 结论

当前“游戏桌”UI 已经形成了比较稳定的基础结构：

- 主桌采用上下镜像的中心对称布局
- 主交互依赖拖拽
- 阶段流程通过右下角面板推进
- 复杂判定通过左侧抽屉处理
- 详细信息通过右侧详情和中央模态补充

如果后续要继续迭代设计，建议把问题拆成三条线：

- 桌面主布局是否要继续保留当前“调试型复合桌面”结构
- 提示系统是否需要收敛成更统一的层级
- 交互是否要从“自由拖拽 + 后端纠正”收敛为“前端引导更强的合法操作流”
