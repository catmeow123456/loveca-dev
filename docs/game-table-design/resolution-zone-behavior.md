# 解决区行为定稿

> 目的：固定“解决区”在当前牌桌中的正式语义，明确它不是一个模糊的“左侧判定区”，而是当前联机模型中的 `RESOLUTION_ZONE`。
>
> 若与 `game-table-ui-current-state.md` 的旧表述冲突，以本文为准。

## 1. 设计结论

当前表演阶段里所谓的“判定区”，正式语义应统一落为解决区（`RESOLUTION_ZONE`）。

也就是说：

- `JudgmentPanel` 只是当前查看和操作解决区的 UI 载体
- 解决区本身是正式牌桌区域，不是面板里的临时草稿容器
- 面板的展开、收起、切换，不会创建、清空或重置解决区

本文中的“判定区”若继续出现，只能理解为“解决区的当前 UI 视图”，不再作为独立规则区域使用。

## 2. 区域语义

解决区的正式角色是：

- 承载 Live 判定中的应援翻牌过程
- 承载这些牌在公开链路中的连续观察关系
- 承载这些牌后续移入手牌、休息室、主卡组顶等操作

它与检视区是两个不同区域：

- 检视区用于隐藏信息流程
- 解决区用于应援翻牌与公开处理流程

解决区是共享区域，但当前常见操作窗口通常围绕“当前正在进行表演判定的玩家”展开。

## 3. 进入解决区的默认过程

当前解决区的基础动作应固定为：

- 从当前表演玩家的主卡组顶翻开 1 张应援牌
- 该牌真实进入解决区

这个动作的正式语义不是“直接从卡组里凭空显示正面”，而是两步公开：

- 先以可连续追踪的对象进入解决区
- 再翻开为双方可见的正面信息

因此后续实现和文档都应遵守以下约束：

- 进入解决区与翻开是正式过程，不是本地预览
- 同一张牌在进入解决区和翻开之后，仍是同一个连续对象
- 不能把“翻开应援牌”描述成只是面板里的局部阅读动作

## 4. JudgmentPanel 的职责边界

`JudgmentPanel` 的职责应理解为：

- 展示当前表演玩家在解决区中的牌
- 提供翻开下一张应援牌的入口
- 提供把解决区牌移出到合法目标区的入口
- 提供 Live 成功 / 失败确认与后续推进入口

它不应承担以下语义：

- 作为解决区的唯一存在形式
- 关闭时自动清空解决区
- 关闭时把解决区牌自动塞回牌库
- 把解决区操作降级成“只在抽屉里看一眼”的局部 UI

因此：

- 打开左侧按钮，只是打开解决区 / 判定面板
- 收起面板，只是收起 UI
- 解决区内的牌是否存在，始终由服务端状态决定

## 5. 解决区内的可操作行为

当前解决区应固定支持以下类型的正式动作：

- 继续翻开 1 张应援牌进入解决区
- 将解决区牌移入手牌
- 将解决区牌移入休息室
- 将解决区牌放回主卡组顶或底
- 将解决区牌移入除外区

这些动作的边界如下：

- 它们虽然可以由拖拽或面板按钮触发
- 但命令层必须走解决区专用语义
- 不应把它们归并成通用 `MOVE_TABLE_CARD`

当前文档里若写“解决区 -> 手牌 / 休息室 / 卡组”，应明确指：

- 这是正式牌桌处理
- 不是本地排序或阅读辅助

## 6. 点击、拖拽与流程控制的分工

为避免解决区行为继续混乱，本文固定如下分工：

- 点击左侧解决区按钮：打开或聚焦解决区 / 判定面板
- 点击“翻开一张”：从当前表演玩家主卡组顶翻开 1 张进入解决区
- 拖拽解决区牌到目标区：表达正式移动处理
- 点击 `LIVE失败` / `LIVE成功` / `成功效果发动完毕`：表达流程确认

不应再出现以下混写：

- 把“打开面板”和“翻开一张应援牌”写成同一动作
- 把“查看解决区”与“处理解决区牌”写成同一层语义
- 把“面板关闭”写成“解决区流程结束”

## 7. 可见性与连续性

解决区中的牌属于共享观察链上的对象。

因此文档层必须固定以下结论：

- 牌进入解决区时，对手应能感知到对象进入了该区域
- 翻开前后应保持同一个连续对象
- 牌从解决区进入手牌、休息室或主卡组顶时，应仍然是正式移动过程

其中：

- 进入手牌后，对非持有者不再持续看到该对象
- 回到主卡组后，对非持有者也不再持续看到该对象
- 但“进入目标区”这一步本身仍然是正式可观察事实

## 8. 对当前术语的纠偏

以下旧说法现在应视为不够准确：

- “左侧判定区”
- “应援区”被当作独立于解决区的正式区域
- “打开 `JudgmentPanel` = 进入判定流程本体”

新的固定理解是：

- 左侧 UI 叫“解决区 / 判定面板”更准确
- 上半部分展示的是当前玩家在解决区中的应援牌
- 真正的正式区域是 `RESOLUTION_ZONE`
- `JudgmentPanel` 只是当前默认的查看与操作入口

## 9. 验收标准

后续实现或重构至少应满足以下场景：

- 在 Performance 判定阶段，点击左侧按钮只打开解决区 / 判定面板，不直接改动解决区状态
- 点击“翻开一张”后，当前表演玩家主卡组顶的 1 张牌进入解决区，并进入公开链路
- 同一张解决区牌在“进入区域”和“翻开正面”前后保持对象连续性
- 解决区牌可以被处理到手牌、休息室或主卡组顶，且这些都是正式状态变化
- 收起 `JudgmentPanel` 后，解决区内已有牌仍然存在，不被隐式清空
- 文案、设计稿、注释中不再把解决区和一个纯 UI 的“左侧判定抽屉”混为一谈

## 10. 本文不处理的内容

本文只冻结解决区的基础语义与交互边界，不展开以下议题：

- 各类应援效果的完整规则裁定
- 解决区中是否允许仅为阅读目的的本地重排
- 后续是否把解决区改为桌面常驻可见区域，而不依赖左侧面板
- 具体代码迁移顺序与组件拆分方案
