# 游戏桌移动端设计改进方向建议

> 目标：在保留当前对局信息结构和规则流转能力的前提下，将现有偏 PC / 调试桌面的交互，重构为更适合手机竖屏与单手操作的移动端游戏桌体验。

## 1. 当前问题概述

基于当前实现，移动端适配会遇到几个核心问题：

- 当前桌面依赖大量 hover 行为，而移动端没有稳定 hover
- 左日志、左判定抽屉、右详情浮窗、右下阶段面板同时存在，边缘悬浮层过多
- 上下双场地同时完整展示，对手机竖屏空间压力过大
- 许多区域尺寸过小，例如能量区、成功区、休息室缩略堆
- 当前以自由拖拽为核心，但移动端拖拽精度和反馈明显弱于 PC
- 多个辅助操作带有 debug 性质，不适合直接暴露在移动端主路径中

因此，移动端不应简单缩放当前桌面，而应重新定义信息层级和操作主路径。

## 2. 设计目标

建议移动端版本优先满足以下目标：

- 单手可完成绝大多数常用操作
- 当前回合、当前可操作对象、当前下一步足够明确
- 主桌只展示“当前最需要看的信息”
- 详情、日志、检视、判定等次级信息收纳进底部抽屉或全屏页
- 尽量减少精细拖拽，增加点击式和半引导式操作
- 仍保留 debug/自由操作能力，但放入次级入口

## 3. 总体方向

### 3.1 从“双方面全展开桌面”改为“主视角聚焦桌面”

建议移动端主桌只让己方区域占据主要空间，对手区域压缩为顶部摘要条，而不是继续维持 PC 的上下对称完整视图。

建议结构：

- 顶部：对手摘要区
- 中部：中央战场区
- 底部：己方手牌与主操作区
- 底部上方：可展开的上下文操作栏

这样做的原因：

- 手机竖屏最稀缺的是垂直空间
- 玩家操作几乎都集中在己方区域
- 对手完整桌面不是每一秒都需要高可见

### 3.2 从“多侧边悬浮层”改为“底部抽屉体系”

移动端建议统一采用 bottom sheet / 全屏模态，而不是左抽屉、右浮窗、右下固定面板并存。

建议统一成三层：

- 主桌层：只放核心对局信息
- 底部抽屉层：放阶段说明、判定、详情、日志、检视
- 全屏任务层：换牌、分数确认、休息室浏览等重任务

## 4. 推荐的信息架构

### 4.1 顶部区域

顶部建议只保留对手摘要信息：

- 对手头像 / 名称
- 手牌数
- 主卡组数量
- 能量数量
- 成功 Live 数
- 当前是否为对手回合

可选增强：

- 顶部摘要条点击后展开“对手场地详情”
- 详情以半屏或全屏面板打开，而不是常驻主桌

### 4.2 中央战场区

中央区域只保留本回合最关键的公开场地：

- 对手成员区摘要
- 中央 Live 区
- 己方成员区

推荐采用“中央战线”布局：

- 上排：对手成员槽和对手当前公开 Live
- 中排：回合 / 阶段 / 当前行动提示
- 下排：己方 Live 区和己方成员槽

移动端上不建议把资源区、成功区、休息室都常驻在主战区两侧。

### 4.3 资源区收纳

建议将以下区域从主桌移出，改为可点击摘要入口：

- 主卡组
- 能量卡组
- 休息室
- 成功区

入口形式可以是：

- 一排小型资源胶囊按钮
- 图标 + 数量
- 点击后拉起对应抽屉

例如己方资源栏可放在手牌上方：

- `主卡组 28`
- `休息室 6`
- `能量 4/7`
- `成功 1`

## 5. 交互改进方向

### 5.1 从“自由拖拽优先”改为“点击选卡 + 目标按钮优先”

移动端不建议继续以精细拖拽作为主交互。

推荐主交互改为：

- 点击卡牌进入选中态
- 底部弹出上下文动作条
- 用户点击目标动作完成操作

例如：

- 点击手牌成员卡 -> 出现“上场到左/中/右”
- 点击手牌 Live 卡 -> 出现“放入 Live 区”
- 点击场上成员 -> 出现“切换状态 / 查看详情 / 移动 / 放入休息室”

拖拽可以保留为高级操作，但不作为主要路径。

### 5.2 增加“合法目标高亮 + 非法动作隐藏”

移动端建议弱化“自由尝试后报错”的模式，改为：

- 选中卡牌后，仅显示当前合理动作
- 不显示明显非法动作
- 必要时在底部说明“当前阶段不能执行此操作”

这会显著降低手机端的认知成本。

### 5.3 用长按替代 hover

当前大量依赖 hover 的能力，需要转为：

- 单击：选中 / 打开动作
- 长按：查看卡牌详情
- 再点空白处：取消

这样可替代：

- 右侧详情浮窗
- hover 时显示快捷按钮
- hover 提示判定信息

### 5.4 减少双击交互

移动端不建议保留双击作为关键操作。

例如成员卡切换活跃/等待状态，建议改为：

- 点击成员卡
- 底部动作条显示“切换为等待 / 切换为活跃”

## 6. 提示与流程改进方向

### 6.1 固定底部流程条替代右下角阶段面板

建议将当前 `PhaseIndicator` 重构为底部固定流程条。

内容建议：

- 当前阶段
- 当前子阶段
- 当前轮到谁
- 当前建议动作一句话
- 一个主按钮
- 一个次按钮

示例：

- `表演阶段 · Live 判定`
- `请先完成应援与 Hearts 判定`
- 主按钮：`打开判定区`
- 次按钮：`撤销`

相比右下角浮块，底部全宽条更适合手机单手点击。

### 6.2 阶段提示横幅收敛为顶部 toast

移动端不建议保留大面积居中阶段横幅。

建议改为：

- 顶部短时 toast
- 或底部流程条内容更新并轻动画提示

原因：

- 中央弹出会遮挡有限主桌空间
- 用户注意力应尽量停留在当前操作区域

### 6.3 判定区改为全屏任务页

当前左侧 `JudgmentPanel` 在移动端建议升级为全屏任务页，而不是侧边抽屉。

推荐结构：

- 顶部：返回 + 阶段说明
- 中部上半：应援区
- 中部下半：Hearts 汇总 + Live 需求
- 底部：`LIVE失败` / `LIVE成功` / `成功效果发动完毕`

原因：

- 判定本身是高专注任务
- 涉及翻牌、统计、结果确认
- 在小屏上应提供完整上下文，不适合挤在窄侧栏里

### 6.4 分数确认保持全屏聚焦

分数确认建议继续使用模态，但移动端应改为接近全屏的底部或全屏确认页：

- 顶部显示双方分数对比
- 中部显示己方可编辑分数
- 底部显示确认按钮

尽量避免小卡片式模态。

## 7. 各元素的移动端改造建议

### 7.1 手牌

手牌建议变为底部横向可滑动卡列，而不是固定扇形。

推荐方式：

- 底部固定一行卡牌缩略图
- 可左右滑动
- 当前选中牌抬高并放大
- 上滑或点按进入动作面板

优点：

- 避免扇形在窄屏上被压缩
- 便于单手横向浏览
- 适合和底部动作条联动

### 7.2 成员区

成员区建议保留 3 槽，但放大交互热区：

- 卡牌本体可点击
- 槽位空位可直接点
- 选中手牌后，空槽变成明显可落点按钮

附加能量建议不要继续做复杂斜向叠堆，而改为：

- 卡牌右下角能量数量徽标
- 点开成员详情后再展示具体附加能量列表

### 7.3 Live 区

Live 区建议保留 3 槽，但强化状态表达：

- 未放置
- 已里侧放置
- 已公开
- 判定成功/失败

可用角标和边框颜色表达，不要依赖细小角标。

### 7.4 成功区与休息室

建议主桌只显示数量摘要，不显示完整缩略堆。

点击后打开全屏浏览页：

- 成功区：纵向列表或卡片带
- 休息室：网格浏览

### 7.5 能量区

建议主桌只显示总量和活跃量：

- `能量 4/7`

点开后进入能量详情抽屉：

- 查看每张能量牌
- 执行单张操作

## 8. 调试与正式操作分层

当前移动端如果直接暴露所有调试能力，会让主流程变得混乱。

建议分两层：

- 正式操作层：只保留规则主路径需要的动作
- 调试操作层：收纳到“更多”菜单或 debug 开关中

应收纳的 debug 型能力包括：

- 自由抽牌 / 放回
- 任意区域手动移动
- 视角切换
- 对墙打 / 调试模式切换
- 检视区高级操作

这样移动端主界面可以更干净。

## 9. 一个推荐的移动端页面草图方向

可参考如下结构：

### 9.1 主桌页

- 顶部状态栏
  - 返回
  - 回合 / 阶段
  - 设置
- 对手摘要栏
  - 名称、手牌数、资源摘要、成功数
- 中央战场
  - 对手场上成员摘要
  - 中央流程提示条
  - 己方 Live 区
  - 己方成员区
- 资源快捷栏
  - 主卡组 / 休息室 / 能量 / 成功区
- 手牌区
  - 横向卡列
- 底部动作条
  - 根据选中对象动态变化

### 9.2 底部动作条

未选卡时：

- `查看判定`
- `结束阶段`
- `更多`

选中手牌成员卡时：

- `上场左`
- `上场中`
- `上场右`
- `详情`

选中 Live 卡时：

- `放入 Live`
- `详情`

选中场上成员时：

- `切换状态`
- `附加能量`
- `移动`
- `详情`

## 10. 分阶段实施建议

### 第一阶段：信息层收敛

先做结构调整，不急着改全部规则交互：

- 取消左右两侧悬浮层
- 阶段面板改底部流程条
- 卡牌详情改长按打开底部详情页
- 对手区改摘要

### 第二阶段：交互重构

- 建立“选卡 -> 动作条 -> 执行”的主路径
- 保留拖拽作为补充能力
- 逐步减少非法操作入口

### 第三阶段：任务页重构

- 判定区改全屏任务页
- 休息室 / 检视 / 成功区改统一浏览页
- 分数确认改全屏确认页

### 第四阶段：调试能力下沉

- 将调试能力收纳到二级菜单
- 区分正式模式和调试模式的 UI 暴露层级

## 11. 结论

移动端改造的关键不是“把桌面缩小”，而是重建交互优先级：

- 主桌只保留当前回合最关键的信息
- 细节通过底部抽屉或全屏任务页展开
- 主操作从拖拽转向点击选中和动作按钮
- 提示系统从分散悬浮层收敛为统一流程条和任务页

如果按这个方向推进，最终会从当前的“PC 调试型对局桌”转成“移动端可控、可读、可单手操作的主视角战斗桌面”。
