# Loveca 对战模式目的与边界

> 文档类型：专题说明
> 适用范围：本地调试、对墙打、正式联机、正式联机观战、远程调试联机的桌面场景目的、能力边界和免费登场语义
> 当前状态：现行模式目的说明；已补充本地对墙打与服务端可记录对墙打边界；后续调整 `GameMode`、对战桌面能力 selector 或联机免费登场语义时应同步维护

本文档只回答“这些模式分别为谁、为哪种场景设计”。具体 UI 组件、命令可用性和投影实现仍以代码和对应设计/编码标准为准。

## 1. 两个不同维度

当前项目容易混淆的概念有两层：

- `GameMode`：规则自动化策略。它决定 `GameSession` 如何自动推进流程，例如 `DEBUG` 走双人完整流程，`SOLITAIRE` 自动处理对手流程。
- 对战桌面场景：UI 和客户端能力模型。它决定桌面是否显示视角切换、本地日志、撤销、模式切换、免费登场按钮等。

`GameMode.DEBUG` 不等于“显示本地调试 UI”。正式联机和远程调试联机的服务端 `GameSession` 也可以使用 `DEBUG` 规则策略，因为它表示“不自动替玩家跳过完整双人流程”。共享桌面组件不能只看 `gameMode` 来判断 UI 能力。

命名上需要保持清楚：`LOCAL_DEBUG`、`SOLITAIRE`、`ONLINE`、`REMOTE_DEBUG`、`SPECTATOR_READONLY` 是桌面场景标签，后续应作为 `BattleSurfaceKind` 这类派生 UI 能力字段使用；当前 `GameMode` 只有 `DEBUG` / `SOLITAIRE`，只表示规则自动化策略。`SOLITAIRE` 在两层里同名只是为了表达“本地对墙打场景由 `GameMode.SOLITAIRE` 派生”，不要因此把 `ONLINE`、`REMOTE_DEBUG` 或 `SPECTATOR_READONLY` 加进 `GameMode`。

## 2. 场景总览

| 场景                              | 权威状态                                        | 主要使用者                       | 目的                                                                             | 免费登场语义                     |
| --------------------------------- | ----------------------------------------------- | -------------------------------- | -------------------------------------------------------------------------------- | -------------------------------- |
| 本地调试 `LOCAL_DEBUG`            | 本地浏览器 `GameSession`                        | 开发者、卡效实现者、规则测试者   | 快速验证规则流程、卡效、UI 行为和双方视角                                        | 本地会话级兜底                   |
| 对墙打 `SOLITAIRE`                | 本地浏览器 `GameSession` 或服务端 `GameSession` | 单人测试玩家、卡效验证者         | 用接近玩家视角的方式推进一人对局，对手流程自动化；服务端可记录入口会进入历史复盘 | 本地会话级兜底或客户端命令级兜底 |
| 正式联机 `ONLINE`                 | 服务端 `GameSession`                            | 真实房间中的两名玩家、联机验收者 | 在隐藏信息和服务端权威下推进可玩的远程对局                                       | 客户端命令级正式兜底             |
| 正式联机观战 `SPECTATOR_READONLY` | 服务端 `GameSession`                            | 受邀观战者、联机验收者           | 按某一参赛玩家视角只读观看进行中对局和公共日志，不提供上帝视角                   | 不显示                           |
| 远程调试联机 `REMOTE_DEBUG`       | 服务端 `GameSession`                            | 开发者、联机投影/传输调试者      | 用服务端权威验证联机同步、投影、命令链路和座位视角                               | 客户端命令级正式兜底             |

## 3. 维护价值与收敛条件

这五个场景目前都有维护价值，但价值来源不同：

- `LOCAL_DEBUG`：高价值。卡效开发、规则复现、双人视角检查和快速误操作回退仍依赖它。只有当自动化测试、回放工具和管理诊断入口足以替代同浏览器双视角调试时，才考虑弱化。
- `SOLITAIRE`：中高价值。它服务单人可测和接近玩家视角的卡组/卡效 smoke test，不应扩展成完整 AI 对战。若后续维护成本升高，应先缩小为“自动确认/跳过对手无输入流程”的轻量模式，而不是删除。
- `ONLINE`：必须维护。它是正式产品链路和联机验收目标，不能被本地调试或远程调试替代。
- `SPECTATOR_READONLY`：正式联机的只读延伸。它复用玩家视角投影验证隐藏信息边界和公共日志可读性，但不能拥有命令、撤销、免费登场或上帝视角能力。
- `REMOTE_DEBUG`：技术调试价值。它用于验证服务端权威、座位视角、隐藏信息投影、命令传输和同步恢复。只有当正式联机入口已经提供等价的开发者诊断、座位视角验证和传输调试能力时，才考虑并入正式联机诊断工具。

“免费登场”不是独立模式，而是当前规则自动化未闭环阶段的桌面能力。随着费用计算、替代登场和卡效自动化补齐，可以缩小它的展示范围或默认状态；但在没有替代流程前，不应从正式联机移除。

## 4. 本地调试

本地调试用于开发和验证，不面向正式对局体验。

它需要：

- 同一浏览器内切换双方视角。
- 查看本地 action log / 调试日志。
- 切换本地 `GameMode`，便于在调试和对墙打策略之间比较。
- 使用本地撤销，快速回退误操作。
- 使用免费登场兜底，绕过尚未自动化或正在验证的费用路径。

本地调试可以暴露更多内部便利能力，但这些能力不能被正式联机继承。

## 5. 对墙打

对墙打用于单人可测，不是完整 AI 对战。

当前对墙打有两种权威来源：

- 本地对墙打：浏览器内 `GameSession`，用于离线 / 开发 fallback；保留本地撤销和本地会话级免费登场，不写入历史记录。
- 服务端可记录对墙打：服务端 `GameSession`，使用 `GameMode.SOLITAIRE` 自动处理对手流程；登录且 API 可用时从对墙打入口创建，写入 match record、系统对手 participant、默认对手卡组快照和 authority checkpoint，后续可在历史对局中复盘。

两种来源的桌面场景仍都是 `SOLITAIRE`。差异应由权威来源与桌面能力 selector 表达，不要把“可记录对墙打”扩展成新的 `GameMode`。

它需要：

- 让玩家只关注己方流程。
- 自动跳过或处理对手侧无输入流程。
- 弱化对手区域展示。
- 保留必要的测试便利能力；本地来源使用本地撤销，服务端来源使用服务端权威撤销。
- 进入桌面前显示单人模拟提示，说明对墙打的默认对手、自由操作兜底、能量与检视处理方式。

服务端可记录对墙打不使用本地撤销；撤销通过服务端即时撤销接口回滚权威状态并写入历史记录。其免费登场按命令级 `freePlay` 兜底提交，和正式联机保持同类安全边界。

对墙打不需要本地双视角调试入口，也不应把调试日志作为主要体验。

## 6. 正式联机

正式联机用于真实远程房间和联机验收。它的核心目标是“服务端权威 + 隐藏信息正确 + 玩家能推进对局”。

正式联机不应显示：

- 本地双视角切换。
- 本地模式切换。
- 本地 action log 面板。
- 本地调试用分数确认透传遮罩。
- 本地撤销。

正式联机可以显示请求式远程撤销。玩家发起请求后由对手拒绝、接受一步，或允许同一操作窗口内连续撤销；服务端负责校验 revision、undo entry、连续授权、随机/洗切边界和参与者身份。连续授权不是整回合授权，换阶段、有新命令、超时或撤销目标不再连续后会失效。若拟撤销操作已经公开隐藏信息，撤销只回滚局面，不能消除对手已经看见的信息。

正式联机也可以显示请求式重开。重开影响双方共享对局状态，不能做成单方即时按钮；玩家发起后由对手接受、拒绝或等待请求超时，发起者可取消。对手接受后，服务端应封存当前 match 记录，并在同一房间回到准备阶段；双方可以沿用已锁定卡组直接准备，也可以重新锁定其他合法云端卡组。双方再次准备后进入开局猜拳；猜拳胜者决定自己先手或后手后，服务端再创建新的 match，前端同步新的 `matchId`。这类入口属于房间 / 对局会话操作，适合与“离开房间”放在同一组操作区，但语义上必须区分：离开是单方退出，重开是双方协商。

正式联机的先后手应属于正式开局流程，而不是房间准备页里的房主提议。双方锁定卡组并都确认准备开始后，房间进入开局状态；双方暗选石头 / 剪刀 / 布，未公开前只能看到对手是否已选择，双方提交后公开结果。若平局则重新猜拳；若分出胜负，则由胜者决定自己先手或后手，然后进入真正的 `GameSession` 对局流程。

正式联机进入桌面后应显示联机对局提示，提醒双方按玩家视角操作、先沟通同一时点效果顺序、理解请求式撤销/重开与隐藏信息边界。提示只服务玩家流程说明，不改变服务端命令校验或玩家视角投影边界。

正式联机的普通观战统一使用房间号入口，不提供参赛玩家专用观战链接。房间号观战默认开放双方玩家视角，玩家可以在对局中关闭或重新开启自己的视角；观战会话只在当前获授权的视角间切换。观战者通过已开放的房间号入口加入后，只读取对应的 `PlayerViewState` 投影和公共日志增量，不读取权威状态、对手隐藏信息或私有事件。观战桌面必须是只读场景：不能提交命令、推进阶段、发起撤销或使用免费登场；undo 投影应显示为禁用。管理员房间监控可以生成指定先攻/后攻的玩家视角只读入口，但这类管理员入口不计入公开观战人数。观战不是上帝视角，也不替代管理员审计。

正式联机必须保留“免费登场”按钮，但它不是调试作弊开关。它的语义是规则自动化缺口的正式兜底开关：

- 动态费用尚未完全被 `costCalculator` 覆盖时，正常登场可能因为费用计算不准失败。
- 被卡效、被动或替代流程登场的成员，不一定应该走普通登场费用。
- 部分复杂卡效尚未自动化时，玩家需要手动把规则结果落实到桌面上。
- 当前项目目标是“规则正确 + 玩家视角可测 + 联机能推进”，不是马上做到所有规则全自动闭环。

正式联机中的免费登场应按命令级生效：客户端提交 `PLAY_MEMBER_TO_SLOT` 时携带 `freePlay: true`，服务端仍校验操作者、自由拖拽窗口、己方手牌、成员卡类型和目标槽位，只跳过本次登场费用检查与支付。它不写入远程房间全局状态，避免双方 UI 偏好漂移。

## 7. 远程调试联机

远程调试联机用于开发联机链路，不是本地调试桌面的远程版本。

它需要：

- 使用服务端权威状态。
- 保留座位视角和隐藏信息投影。
- 验证远程命令提交、快照同步、公开事件、恢复帧和投影边界。
- 允许调试入口外层提供诊断信息。

远程调试联机不应继承本地调试桌面的双视角切换、本地日志、模式切换或本地撤销。若某些实现需要区分远程调试传输行为，应通过远程会话/传输层 selector 表达，不要把它混入普通桌面调试能力。

## 8. 能力模型要求

共享桌面组件应读取一个派生的桌面能力对象，而不是在组件里重复组合 `gameMode`、`remoteSession` 和 `remoteSession.source`。

建议能力至少表达：

- 权威来源：本地或远程。
- 桌面场景：本地调试、对墙打、正式联机、正式联机观战、远程调试联机。
- 是否可切换视角。
- 是否可切换本地模式。
- 是否显示本地调试日志。
- 撤销策略：本地立即撤销、远程立即撤销、远程请求撤销或禁用。
- 是否显示免费登场。
- 免费登场策略：本地会话级或命令级。
- 是否使用对墙打展示。
- 分数确认弹窗展示策略。
- 是否只读。

能力模型只服务 UI 展示和命令提交策略，不作为服务端合法性判断。服务端仍必须由 `GameSession` / command 层校验身份、阶段、区域、卡种和目标。

## 9. 维护规则

- 新增桌面入口时，先明确它属于哪种权威来源和哪种对战桌面场景。
- 新增 UI 控制项时，先判断它是本地调试能力、单人测试便利、正式联机兜底，还是远程调试诊断能力。
- 涉及正式联机免费登场时，不得把它描述为作弊或调试开关；它是当前规则自动化未闭环阶段的正式兜底。
- 如果以后某类规则自动化补齐，可以缩小免费登场的使用场景，但不应在没有替代流程前移除正式联机兜底。
- `GameMode` 的注释和使用应逐步收敛为规则自动化策略，不再承担完整 UI 场景标识。
- 后续触碰 `src/shared/types/enums.ts`、`src/application/game-session.ts`、`docs/solitaire-mode-requirements.md` 等旧注释或旧需求文档时，应同步把 `GameMode.DEBUG` 的说明改成“完整双人流程/无对手自动化策略”，避免继续暗示它等于本地调试 UI。
