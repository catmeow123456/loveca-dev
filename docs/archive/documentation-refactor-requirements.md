# Loveca 文档重构需求文档

> 文档类型：需求文档
> 适用范围：Loveca 项目文档体系重构
> 目标读者：项目维护者、功能开发者、文档维护者
> 当前状态：已归档；不作为项目实现事实来源

## 1. 背景

Loveca 当前已经形成了较完整的功能实现和专题文档，包括总体需求、系统设计、联机模式、卡组管理、卡牌数据管理、UI 改进、编码标准等内容。

随着项目持续迭代，部分文档出现以下问题：

- 历史方案、当前实现和未来计划混在同一文档中，读者难以判断可信度。
- 部分文档仍保留旧技术栈、旧路径或旧状态描述，容易误导开发。
- 文档入口分散，缺少清晰的主干导航。
- 同一主题在多个文档中重复描述，且更新节奏不一致。
- 临时任务记录和长期项目文档边界不清。

因此，需要对文档体系进行一次面向长期维护的重构。

## 2. 重构目标

文档重构应达成以下目标：

- 建立清晰、稳定的文档主干，让新维护者能快速理解项目现状。
- 明确区分需求文档、设计文档、编码标准、历史方案和临时记录。
- 删除或归档过时描述，降低文档与代码实现不一致的风险。
- 让每份文档都有明确用途、适用范围、当前状态和维护责任边界。
- 减少重复描述，使同一事实尽量只在一个权威位置维护。
- 保留对开发有价值的背景与约束，但避免文档成为实现细节堆积。
- 控制代码变更引发的文档修改范围，避免一次局部实现调整牵动多份文档同步修改。

## 3. 范围

本次文档重构覆盖：

- 根目录文档，包括 `README.md`、`game_system_design.md`、`progress.md` 等。
- `docs/` 下的需求文档、设计文档、专题说明、编码标准和 UI 文档。
- 与文档入口、文档命名、文档状态、文档引用关系相关的内容。
- 明确已经过时但仍有历史参考价值的文档。

本次文档重构不覆盖：

- 业务代码重构。
- 游戏规则、产品功能或 UI 方案本身的重新设计。
- 数据库 schema、API、组件或测试的实现变更。
- 外部资料本身的改写，但需要标注来源和当前用途。

## 4. 文档分类需求

重构后的文档体系应至少区分以下类型：

- 总览文档：说明项目是什么、如何运行、主要模块在哪里、文档从哪里读起。
- 需求文档：描述产品、规则或工程目标，不包含具体代码实现。
- 设计文档：描述当前实现架构、数据流、模块职责和关键代码路径。
- 编码标准：描述开发约束、边界规则、代码风格和评审标准。
- 专题说明：描述某一局部问题的当前事实、边界或约束。
- 历史/计划文档：保留尚未落地或已经被替代的方案，并清楚标注状态。
- 临时记录：只用于短期任务交接，不应作为长期事实来源。

每份长期文档都应能被归入上述某一类。

## 5. 内容质量需求

重构后的文档应满足以下质量要求：

- 文档开头应说明文档类型、适用范围和当前状态。
- 描述当前实现时，应与代码路径和实际行为保持一致。
- 描述未来计划时，应明确标注为计划、建议或未实现能力。
- 需求文档不得写入具体实现代码、文件级改造步骤或迁移脚本。
- 设计文档可以列出关键代码路径，但不应复制大段源码。
- 外部资料、第三方接口和迁移来源应标明原始来源或维护依据。
- 已知限制应集中描述，避免散落在多个文档中造成冲突。
- 删除过时描述时，应优先保留仍能解释当前设计取舍的背景。

## 6. 文档更新半径控制需求

重构后的文档体系应降低“代码一处修改、文档多处连锁修改”的维护成本。文档职责边界应让维护者能快速判断某次代码变更是否需要改文档，以及只需要改哪一份文档。

整理时应遵循以下要求：

- 同一当前事实只能有一个权威维护位置；其他文档如需提及，应使用链接或简短引用，不重复展开。
- 每份长期文档应说明自身负责维护的事实类型，以及不负责维护的内容边界。
- 文档应区分稳定事实和易变事实。稳定事实包括产品目标、规则约束、模块职责、外部接口契约、部署边界；易变事实包括内部实现步骤、局部组件结构、临时任务状态、具体排期和一次性 checklist。
- 设计文档引用代码路径时，应优先引用模块级或入口级路径，避免复制局部实现细节。只有当文件路径本身构成关键架构边界时，才应作为长期事实维护。
- 专题文档只维护本专题的权威事实；跨专题限制、部署差异和已知实现偏差应集中到专门的限制文档或主干设计文档中。
- 计划文档、方案建议和历史记录不得重复维护当前实现事实；若需要说明当前状态，应链接到当前权威文档。
- 临时进度、阶段实施记录和评审意见不应成为代码变更时必须同步更新的对象。

代码变更触发文档更新的规则应尽量明确：

- 若变更影响产品行为、规则边界、用户流程、API 契约、数据 schema、部署方式、鉴权模型、联机同步边界或已知限制，应更新对应权威文档。
- 若变更只是局部重构、命名调整、组件内部实现优化、测试补充或不改变外部行为的代码整理，通常不应要求同步修改长期文档。
- 若变更只影响某一专题，应只更新该专题权威文档；不应同时在总览、需求、设计和计划文档中重复改同一事实。
- 若发现需要修改多份文档表达同一事实，应优先重构文档职责或抽取权威说明，而不是继续同步复制。

验收时应能够回答：

- 某个事实的权威来源是哪一份文档。
- 某次代码变更最多需要同步哪些文档。
- 哪些文档只是背景或历史参考，不需要随当前实现细节变化而更新。

## 7. 长期文档与临时记录边界

重构后的长期文档应重点保留需求背景、设计约束、当前能力边界和关键取舍原因，不应承担开发日志、阶段任务记录、评审记录或问题追踪工具的职责。

整理时应遵循以下要求：

- 删除已经完成或失效的阶段实施记录、一次性评审记录、临时问题列表和任务交接碎片。
- 已完成的 checklist 不应作为长期事实保留；若仍用于回归测试或人工验收，应标注文档类型、适用范围、当前状态和维护边界。
- 对仍有价值的历史记录，应提炼成“背景原因”“设计约束”或“当前限制”，而不是保留执行过程本身。
- 临时待办、问题列表和评审意见不应混入需求文档或设计文档；未解决事项应改写为明确的“当前限制”或“后续计划”。
- 主阅读路径不应要求读者理解历史任务过程才能获得当前项目事实。

## 8. 一致性需求

文档体系应与当前项目事实保持一致，重点包括：

- 技术栈、运行方式、数据库、对象存储和鉴权模型。
- 对局引擎、联机模式、卡组管理、卡牌数据管理的当前能力边界。
- 正式能力、调试能力、历史方案和后续计划之间的区别。
- 文件路径、组件名称、服务名称、API 入口和状态字段。
- 已实现功能与未实现能力的边界。

当代码实现发生变化时，相关权威文档应同步更新。

## 9. 导航需求

重构后的文档应提供清晰导航：

- 根目录 `README.md` 应作为项目文档入口。
- 重要长期文档应能从 `README.md` 或主干文档中被找到。
- 专题文档之间应避免互相形成复杂引用链。
- 历史或计划文档应避免出现在主阅读路径中，除非明确需要背景说明。
- 临时进度记录不应被主文档当作权威来源引用。

## 10. 验收标准

文档重构完成后，应满足以下验收标准：

- 新开发者能通过根目录入口找到项目总览、系统设计、主要专题和编码标准。
- 主要文档不再包含已知过时技术栈、旧路径或已被实现替代的状态描述。
- 每份长期文档都能说明自身类型、范围和用途。
- 同一主题不存在多个互相冲突的当前事实描述。
- 历史方案、计划事项和当前实现边界能够被明确区分。
- 文档体系不要求读者阅读临时任务记录才能理解项目现状。
- 长期文档中不再混入已完成阶段记录、一次性评审记录、临时问题列表或任务交接碎片。
- 对一个典型局部代码变更，维护者能判断是否需要改文档；若需要，通常只需更新对应权威文档，而不是多份文档同步复制同一事实。
- 同一当前事实不存在多个必须同时维护的重复描述。

## 11. 非目标

本次文档重构不追求：

- 把所有历史文档全部删除。
- 把所有专题文档合并成单一大文档。
- 用文档替代代码、测试或类型定义作为实现真相。
- 为尚未决定的产品方向提前写成确定承诺。
- 在需求文档中指定具体改造步骤、负责人或排期。
