# Loveca 对局记录、复盘与可复现回放需求文档

> 文档类型：需求文档
> 适用范围：对局记录保存、用户复盘、回放观看、卡组快照和确定性重演能力
> 当前状态：现行需求基线；正式联机与服务端可记录对墙打的记录式回放已落地，完整随机记录、完整决策覆盖、确定性重演和公开分享回放仍未实现
> 最后更新：2026-07-24

## 1. 文档目的

本文档用于定义 Loveca 下一阶段“卡效可复盘、卡组可复现、对局可回放”的产品与工程需求。

目标不是立刻实现完整事件源化系统，也不是一次性完成全卡池自动裁判。第一步要先把需求边界说清楚，让后续设计和实现围绕同一个目标推进：

> 用户可以保留自己的对局记录，并在对局结束后随时回过头查看、复盘和回放该局。

长期目标还包括：

- 开发者能够复现线上卡效 bug。
- 玩家能够按自己的视角回看关键选择。
- 裁判或管理员在授权范围内能够辅助争议复盘。
- 未来 AI、观战、教学和战报能力可以复用同一套记录基础。

## 2. 核心结论

最终可以实现“用户保留对局记录并随时回放”的效果，但需要区分两个层级：

1. **记录式回放**
   保存对局过程中的状态快照、公共事件、玩家视图和关键选择，让用户可以像看录像一样回看。该能力更适合第一阶段落地。

2. **确定性重演**
   从初始状态、卡组快照、随机种子、命令序列和卡效决策日志重新运行引擎，并得到同一局结果。该能力更适合第二阶段逐步完善。

第一阶段不必等待完整确定性重演完成，也不必等待所有卡效自动化完成。只要记录足够完整，用户就可以先获得可用的复盘体验；同时，为后续确定性重演保留必要数据。

第一阶段的记录生命周期应从对局开始，而不是对局结束才补建记录：

- 对局创建时锁定参与者、座位、卡组快照和记录 ID。
- 对局初始化和推进时持续追加命令、事件、快照、决策和审计事实。
- 对局正常结束、投降、异常中断或过期清理时，将记录标记为对应状态。
- 用户在对局结束后看到的是已经封存或部分封存的历史记录。

这能避免用户后续改卡组、进程重启、异常断线或卡牌数据更新导致历史事实丢失。

2026-06-19 补充：普通用户历史记录的第一版范围已从“正式联机”扩展到“正式联机 + 服务端可记录对墙打”。本地 / 离线对墙打仍是本地 `GameSession` fallback，不承诺保存历史，也不会上传本地权威状态形成历史记录。

早期实现可以先落地一个更低风险的 E0 阶段；当前主线已经进入正式持久化记录和玩家视角只读回放的 P0-P1 阶段：

- E0：管理员专用临时导出 / 导入调试回放包。
- E0 只服务开发调试、卡效 bug 复现和 replay 格式验证，不进入普通用户历史对局列表。
- E0 可以包含权威检查点、双方隐藏信息和密封审计，因此必须只开放给管理员或开发审计入口。
- E0 导出文件必须带 `recordSchemaVersion`、项目版本或 git commit、卡牌数据 hash、导出时间和能力标记；导入时允许因版本不兼容而拒绝或降级为仅审计查看。
- E0 不承诺长期跨版本兼容，不触发线上用户数据 migration 义务；当前正式记录链路已经复用同一套逻辑边界沉淀到 P0-P1 的持久化记录。
- 2026-06-28 起，同一 `DebugReplayBundle` envelope 也可承载从正式历史记录导出的高权限回放包；这类包以 `sourceMatch.exportedStatus=HISTORY_RECORD` 标识，不标记 `NOT_USER_HISTORY_RECORD`，但仍只允许管理员下载 / 导入，不改变普通玩家按视角读取历史回放的权限边界。

第一阶段必须拆成可独立验收的小步推进，而不是把历史列表、回放 UI、决策解释、审计导出和确定性重演一次性打包。当前状态是：P0-P1 已有阶段性用户闭环，P2/P3 与确定性重演仍需继续推进。推荐优先级仍是：

- E0：管理员专用调试回放包导出 / 导入。
- P0：记录创建、卡组快照、初始检查点和结束封存。
- P1：时间线、检查点读取和玩家视角只读回放。
- P2：卡效决策、手动处理原因和能力标记。
- P3：回放 UI 体验增强、关键节点说明和开发/审计导出。

E0 不构成普通用户可交付 MVP；它是正式持久化前的管理员调试入口。P0-P1 构成第一阶段首个用户可交付 MVP：用户能在对局结束后看到历史记录，并以自己的当时视角查看离散检查点。当前实现已接入正式联机与服务端可记录对墙打的历史根记录、卡组快照、timeline、authority checkpoint、public/private event 明细、普通玩家历史列表/详情/timeline/checkpoint 投影读取和只读 GameBoard 桌面展示。P2 是第一阶段的复盘解释增强闭环，应在 P0-P1 稳定后继续推进，不应阻塞第一版历史列表和只读回放落地。P3 可在核心闭环稳定后继续推进。完整确定性重演仍属于第二阶段，不应被第一阶段产品文案承诺为已完成能力。

P0/P1 实现与后续扩展仍应遵守以下工程边界，避免把运行时恢复能力误当成长期历史记录：

- 记录层必须拥有独立的 `timelineSeq` 与 `checkpointSeq`；现有公共事件序号可以作为关联字段，但不能作为历史检查点的唯一身份。
- 记录层应以 `RecordFrame` 或等价时间线账本统一排序命令、事件、决策、随机事实和检查点；`PublicEvent.seq`、`commandSeq`、`checkpointSeq` 都只能作为关联字段，不能各自承担全局回放顺序。
- 记录层应同时区分规则事件与公开展示事件：`GameState.eventLog` / `GameEvent` 是当前规则层正在收口的内部事件事实来源，`PublicEvent` / `PrivateEvent` / `SealedAudit` 分别服务公共展示、玩家私密视角和高权限审计。
- 若第一阶段要持久化规则事件增量，应先提供稳定的 `GameEvent` 增量读取或 recorder hook；不应把当前运行时调试用的完整权威状态读取当作长期记录接口。
- checkpoint 和 debug bundle 的 authority payload 必须遵守 [checkpoint / bundle 序列化与复水契约](serialization-contract.md)：写入前转换为 JSON-safe payload，读取时校验版本、hash、能力标记并复水；不得直接把原始 `GameState` 做裸 JSON 存储或返回普通玩家接口。
- 卡效 runner、trigger matcher 和 steps 后续仍会继续重构；记录层应保存稳定语义事实和带版本的决策摘要，不应把 `pendingAbilities`、`activeEffect` 或 runner 内部 payload 原样作为长期存储格式。
- 卡组快照输入必须在锁卡或开局时形成，不能只保存运行时 `DeckConfig` 或只引用用户当前 deck 外键。
- 若当前卡牌数据尚无正式版本号，卡组快照应至少保存锁卡时的卡牌摘要与规范化 hash，作为当局展示事实和后续兼容判断的来源标记。
- 正式对局启动链路需要支持 recorder 前置写入；创建历史根记录、参与者、卡组快照或初始检查点失败时，不应进入正常 `IN_GAME` 状态。
- 正式联机默认不能无 recorder 启动。开发或本地调试若需要临时跳过 recorder，必须使用显式配置开关，并在日志或调试 UI 中标明该局不是可持久回放记录。
- P0-P1 的追加写入失败必须有明确降级策略：记录标记为部分记录或不完整记录，并保留可排查的服务端错误；不能让用户看到伪装成完整的历史记录。
- 当前若尚未实现投降命令或 API，P0 首版不应被投降 UI 阻塞；但记录模型必须预留投降结束状态，一旦后续加入投降入口，该入口必须调用封存流程。

## 3. 背景与动机

Loveca 当前已经具备正式联机对局基础闭环、阶段性历史记录/只读回放能力和第一阶段卡效自动化能力，但仍存在以下限制：

- 历史根记录、timeline、authority checkpoint、public/private event 明细和玩家视角只读 checkpoint 回放已阶段性落地。正式联机进程重启后的运行中对局恢复尚未落地；服务端可记录对墙打在运行态缺失时已能从最新 authority checkpoint 与公共事件尾部恢复到最近保存点，恢复可能回退到该保存点，且本地撤销栈不会跨复水保留。
- `GameState.eventLog` / `GameEvent` 已开始承载规则事件事实，当前已覆盖登场、离场、成员状态变化、成员槽位移动、LIVE 开始、LIVE 成功和声援等 proving path；运行中产生的 game event 已可作为 recorder 关联事实写入 timeline 游标，但仍没有覆盖所有卡效和公共可见变化。
- 部分卡效内部动作仍处于 snapshot/audit 过渡语义。
- 自动卡效涉及检视顶牌、声援、洗牌、选择顺序和玩家决策，缺少复盘数据会导致线上问题难以重现。
- 自由拖拽与信任玩家模型需要更清楚地记录“为什么发生这次移动”。

因此，对局记录和回放不是锦上添花，而是后续卡效自动化、联机稳定性、争议复盘和开发调试的共同底座。

## 4. 使用角色

### 4.1 普通玩家

普通玩家希望：

- 查看自己参与过的历史对局。
- 回看某局的完整流程。
- 复盘关键选择，例如换牌、Live 设置、卡效选择、成功 Live 选择。
- 对照当时的手牌、场面和卡组剩余信息，理解自己为什么赢或输。
- 在不泄露额外隐藏信息的前提下分享对局记录。

### 4.2 对局双方

对局双方希望：

- 对局结束后都能保存该局记录。
- 回放时只能看到自己在当时本应看到的信息。
- 如果双方同意，允许打开完整公开复盘或共享复盘。

### 4.3 管理员或裁判角色

管理员或裁判希望：

- 在明确权限和场景下辅助处理争议。
- 查看足够的审计信息，判断某一步是玩家操作、系统规则处理、卡效自动处理还是手动修正。
- 不在普通观测页面泄露玩家隐藏信息。

### 4.4 开发者

开发者希望：

- 用某局记录复现卡效、阶段推进、联机同步或投影问题。
- 从日志中定位导致错误的命令、卡效决策、随机结果和状态变化。
- 用历史对局作为回归测试样本。

## 5. 产品目标

### 5.1 用户可保留对局记录

系统应支持用户保留自己参与过的历史对局。记录应在对局开始时创建，在对局过程中持续追加事实，并在结束后形成可查看的历史记录。

记录至少应包含：

- 对局基本信息；
- 对局双方；
- 使用卡组；
- 开始和结束时间；
- 胜负结果；
- 回合数；
- 关键公共事件摘要；
- 是否支持回放；
- 是否支持完整确定性重演。

用户应能在自己的历史对局列表中找到这些记录。

历史记录必须明确表达来源模式：

- `matchMode=ONLINE`：正式联机对局。
- `matchMode=SOLITAIRE`：服务端可记录对墙打。

对墙打记录中的系统对手必须以 `participantKind=SYSTEM` 表达，不能伪装成真实用户；普通历史读取授权只基于真实用户 participant。

如果对局异常结束或记录不完整，历史列表也应明确标记记录状态，而不是静默隐藏或伪装为完整记录。

### 5.2 用户可按视角回看对局

用户回看对局时，默认应按其参与该局时的视角展示：

- 当时可见的手牌、场面、公开卡、解决区和检视区信息。
- 当时不可见的信息仍不可见。
- 已公开后又进入隐藏区的卡牌，应遵循当时的可见性边界。
- 回放不应因为对局已经结束就默认暴露所有隐藏信息。

### 5.3 用户可复盘关键节点

回放不应只是连续播放状态。系统应支持用户定位关键节点，例如：

- 开局和换牌；
- 每回合开始；
- 每次主要阶段开始和结束；
- 成员登场；
- 起动能力发动；
- 自动能力入队和结算；
- Live 设置；
- 表演、声援和判定；
- 成功 Live 选择；
- 卡效手动处理；
- 规则修正或系统纠偏；
- 胜负结算。

### 5.4 用户可暂停、跳转和逐步查看

回放体验应支持：

- 从头开始观看；
- 暂停；
- 前进到下一事件；
- 回退到上一事件或上一快照；
- 跳转到指定回合、阶段或关键事件；
- 查看当前节点的事件说明和操作来源。

第一阶段可以先支持离散事件跳转，不要求完整动画时间轴。

### 5.5 卡组可复现

对局记录必须保留对局开始时双方使用的卡组快照，而不是只引用用户当前卡组。

原因：

- 用户之后可能修改或删除卡组。
- 卡牌数据可能更新。
- 同一基础编号不同罕度、同名规则和卡图信息需要保持当局一致性。
- 回放和重演必须基于当时锁定的卡组内容。

卡组快照应满足：

- 能还原主卡组和能量卡组的卡牌构成。
- 能说明当时使用的卡牌数据版本或快照来源。
- 能区分卡牌编号和对局中的卡牌实例。
- 不因用户之后编辑卡组而改变历史对局。

### 5.6 卡效可复盘

卡效自动化和手动处理都必须能在复盘中解释。

对自动卡效，记录应表达：

- 哪张卡的哪个能力触发或发动；
- 触发时点；
- 来源区域；
- 是否可选；
- 玩家进入决策窗口时看到的候选、步骤文本和限制摘要；
- 玩家是否选择发动或跳过；
- 玩家选择了哪些目标；
- 该选择关联到哪些时间线位置、事件范围或检查点范围；
- 产生了哪些公开结果；
- 该效果是否属于已自动化范围。

对手动卡效或自由拖拽处理，记录应表达：

- 操作者；
- 操作原因，例如规则流程、卡效处理、桌面整理、误操作修正、双方约定修正或系统纠偏；
- 是否来自某张卡的效果；
- 若来自卡效，来源卡、能力或玩家声明的效果说明；
- 是否属于误操作修正；
- 移动或调整了哪些对象；
- 服务端是否接受；
- 是否存在只供服务端审计的隐藏信息。

如果第一阶段尚不能完整结构化某次手动处理原因，记录也不能静默丢失该事实。该节点至少应标记为“手动处理 / 原因未结构化 / 记录式回放可看但不可确定性重演”，并保留操作者、命令、前后检查点和必要审计信息。

### 5.7 可用于开发复现

面向开发者的记录应能支持问题定位：

- 某一步前后的权威状态；
- 玩家提交的命令；
- 命令接受或拒绝原因；
- 当时可用命令或操作窗口；
- 卡效决策；
- 随机、洗牌或公开结果；
- 相关公共事件和私密事件。

这些信息不一定都暴露给普通玩家，但应在权限允许的开发或审计场景中可用。

## 6. 回放类型

### 6.1 玩家视角回放

玩家视角回放是第一优先级。

特点：

- 使用玩家当时视角。
- 不额外泄露隐藏信息。
- 面向用户复盘和收藏。
- 允许展示该玩家当时的手牌、选择和提示。

### 6.2 公共视角回放

公共视角回放用于分享和观战式复盘。

特点：

- 只展示双方都能看到的信息。
- 不展示任一方隐藏手牌或牌库顺序。
- 适合分享给第三方。
- 可作为未来战报、观战和教学功能基础。

### 6.3 完整审计回放

完整审计回放用于管理员、裁判或开发者。

特点：

- 可包含双方隐藏信息、卡组顺序、随机结果和密封审计。
- 必须受权限控制。
- 不应作为普通玩家默认能力。
- 主要用于争议处理、bug 复现和自动化测试。

### 6.4 确定性重演

确定性重演是从初始状态和命令/决策序列重新运行规则引擎。

特点：

- 用于验证记录是否能重现同一局。
- 用于发现规则实现变化导致的行为差异。
- 对自动卡效和随机过程要求更高。
- 可作为未来回归测试和 AI 模拟基础。

第一阶段可以记录重演所需数据，但不要求所有历史记录都能通过确定性重演校验。

## 7. 功能范围

### 7.1 第一阶段 MVP 必须支持

第一阶段可按 E0-P3 顺序落地。E0 是管理员验证阶段，不算普通用户 MVP；P0-P1 是首个用户可交付 MVP，P2 是第一阶段增强闭环，P3 是体验和审计增强。进入下一优先级前，上一优先级应能独立验收，避免把所有能力绑定成一次性大改。

#### E0：管理员专用临时导出 / 导入调试回放包

E0 目标是在不上正式对局存储、不承诺用户历史记录兼容性的前提下，先验证 recorder、检查点、时间线和只读回放读取模型。

必须支持：

- 仅管理员或开发审计入口可以导出和导入调试回放包，普通用户不可见、不可下载、不可导入。
- 导出对象可以是仍在内存中的运行中对局或刚结束对局；首版不要求写入 `match_records` / `match_timeline_entries` 等正式历史表。
- 导出包应包含 `recordSchemaVersion`、项目版本或 git commit、规则版本、卡牌数据 hash、导出时间、能力标记、参与者、座位、卡组快照、初始 / 当前权威检查点、可用时间线、命令记录、公共事件、私密事件、密封审计摘要、可用 `GameEvent` 和已能采集的决策记录。
- 导出包可以包含双方隐藏信息、牌库顺序和密封审计；文件本身应被视为高权限材料，不应直接分享给普通玩家。
- 导入时必须校验 schema version、项目版本或兼容范围、卡牌数据 hash 和能力标记；不兼容时可以拒绝导入，或仅以审计查看模式打开。
- E0 回放界面必须只读，不允许提交正式对局命令，也不反向修改线上对局。
- E0 的格式允许随 recorder 迭代调整；只需对管理员说明“同版本或兼容版本可读”，不对普通用户承诺长期保存和迁移。
- E0 稳定后，其 bundle 逻辑边界应可迁移到 P0-P1 的正式持久化模型，而不是成为另一套平行 replay 格式。
- 已持久化历史记录的管理员导出可复用同一 bundle envelope，导出 `.replay.json` 包含历史 timeline、authority checkpoints、public/private events 与 decision records。若当前历史记录没有完整 command log，导出必须如实保留空 `commands`，不能为确定性重演伪造命令序列。

#### P0：记录创建与封存

P0 目标是让每局联机对局从开始就有可持久化的历史根记录。

P0 可以按数据库与 recorder 底座、开局写入闭环、封存闭环分段验收；但只有三者都完成后，才表示 P0 用户侧基础能力完成。

必须支持：

- 联机对局开始时创建历史记录，并锁定记录 ID、参与者、座位、先后手、房间来源和开始时间。
- 对局开始时保存双方卡组快照，而不是只引用用户当前卡组。卡组快照至少包含当局 deck ID / deck name / 构筑来源、主卡组与能量卡组构成、卡牌编号、基础展示摘要、卡牌数据版本或等价来源标记。基础展示摘要至少应覆盖历史展示所需的卡牌编号、名称、类型、费用/分数、图片文件名、罕度、收录商品和必要的效果文本。实现上不能只保存运行时牌组列表；锁卡结果或开局参数必须携带足够的快照输入。
- 对局初始化后保存初始权威检查点或等价记录，能说明当局卡牌实例、初始牌库顺序、起手和能量卡组顺序。
- 记录中必须存在独立于公共事件序号的时间线或检查点游标。该游标应由记录层生成；若某些状态变化没有产生公共事件，也不能覆盖或丢失其检查点位置。
- 对局正常规则结束、投降、异常中断或被清理时，保存结束状态、胜负结果、回合数、结束原因、记录完整性状态和封存时间。若当前版本尚未提供投降入口，P0 首版至少应覆盖正常规则结束与清理前部分封存，并在模型中预留投降状态。
- 异常中断或被清理前只能形成部分记录时，应明确标记为“部分封存”或等价状态。

P0 不要求完整回放 UI，但应能通过内部或开发入口确认记录、卡组快照、初始检查点和结束状态已落库。

#### P1：时间线与玩家视角回放读取

P1 目标是让玩家能按自己当时视角只读查看历史节点。

必须支持：

- 对局推进过程中持续保存命令、公共事件、私密事件、密封审计、随机结果摘要或等价审计事实，以及必要回放检查点。P1 可以先由权威检查点和审计事实兜底随机结果解释；独立 `Randomness Record` 的完整结构化不应阻塞首版玩家视角回放。
- 私密事件应按参与 seat 记录双方增量；普通玩家读取时只返回其所属 seat 在当时可见的私密事件，不能因为命令由某一方发起而漏掉另一方后续回放所需的私密事实。
- 时间线条目应能串联命令、事件、系统推进、检查点和记录状态变化，不能只依赖 `PublicEvent.seq` 作为唯一节点。
- P1 首版必须保留压缩、采样或关键节点 checkpoint 策略的入口，避免把高存储成本策略固化为长期唯一方案。当前实现已采用稀疏 authority checkpoint：普通高频已接受命令可以只追加 timeline/event/summary，每 5 帧或遇到关键命令、系统转移、撤销、结算/阶段类命令时写 checkpoint；无 checkpoint 帧通过 `stateSummary` 保持 timeline 的回合、阶段和子阶段可读。
- 普通玩家时间线必须经过玩家视角过滤，不能直接复用管理员调试回放中的 private / audit / admin 级 timeline 摘要。
- 玩家可以在历史列表中查看自己参与过的对局。
- 玩家可以打开对局详情，查看双方、先后手、卡组快照摘要、胜负、回合数、结束原因和记录能力状态。
- 玩家可以按自己的当时视角打开只读回放。
- 回放支持基于时间线或检查点的前进、后退和跳转到关键节点；第一版可以是离散节点跳转，不要求动画式播放。
- 回放中可以查看当前节点的基础事件说明和操作来源。
- 普通玩家回放接口不能返回权威检查点、密封审计、完整随机顺序或对手隐藏信息后再交给前端隐藏；隐藏信息过滤必须在服务端读取模型或投影层完成。

#### P2：决策记录、手动处理原因与能力标记

P2 目标是让回放不只“能看状态”，还要能解释关键选择和过渡语义。P2 属于第一阶段增强闭环，不应成为 P0-P1 历史列表和玩家视角只读回放的上线前置。

必须支持：

- 卡效自动处理的决策记录至少保存 `decision opened` 与 `decision submitted` 两类事实。
- `decision opened` 至少包含来源能力、来源卡、步骤 ID、步骤文本、等待玩家、当时可见候选摘要、选择上下限、可跳过状态和关联检查点。
- `decision submitted` 至少包含玩家最终选择、跳过、确认、顺序发动或无选择继续，以及关联命令、事件范围和后续检查点。
- 换牌、Live 设置、成功 Live 选择、起动能力发动、自动能力顺序选择、可选卡效、卡效目标/选项/确认步骤都应逐步进入同一决策记录模型。
- 卡效手动处理和自由拖拽至少保存操作者、移动对象、操作原因；若原因尚未结构化，应明确标记为“手动处理 / 原因未结构化 / 记录式回放可看但不可确定性重演”。
- 每局记录明确标记是否支持摘要、玩家视角回放、审计读取、调试导出和完整确定性重演。
- 卡效自动处理、卡效手动处理和自由拖拽至少以可读事件、决策说明或过渡说明出现在记录中；若某段仍只有 snapshot/audit 语义，应明确标记。

#### P3：回放体验增强与审计辅助

P3 属于第一阶段核心闭环稳定后的增强项。

可以继续支持：

- 更完整的回放 UI 时间线、关键节点索引和事件说明。
- 对局结束后自动展示“查看复盘”入口。
- 对局中断后展示“部分记录可查看”的提示。
- 开发者或高权限审计场景能够按记录 ID 读取足够信息定位某一步命令、卡效选择、随机结果或状态变化。
- 管理员或开发者导出脱敏后的调试记录。

### 7.2 第一阶段应该支持

P0-P2 稳定后，如果成本可控，应继续支持：

- 用户为对局添加备注或收藏标记。
- 对局记录按时间、对手、胜负、使用卡组筛选。
- 普通玩家在授权范围内下载自己的玩家视角记录文件。
- 为未来分享链接预留授权状态和撤销状态。
- 为记录保留周期、收藏优先保留和清理策略预留能力标记。

### 7.3 第一阶段暂不要求

第一阶段不要求：

- 完整动画式录像播放。
- 第三方公开分享页面。
- 全卡效确定性重演。
- 所有历史记录跨版本完全重演成功。
- 基于回放的自动战报生成。
- AI 自动评价玩家选择。
- WebSocket 或实时观战。

这些能力可以作为后续阶段建设。

## 8. 数据与保留需求

### 8.1 对局元信息

对局记录应保留：

- 对局 ID；
- 对局模式；
- 参与玩家；
- 先后手；
- 开始时间；
- 结束时间；
- 当前记录状态；
- 胜负结果；
- 结束原因；
- 回合数；
- 使用的规则版本或项目版本标记；
- 记录创建时间、最后追加时间和封存时间；
- 记录完整性状态；
- 回放能力状态。

记录状态至少应能区分：

- 进行中；
- 正常完成；
- 玩家投降或规则胜负；
- 异常中断；
- 被清理前部分封存；
- 记录损坏或不可回放。

### 8.2 卡组快照

对局记录应保留双方开局锁定的卡组快照。

卡组快照应独立于玩家当前卡组，至少满足：

- 能还原当局主卡组和能量卡组构成。
- 能说明每张卡所用的卡牌编号。
- 能保留当局展示所需的基础卡牌信息。
- 能用于未来确定性重演或问题复现。

### 8.3 初始状态

记录应保留足以还原对局起点的信息。

这包括：

- 双方卡组快照；
- 先后手；
- 初始随机或洗牌结果；
- 初始手牌、能量和牌库顺序的权威信息；
- 对普通玩家隐藏的内容应按权限控制。

### 8.4 命令与决策记录

记录应保留对局推进中的玩家命令和关键系统决策。

至少应区分：

- 玩家主动命令；
- 卡效确认；
- 卡效目标选择；
- 可选效果接受或跳过；
- 手动效果处理；
- 自由拖拽或桌面整理；
- 系统自动推进；
- 系统规则处理；
- 命令拒绝。

命令记录和决策记录不是同一个概念。命令记录回答“客户端或系统提交了什么意图”；决策记录回答“玩家在当时的选择窗口里看到了什么、为什么需要选择、最终如何选择”。第一阶段至少应为以下场景建立稳定决策记录：

- 换牌选择；
- Live 设置与成功 Live 选择；
- 起动能力发动；
- 自动能力顺序选择；
- 可选卡效发动或跳过；
- 卡效目标、多选、顺序、选项和确认步骤；
- 手动效果处理确认。

决策记录至少应包含：

- 决策 ID 和时间线游标；
- 决策来源，例如命令、卡牌能力、系统流程或手动处理；
- 决策参与者；
- 当时展示给该参与者的候选摘要、限制和步骤文本；
- 最终提交结果、跳过或确认；
- 关联的命令、公共事件、私密事件、审计记录和前后检查点范围；
- 普通玩家可见字段与高权限审计字段的边界。

### 8.5 事件记录

记录应保留用于回放展示的事件。

事件应区分规则事实、展示事实和审计事实。当前 `GameState.eventLog` / `GameEvent` 已开始表达规则层事实；`PublicEvent`、`PrivateEvent` 和 `SealedAudit` 分别表达公共展示、单侧可见事实和服务端审计。记录层不应把这些事件源混成一个无权限边界的日志。

事件应至少区分：

- 公共事件；
- 私密事件；
- 密封审计；
- 权威规则事件；
- 系统事件；
- 玩家声明或确认；
- 卡效相关事件。

并非所有事件都需要第一阶段完整标准化，但凡会出现在用户复盘中的事实，都应有稳定、可解释的记录方式。

### 8.6 快照记录

记录应保留足够的快照，使用户可以跳转和回退。

快照可以按以下粒度之一或组合保存：

- 每个公共事件后；
- 每个命令后；
- 每个阶段或子阶段开始；
- 每 N 个事件保存一次；
- 对关键节点保存完整快照。

第一阶段可优先采用简单、可靠的快照策略，以换取稳定回放体验。

快照与时间线位置应有稳定游标。该游标不应只依赖前端播放时间，也不应把公共事件序号当成唯一检查点身份；如果某些状态变化没有产生新的公共事件，也仍应能在记录内部定位到唯一检查点或命令位置。

快照需要区分权限和用途：

- 权威快照：保存完整 `GameState` 或等价权威状态，只用于审计、开发定位和未来确定性重演对照。
- 玩家视角快照：保存某个参与玩家当时可见的 `PlayerViewState` 或等价投影，用于普通玩家回放。
- 公共视角快照：只保存双方共同可见信息，用于未来分享或公共复盘。

普通玩家回放接口不得直接返回权威快照。若第一阶段只落地权威快照，也必须在读取时经过投影层生成玩家视角后再返回。权威快照、密封审计和随机顺序数据应有独立权限控制、保留策略和导出限制。

### 8.7 随机与洗牌记录

为了支持卡组可复现和未来确定性重演，记录应保留随机相关信息。

至少应能解释：

- 开局洗牌结果；
- 换牌后的洗牌或回洗结果；
- 牌库刷新后的顺序；
- 卡效导致的洗牌、重排或随机选择；
- 声援、检视顶牌等公开结果。

生产对局可以继续使用安全随机。第一阶段应优先记录随机结果、洗牌后顺序或等价权威快照，以便解释历史事实；P1 首版可以通过权威检查点和审计事实兜底，不要求立即建立完整独立随机流。第二阶段再进一步要求 seed、随机调用序号、随机流和命令重演一致性。

## 9. 隐私与权限需求

### 9.1 默认不泄露隐藏信息

普通玩家回放默认只能看到其当时可见的信息。

即使对局已经结束，也不应默认展示：

- 对手当时手牌；
- 对手未公开的牌库顺序；
- 对手未公开的能量卡组顺序；
- 对手私密选择候选；
- 密封审计内容。

普通玩家接口不应因为实现方便而返回完整权威状态、权威检查点、密封审计或随机顺序日志后再交由前端隐藏。隐藏信息过滤必须在服务端读取模型或投影层完成。

### 9.2 对局双方的权限

对局双方都应能查看自己参与的对局记录。

默认规则：

- 玩家 A 以玩家 A 当时视角回放。
- 玩家 B 以玩家 B 当时视角回放。
- 公共分享只使用公共视角。
- 完整审计视角需要额外权限。

### 9.3 管理员与裁判权限

管理员或裁判查看完整记录必须有明确权限边界。

需求包括：

- 普通管理员观测不默认开放完整隐藏信息。
- 争议处理或开发排障场景可开启更高权限。
- 高权限访问应留有审计。

### 9.4 用户控制与保留周期

系统应支持对历史对局设置合理保留策略。

后续可讨论：

- 默认保存最近若干局；
- 用户收藏的对局长期保存；
- 用户可删除自己的对局记录入口；
- 管理员按存储策略清理旧记录；
- 分享链接可撤销。

第一阶段可以先明确记录归属和保留状态，不要求完整用户自助清理。

## 10. 版本与兼容需求

### 10.1 历史记录不应被当前卡组影响

历史对局必须使用当局卡组快照展示。用户修改当前卡组不应改变历史记录。

### 10.2 历史记录不应被当前卡牌数据静默改写

如果卡牌数据后续更新，历史对局至少应保留当局展示所需信息。
未来可以支持显示“当前卡牌文本已更新”的提示，但不能静默改变历史事实。

### 10.3 回放能力需要标记兼容状态

每条对局记录应能表达自身回放能力状态，例如：

- 可查看摘要；
- 可按视角回放；
- 可完整回放；
- 可导出调试记录；
- 可确定性重演；
- 因版本过旧只能部分回放。

这样可以避免未来版本演进后把旧记录误认为完全可重演。

## 11. 用户体验需求

### 11.1 历史对局列表

用户应能从个人入口进入历史对局列表。

列表应帮助用户快速识别：

- 对局时间；
- 对手；
- 使用卡组；
- 胜负；
- 回合数；
- 记录完整状态；
- 是否已收藏；
- 是否可回放。

### 11.2 对局详情

对局详情应展示：

- 双方玩家；
- 先后手；
- 使用卡组快照摘要；
- 胜负结果；
- 关键事件时间线；
- 进入回放入口；
- 用户备注或收藏状态。

### 11.3 回放界面

回放界面应尽量复用当前对局桌面体验，但需要明确处于回放模式。

回放模式应至少提供：

- 当前回合、阶段、事件位置；
- 播放/暂停；
- 上一步/下一步；
- 跳转关键节点；
- 事件说明；
- 视角标识；
- 当前节点的可见信息。

回放界面不应允许提交正式对局命令。

### 11.4 复盘辅助信息

回放中应逐步补充复盘辅助信息：

- 当前事件来自玩家、系统、卡效还是手动处理；
- 卡效文本或能力摘要；
- 选择项和最终选择；
- 命令失败或被拒绝的原因；
- 规则修正说明；
- 关键公开随机结果。

第一阶段可以先展示基础事件说明，不要求复杂分析。

## 12. 非功能需求

### 12.1 正确性

对局记录必须以服务端权威事实为准。客户端本地动画、拖拽预览和临时 UI 状态不能成为历史记录事实来源。

### 12.2 可复现性

记录应尽量保存足够信息，使开发者能够复现对局关键过程。
第一阶段可以先做到记录式复盘，后续逐步达到确定性重演。

### 12.3 可审计性

记录应能区分：

- 玩家命令；
- 系统处理；
- 卡效自动化；
- 手动效果处理；
- 误操作修正；
- 拒绝命令。

### 12.4 可扩展性

需求应支持未来扩展：

- 更完整卡效事件；
- 观战；
- 分享；
- 自动战报；
- AI 分析；
- 回归测试样本。

### 12.5 性能与存储

对局记录会增加存储成本。系统应避免无边界保存过大数据。

需求层面应允许：

- 快照压缩；
- 摘要化公共事件；
- 按保留策略清理；
- 收藏对局优先保留；
- 调试记录和普通用户记录采用不同保留策略。

权威快照与玩家视角快照可以采用不同粒度和保留周期。第一阶段可以优先保存更多快照保证正确性，但必须为后续压缩、抽样、冷存储或按收藏保留预留能力标记。

### 12.6 安全与隐私

任何回放、分享和导出都必须遵守可见性和权限规则。
隐藏信息、密封审计和完整权威状态不能被普通接口意外暴露。

审计视角访问完整权威状态、双方隐藏信息或密封审计时，应作为高权限操作处理，并在后续实现中保留访问审计能力。管理员历史 `.replay.json` 导出属于这种高权限材料；普通玩家下载或分享记录时，只能导出其有权看到的视角数据。

## 13. 第一阶段验收标准

第一阶段按分层验收。P0-P1 达成后即可视为首个可交付 MVP；P2-P3 达成后，第一阶段复盘解释能力才算完整。

E0 验收：

1. 管理员可以从一个运行中或已结束的联机对局导出调试回放包。
2. 导出包包含 schema version、项目版本或 git commit、卡牌数据 hash、能力标记和足够的权威检查点 / 事件 / 命令事实；authority checkpoint 遵守序列化与复水契约，不保存未版本化裸 `GameState`。
3. 管理员可以在只读回放入口导入兼容导出包并查看历史节点。
4. 普通用户不能访问 E0 导出 / 导入入口，也不能下载包含隐藏信息的管理员调试包。
5. E0a 若只有单点或有限 checkpoint，必须用能力标记说明 `SINGLE_CHECKPOINT_ONLY` / `LIMITED_TIMELINE` / `NO_DETERMINISTIC_REPLAY` 或等价限制。
6. 版本不兼容、payload hash 不匹配或 serializer 不受支持的导出包会被明确拒绝或降级为仅审计查看，不伪装成完整可回放记录。
7. 从正式历史记录导出的管理员回放包使用 `sourceMatch.exportedStatus=HISTORY_RECORD`，不要求 `NOT_USER_HISTORY_RECORD`，但仍不能通过普通用户接口下载或读取 authority payload。

P0 验收：

1. 用户开始一局正式联机对局时，系统创建对应历史记录并锁定双方卡组快照；硬写入失败时该局不进入 `IN_GAME`。
2. 卡组快照不受之后卡组编辑影响，且保留当局展示所需的卡牌摘要和卡牌数据版本或等价来源标记。
3. 对局初始化后存在初始检查点，能还原当局卡牌实例、初始牌库顺序、起手和能量卡组顺序。
4. 记录使用独立时间线或检查点游标；没有公共事件的状态变化也不会覆盖或丢失检查点。
5. 正常规则结束、异常中断或清理前部分封存都会写入明确记录状态；投降状态在模型中预留，若当前版本新增投降入口，该入口也必须写入明确记录状态。

P1 验收：

6. 用户完成一局联机对局后，可以在历史对局列表看到该局。
7. 用户可以打开该局详情，看到对手、卡组、胜负、回合数、结束原因和记录能力状态。
8. 用户可以进入回放界面，以自己的当时视角查看对局过程。
9. 回放可以基于 `RecordFrame` / `timelineSeq` 或 checkpoint 前进、后退和跳转到关键节点。
10. 回放不会泄露用户当时不可见的信息。
11. 普通玩家回放接口不会返回权威快照、密封审计、完整随机顺序或对手隐藏信息。
12. 不完整记录会在详情和回放读取中返回脱敏后的部分记录状态或原因摘要；服务端保留可排查的追加失败信息。

P2 验收：

13. 卡效自动处理的决策记录能展示来源能力、当时候选、玩家选择、跳过或确认结果。
14. 决策记录至少区分 `decision opened` 和 `decision submitted`，不能只从最终命令结果倒推选择窗口。
15. 手动处理和自由拖拽至少能展示操作者、移动对象、操作原因；若原因未结构化，应明确标记为过渡语义。
16. 异常中断或不完整记录会显示明确状态，不被伪装为完整可回放记录。
17. 记录明确标记自身是否支持摘要、玩家视角回放、审计读取、调试导出和完整确定性重演。
18. 当前未完整支持的重演能力不会被产品文案描述为已完成。

P3 验收：

19. 回放界面能展示更完整的关键节点说明和事件来源。
20. 开发者或高权限审计场景可以基于记录 ID 定位某一步命令、卡效选择、随机结果或状态变化。

## 14. 后续阶段方向

### 14.1 第二阶段：确定性重演

第二阶段目标是从初始状态和命令/决策序列重新运行规则引擎，验证结果一致。

重点包括：

- 更完整的随机种子或随机结果记录；
- effect decision log；
- command replay；
- 自动卡效事件标准化；
- 重演差异报告。

### 14.2 卡效 runner 重构与存储格式

最近的卡效重构不会改变 P0-P1 的核心存储方向。记录创建、卡组快照、权威检查点、`timelineSeq`、`checkpointSeq`、命令记录、`PublicEvent`、`PrivateEvent`、`SealedAudit` 和可用的 `GameEvent` 事实都应独立于 runner 内部拆分方式。

需要防止的是 P2 决策记录直接绑定运行时结构。长期存储应优先保存：

- 来源卡实例、卡牌编号、基础编号、控制者、来源区域和来源槽位；
- `abilityId`、`triggerCondition`、能力分类、卡面效果文本快照和步骤 ID；
- 决策打开时的步骤文本、候选摘要、选择上下限、可跳过状态和等待玩家；
- 决策提交时的选择结果、跳过或确认结果，以及关联命令、事件范围和检查点；
- 决策记录 schema version、记录能力标记和必要的兼容说明。

`abilityId` 可以作为关联键，但不应成为唯一可读来源。若未来 runner 重命名 helper、拆分 resolver 或迁移 steps DSL，只要上述语义字段和版本标记稳定，历史记录通常只需要读模型兼容，不应频繁触发数据库迁移。只有新增一类必须索引或查询的长期事实，例如新的高频决策类型、随机流字段或权限维度，才应考虑 schema migration。

### 14.3 第三阶段：分享与教学

第三阶段可以扩展：

- 公共视角分享链接；
- 关键回合剪辑；
- 战报摘要；
- 卡效说明叠层；
- 教学标注。

### 14.4 第四阶段：AI 与自动分析

在回放数据稳定后，可以扩展：

- AI 复盘建议；
- 胜负转折点分析；
- 卡组表现统计；
- 自动测试样本库；
- Monte Carlo 或对局模拟数据积累。

## 15. 非目标

本需求文档当前不要求：

- 立刻实现完整数据库 schema。
- 立刻定义全部 API 契约。
- 立刻替换当前轮询传输。
- 立刻实现 WebSocket/SSE。
- 立刻实现全卡池自动裁判。
- 立刻实现完整卡文解释器。
- 立刻开放第三方公开分享。
- 立刻向普通玩家开放完整隐藏信息回放。
- E0 管理员调试导出不作为普通用户历史对局功能，也不承诺长期跨版本兼容。

## 16. 已决策与待讨论问题

第一版 checkpoint 粒度已经确定为“定期间隔 + 关键状态转移”的稀疏 authority checkpoint；没有 checkpoint 的时间线节点仍需保留可读状态摘要。具体默认间隔和关键命令分类属于设计/实现参数，可以根据记录体积和跳转性能调整，但不再作为第一阶段未决策项。

以下问题仍需后续迭代继续明确：

- 历史对局默认保存多久？
- 用户是否可以手动删除历史对局？
- 对局双方是否都能永久保留同一局？
- 分享回放是否需要双方同意？
- 完整审计回放的权限如何定义？
- 对局中途异常结束时记录如何标记？
- 手动处理原因是否先采用固定枚举，是否允许玩家补充短说明？
- 决策记录中的候选摘要需要保存完整候选列表，还是保存候选摘要并依赖检查点还原？
- 是否需要支持本地下载复盘文件？
- 调试导出是否应脱敏？
- 旧版本记录在规则更新后如何展示兼容提示？

## 17. 维护规则

本文档是需求基线。后续讨论中：

- 产品目标变化应更新本文档。
- 具体数据结构、接口、表设计和代码路径应进入后续设计文档。
- 已完成阶段不再维护平行实施流水账；需要追溯施工拆分时使用 Git 历史。后续需求优先更新本文档，当前架构和实施状态优先更新 `design.md`。
- checkpoint、debug bundle、payload envelope、导入复水和普通回放读取的安全边界以 [checkpoint / bundle 序列化与复水契约](serialization-contract.md) 为 review 基线。
- 当前实现限制仍以 `docs/current-limitations.md` 集中维护。
- 对局回放的产品范围或实施状态变化时，应同步检查 `docs/PROJECT_REQUIREMENTS.md`、`docs/system-design.md` 和相关专题文档。
