## 1. 设计目标与当前实现取向

从 `docs/system-design.md` 和当前实现看，桌面 UI 同时服务规则对局与人工调试：

- 让前端作为一个高可视化的对局操作台
- 将规则判断尽量留给 `GameSession` / `GameService`
- 新对局默认 `RULES`；前端按权限提示合法交互，权威命令政策拒绝普通手动区域移动和与当前流程无关的输入
- 显式进入 `FREE` 后，保留较自由的己方拖拽和手动修正，用于尚未自动化规则的人工处理
- 用子阶段、浮层和局部操作面板去补足流程提示

当前桌面因此呈现出一种“规则驱动状态机 + 可切换自由模式的调试型交互桌面”风格；正式线上对局默认使用规则模式。

## 2. 整体桌面结构

整体由 `GameBoard` 组织，采用纵向三段式结构：

- 顶部：对手区域
- 中部：`VS` 分隔条
- 底部：己方区域

外层还有几层全局视觉和功能层：

- 整体背景使用卡背图铺底
- 其上叠加若干全屏遮罩和光效渐变
- 右上角固定主题切换按钮
- 顶部中央固定调试控制条
- 左侧固定日志抽屉
- 左侧固定解决区 / 判定面板及其唤起按钮
- 右下角固定阶段指示器
- 右侧中部固定卡牌详情浮窗
- 中央根据阶段弹出换牌框、分数确认框、阶段横幅、拖拽覆盖层等
- 全屏对局动作反馈层，用于展示拖拽语义、命令失败和局部成功反馈
- 全屏对局动画层，用 `PlayerViewState` 前后差异展示卡牌移动、翻面、方向变化或区域脉冲

这意味着当前桌面不是“单一棋盘式平面布局”，而是“游戏桌 + 多个固定悬浮控制层”的复合界面。

## 3. 对局区域布局现状

### 3.1 上下两侧采用镜像布局

`PlayerArea` 明确采用“中心对称”设计：

- 己方在底部，按自己视角展示 LEFT / CENTER / RIGHT 成员槽
- 对手在顶部，成员槽顺序镜像为 RIGHT / CENTER / LEFT
- 己方资源区在画面右侧，对手资源区镜像后在画面左侧
- 己方成功 Live 区在画面左侧，对手成功 Live 区镜像后在画面右侧

布局目标是让双方“面对面坐在桌子两侧”的感受成立。

### 3.2 己方区域布局

己方区域从上到下分为三层：

- 主战区顶部
- 玩家信息条
- 手牌/能量区底部

其中主战区内部又是三列绝对定位：

- 左列：成功 Live 区
- 中列：Live 区 + 成员槽
- 右列：资源区（主卡组、休息室、能量卡组）

中列是主要视觉焦点：

- 上方是 3 个固定 Live 槽位
- 下方是 3 个成员槽位

成员槽下方还可能叠有附加能量牌，形成“成员卡在上，能量卡向左下偏移叠放”的局部堆叠结构。

底部区域：

- 左下角是能量区
- 中间是扇形展开的手牌
- 手牌最右侧外挂两个小按钮：抽一张、放回一张

### 3.3 对手区域布局

对手区域与己方相反，从上到下分为：

- 手牌/能量区顶部
- 玩家信息条
- 主战区底部

主战区内部仍然是三列，但左右互换：

- 左列：资源区
- 中列：成员槽 + Live 区
- 右列：成功 Live 区

中列内部也镜像：

- 上方是成员槽
- 下方是 Live 区
- 成员顺序为 RIGHT / CENTER / LEFT

这种设计保证了双方的“主战区”都靠近中央 VS 分隔条。

## 4. 各游戏元素的具体布局现状

### 4.1 成员区

成员区固定 3 槽：

- LEFT
- CENTER
- RIGHT

单槽表现：

- 空槽为虚线框
- 有卡时显示标准成员卡
- `FREE` 下可双击切换活跃/等待状态；`RULES` 下只展示规则流程或卡效产生的状态
- 被选中卡会进入 store 的 `selectedCardId`

能量附着表现：

- 附加能量卡与成员卡同尺寸
- 以绝对定位叠放在成员卡下方
- 每张能量卡向左下偏移 10%
- 槽位底部用小胶囊显示“附加能量 xN”

成员下方堆叠表现（详见 `docs/special-member-stacking.md`）：

- 已登记卡牌效果可在舞台主成员下方堆叠成员卡（`memberBelow`）
- 堆叠成员卡向右下偏移 8%，与能量附着相区分
- 使用 amber 色边框，与能量附着的 indigo 色相区分
- 槽位底部用小胶囊显示“堆叠成员 xN”

### 4.2 Live 区

Live 区固定 3 槽，与成员槽横向对齐。

表现特点：

- Live 卡横置显示
- 未翻开时按背面/暗面状态处理
- 已翻开的 Live 卡底部会显示所需 Hearts
- 判定结果会在卡片右上角显示绿色 `✓` 或红色 `✗`

己方 Live 区是可拖放区，对手 Live 区只是展示区。

### 4.3 成功 Live 区

成功区采用纵向 3 槽堆叠：

- 每张卡横置展示
- 槽位固定高度
- 上下有遮挡覆盖感
- 空槽显示虚线框占位

当前没有成功区专属展开浮层，信息密度较低，但展示直接。

### 4.4 资源区

资源区由三项横向排布组成：

- 主卡组
- 休息室
- 能量卡组

己方顺序：

- 主卡组 -> 休息室 -> 能量卡组

对手顺序镜像：

- 能量卡组 -> 休息室 -> 主卡组

卡组表现：

- 主卡组显示背面缩略牌堆和数量徽标
- `FREE` 的自由拖拽窗口内点击己方主卡组：将主卡组顶 1 张翻入检视区
- 点击己方能量卡组：默认无动作
- 能量卡组顶牌可在 UI 中拖起；只有 `FREE` 的自由拖拽窗口会接受放入能量区等兼容操作

休息室表现：

- 缩略叠堆显示前几张卡
- 点击后打开休息室弹窗
- 弹窗内使用网格展示所有卡牌
- 弹窗标题栏显示基于可见正面卡牌的判心统计入口，桌面端可 hover/focus 或点击查看详情，移动端点击后在标题栏下方展开
- 弹窗内卡牌仍可拖起；`FREE` 的自由拖拽窗口可提交当前兼容区域移动，`RULES` 不接受普通手动移动

### 4.5 能量区

能量区在玩家头像行附近的小角落里：

- 己方在左下角
- 对手在右上角

表现形式为横向小缩略图列表：

- 最多先显示 12 张
- 活跃/等待状态通过透明度和灰度表现
- 标题同时显示 `(活跃数/总数)`
- `FREE` 的自由拖拽窗口内可将己方能量区中的能量拖回能量卡组顶；`RULES` 不开放该手动操作

### 4.6 手牌区

对手手牌：

- 只显示背面
- 扇形散开
- 超过 10 张后用 `+N` 提示

己方手牌：

- 正面扇形散开
- 相邻卡牌负 margin 重叠
- hover 向上抬起并放大
- 可点击选中
- 可拖拽到其它区域

己方手牌右侧有两个快捷按钮：

- `←` 从主卡组顶抽一张到手牌
- `→` 将手牌最右一张放回主卡组顶

这两个操作明显偏调试/辅助操作性质，只在 `FREE` 的命令窗口内作为兼容能力生效。

## 5. 提示框与浮层布局现状

当前提示不是集中在一个系统内，而是按用途分散为多种悬浮层。

### 5.1 阶段提示横幅

`PhaseBanner` 为全屏居中提示：

- 阶段切换时短暂出现
- 使用深色背景 + 亮色描边
- 仅负责“阶段名提示”
- 不承载交互

位置：

- 全屏居中
- `pointer-events: none`

### 5.2 阶段指示器

`PhaseIndicator` 固定在右下角，是当前最核心的“流程操作面板”。

展示内容：

- 当前回合数
- 当前主阶段名
- 当前子阶段名、图标、是否“需要操作”
- 当前是否轮到自己

操作内容：

- 主按钮：进入下一阶段 / 确认完成 / Live 判定
- 次按钮：部分子阶段可撤销

已知语义重复：右下角在 Live 结算阶段（`LIVE_RESULT_PHASE`）显示的"进入下一回合"按钮应只出现在分数确认、成功 Live 结算等后续流程完成后。表演成功后的效果处理现在属于 `LIVE_RESULT_PHASE` 的前置正式子阶段，按先攻成功效果、后攻成功效果依次处理。

它既是状态提示，也是主要流程推进入口。

### 5.3 解决区 / 判定面板

`JudgmentPanel` 是当前查看和操作解决区的左侧固定面板：

- 默认可由左侧窄按钮唤起
- 打开后占据左侧最大约 420px
- 收起按钮挂在面板右边缘

面板内部结构：

- 上半部：解决区中的应援牌区域
- 下半部：Live 判定区
- 底部：根据阶段变化的操作按钮

它当前兼具两层职责：

- 作为解决区的主要查看与处理入口
- 作为 Live 判定阶段的流程辅助面板

因此它是当前最复杂的交互提示容器，兼具：

- 阶段说明
- 解决区查看
- `FREE` 下的应援翻牌与手动处理
- 成功/失败确认
- 辅助说明文案

补充说明：

- 正式区域是 `RESOLUTION_ZONE`
- `JudgmentPanel` 只是这个区域当前的 UI 载体，不是独立规则区域
- 表演判定成功后不会立刻进入成功效果窗口；双方表演完成后才进入先攻成功效果、后攻成功效果
- 当前实现将双方表演后的成功效果窗口挂到 `LIVE_RESULT_PHASE` 的 `RESULT_*_SUCCESS_EFFECTS` 子阶段；这是现行状态机表达，UI 按该状态渲染成功效果窗口
- 相关定稿见 `docs/ui-design/game-table/resolution-zone-behavior.md`

### 5.4 分数确认弹窗

`ScoreConfirmModal` 为中央模态框：

- 按规则目标模型，仅在 `LIVE_RESULT_PHASE` 的分数确认步骤中出现
- 展示双方分数和确认状态
- `RULES` 下只读确认自动计算的己方分数；`FREE` 下允许调整后确认

位置：

- 全屏蒙层中央

在调试模式下：

- 蒙层弱化
- 不完全阻断背景视觉

### 5.5 换牌弹窗

`MulliganPanel` 为中央大型模态框：

- 仅在换牌阶段出现
- 展示当前手牌
- 点击卡牌切换是否换掉
- 窄屏与触屏可长按查看卡牌详情
- 底部有“保留手牌”和“换 N 张”确认操作

该面板是一个很明确的单阶段专用任务弹窗。

### 5.6 隐藏区与检视区

当前代码中没有独立的隐藏区浏览器组件，也没有“打开完整主卡组/能量卡组浏览器”的弹窗入口。

当前与隐藏区相关的主要交互是：

- `FREE` 的自由拖拽窗口内点击己方主卡组会把牌加入检视区流程；`RULES` 不开放任意检视。
- 检视区是真实游戏状态的一部分，由 `gameStore.openInspection()` 与 `PlayerArea` 内的检视区 UI 承接。
- 点击己方能量卡组没有默认检视行为。
- 若后续需要完整隐藏区浏览器，需要重新设计权限、拖拽提交和与检视区的关系。

因此，当前不能把隐藏区浏览器视为已有弹窗组件；它也不是点击卡组后的默认结果。

### 5.7 休息室弹窗

休息室弹窗从 `PlayerArea` 内创建 portal：

- 点击休息室牌堆打开
- 居中显示
- 标题栏提供判心统计入口，统计休息室可见卡牌的 `bladeHearts` 彩心、加分判与抽卡标
- 网格列出所有休息室卡牌
- 支持从弹窗内继续拖起卡牌；普通区域移动只在 `FREE` 的自由拖拽窗口内生效

### 5.8 卡牌详情浮窗

`CardDetailOverlay` 固定在右侧中部：

- 鼠标悬停任意卡牌时触发
- 展示大图、卡名、编号、效果、成员/Live 专属属性
- 不可交互，只用于阅读

位置固定意味着它不是贴近卡牌的 tooltip，而是独立详情侧栏。

### 5.9 游戏日志

`GameLog` 是左侧可展开日志抽屉：

- 仅在 DEBUG 模式下显示
- 默认收起，左边留一条窄按钮
- 展开后宽 280px
- 自动滚到底部

日志类型：

- info
- action
- phase
- error

## 6. 用户交互现状

### 6.1 核心交互模式

当前桌面的核心交互模式有 4 类：

- 拖拽
- 点击/双击
- 固定面板按钮推进流程
- 模态框内局部任务操作

### 6.2 拖拽交互

拖拽是主要桌面交互方式，基于 `dnd-kit`。当前实现采用“UI 可尝试、服务端定真值”的模型：前端可能允许己方卡牌拖起，但是否成为正式移动由命令窗口、卡种、区域和服务端校验决定。

特点：

- PointerSensor 需要移动 5px 才启动，避免误触双击
- 拖拽时显示 `DragOverlay`
- 拖拽悬停时会显示当前落点语义，例如“登场”“在此登场”“加入手牌”“放回卡组顶”
- `DroppableZone` 支持三种视觉状态：
  - 推荐目标高亮
  - 当前悬停高亮
  - 其他区域轻度变暗

当前推荐目标和拖拽语义只做提示，不做强限制。命令是否成立仍由本地 `GameSession` 或远程服务端权威校验决定；命令失败时，桌面会在目标附近显示短回执，同时保留日志记录。

以下除 Live 设置阶段的专用放置外，均描述 `FREE` 保留的兼容拖拽；`RULES` 只接受当前阶段或 pending 明确开放的语义化命令：

- `FREE` 的主阶段、表演自由交互窗口、结果成功效果窗口：手牌成员登场、成员换位、能量附着、公开卡移动、手牌/休息室等己方桌面移动。
- Live 设置阶段：`RULES` 与 `FREE` 都可通过专用流程将手牌放置到 Live 区；放置能量、公开卡回手/回能量卡组及其它己方桌面移动仅由 `FREE` 兼容。
- `FREE` 的 Live 卡桌面豁免：Live 卡从 Live 区回手/进休息室，或在 Live 区/成功区相关公开桌面目标之间移动。
- `FREE` 的检视流程：检视区排序、检视区 -> 手牌 / 休息室 / 除外区 / 卡组顶底，以及己方手牌/休息室 -> 当前检视区。
- `FREE` 的判定/解决窗口：解决区 -> 手牌 / 休息室 / 主卡组顶。
- `FREE` 下能量卡组顶牌 -> 能量区。
- `FREE` 下能量区 -> 能量卡组。
- `FREE` 下休息室弹窗内卡 -> 手牌 / 其它当前允许区域。

补充约束：

- `手牌 -> 休息室` 由拖拽触发，命令层走 `MOVE_OWNED_CARD_TO_ZONE`。
- `Live 区 -> 手牌` 在当前实现中走公开卡回手路径；Live 卡从 Live 区回手有阶段豁免，非 Live 卡不应按该路径成立。
- `Live 区 -> 成功区` 在 `RESULT_SETTLEMENT` 走 `SELECT_SUCCESS_LIVE`；其它非检视流程中如果是 Live 卡相关桌面整理，则走 Live 卡桌面移动豁免并继续由服务端校验目标是否合法。
- `检视区 -> 手牌 / 休息室 / 卡组顶底` 虽然可以由拖拽触发，但命令层应走检视区专用命令。
- `手牌 / 休息室 -> 检视区` 只有在自己的检视流程中成立，命令层走 `MOVE_CARD_TO_INSPECTION`。
- `解决区 -> 手牌 / 休息室 / 卡组` 虽然可以由拖拽触发，但命令层应走解决区专用命令。
- `休息室 -> 手牌` 虽然可以由拖拽触发，但命令层应走公开区进入私有区的专用回手命令。
- `能量卡组顶牌 -> 能量区` 虽然由拖拽触发，但命令层应走专用”放置能量”命令，而不是通用 `MOVE_TABLE_CARD`。
- `能量区 -> 能量卡组` 虽然由拖拽触发，但命令层应走公开区进入隐藏区的专用回收命令；它只在当前命令窗口允许时成立，不视为撤销。

### 6.3 对局反馈与状态变化动画

`GameBoard` 当前挂载两层只读展示增强：

- `BattleActionFeedbackLayer`：消费 `gameStore.ui.dragActionHint` 与 `battleFeedbackEvents`，显示拖拽意图、命令失败和少量局部回执。
- `BattleAnimationLayer`：比较前后 `PlayerViewState`，根据 DOM 中的 `data-object-id` / `data-animation-zone-id` 锚点生成卡牌移动、翻面、方向变化和区域脉冲动画。

边界：

- 动画和反馈都不改变权威状态，不生成命令，不写入对局历史。
- 卡牌移动动画期间，`battleAnimationOcclusions` 会临时隐藏已经由 React 渲染到新位置的真实卡牌，避免飞牌和真实卡牌重叠。
- 大量移动超过单卡动画上限时，动画层降级为区域脉冲。
- 只读回放、观战或其它只读桌面不会生成状态 diff 飞牌动画。
- 登场触发 active effect 时，效果窗口会短暂延后显示，让来源卡移动动画先完成；延后状态只属于本地 UI。

### 6.4 点击交互

主要点击行为包括：

- 点击己方手牌：选中卡牌
- `FREE` 的自由拖拽窗口内点击主卡组：将主卡组顶 1 张翻入检视区
- 点击能量卡组：默认无动作
- 点击休息室：打开休息室弹窗
- 点击左侧解决区按钮：打开解决区 / 判定面板
- 点击阶段指示器按钮：推进流程
- 点击中央模态按钮：换牌/确认分数

### 6.5 双击交互

当前明确存在的双击交互：

- `FREE` 下双击己方成员卡：切换活跃/等待状态
- `RULES` 下不提供普通手动横置/重置；成员状态由规则流程或卡效命令改变

### 6.6 Hover 交互

hover 在当前桌面中占比很高：

- 卡牌 hover 时在右侧显示详情浮窗
- 己方手牌 hover 时上抬
- 缩略牌 hover 时放大
- 检视区/解决区卡牌 hover 时显示快捷操作条

因此当前桌面是明显偏 PC 鼠标环境的设计。

## 7. 典型流程下的交互路径

### 7.1 换牌阶段

- 中央出现 `MulliganPanel`
- 当前轮到的玩家点击要换的牌
- 点击“确认换牌”或“不换牌”
- 面板关闭，进入下阶段

### 7.2 主要阶段

- 当前行动玩家可将手牌成员拖到成员槽，走正式登场/换手语义及费用校验
- 同一成员区域一回合一次的限制跟随当前占用成员实例；区域内成员本回合已登场时不可再次被普通登场覆盖
- `RULES` 不允许普通手动换位、横置、能量操作或其它区域移动；`FREE` 才保留这些桌面整理能力
- 完成后在右下角点击 `Live Start!`

主要阶段是规则模式下正式成员登场的窗口，也是自由模式下普通桌面整理的主窗口；活跃、能量、抽卡等自动阶段不开放普通自由拖拽。

### 7.3 Live 设置阶段

- 从手牌拖牌到 Live 区
- 系统在该阶段把放入 Live 区视为“里侧放置”
- 右下角按钮用于确认进入下一流程

`RULES` 下该阶段只允许当前 Live 放置流程及明确 pending 的输入，不允许放置能量、公开卡回手/回能量卡组等普通手动区域移动。`FREE` 可保留兼容桌面整理。非 Live 卡拖入 Live 区只在规则允许的“手牌放置 Live”路径中成立。

### 7.4 Performance 判定阶段

- 右下角按钮变为 `Live 判定`
- 点击后打开左侧 `JudgmentPanel`
- 用户在解决区 / 判定面板中翻开应援牌、查看 Hearts 汇总
- `RULES` 下不能手动切换成员状态
- 提交空判定输入，由服务端自动计算 `LIVE失败` / `LIVE成功`
- 若成功，继续点击“成功效果发动完毕”

`FREE` 的表演自由交互窗口包括 `PERFORMANCE_LIVE_START_EFFECTS` 和 `PERFORMANCE_JUDGMENT`。其中 `PERFORMANCE_JUDGMENT` 保留应援、解决区移动、判定提交和成功 Live 选择等兼容操作；`RULES` 只接受该时点的专用流程输入。

### 7.5 结果结算阶段

- 中央 `ScoreConfirmModal` 打开
- 用户查看双方分数
- `RULES` 下分数只读并确认自动计算结果；`FREE` 可人工调整
- 点击确认

当前实现中，`LIVE_RESULT_PHASE` 的 `requiresUserAction: true` 只表示存在玩家流程输入，不代表 `RULES` 开放己方桌面自由拖拽。分数提交、确认推进和成功 Live 选择分别走 `SUBMIT_SCORE`、`CONFIRM_STEP`、`SELECT_SUCCESS_LIVE` 等专用命令；普通区域整理仅在 `FREE` 中开放。

`RESULT_TURN_END` 是自动推进子阶段，不开放普通桌面拖拽。

- 双方都确认后进入下一回合

## 8. 当前设计的明显特征

### 8.1 优点

- 桌面信息量高，几乎所有重要区域都在主视图中可见
- 上下镜像结构清楚，主战区始终聚焦中央
- 流程按钮集中在右下角，不需要四处找“下一步”
- 左侧解决区 / 判定面板把 Live 相关复杂交互集中管理
- hover 详情、日志、检视面板等辅助阅读能力比较强

### 8.2 局限与设计债务

- 当前交互仍保留调试桌面的视觉与控件密度，正式对战体验尚未完全收敛
- 部分组件仍允许拖起最终会被权威命令政策拒绝的卡牌，前端合法落点提示尚可进一步收紧
- 提示系统分散，横幅、抽屉、模态、固定面板各自承担部分提示责任
- 左侧日志、左侧解决区面板、右侧详情浮窗、右下角阶段面板同时存在时，界面边缘负担较重
- 许多操作依赖 hover，移动端和触屏适配会比较困难
- 资源区、能量区、成功区尺寸较小，某些状态需要靠 hover 才能读清
- 卡组检视、手牌快捷抽放和自由移动带有明显 debug 属性，现已通过 `RULES/FREE` 区分权威能力，但 UI 提示边界仍可继续精修

## 9. 结论

当前“游戏桌”UI 已经形成了比较稳定的基础结构：

- 主桌采用上下镜像的中心对称布局
- 主交互依赖拖拽
- 阶段流程通过右下角面板推进
- 复杂判定通过左侧抽屉处理
- 详细信息通过右侧详情和中央模态补充

如果后续要继续迭代设计，建议把问题拆成三条线：

- 桌面主布局是否要继续保留当前“调试型复合桌面”结构
- 提示系统是否需要收敛成更统一的层级
- 规则模式的前端提示是否要进一步贴合中央命令政策，减少“可以拖起但提交后被拒绝”的尝试
