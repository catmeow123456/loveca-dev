# Loveca 联机模式自由拖拽重点测试场景

> 文档类型：联机手工测试场景
> 适用范围：重点验证“信任玩家自由拖拽”在联机模式下是否真的可操作，而不只是规则上允许
> 编写依据：`docs/online-mode-preparation.md`、`docs/game-table-design/game-table-ui-current-state.md`、`docs/coding-standard/online-mode-boundary.md`、`detail_rules.md`、当前 `GameBoard` / `PlayerArea` / `HiddenDeckBrowserModal` / `JudgmentPanel` / `gameStore` / `GameSession`

## 1. 这份文档要解决什么问题

上一份联机场景文档更关注“规则边界和可见性是否正确”。这份文档只盯一个更容易被忽略、但严重性很高的维度：

- 文档或设计语义中说玩家可以靠自由拖拽完成的事，玩家是否真的拖得出来。
- 拖拽入口是否真的存在，而不是“理论上允许，实际只能点按钮”。
- 拖拽后是否真的落到正确的联机命令，而不是看起来拖动了，实际没有正式提交。

对“信任玩家”方案来说，很多复杂效果和桌面整理都依赖玩家自行操作。如果某个本该允许的动作在联机里根本做不出来，这不是普通交互瑕疵，而是严重问题，因为它会直接让玩家无法按文档要求完成对局流程。

---

## 2. 缺陷判定标准

测试这份文档时，不要只问“有没有替代方案”，而要问“文档承诺的拖拽路径是否真实存在”。

以下情况都应记为高优先级问题：

- 文档或界面文案表达“可拖拽完成”，但实际无法开始拖拽。
- 卡牌能被抓起来，但没有任何可落点，或落下后没有正式效果。
- 只能靠按钮、快捷操作或别的隐藏入口完成，但拖拽本身做不到。
- 自己的公开区本该可自由拖拽，结果联机模式下被错误禁用。
- 拖拽动作在某个关键阶段本该可用，但只有本地模式可用、联机调试模式不可用。

以下情况也值得记录，但优先级次于上面几类：

- 明明当前阶段不允许该动作，但 UI 仍允许先拖起来再报错。
- 推荐高亮和真实可落点不一致。
- 拖拽动作可用，但成功后双方看到的公共过程不完整。

---

## 3. 测试原则

- 一律按“真实玩家操作”测试，不要用控制台、手工改状态来补做缺失入口。
- 如果文档描述的是拖拽，就必须真的用鼠标拖拽通过，不要用按钮代替判通过。
- 每个场景至少同时看三件事：是否能抓起、是否有可落点、落下后是否形成正式联机事实。
- 若某动作只在一个特殊面板里能通过按钮完成，但桌面主交互文档把它描述成拖拽，也应记为文档与实现不一致。

---

## 4. 重点场景

### 场景 1：己方手牌在主阶段必须能直接拖到成员槽

“手牌 -> 成员槽”是最核心的自由拖拽路径之一。若这一条不通，联机模式的“信任玩家”桌面心智就已经断了。

重点观察：

- 手牌能被抓起，不需要先点额外按钮进入特殊模式。
- 三个成员槽都能作为落点响应。
- 放下后，成员应正式进入目标槽位，而不是只出现本地预览。
- 若目标槽已有成员，替换流程也应能继续成立。

### 场景 2：己方手牌在 Live 设置阶段必须能直接拖到 Live 区

Live 设置阶段的“手牌 -> Live 区”是当前桌面文档明确列出的典型拖拽。这里不应要求玩家改走别的非拖拽入口。

重点观察：

- 手牌在该阶段可拖。
- Live 区在未满时可作为有效落点。
- 拖放后立即形成正式事实，不是本地草稿。
- 放进去后，对手立刻看到相应的公开对象变化。

### 场景 3：Live 设置阶段刚放进去的 Live，必须还能直接拖回手牌

文档和当前联机命令语义都允许这类调整，而且强调这是新的正式动作，不是撤销。如果拖不回去，会直接卡住“信任玩家自由调整”的承诺。

重点观察：

- Live 区中的己方卡牌能再次抓起。
- 拖到手牌区域后，动作正式成立。
- 不是只有按钮能回手，而是拖拽本身可行。
- 回手后历史上仍保留前一次放置过程。

### 场景 4：成功效果发动阶段和判定阶段本地成功窗口里，公开区回手必须能直接拖出来

联机规范里，成功后的自由调整属于双方表演完成后的 `RESULT_FIRST_SUCCESS_EFFECTS` / `RESULT_SECOND_SUCCESS_EFFECTS`。这里最容易出现“命令层允许，但 UI 没开放拖拽”的断层。

重点观察：

- 在成功效果发动阶段中，成功区或 Live 区卡牌可以实际抓起。
- 在判定阶段本地成功窗口中，相关公开区卡牌也能直接拖回手牌。
- 不应出现只能看到高亮、但真正放手无效果的情况。

补充说明：

- 成功效果发动阶段还应允许打开自己的检视区或判定区，并允许检视区与判定区的正式拖拽处理。

### 场景 5：能量卡组顶牌必须能直接拖到能量区

这是典型的“拖拽触发，但命令层走专用命令”的场景。对测试来说，重点不是协议层用什么命令，而是玩家是否真的能用拖拽完成这个动作。

重点观察：

- 只有顶牌可抓起，不能随意抓中间牌。
- 顶牌拖到能量区后会正式落入能量区。
- 若拖到别的区，系统要明确拒绝，不应悄悄消失或静默失败。
- 对手在它进公开区前不应被泄露牌面。

### 场景 6：能量区或成员下方的能量，必须能直接拖到成员槽位下方完成附着

“玩家通过显式桌面操作自行完成资源处理”是当前信任玩家模型的重要部分。若能量附着做不到，很多实际对局步骤就会断。

重点观察：

- 能量区的能量可以抓起并放到目标成员槽位。
- 已经在成员下方的能量也可以再抓起并挪到另一个成员下方。
- 落点必须准确落到指定槽位，而不是模糊丢到成员区。
- 附着后叠放顺序和显示位置合理，双方都能看到结果。

### 场景 6A：能量区中的能量，必须能随时直接拖回能量卡组

这条路径不能被当成“特殊按钮操作”或“只在个别阶段开放”。在当前允许玩家进行桌面操作的窗口里，己方能量区中的能量应始终可以直接拖回能量卡组。

重点观察：

- 能量区中的己方能量可以直接抓起并落到能量卡组。
- 落下后应正式进入能量卡组顶，而不是停留在本地预览态。
- 这一步应作为新的正式公共移动存在，不应被表现成撤销。
- 若当前窗口还能做其它桌面整理动作，这条回收路径不应被额外禁用。

### 场景 7：成员槽位之间必须能直接拖拽换位

这是标准公开区内部自由移动。如果文档声称桌面支持自由拖拽，但成员换位还要走别的隐蔽操作，说明主交互设计已经和实现脱节。

重点观察：

- 成员卡可以从一个槽位抓起并放到另一个槽位。
- 换位后的新位置立即生效。
- 下方附着能量是否按既定规则一起处理，要和当前规则/实现一致。
- 同槽位放下不应产生异常动作。

### 场景 8：休息室弹窗中的牌，必须能真的拖到手牌或其它当前允许区域

当前桌面文档把“休息室弹窗内卡 -> 手牌 / 其它允许区域”列为典型拖拽行为。这个场景专门用来抓“弹窗里看起来可拖，实际落不到主桌”的问题。

重点观察：

- 休息室弹窗内的卡牌确实可抓起。
- 抓起后能跨弹窗边界落到主桌区域。
- 若当前时机允许回手，则可以直接落到手牌。
- 也可以落到文档和规则当前允许的其它目标区。
- 不应出现拖拽遮罩、层级或 portal 问题导致“卡能拿起但任何区域都接不到”。

### 场景 9：手牌必须能直接拖到休息室

既然当前边界已经明确把 `手牌 -> 休息室（通用自由拖拽）` 视为允许动作，那这条就不能再停留在“也许有别的按钮能做”。玩家必须能在桌面上直接完成。

重点观察：

- 手牌可以直接抓起并落到休息室。
- 落下后形成正式事实，而不是只在本地消失或回弹。
- 双方都能看到这次移动的结果。
- 如果当前 UI 仍然挡掉这条拖拽，应直接记为严重问题。

### 场景 10：隐藏区中的非顶牌若当前设计允许被自由选取，也必须真的拖得出来

你已经明确要求“隐藏区任意中间牌 -> 公开区”应当允许。因此测试不能只验证顶牌，要专门验证中间牌是否也有真实入口。

重点观察：

- 主卡组和能量卡组中，不只是顶牌，中间牌也能被玩家真实选取并抓起。
- 被抓起后可以拖到当前允许的公开区。
- 该过程不会因为旧的“只允许顶牌”假设而被 UI 限死。
- 若当前实现根本没有中间牌拖拽入口，这不是小问题，而是能力缺口。

### 场景 11：跨公开/隐藏边界的万能拖拽既然被定义为允许，就必须重点验证它是否真的被放开

你已经明确把“手牌 / 主卡组 / 能量卡组 跨公开隐藏边界的万能拖拽”改为允许。因此测试重点要从“会不会被拒绝”改成“是否真的打得通，以及打通后是否正确”。

重点观察：

- 从手牌、主卡组、能量卡组拖到公开区时，桌面是否真的接受这些落点。
- 不同来源区进入公开区时，公开方式和牌面信息是否符合当前设计。
- 旧的限制提示、旧的前端早退逻辑、旧的推荐高亮，是否还在错误地阻止这类拖拽。
- 如果文档允许、但当前实现仍沿用旧边界拒绝，应直接记为严重问题。

### 场景 12：检视区内的顺序调整必须真的是拖拽，不只是按钮排序

检视区顺序是正式事实。既然文档把检视区重排当作拖拽表达的一部分，那测试就要确认玩家真的能拖动排序。

重点观察：

- 检视区中的牌可以直接抓起并改变前后顺序。
- 重排后顺序立即生效。
- 关闭再打开、或刷新后顺序不丢。
- 对手虽然看不到正面，也应感知到顺序变化过程。

### 场景 13：检视区里的“移入手牌 / 休息室 / 放回底部”必须能直接拖拽成立

这是一组已经明确承诺的拖拽路径。检视牌进入手牌、休息室或回到底部，交互层都应允许真实拖拽；协议层再分别落到专用检视命令。

重点观察：

- 检视牌能否直接拖到手牌区域。
- 检视牌能否直接拖到休息室区域。
- 检视牌能否直接通过拖拽表达“放到底部”之类的操作。
- 如果只能点击面板里的小按钮完成，而无法真实拖拽，应直接记为实现缺口。

### 场景 14：解决区中的应援牌必须能直接拖到手牌、休息室或卡组

判定应援区是高风险区，但当前文档已把“解决区 -> 手牌 / 休息室 / 卡组”视为正式拖拽路径。交互层可以是拖拽触发，命令层再分别落到解决区专用命令。

重点观察：

- 解决区中的己方应援牌能否直接抓起。
- 能否通过真实拖拽移到手牌、休息室、卡组顶部。
- 若只能通过判定面板里的按钮处理，而拖拽本身不可行，要记为拖拽承诺未兑现。

### 场景 15：等待状态、成功区、Live 区这些公开区域之间的“桌面整理”必须至少保留文档承诺的那一部分

“信任玩家”桌面最怕的是一部分区可拖，一部分区突然只能点按钮，导致玩家完全不知道哪类操作还能用原来的桌面心智。

重点观察：

- 成功区中的牌可以抓起并移到当前允许的目标区。
- Live 区中的牌在允许的窗口下可以抓起并移到当前允许的目标区。
- 休息室到成功区、公开区到休息室等典型公开移动，拖拽本身要打得通。
- 拖拽成功后双方看到的是公共过程，而不是只有当前端局部刷新。

### 场景 16：自己的区应可拖，对手的区必须不可拖，不能混淆

“不能拖对手区域”是自由拖拽模型里少数明确的硬边界。这个场景用来确认系统没有因为联机投影或 viewer seat 判断错误，把己方区误禁用，或者把对手区误放开。

重点观察：

- 自己区域能抓起的牌，对手区域不能抓起。
- 己方和对方在镜像布局下，不应因为 seat 判断反了导致拖拽权限颠倒。
- 刷新后拖拽权限不能错乱。

### 场景 17：拖拽提示高亮只是提示，不应把真正可做的合法操作挡掉

当前桌面高亮是推荐而非强限制。因此测试时要看那些“非推荐但文档允许”的落点是否仍可落下。

重点观察：

- 即便某个目标没有被推荐高亮，只要它在当前设计中被允许，就应可以落下。
- 不应出现视觉上只有一个推荐区，结果其它本来合法的区完全接不到的情况。
- 若推荐高亮和真实可放置集合不一致，要单独记录。

### 场景 18：拖拽允许和命令允许必须基本对齐，不能长期依赖“拖起来再报错”

虽然这份文档主要抓“该能拖却不能拖”，但另一个相关风险是“任何东西都能拖，结果全靠后端拒绝”。这种情况会让玩家误以为支持自由拖拽，实际体验却很差。

重点观察：

- 当前阶段明显不允许的动作，不应长期表现成“先拖得起来，再统一报错”。
- 如果某个阶段文档明确允许该自由操作，那 UI 应尽量直通，不该让玩家反复撞后端错误。
- 如果文档收口了某类拖拽，应同步把桌面入口收口，不要继续假装可拖。

## 5. 特别容易出问题的错位点

这几类场景建议测试工程师在执行时额外敏感，因为它们最容易出现“文档有，玩家做不到”的严重断层。

### 错位点 1：检视面板的描述和真实交互是否一致

需要特别核对：

- 文档、注释、UI 文案是否把检视区操作表述成“可拖到其它区域”。
- 实际上是否只有排序支持拖拽，而移入手牌、休息室、卡组底只能靠按钮。

如果答案是后者，就必须明确记录。因为这不只是一个实现细节，而是玩家心智模型被写错了。

### 错位点 2：解决区处理是否仍然被文档描述成典型拖拽

需要特别核对：

- 文档是否仍把“解决区 -> 手牌 / 休息室 / 卡组”当作典型拖拽。
- 实际实现是否已经收口成判定面板里的按钮处理。

如果当前正式体验不再支持拖拽，文档就不能继续写成“典型拖拽行为”。

### 错位点 3：休息室弹窗跨层拖拽是否真的打通

需要特别核对：

- 弹窗里的 Draggable 卡牌能否真的跨 portal 落到主桌。
- 如果因为层级、碰撞检测、遮罩、hover 区域等原因根本落不到主桌，这是非常典型的“看起来支持，实际不可用”问题。

---

## 6. 建议执行顺序

若时间有限，建议优先做这几组：

1. 场景 1、2、3、5、6，先把主桌最核心的自由拖拽链路测通。
2. 场景 9、10、11、13、14，重点抓“边界已经放开，但实现仍沿用旧限制”以及“文档说能拖，实际可能只有按钮”的错位。
3. 场景 16、17、18，再补拖拽权限与提示系统是否合理。

---

## 7. 结论标准

对这份文档而言，“规则允许”不等于“功能通过”。只有满足下面条件，才应判定该自由拖拽能力成立：

- 玩家知道可以这么做。
- 玩家能真实把牌抓起来。
- 玩家能把牌放到正确目标。
- 服务端把这次拖拽当作正式联机事实接收。
- 双方都看到和该动作对应的正确结果。

只要其中任何一步断掉，这条“信任玩家自由拖拽”链路就不成立。
