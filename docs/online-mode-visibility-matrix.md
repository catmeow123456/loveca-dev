# Loveca 联机模式可见性与公开对象矩阵

> 文档类型：联机设计文档
> 适用范围：定义 Loveca 联机首版中容器可见性、卡牌可见面、公开对象追踪和动作公开规则。
> 最后更新：2026-04-02

---

## 1. 文档目标

本文档只回答一个问题：

“同一张牌或同一个区域，在不同座位视角下，到底应该看到什么，以及公开世界里应追踪什么对象。”

为了避免旧版矩阵的混乱，本文档不再把以下概念混写：

- 区域张数是否公开
- 单卡对象是否存在于某视角
- 某视角看到正面、背面还是看不到
- 动作是否写入公共日志
- 服务端是否需要保存完整审计

这些概念现在分成四条独立轴：

1. 容器可见性
2. 单卡对象可见性
3. 卡面可见性
4. 动作输出层级

---

## 2. 统一模型

### 2.1 容器层

每个容器至少定义以下属性：

- `countVisibility`
  哪些座位知道该容器张数。
- `occupancyVisibility`
  哪些座位会收到该容器的对象级投影，而不只是容器摘要。
- `orderPolicy`
  `NONE | PUBLIC_ORDERED | OWNER_ORDERED_HIDDEN`

首版本文中的常用取值含义：

- `BOTH`
  双方都可见。
- `OWNER_ONLY`
  仅该区域所属 seat 可见。

含义区分：

- `countVisibility` 解决“对手知道你有几张手牌吗”。
- `occupancyVisibility` 解决“对手是否拿到该容器的对象级投影，而不只是张数摘要”。
- `orderPolicy` 解决“这些对象的相对顺序是否由系统维护，以及该顺序是否属于双方都可感知的牌桌事实”。

本轮讨论后，联机底层牌桌模型统一采用以下约束：

- 所有卡牌对象都属于共享牌桌世界的一部分。
- 但首版不再要求所有隐藏区对象都对双方持续可追踪。
- 首版采用“局部强跟踪，进入混淆态后断跟踪”。

### 2.2 对象层

对每张权威卡牌，系统按座位投影以下结果：

- `surfaceBySeat[seat] = NONE | BACK | FRONT`
- `trackableBySeat`
  哪些座位能把它当作一个持续存在的具体公开对象跟踪。
- `publicObjectId`
  当该对象在共享牌桌世界中可追踪时使用的对象 ID。

关键约束：

- `publicObjectId` 不是 `authorityCardId`。
- `publicObjectId` 是每张牌在一局对战内全局稳定的共享对象 ID。
- `trackableBySeat` 和 `surfaceBySeat` 不是同一件事。
- 并非所有区域中的单卡对象都要对双方持续可追踪；对象位于隐藏区期间，对非持有者可以暂时不投影该对象，但对象重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。

### 2.3 三种卡面

- `NONE`
  该座位看不到这个单卡对象。
- `BACK`
  该座位知道这里有一个具体对象，但看不到正面。
- `FRONT`
  该座位能看到正面公开信息。

补充约束：

- 在本轮确定的共享牌桌模型下，`NONE` 的适用范围有所缩小，但不会在所有隐藏区被 `BACK` 完全替代。
- 检视区、Live 放置区、解决区中的连续可观察对象，对非持有者通常更接近 `BACK`。
- 手牌、主卡组、能量卡组等隐藏混淆区，对非持有者通常只保留摘要，不默认投影为可持续跟踪的 `BACK` 单卡对象。

## 3. 设计结论

### 3.1 “张数公开”不是对象级投影

`detail_rules.md` 中“各领域卡牌数量始终公开”的条文，在联机设计里应落在：

- `countVisibility = BOTH`

它不等于“把每张牌都投影给对手”。

补充说明：

- 对手可以知道某容器有几张牌，但仍只拿到容器摘要。
- 只有 `occupancyVisibility` 允许的场合，对手才会拿到对象级投影和对应的 `publicObjectId`。

### 3.2 Live 盖牌是联机建模里的关键特殊点

同一张盖放 Live 在联机里通常同时满足：

- 操作者看到 `FRONT`
- 对手看到 `BACK`
- 双方都能追踪该对象在 Live 区中的公开占位

因此 Live 区不能再用简单的“公开区/非公开区”二分法描述。

### 3.3 首版采用“局部强跟踪，进入混淆态后断跟踪”

本轮讨论后，联机首版不再采用“手牌、牌库、能量卡组都对双方持续显示为背面对象”的模型。

更准确的结论是：

- 检视区、解决区、Live 放置区和公开区中的连续桌面过程，允许对象级强跟踪。
- 手牌、主卡组、能量卡组等隐藏混淆区，对非持有者默认只保留张数或必要摘要。
- 对象进入手牌后，对非持有者默认不再持续投影该对象。
- 对象进入主卡组、能量卡组等隐藏混淆容器后，对非持有者默认不再持续投影该对象；若之后重新进入公开链路，首版可继续沿用同一个 `publicObjectId`。

### 3.4 审计不属于可见性枚举

任何动作的审计和可见性应分开处理：

- `Public Event`
- `Private Event`
- `Sealed Audit`

本文档只定义前两者的可见性依据，不定义审计字段细节。

## 4. 区域矩阵

| 区域 | `countVisibility` | `occupancyVisibility` | 自己通常看到 | 对手通常看到 | `orderPolicy` |
| --- | --- | --- | --- | --- | --- |
| 成员区 | BOTH | BOTH | `FRONT` | `FRONT` | `NONE` |
| 成员下方能量 | BOTH | BOTH | `FRONT` | `FRONT` | `PUBLIC_ORDERED` |
| 能量区 | BOTH | BOTH | `FRONT` | `FRONT` | `NONE` |
| Live 放置区（里侧） | BOTH | BOTH | `FRONT` | `BACK` | `NONE` |
| Live 放置区（翻开后） | BOTH | BOTH | `FRONT` | `FRONT` | `NONE` |
| 主卡组 | BOTH | OWNER_ONLY | 持有者依效果决定是否可查看 `FRONT`，否则通常 `BACK` | `NONE` | `OWNER_ORDERED_HIDDEN` |
| 能量卡组 | BOTH | OWNER_ONLY | 持有者依效果决定是否可查看 `FRONT`，否则通常 `BACK` | `NONE` | `OWNER_ORDERED_HIDDEN` |
| 手牌 | BOTH | OWNER_ONLY | `FRONT` | `NONE` | `OWNER_ORDERED_HIDDEN` |
| 成功 Live 区 | BOTH | BOTH | `FRONT` | `FRONT` | `PUBLIC_ORDERED` |
| 休息室 | BOTH | BOTH | `FRONT` | `FRONT` | `NONE` |
| 除外区 | BOTH | BOTH | `FRONT` | `FRONT` | `NONE` |
| 解决区 | BOTH | BOTH | 通常 `FRONT`，也允许短暂 `BACK` -> `FRONT` | 通常 `BACK`，翻开后 `FRONT` | `NONE` |
| 检视区 | BOTH | BOTH | 通常 `FRONT` | 通常 `BACK` | `PUBLIC_ORDERED` |

补充说明：

- `OWNER_ORDERED_HIDDEN` 表示服务端可维护内部顺序真值，但该顺序不属于对手可观察的公共事实。
- 检视区仍保留为正式过程区，因此继续使用 `PUBLIC_ORDERED`。
- “成员下方能量”在事件和公共位置引用里，不建议再单独视为新的顶层区域；更准确的表达是“成员槽位上的 overlay 序列”。

## 5. 典型动作如何投影

### 5.1 抽牌

- 公共世界：通常只需要知道抽了几张牌，或某张数发生了从主卡组到手牌的变化。
- 抽牌玩家：该对象进入手牌后对自己 `FRONT`。
- 对手：不默认获得对象级持续跟踪；进入手牌后通常不再持续看到该对象，但该牌之后重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。

### 5.2 检视牌库顶 N 张

- 公共世界：这些对象真实从牌库移动到 `INSPECTION_ZONE`。
- 检视者：这些牌在检视期间通常为 `FRONT`。
- 对手：这些对象在检视期间通常为 `BACK`。
- 若检视者主动公开其中某张牌，则该对象应在检视区内直接变为双方都可见的 `FRONT`，而不是必须先离开检视区。
- 若检视区内发生拖动、放顶、放底、移入其他区域，对手也应能感知这些过程变化。

### 5.3 从手牌盖放 Live

- 操作者：该牌进入 Live 区后继续 `FRONT`。
- 对手：新出现一个 `BACK` 公开对象。
- 公共世界：应生成可追踪的 `publicObjectId`。
- 在 `LIVE_SET_PHASE` 内，若玩家将自己刚放入 Live 区的牌拖回手牌，这应视为新的正式公共移动，不是“撤销”。

### 5.4 翻开 Live

- 对手：对应对象 `BACK -> FRONT`。
- 操作者：通常仍为 `FRONT`。
- 公共世界：对象延续同一个 `publicObjectId`，只是卡面公开信息发生变化。

### 5.5 从手牌 play 成员到成员区

- 双方：该牌进入公开区并为 `FRONT`。
- 公共世界：对象可直接作为公开牌对象追踪。

### 5.6 公开区移动到私有区

- 公共世界：该对象从原区域移动到目标私有区。
- 目标私有区：服务端内部仍有权威对象真值。
- 持有者：通常看到 `FRONT`。
- 对手：到进入目标私有区这一刻为止可观察到该移动；这一步公共移动仍沿用原 `publicObjectId`。在该事件之后，进入手牌或其它隐藏混淆容器的对象通常不再持续显示给对手，但该牌之后重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。

补充锁定规则：

- 若某张已公开牌进入手牌，则“进入手牌”这一步仍可作为带原 `publicObjectId` 的公共移动被观察到；在该事件之后，对非持有者通常不再持续看到该对象。该牌之后重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。
- 若某张牌进入主卡组、能量卡组等隐藏混淆容器，则“进入目标区”这一步仍可作为带原 `publicObjectId` 的公共移动被观察到；在该事件之后，对非持有者通常不再持续看到该对象。该牌之后重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。
- `publicObjectId` 可在服务端和公共链路中稳定存在，但对象位于隐藏区期间，前端不一定持续收到该对象的投影。

### 5.7 成员下方能量的公共位置表达

- 成员下方能量在公共语义上视为附着在某个成员槽位下的公开叠放对象。
- 在公共事件中，不建议再用独立 `zone` 去表达“成员下方能量区”。
- 若一张牌被放到某成员下方，应使用“成员区槽位 + overlay 位置”表达其落点。
- 也就是说，公共位置引用更适合写成：
  - `zone = MEMBER_ZONE`
  - `ownerSeat = FIRST | SECOND`
  - `slot = LEFT | CENTER | RIGHT`
  - `overlayIndex = 0..n`

### 5.8 从能量卡组放置能量到能量区

- 这是一条“隐藏区 -> 公开区”的正式过程动作。
- 前端可表现为从能量卡组顶拖到能量区，但协议层不应把它归入 `MOVE_TABLE_CARD` 一类的万能公开区移动。
- 更准确的建模是：使用专用命令声明“从己方能量卡组顶放置 1 张能量到能量区”。
- 公共世界应记录该对象进入 `ENERGY_ZONE` 的移动；对象进入公开区后继续沿用同一个 `publicObjectId`。

### 5.9 从能量区回到能量卡组

- 这是一条“公开区 -> 隐藏区”的正式过程动作。
- 己方能量区中的能量，在当前可操作窗口内可随时拖回能量卡组。
- 前端可表现为把能量区中的公开能量直接拖回能量卡组，但协议层不应把它归入 `MOVE_TABLE_CARD` 一类的万能公开区移动。
- 更准确的建模是：使用专用命令声明“把己方能量区中的 1 张公开能量移回能量卡组”。
- 公共世界应记录该对象进入 `ENERGY_DECK` 的移动；这一步仍沿用原 `publicObjectId`。在该事件之后，对非持有者通常不再持续看到该对象。

## 6. 动作输出的统一层级

任何已提交动作都应按三层输出理解：

1. `Public Event`
   记录双方都应知道的公共过程。
2. `Private Event`
   仅补给相关座位的私密信息。
3. `Sealed Audit`
   服务端保存完整候选、顺序、随机结果和选择细节。

示例：

### 6.1 洗牌

- `Public Event`
  “某玩家洗了某个合法牌群。”
- `Private Event`
  通常为空。
- `Sealed Audit`
  保存原顺序、随机源和结果顺序。

### 6.2 Mulligan

- `Public Event`
  “某玩家已提交 Mulligan。”
- `Private Event`
  保存该玩家换出的牌和重抽结果。
- `Sealed Audit`
  保存完整换牌集合与洗回后顺序。

### 6.3 私密检视后公开加入手牌

- `Public Event`
  检视区对象移动过程 + 公开加入手牌事实。
- `Private Event`
  检视候选完整列表和选择结果。
- `Sealed Audit`
  保存完整候选、未选牌去向和顺序。

### 6.4 应援翻牌与解决区移动

- `Public Event`
  先记录对象以 `BACK` 进入解决区，再记录翻开；之后只有真正跨区移动才继续记为正式过程。
- `Private Event`
  通常为空，除非某效果额外要求仅对一侧补充私密上下文。
- `Sealed Audit`
  保存应援来源、翻牌顺序、后续选择和目标去向。

## 7. 与联机首版规则边界的关系

当前可见性模型默认建立在以下联机边界上：

- 服务端必须拒绝非法信息泄露。
- 服务端必须维护公开/非公开边界。
- 服务端必须维护有序私密区和随机真值。
- 服务端不得通过静默纠错改变公共事实。
- 规则文本明确要求自动发生的客观处理，应作为显式系统事件出现。

也就是说：

- 可见性模型不是 UI 建议。
- 它是服务端校验和投影的硬边界。

## 8. 前端渲染约束

前端不应自行猜测一张牌该显示什么，而应直接按投影结果渲染：

- `NONE`
  不渲染单卡对象，只显示容器摘要。
- `BACK`
  渲染背面对象，并使用 `publicObjectId` 做动画和跟踪。
- `FRONT`
  渲染公开牌面信息。

如果前端需要知道“这是不是同一个背面对象”，也应依赖 `publicObjectId`，而不是猜测索引或权威卡牌 ID。

补充约束：

- 前端不应再把所有隐藏区一律当成“可持续跟踪的背面对象列表”。
- 检视区、解决区、Live 放置区中的对象可按 `publicObjectId` 渲染和跟踪。
- 手牌、主卡组、能量卡组对非持有者通常只渲染容器摘要，不默认渲染对象级背面牌。

## 9. 仍待确认的问题

1. 观战模式是否允许比普通对手看到更多的摘要信息。
2. 是否需要在投影层进一步把 `FRONT` 拆成更细的信息可读级别，以表达“看到正面朝上但不可读取完整信息”的未来场景。

这些问题暂不影响当前矩阵的基本成立。
