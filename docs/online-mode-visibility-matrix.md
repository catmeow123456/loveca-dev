# Loveca 联机模式可见性与公开对象矩阵

> 文档类型：联机设计文档
> 适用范围：定义联机首版的容器可见性、卡牌可见面、公开对象跟踪和前端投影规则
> 最后更新：2026-04-22

## 1. 文档目标

本文档只回答可见性问题：

- 一个区域对谁公开张数、对象和顺序。
- 一张牌在不同座位下显示 `NONE`、`BACK` 还是 `FRONT`。
- 一个对象什么时候能用 `publicObjectId` 持续跟踪。
- 前端应按什么投影结果渲染。

公共事件字段、私密事件字段和审计字段不在本文档展开。

## 2. 统一模型

### 2.1 容器层

每个容器按三项理解：

- `countVisibility`：哪些座位知道该容器张数。
- `occupancyVisibility`：哪些座位收到对象级投影。
- `orderPolicy`：是否维护顺序，以及该顺序是否属于公共事实。

常用取值：

- `BOTH`：双方可见。
- `OWNER_ONLY`：仅区域所属座位可见。
- `NONE`：不维护公共顺序。
- `PUBLIC_ORDERED`：顺序是双方可感知的公共事实。
- `OWNER_ORDERED_HIDDEN`：服务端维护顺序，但只对持有者或审计可见。

### 2.2 对象层

每张权威卡牌按座位投影为：

- `surface = NONE | BACK | FRONT`
- `trackableBySeat`
- `publicObjectId`

约束：

- `publicObjectId` 不是权威卡牌实例 ID。
- `surface` 和“是否可持续跟踪”不是同一件事。
- 隐藏区对象可以在服务端保持稳定 `publicObjectId`，但对非持有者不持续投影。
- 对象重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。

### 2.3 三种卡面

- `NONE`：该座位看不到单卡对象，只能看到容器摘要。
- `BACK`：该座位知道这里有一个具体对象，但看不到正面。
- `FRONT`：该座位能看到正面公开信息。

## 3. 核心结论

### 3.1 张数公开不等于对象公开

规则中“各领域卡牌数量公开”在联机实现中对应 `countVisibility = BOTH`。

它不表示对手一定能拿到每张牌的对象级投影，也不表示对手能持续跟踪隐藏区对象。

### 3.2 首版采用局部强跟踪

可持续对象跟踪主要用于：

- 成员区。
- 成员下方能量。
- 能量区。
- Live 放置区。
- 成功 Live 区。
- 休息室。
- 除外区。
- 解决区。
- 检视区。

进入以下隐藏混淆区后，对非持有者默认断开持续投影：

- 手牌。
- 主卡组。
- 能量卡组。
- 洗回或洗牌后的牌堆。

### 3.3 Live 盖牌是特殊公开占位

盖放 Live 在联机中通常同时满足：

- 操作者看到 `FRONT`。
- 对手看到 `BACK`。
- 双方都能跟踪该对象在 Live 区中的公开占位。

因此 Live 区不能用简单的公开/非公开二分法处理。

## 4. 区域矩阵

| 区域 | `countVisibility` | `occupancyVisibility` | 自己通常看到 | 对手通常看到 | `orderPolicy` |
| --- | --- | --- | --- | --- | --- |
| 成员区 | BOTH | BOTH | `FRONT` | `FRONT` | NONE |
| 成员下方能量 | BOTH | BOTH | `FRONT` | `FRONT` | PUBLIC_ORDERED |
| 能量区 | BOTH | BOTH | `FRONT` | `FRONT` | NONE |
| Live 放置区（里侧） | BOTH | BOTH | `FRONT` | `BACK` | NONE |
| Live 放置区（翻开后） | BOTH | BOTH | `FRONT` | `FRONT` | NONE |
| 主卡组 | BOTH | OWNER_ONLY | 通常 `BACK`，依效果可 `FRONT` | `NONE` | OWNER_ORDERED_HIDDEN |
| 能量卡组 | BOTH | OWNER_ONLY | 通常 `BACK`，依效果可 `FRONT` | `NONE` | OWNER_ORDERED_HIDDEN |
| 手牌 | BOTH | OWNER_ONLY | `FRONT` | `NONE` | OWNER_ORDERED_HIDDEN |
| 成功 Live 区 | BOTH | BOTH | `FRONT` | `FRONT` | PUBLIC_ORDERED |
| 休息室 | BOTH | BOTH | `FRONT` | `FRONT` | NONE |
| 除外区 | BOTH | BOTH | `FRONT` | `FRONT` | NONE |
| 解决区 | BOTH | BOTH | 通常 `FRONT`，可短暂 `BACK` | 通常 `BACK`，翻开后 `FRONT` | NONE |
| 检视区 | BOTH | BOTH | 通常 `FRONT` | 通常 `BACK` | PUBLIC_ORDERED |

补充：

- 成员下方能量在公共位置表达中应视为成员槽位 overlay，不建议建成新的顶层区域。
- 检视区是正式过程区，区内排序属于可感知过程。
- 解决区中仅为阅读和排布的本地拖拽不属于正式公共事实。

## 5. 典型动作投影

### 5.1 抽牌

- 持有者：进入手牌后看到 `FRONT`。
- 对手：知道张数变化，不持续收到该手牌对象投影。
- 公共链路：可记录抽牌数量或从主卡组到手牌的摘要变化。

### 5.2 检视牌库顶 N 张

- 对象真实进入检视区。
- 检视者通常看到 `FRONT`。
- 对手通常看到 `BACK`。
- 检视区内重排、移出、放顶、放底等正式操作，应作为过程变化可感知。
- 若检视者公开其中某张牌，该对象可在检视区内从 `BACK` 变为 `FRONT`。

### 5.3 从手牌盖放 Live

- 操作者继续看到 `FRONT`。
- 对手看到新的 `BACK` 公开占位。
- 公共世界应生成或继续使用可追踪 `publicObjectId`。
- Live 设置阶段把该牌拖回手牌，是新的正式移动，不是撤销。

### 5.4 翻开 Live

- 对手看到同一对象从 `BACK` 变为 `FRONT`。
- 操作者通常仍为 `FRONT`。
- 对象延续同一个 `publicObjectId`。

### 5.5 公开区移动到隐藏区

- 进入目标隐藏区这一步仍可作为公共移动被观察。
- 该移动可继续带原 `publicObjectId`。
- 事件之后，对非持有者通常不再持续投影该对象。
- 之后重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。

### 5.6 能量卡组到能量区

- 这是隐藏区到公开区的正式过程。
- 前端可表现为拖拽能量卡组顶牌到能量区。
- 服务端应以专用命令语义处理，不应退化为无语义万能移动。
- 对象进入能量区后，双方看到 `FRONT` 并可跟踪。

### 5.7 能量区回能量卡组

- 这是公开区到隐藏区的正式过程。
- 公共世界可观察该对象进入能量卡组。
- 进入目标区后，对非持有者不再持续投影该对象。

## 6. 前端渲染约束

前端不应自行猜测卡牌可见性，应直接按投影结果渲染：

- `NONE`：不渲染单卡对象，只显示容器摘要。
- `BACK`：渲染背面对象，使用 `publicObjectId` 做动画和跟踪。
- `FRONT`：渲染公开牌面信息。

要求：

- 组件若需要名称、分数、心数、详情浮层，必须先确认对象是 `FRONT`。
- 不得从兼容态或权威态偷读 `BACK` / `NONE` 对象的真实卡牌信息。
- 检视区、解决区、Live 放置区中的对象可以按 `publicObjectId` 渲染和跟踪。
- 手牌、主卡组、能量卡组对非持有者通常只渲染容器摘要。

## 7. 仍待确认

以下问题不影响当前矩阵执行，但会影响未来扩展：

1. 观战模式是否允许比普通对手看到更多摘要。
2. 是否需要把 `FRONT` 继续拆成更细的信息可读级别。
3. 未来公共事件是否需要单独区分裁判级和玩家级投影。
