# llocg_db 卡牌同步

> 更新时间: 2026-06-27
> 文档类型: 专题说明
> 适用范围: `llocg_db` 子模块输入、中文优先合并和字段映射边界
> 当前状态: 以 `src/scripts/sync-cards-llocg.ts` 为准

本文档说明 `llocg_db` 同步专题的关键规则，不维护命令、SQL、JSON 示例或终端输出细节。

## 1. 同步定位

`llocg_db` 是当前批量同步卡牌主记录和规则结构化字段的外部数据源。同步脚本读取 JP 主数据和 CN 补充数据，生成项目内部的卡牌记录。

该专题文档用于解释 `llocg_db` 外部数据合并策略；两条同步脚本的完整职责边界、运行顺序和风险控制见 [卡牌数据同步管线](./design.md) 与 [卡牌数据同步需求](./requirements.md)。

## 2. 输入文件

同步依赖 `llocg_db/json/` 下的 JP 与 CN 卡牌数据文件。缺少输入时，同步应停止，并提示维护者初始化或更新子模块。

输入文件只作为同步脚本的数据源，不应被前端或服务端运行时代码直接依赖。

## 3. 合并逻辑

### 3.1 主从数据源

- JP 数据是主数据源，负责卡牌编号、类型和多数结构化规则字段。
- CN 数据是补充数据源，负责中文名称、中文效果文本和部分翻译信息。
- 两边数据都必须先经过卡号标准化，再按标准化后的编号匹配。

### 3.2 中文优先

对于 JP/CN 同时存在的卡牌：

- 名称和效果文本分列保存为 `name_jp` / `name_cn` 与 `card_text_jp` / `card_text_cn`。
- 运行时展示名和展示文本可以中文优先派生，但数据库不再保存重复的单一 `name` / `card_text` 列。
- 当中文名称过于泛化、会降低识别度时，应保留 JP 名称或更稳定的名称来源。
- 中文优先不能改变卡牌类型、卡号或规则字段含义。

### 3.3 CN-only

CN-only 卡牌可以生成记录，但需要明确风险：

- 结构化规则字段可能不完整。
- 组合、小组、收录商品或图片字段可能缺失。
- 进入 PUBLISHED 状态前需要人工检查。

## 4. 字段映射范围

同步字段分为几类：

| 类别 | 内容 |
| --- | --- |
| 基础身份 | 卡号、类型、中日名称 |
| 展示信息 | 中日效果文本、图片文件名、稀有度、收录商品 |
| MEMBER 规则信息 | 费用、应援棒、基础心、BLADE 心效果 |
| LIVE 规则信息 | 分数、需求心、BLADE 心效果 |
| 归属信息 | 作品数组、小组 |
| 状态信息 | 发布状态 |

Loveca Excel 负责补强真实团体、真实小队原文、商品编号、图片来源 URI 和外部来源标识；这些字段不在本专题文档重复维护。字段转换的具体实现只在同步脚本中维护。文档只描述映射范围和风险，不复制外部 JSON 结构。

## 5. 结构化字段

hearts、requirements 和 bladeHearts 是对局与构筑会使用的结构化字段。同步时需要：

- 把外部颜色 key 转换为内部 Heart 颜色枚举。
- 把 BLADE 心效果转换为内部效果类型。
- 对无法识别的 key 输出 warning。
- 避免用默认值掩盖输入数据缺失。

## 6. 数据库写入策略

同步脚本按标准化卡号比较现有卡牌：

- 不存在的卡牌可以插入。
- 已存在且无差异的卡牌跳过。
- 已存在且有差异的卡牌进入人工审核。

人工审核通过后才允许更新已有卡牌。被跳过的卡牌不得发生数据库变更。

## 7. 发布状态注意事项

当前同步记录会进入 PUBLISHED 状态。这意味着：

- 新插入卡牌会直接对构筑和对局可见。
- 已有 DRAFT 卡牌在被同步更新后可能变为 PUBLISHED。
- 正式同步后需要检查管理端与构筑端可见性。

这项行为如果改变，需要同步更新脚本、卡牌管理流程和相关文档。
