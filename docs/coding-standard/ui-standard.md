# Loveca 前端 UI 开发规范

> 本文档基于本轮 UI 重构中的实际问题整理，目标不是“写设计理念”，而是降低后续页面开发、重构和维护成本，同时保证用户体验一致。

## 1. 总原则

### 1.1 先统一结构，再做视觉

后续 UI 开发优先级应固定为：

1. 信息结构正确
2. 交互路径稳定
3. 组件职责清晰
4. 视觉风格统一
5. 局部装饰与氛围强化

禁止为了“看起来更特别”破坏结构一致性。

### 1.2 用户语言优先，禁止实现说明文案进入产品界面

界面文案只回答用户关心的三件事：

- 这里是什么
- 我现在能做什么
- 这个操作会带来什么结果

不要在 UI 中出现以下类型文案：

- “为下一场 Live 准备你的桌面”
- “当前入口已经迁到统一主题体系”
- “这个页面使用新版工作台布局”

这类内容属于设计说明，不属于产品语言。

### 1.3 同一功能必须有稳定锚点

如果一个面板从屏幕左侧打开，那么收起入口也应尽量保留在左侧同一锚点附近。

禁止出现：

- 打开按钮在左侧中部
- 收起按钮跑到右上角

这会破坏用户的空间记忆，增加操作成本。

## 2. 视觉系统规范

### 2.1 必须使用语义化 token，不允许业务组件继续写旧主题硬编码

优先使用：

- `var(--bg-*)`
- `var(--text-*)`
- `var(--border-*)`
- `var(--accent-*)`
- `var(--semantic-*)`

优先复用：

- `surface-panel*`
- `workspace-*`
- `modal-*`
- `button-*`
- `input-field`
- `status-pill`
- `chip-badge`

禁止在业务组件里继续扩散：

- `bg-[#2d2820]`
- `bg-[#3d3020]`
- `text-orange-*`
- `bg-slate-*`
- 旧 `cute-*` 风格

规则：

- 新页面不得新增一套私有配色语言
- 旧页面改造时优先先替换硬编码颜色，再处理视觉细节
- 同类组件必须共享同一套按钮、输入框、badge、panel 语义类

### 2.2 计数、标签、状态提示必须统一样式

数量 badge、状态 pill、顶部标签、面板标题不能各写一套。

建议：

- 数量标签统一用浅色 surface 胶囊
- 状态提示统一用语义色
- 卡组数量、休息室数量、卡牌数量不要混用黑底、小灰框、纯文本三种不同做法

## 3. 布局规范

### 3.1 工具栏默认紧凑，避免“大而空”的控制区

编辑器、管理页、检索页的头部工具区要默认走“紧凑工作台”布局。

原则：

- 高频输入项放同一行
- 低频扩展项折叠到下方或展开区
- 类型切换、搜索、筛选优先压缩到 1 到 2 行
- 不要让搜索框、类型 tabs、筛选器各自独占一整行，除非信息量确实过大

经验：

- “卡组名称 + 描述 + 操作按钮”可以同一行
- “类型 tabs + 搜索框”可以同一工具条
- “筛选项”应作为次级工具条，而不是主要视觉块

### 3.2 布局稳定性依赖显式高度链

涉及 `flex`、侧栏、抽屉、卡牌列表、滚动区域时，必须明确高度关系。

要求：

- 需要滚动的容器自己声明 `overflow`
- `flex` 子元素缺省检查 `min-h-0`
- 抽屉和浮层内部滚动区单独拆出，不要让整个容器既承担定位又承担滚动
- 面板外露把手时，外层容器必须允许 `overflow-visible`

### 3.3 不要依赖“看上去差不多”的定位

如果一个浮层应当居中，就必须使用稳定的居中结构，而不是靠试偏移值。

推荐结构：

```tsx
<div className="fixed inset-0 flex items-center justify-center">
  <motion.div />
</div>
```

不要把“居中定位”和“动画 transform”写在同一个 `motion.div` 上。

原因：

- Framer Motion 会覆盖 `transform`
- `translate(-50%, -50%)` 很容易失效
- 最终表现就是弹窗跑到屏幕下方或右下角

## 4. 弹窗与浮层规范

### 4.1 弹窗默认使用 portal

只要是全屏遮罩弹窗、居中 modal、跨区域浮层，默认挂到 `document.body`。

适用场景：

- 检视卡组顶
- 休息室展开
- 详情模态框
- 中央确认框

原因：

- 组件挂在局部树内时，容易被父级布局、`overflow`、`transform`、层叠上下文影响
- `fixed` 不一定真的相对视口定位

### 4.2 外层负责定位，内层负责动画

规范：

- 外层容器负责 `fixed`、`inset-0`、`items-center`、`justify-center`
- 内层 `motion.div` 只做 `opacity/scale/y` 动画

不要反过来。

### 4.3 浮层要分级，不要所有东西都做成 modal

按层级分为：

- `overlay`: 卡牌 hover 详情
- `drawer`: 判定区、详情抽屉
- `modal`: 检视卡组顶、休息室、确认框
- `workspace panel`: 编辑器、管理页、侧栏

不同层级不要混用同一交互逻辑。

## 5. 交互规范

### 5.1 模式隔离必须是条件渲染，不是“显示但弱化”

如果一个功能只属于调试模式，就在非调试模式下直接不渲染。

例如：

- 游戏日志
- 调试控制条

不要采用：

- 透明度降低
- disabled 但还占位置
- 文案提示“当前模式不可用”

这会增加视觉噪音。

### 5.2 面板开关必须成对设计

任何可展开/收起结构都要同时检查：

- 打开入口在哪
- 收起入口在哪
- 两者是否处于同一空间锚点
- 收起按钮是否会被裁切、遮挡、覆盖

### 5.3 hover 操作不能成为唯一操作路径

卡牌操作悬浮菜单可以存在，但关键操作不能只依赖 hover 才能发现。

对于移动端或窄屏：

- 必须有替代入口
- 或至少保证核心操作仍可通过按钮、面板完成

## 6. 表单规范

### 6.1 高频字段优先可见

用户最常修改的字段必须直接可见，不要藏在折叠区或滚动外。

例如卡组编辑：

- 卡组名称
- 卡组备注/描述
- 保存操作

这些必须始终在编辑入口附近看得见。

### 6.2 描述字段不要被压没

经验教训：

- 单行输入很多时，描述字段最容易被操作按钮挤掉
- 解决方式不是删掉描述，而是重排头部布局

标准：

- 如果与名称同一行，必须保证最小宽度
- 如果宽度不够，应自动换行，而不是视觉上“消失”

## 7. 文案与图标规范

### 7.1 少用 emoji 作为界面主表达

emoji 可以用于临时调试，不应作为正式 UI 的主图标系统。

优先使用统一图标库，例如 `lucide-react`。

### 7.2 图标只辅助，不替代含义

图标要辅助理解，不要让用户猜测。

正确做法：

- 图标 + 明确按钮文本
- 或图标 + `title` / `aria-label`

错误做法：

- 单独用 `🎯` 表示模式切换
- 单独用 `🔄` 表示视角切换

## 8. 维护性规范

### 8.1 组件内先抽结构，再调样式

重构顺序建议：

1. 去掉硬编码颜色和旧主题类
2. 替换成统一 token / 语义类
3. 修布局层级和滚动关系
4. 最后才调间距、hover、动画和装饰

如果顺序反过来，通常会重复返工。

### 8.2 发现同类问题时要整片收口，不要点修

例如某一轮发现：

- `BrowserCardCell`
- `DeckSectionList`
- `DeckSidebarEntry`
- `FilterPanel`

都还在用旧橙色样式

那么应该整组一起收口，而不是只改用户刚提到的单个组件。

否则后续维护成本更高，视觉也会长期半新半旧。

### 8.3 所有 UI 修复都要验证三件事

每次改动后，至少检查：

1. `npm run build` 是否通过
2. 相关组件是否仍有旧配色硬编码残留
3. 交互锚点、定位和滚动是否符合预期

## 9. 发布与版本规范

### 9.1 版本号必须单一来源

前端版本号应只来自一个地方，推荐 `client/package.json`。

版本号应同时驱动：

- 首页显示
- 页脚显示
- PWA 缓存版本

### 9.2 缓存更新必须与版本联动

只改显示版本号没有意义。

要求：

- Service Worker cache key 带版本号
- 升级版本后清理旧缓存
- 新 SW 接管后自动刷新页面

否则用户看到的版本号和实际代码可能不是同一版。

## 10. 开发检查清单

提交 UI 代码前，至少自检以下问题：

- 是否新增了硬编码颜色，而不是复用 token
- 是否有对用户无意义的说明性文案
- 是否把高频字段藏起来了
- 是否让工具栏过高、过散
- 是否出现“打开在 A，关闭在 B”的交互锚点漂移
- 是否把 modal 挂在局部容器里导致定位异常
- 是否在 `motion.div` 上同时写定位 transform 和动画 transform
- 是否在非目标模式下仍渲染无关控件
- 是否仍残留 emoji 作为正式主交互图标
- `npm run build` 是否通过

---

这份规范的核心目标只有两个：

- 降低后续 UI 修改的连锁成本
- 让用户在不同页面和不同模式下，始终得到一致、可预测的交互体验
