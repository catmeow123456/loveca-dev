## 2. 重设计背景

旧 UI 的主要问题是视觉身份割裂、原型感明显、游戏板缺少题材氛围、组件风格不统一，以及功能页面和氛围页面没有分层。

重设计的目标不是让所有页面都更“华丽”，而是建立一套统一的视觉语言：

- 游戏板、主页和关键反馈承载题材氛围。
- 登录、卡组管理、卡牌管理和后台类页面优先清晰、稳定和高效。
- 主题 token、组件状态和交互反馈保持一致。
- Love Live / Loveca 题材元素只在合适场景使用，不污染工具型页面的信息密度。

## 3. 分层双主题

分层双主题的核心是“一套语义令牌，两种色彩表达，不同页面使用不同强度的视觉表达”。

| 页面层级 | 适用页面 | 设计重心 |
| --- | --- | --- |
| 氛围层 | 首页、主页入口、对战界面、阶段反馈、结果反馈 | 题材表达、舞台感、Heart 色和沉浸感 |
| 功能层 | 登录注册、卡组编辑、卡牌管理、设置、后台页面 | 可读性、操作效率、布局稳定性和维护成本 |

深色主题偏“夜之舞台”，浅色主题偏“樱之午后”。两者共享语义 token，但背景、表面、边框、文字和强调色的具体表现由当前 CSS 维护。

## 4. 视觉素材原则

题材素材按场景使用：

- 游戏板和少量首页区域可以使用 `deck.png`、舞台光、樱花氛围和 Heart 色。
- 卡背、图标和卡牌图片应作为可识别的产品信号出现。
- 登录、卡组编辑、卡牌管理等功能页面不使用大面积插画背景或粒子装饰。
- 功能页可以保留少量品牌强调色，但布局和对比度优先。

## 5. 组件语言

统一的是组件语言，而不是所有页面的装饰强度：

- 面板、输入框、按钮、边框、阴影、滚动条和状态反馈使用主题 token。
- 图标优先使用统一图标库，不使用 emoji 作为主要功能图标。
- 游戏区域使用实线边框和状态发光表达交互状态。
- 模态框、阶段横幅和 Live 结果可以有更强的情绪表达，但不改变基础交互模式。
- 功能页按钮、筛选、分页、列表和编辑器控件优先保持克制、可扫描和可复用。

## 6. Heart 色彩

Heart 色是游戏语义色，不是全站装饰色。

优先使用场景：

- 阶段状态、Live 结果、判定统计和游戏内 Heart 计数。
- 主页入口或关键操作的小面积强调。
- 活跃玩家、成功 Live、判定结果等规则相关反馈。

避免使用场景：

- 登录注册页的大面积背景。
- 卡牌管理和卡组编辑中的高饱和大面积装饰。
- 与游戏语义无关的后台状态滥用 Heart 六色。

## 7. 动效原则

动效应服务于反馈，不应干扰高频操作：

- 日常交互保持短促，主要用于 hover、按压、区域高亮和面板开合。
- 阶段横幅、Live 成功和游戏结束可以使用较强动效。
- 待机动效只用于氛围层页面，并保持低强度。
- 动画优先使用 transform 与 opacity，避免引发布局抖动。

## 8. 页面设计方向

| 页面/区域 | 设计方向 |
| --- | --- |
| 认证页 | 功能型账号入口，稳定表单、清晰状态和低装饰 |
| 主页 | 保留品牌识别和入口层级，使用适度氛围 |
| 游戏板 | 保留布局与交互逻辑，强化背景遮罩、区域状态和阶段反馈 |
| PhaseIndicator / GameLog | 信息密度优先，使用轻量磨砂或实体面板 |
| 游戏模态框 | 统一遮罩、容器、标题层级和语义强调条 |
| 卡组管理 / 卡组编辑 | 工具型页面，高频操作舒适、筛选清晰、卡牌预览稳定 |
| 卡牌管理 | 后台管理体验，强调批量扫描、编辑效率和错误反馈 |

## 9. 遗留清理原则

历史实现中存在硬编码颜色、旧命名样式、emoji 图标、虚线区域边框和页面内联样式。清理时遵循以下原则：

- 用语义 token 替代页面级硬编码颜色。
- 将兼容类逐步收敛到统一 token，而不是继续扩展旧样式体系。
- 删除只服务于旧视觉稿的装饰类。
- 功能页不为了题材化而引入高复杂度特效。
- 清理记录进入代码评审或规范文档，不在本文维护逐项替换清单。
