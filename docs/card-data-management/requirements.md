# 卡牌数据管理系统 - 需求文档

> 版本: 1.0.0
> 创建日期: 2026-03-12
> 状态: 已实现

## 1. 背景

LoveLive! Official Card Game 拥有数百张卡牌，每张卡牌包含编号、类型、名称、效果文本、心图标、费用等大量结构化数据。这些数据是游戏引擎运行和用户组卡的基础。

项目初期，卡牌数据以本地 JSON 文件形式管理，存在以下问题：

- **数据更新困难**：修改一张卡牌需要手动编辑 JSON，容易引入格式错误
- **无权限控制**：任何接触代码仓库的人都可以修改卡牌数据
- **无版本状态**：新卡牌一旦提交就对所有用户可见，缺少审核发布流程
- **缺少可视化编辑**：管理员需要直接操作 JSON 文件，没有友好的编辑界面

## 2. 目标

建立一套基于数据库的卡牌数据管理系统，满足：

1. 管理员能通过网页界面便捷地浏览、创建、编辑和删除卡牌数据
2. 卡牌数据具备 DRAFT（草稿）和 PUBLISHED（已上线）两种状态，新数据默认不对普通用户可见
3. 普通用户和游戏引擎只能访问已上线的卡牌
4. 前端应用无需每次都请求服务器，有合理的缓存机制减轻负载

## 3. 用户角色

### 管理员

- 可查看所有卡牌（包括 DRAFT 和 PUBLISHED 状态）
- 可创建、编辑、删除卡牌
- 可控制卡牌上线/下线状态
- 可导出和导入卡牌数据

### 普通用户

- 只能查看 PUBLISHED 状态的卡牌
- 无法修改任何卡牌数据
- 通过游戏界面和组卡界面间接使用卡牌数据

## 4. 功能需求

### 4.1 卡牌浏览与筛选

管理员需要在大量卡牌中快速定位目标卡牌。系统应提供：

- **类型过滤**：按成员卡、Live 卡、能量卡筛选
- **状态过滤**：按 DRAFT、PUBLISHED 筛选，便于集中处理待审核的草稿卡牌
- **关键词搜索**：按卡牌名称或编号搜索
- **分页浏览**：卡牌数量较多时分页展示，避免一次性加载导致性能问题

### 4.2 卡牌创建与编辑

三种卡牌类型拥有不同的字段结构：

- **成员卡（MEMBER）**：费用、应援棒值、心图标（颜色+数量的数组）、应援棒心效果
- **Live 卡（LIVE）**：基础分数、心需求（颜色+数量的数组）、应援棒心效果
- **能量卡（ENERGY）**：仅基础字段（编号、名称、效果文本）

所有卡牌共享的字段：编号、名称、组合名、小组名、效果文本、图片

编辑界面需要根据卡牌类型动态展示对应字段，隐藏不相关的字段。

### 4.3 双模式编辑

为满足不同场景下的编辑效率，提供两种编辑模式：

1. **表单模式**：每个字段有独立的输入控件，心图标等复杂数据有专用的可视化编辑器。适合逐字段核对和修改
2. **YAML 模式**：将卡牌数据序列化为 YAML 文本，管理员可直接编辑文本。适合一次性修改多个字段或从其他来源粘贴数据

两种模式之间可自由切换，切换时自动同步数据。YAML 解析失败时应阻止切换并提示错误，防止数据丢失。

### 4.4 DRAFT/PUBLISHED 上线机制

卡牌的生命周期分为两个阶段：

1. **DRAFT（草稿）**：新创建或由同步管线导入的卡牌默认为此状态。仅管理员可见，普通用户和游戏引擎不可访问。管理员可在此阶段补充和校验数据
2. **PUBLISHED（已上线）**：管理员确认数据无误后将卡牌上线。所有用户可见，可用于游戏和组卡

管理员可随时将卡牌在两种状态之间切换（上线/下线），便于临时撤回有问题的数据。

### 4.5 AI 辅助效果文本提取

部分卡牌的效果文本需要从卡牌图片中人工转录，工作量大且易出错。系统应提供 AI 辅助功能：

- 对已上传图片的卡牌，管理员可一键调用 AI 识别图片上的日文效果文本
- AI 将识别结果转换为项目约定的中文标记格式
- 提取结果回填到效果文本字段，管理员可在此基础上人工校对

此功能作为辅助手段，最终结果仍需管理员确认。

### 4.6 图片管理

每张卡牌可关联一张图片。系统应提供：

- 在编辑界面中直接上传图片
- 上传过程中展示进度信息
- 上传完成后自动关联到当前卡牌

### 4.7 数据导出与导入

- **导出**：将全部卡牌数据导出为 JSON 文件，用于备份或离线分析
- **导入**：批量导入卡牌数据，返回成功/失败统计，便于从外部数据源迁移
  > **实现状态**：后端 API（`cardService.importCards`）已实现，Admin UI 导入界面待实现

### 4.8 游戏数据加载

普通用户使用的游戏界面和组卡界面需要加载卡牌数据：

- 只加载 PUBLISHED 状态的卡牌
- 加载后的数据存入前端状态管理的注册表中，供各游戏组件按编号查询
- 提供图片路径解析功能

## 5. 数据需求

### 5.1 卡牌基础信息

每张卡牌需要存储以下信息：

| 信息 | 说明 | 适用类型 |
|------|------|---------|
| 编号（card_code） | 全局唯一标识，如 PL-sd1-001 | 全部 |
| 类型（card_type） | MEMBER / LIVE / ENERGY | 全部 |
| 名称（name） | 卡牌显示名称 | 全部 |
| 组合名（group_name） | 所属偶像组合 | 全部（可选） |
| 小组名（unit_name） | 所属小组 | 全部（可选） |
| 效果文本（card_text） | 卡牌能力描述 | 全部（可选） |
| 图片（image_filename） | 关联图片文件 | 全部（可选） |
| 稀有度（rare） | R, R+, P, AR, SEC 等（目前仅由同步管线写入，前端暂不支持编辑） | 全部（可选） |
| 收录商品（product） | 来自哪个卡包（目前仅由同步管线写入，前端暂不支持编辑） | 全部（可选） |
| 状态（status） | DRAFT 或 PUBLISHED | 全部 |

### 5.2 类型特有字段

| 信息 | 说明 | 适用类型 |
|------|------|---------|
| 费用（cost） | 出场所需的能量数 | MEMBER |
| 应援棒值（blade） | 应援棒数值 | MEMBER |
| 心图标（hearts） | 颜色与数量的数组 | MEMBER |
| 应援棒心效果（blade_hearts） | 效果类型与可选颜色的数组 | MEMBER, LIVE |
| 基础分数（score） | Live 成功时获得的分数 | LIVE |
| 心需求（requirements） | 完成 Live 所需心的颜色与数量 | LIVE |

### 5.3 心颜色

系统支持 7 种心颜色：PINK、RED、YELLOW、GREEN、BLUE、PURPLE、RAINBOW（万能色，可替代任意颜色）。

## 6. 安全需求

- 卡牌数据的写操作（创建、修改、删除）仅限管理员
- 读操作按状态分级：普通用户只能读 PUBLISHED，管理员可读全部
- 权限控制在数据库层面通过 RLS 策略实现，而非仅依赖前端检查
- AI 服务的 API Key 不应暴露在前端代码中，需通过后端代理

## 7. 缓存需求

卡牌数据在短时间内不会频繁变化，但读取频率较高（游戏界面每次加载都需要）。因此：

- 前端应缓存卡牌数据，避免每次操作都请求服务器
- 缓存应有合理的过期时间，确保数据最终一致
- 管理员操作（创建/编辑/删除/上下线）后应及时清除缓存，保证管理界面数据实时性
- 状态过滤查询不应污染全量缓存，避免 DRAFT 数据泄漏到普通用户视图

## 8. 非功能需求

- **性能**：卡牌列表应分页加载；缩略图应支持预加载以提升浏览体验
- **可用性**：编辑界面应即时反馈操作结果（成功/失败提示）；图片上传应展示进度
- **数据安全**：编辑模式切换时数据不丢失；YAML 解析失败时阻止切换而非丢弃数据
- **兼容性**：新增的数据库字段应使用可空或默认值，不影响已有数据
