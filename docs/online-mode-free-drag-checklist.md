# Loveca 联机模式自由拖拽核对表

> 文档类型：联机手工测试 checklist
> 适用范围：测试工程师快速核对“信任玩家自由拖拽”是否兑现
> 配套文档：`docs/online-mode-free-drag-test-scene.md`
> 最后更新：2026-04-10

---

## 1. 使用方式

这份文档是给测试工程师快速打勾用的，不替代详细场景说明。

每一项都按下面标准判断：

- `通过`：玩家能真实拖起、拖到目标、动作正式成立、双方结果正确。
- `失败`：拖不起来、落不下、只能靠按钮替代、或结果没有正式提交。
- `存疑`：规则和实现都不够清楚，需要产品/规则/开发一起确认边界。

---

## 2. 必须可拖

这些是当前“信任玩家自由拖拽”模式下，若做不到就应视为严重问题的能力。

| 编号 | 操作 | 当前预期 |
| --- | --- | --- |
| D-01 | 手牌 -> 成员槽 | 必须可拖，正式登场 |
| D-02 | 手牌 -> Live 区 | 必须可拖，正式放置 |
| D-03 | Live 区 -> 手牌（Live 设置阶段） | 必须可拖，作为新的正式动作 |
| D-04 | 表演阶段成功效果窗口中的公开区 -> 手牌 | 必须可拖，不应只剩按钮入口 |
| D-05 | 判定阶段本地成功窗口中的公开区 -> 手牌 | 必须可拖，不应只剩按钮入口 |
| D-06 | 能量卡组顶牌 -> 能量区 | 必须可拖，但命令层应走专用命令 |
| D-07 | 能量区 -> 成员槽位下方 | 必须可拖，正式附着 |
| D-07A | 能量区 -> 能量卡组 | 必须可拖，且在当前可操作窗口内始终允许 |
| D-08 | 成员下方能量 -> 另一个成员槽位下方 | 必须可拖，正式重附着 |
| D-09 | 成员槽位 A -> 成员槽位 B | 必须可拖，正式换位 |
| D-10 | 休息室弹窗内卡 -> 其它公开区域 | 若文档仍承诺此路径，则必须真能跨层拖拽成功 |
| D-11 | 检视区内拖拽排序 | 必须可拖，顺序即时生效 |
| D-12 | 成功区中的公开牌 -> 当前允许的公开目标区 | 必须能按当前窗口语义拖拽成立 |
| D-13 | 手牌 -> 休息室（通用自由拖拽） | 必须可拖，正式进入休息室 |
| D-14 | 隐藏区任意中间牌 -> 公开区 | 必须可拖，若当前设计允许玩家自由从隐藏区取牌到公开区 |
| D-15 | 手牌 / 主卡组 / 能量卡组 跨公开隐藏边界的万能拖拽 | 必须允许，重点验证落点、公开方式和正式提交是否正确 |
| D-16 | 检视区牌 -> 手牌 | 必须可拖，命令层走 `MOVE_INSPECTED_CARD_TO_ZONE` |
| D-17 | 判定应援区牌 -> 手牌 | 必须可拖，命令层走 `MOVE_RESOLUTION_CARD_TO_ZONE` |
| D-18 | 休息室弹窗内卡 -> 手牌 | 必须可拖，命令层走 `MOVE_PUBLIC_CARD_TO_HAND` |

## 3. 明确不可拖，或不应走通用拖拽

这些能力即使玩家在桌面上做了类似拖拽尝试，也不应通过“通用自由拖拽”直接成立。

| 编号 | 操作 | 当前预期 |
| --- | --- | --- |
| N-01 | 对手区域的任意拖拽 | 不应可拖 |
| N-02 | 能量牌 -> 手牌 | 不应可拖 |
| N-03 | 能量牌 -> Live 区 | 不应可拖 |
| N-04 | 能量牌 -> 成功区 | 不应可拖 |
| N-05 | 能量牌 -> 休息室 | 不应可拖 |
| N-06 | Live 卡 -> 成员区 | 不应可拖 |
| N-07 | Live 卡 -> 能量区 / 能量卡组 | 不应可拖 |
| N-08 | 成员卡 -> 能量区 / 能量卡组 | 不应可拖 |
| N-09 | 检视流程进行中，非检视拥有者做普通桌面拖拽 | 不应通过 |
| N-10 | 主要阶段成员手牌 -> Live 区 | 不应通过；Live 区放置不应由主要阶段成员拖拽达成 |

## 4. 重点核对文档与实现是否一致

这些项最容易出现“文档写得像支持拖拽，实际只有按钮或根本没入口”的错位。测试时不要自动放过。

| 编号 | 操作或描述 | 当前要核对的问题 |
| --- | --- | --- |
| A-02 | 检视区牌 -> 休息室 | 现在是否真的支持拖拽，还是只能点按钮 |
| A-03 | 检视区牌 -> 卡组底 / 顶 | 现在是否真的支持拖拽，还是只能点按钮 |
| A-05 | 解决区牌 -> 休息室 | 现在是否真的支持拖拽，还是只能在判定面板按钮处理 |
| A-06 | 解决区牌 -> 卡组顶部 | 现在是否真的支持拖拽，还是只能在判定面板按钮处理 |
| A-07 | 休息室弹窗跨层拖拽 | 是否真的能穿过弹窗/遮罩落到主桌 |
| A-08 | 手牌 / 主卡组 / 能量卡组跨边界拖拽 | 既然当前边界改为允许，要核对桌面是否真的放开，而不是仍被旧限制拦住 |
| A-09 | 推荐高亮 | 是否只是提示；没有高亮的合法目标是否仍能落下 |
| A-10 | 表演阶段成功效果窗口里的自由拖拽 | 是真开放，还是权限列表允许但 UI 入口缺失 |
| A-11 | 判定阶段本地成功窗口里的自由拖拽 | 是真开放，还是权限列表允许但 UI 入口缺失 |

补充约束：

- 本文中的“成功效果窗口”默认指 `PERFORMANCE_PHASE` 内的正式流程窗口。
- 若当前实现仍以 `RESULT_FIRST_SUCCESS_EFFECTS` / `RESULT_SECOND_SUCCESS_EFFECTS` 暴露该窗口，应记为实现与规范未收敛，而不是把该命名当成规范前提。

## 5. 结果记录建议

建议测试时每项至少记录这四列：

| 编号 | 结果 | 现象 | 结论 |
| --- | --- | --- | --- |
| 示例：D-03 | 失败 | Live 区卡牌能抓起，但拖回手牌后无正式变化 | 严重问题，文档承诺未兑现 |

推荐结论用语：

- `严重问题`：文档承诺可拖，但玩家实际做不到。
- `实现缺口`：命令层可能支持，但 UI 没开放真实拖拽入口。
- `文档需修正`：当前实现已明确不是拖拽路径，但文档仍写成典型拖拽。
- `可接受限制`：规则或联机边界明确禁止，当前拒绝行为符合预期。

## 6. 最小回归集

如果时间非常有限，至少先测这 10 项：

- `D-01` 手牌 -> 成员槽
- `D-02` 手牌 -> Live 区
- `D-03` Live 区 -> 手牌
- `D-13` 手牌 -> 休息室
- `D-06` 能量卡组顶牌 -> 能量区
- `D-07` 能量区 -> 成员槽位下方
- `D-07A` 能量区 -> 能量卡组
- `D-09` 成员槽位换位
- `D-15` 跨公开隐藏边界的万能拖拽
- `D-16` 检视区牌 -> 手牌
- `D-17` 判定应援区牌 -> 手牌

如果这 10 项里有任意一项失败，就说明当前“信任玩家自由拖拽”能力还不能视为稳定可用。
