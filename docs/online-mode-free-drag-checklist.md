# Loveca 联机模式自由拖拽核对表

> 文档类型：联机手工测试 checklist
> 适用范围：核对当前自由拖拽权限是否和暂存区实现一致
> 最后更新：2026-04-22

## 1. 使用方式

本文档只记录当前实现承诺的拖拽能力，不再把未来设想写成必测项。

判定标准：

- `通过`：玩家能拖起、能落到目标，服务端接受命令，双方视图结果正确。
- `失败`：拖不起来、落不下、服务端拒绝，或结果没有正式提交。
- `文档需修正`：实现明确不支持，但其它文档仍写成支持。
- `可接受限制`：实现按当前规则拒绝，符合本文档边界。

相关代码路径：

- `client/src/components/game/GameBoard.tsx`
- `client/src/components/game/PlayerArea.tsx`
- `client/src/store/gameStore.ts`
- `client/src/lib/zoneUtils.ts`
- `src/application/game-commands.ts`
- `src/application/game-session.ts`

## 2. 当前权限模型

当前实现的自由拖拽按“己方区域 + UI 可拖拽状态 + 命令窗口 + 卡种限制 + 服务端校验”共同决定。

需要区分两层：

- UI 层有时会让己方区域的牌拖起来，用服务端拒绝来纠正非法动作。
- 本文档的“应允许”指服务端接受后形成正式事实，不等同于所有阶段都应能成功。

通用规则：

- 对手区域不可拖。
- 推荐高亮只是提示，不是合法性真值。
- 客户端允许的拖拽仍可能被服务端拒绝，服务端结果是正式事实。
- 常规己方桌面操作由服务端限制在主阶段、表演自由交互窗口、结果成功效果窗口。
- Live 设置阶段只额外开放 Live 放置、放置能量、公开卡回手/回能量卡组、己方卡移动等当前命令覆盖的动作，不等于开放全部普通桌面操作。
- Live 卡进出 Live 区/成功区有阶段和操作者豁免，但仍受卡种、来源、目标和服务端命令校验。
- 检视区命令走独立检视上下文，不按普通主阶段拖拽理解；己方手牌/休息室移入检视区也必须存在自己的检视流程。

当前关键命令边界：

- `MOVE_CARD_TO_INSPECTION`：己方手牌/休息室 -> 当前检视区，仅在自己的检视流程中成立。
- `MOVE_INSPECTED_CARD_TO_ZONE`：检视区 -> 手牌/休息室/除外区。
- `MOVE_INSPECTED_CARD_TO_TOP` / `MOVE_INSPECTED_CARD_TO_BOTTOM`：检视区 -> 主卡组顶/底。
- `REORDER_INSPECTED_CARD`：检视区内排序。
- `MOVE_RESOLUTION_CARD_TO_ZONE`：解决区 -> 手牌/休息室/主卡组顶。
- `MOVE_PUBLIC_CARD_TO_ENERGY_DECK`：公开能量回能量卡组。
- `MOVE_TABLE_CARD` / `MOVE_OWNED_CARD_TO_ZONE`：普通桌面移动，仍受阶段、卡种和区域限制。

| --- | --- | --- |

## 6. 常见错位检查

测试时重点记录以下错位：

- UI 显示可拖，但服务端阶段窗口长期拒绝。
- store 已有命令入口，但组件没有真实拖拽落点。
- 弹窗内卡牌能拖起，但遮罩或层级导致主桌无法接收。
- 推荐高亮遗漏合法落点，或把非法落点误提示为推荐。
- 组件层写了阶段判断，但 `GameSession` 与 `PlayerViewState` 权限投影没有同步。
- 成功日志和失败日志来自不同层，导致一次命令出现冲突提示。
