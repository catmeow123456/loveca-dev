# 卡组分享功能方案

> 版本: 0.1.0
> 创建日期: 2026-03-31
> 状态: 草案

## 1. 目标

为现有卡组管理系统补齐“分享链接 + 访客预览 + 登录后另存为我的卡组”的完整闭环，满足以下用户诉求：

- 用户可以对自己的卡组生成稳定分享链接，并发送给好友。
- 好友未登录时，可以直接打开链接预览卡组内容，但不能修改原卡组。
- 好友登录后，可以从分享页进入卡组预览/编辑界面，并通过“保存到我的卡组”复制一份到自己的卡组列表。
- 保存后的卡组与原卡组彻底解耦，后续修改只影响自己的副本。

## 2. 现状与缺口

当前系统已经具备一部分基础能力：

- 数据库 `decks` 表已有 `is_public` 字段，可标记卡组是否公开。
- 后端已有公开列表接口 `GET /api/decks/public`。
- 后端已有 `GET /api/decks/:id`，允许拥有者、管理员或公开卡组访问。

但现有能力还不足以支撑“可分享”的产品体验，主要缺口如下：

- 公开能力依赖卡组主键 `id`，缺少明确的分享链接语义。
- 没有单独的分享页路由，访客无法通过稳定 URL 直接进入预览页。
- 没有“另存为我的卡组”能力，无法在分享卡组基础上继续构筑。
- 没有分享状态、分享文案、复制链接等前端交互。
- 没有针对分享链路的权限边界和审计约束说明。

## 3. 产品方案

### 3.1 核心用户流程

#### 分享者

1. 在“我的卡组”列表或卡组编辑页点击“分享”。
2. 若该卡组尚未开启分享，系统先开启分享状态。
3. 系统生成并展示分享链接，例如 `/decks/share/{shareId}`。
4. 用户可复制链接发送给好友，也可随时关闭分享。

#### 访客/好友（未登录）

1. 打开分享链接。
2. 进入只读预览页，查看卡组名称、作者、描述、成员卡 / Live 卡 / 能量卡列表、卡组统计与合法性状态。
3. 页面显示“登录后保存到我的卡组”按钮。
4. 点击后跳转登录页；登录成功后回到原分享页。

#### 访客/好友（已登录）

1. 打开分享链接。
2. 进入同一个只读预览页。
3. 点击“保存到我的卡组”。
4. 系统复制该卡组内容，创建一条新的 `decks` 记录归属于当前登录用户。
5. 创建成功后跳转到卡组编辑页，用户可以基于副本继续修改。

### 3.2 页面与路由

建议新增以下前端路由：

- `/decks/share/:shareId`
  用于公开分享卡组的只读预览页，未登录可访问。
- `/decks/share/:shareId/save`
  不单独渲染页面，可作为登录跳转回调后的动作入口，最终仍回到分享预览页或编辑页。

建议复用现有组件能力：

- 卡组详情展示尽量复用 `CardEditor` 右侧的 `DeckSidebar`、`DeckStats`、卡牌详情弹窗等已有只读展示模块。
- 只读预览页不展示加减卡按钮，不允许修改卡组内容。
- “保存到我的卡组”成功后，再跳转到现有编辑流程。

## 4. 数据模型方案

### 4.1 方案选择

有两种实现方向：

#### 方案 A：直接复用 `decks.id` 作为分享链接标识

优点：

- 改动最小，后端几乎不用新增数据列。
- 现有 `GET /api/decks/:id` 基本可复用。

缺点：

- 分享链接语义不清晰，资源标识和分享标识耦合。
- 未来若要支持“关闭后重新生成链接”“分享链接轮换”“分享访问统计”，扩展性差。

#### 方案 B：新增独立 `share_id` 与分享状态字段

优点：

- 分享链接与数据库主键解耦，语义更清晰。
- 后续支持链接轮换、失效、访问统计、导入来源追踪都更自然。
- 前端路由也更稳定，避免直接暴露内部主键语义。

缺点：

- 需要数据库迁移和接口调整。

建议采用 **方案 B**。

### 4.2 推荐字段

建议在 `decks` 表新增以下字段：

```sql
ALTER TABLE public.decks
  ADD COLUMN share_id UUID UNIQUE DEFAULT gen_random_uuid(),
  ADD COLUMN share_enabled BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN shared_at TIMESTAMPTZ,
  ADD COLUMN forked_from_deck_id UUID REFERENCES public.decks(id) ON DELETE SET NULL,
  ADD COLUMN forked_from_share_id UUID,
  ADD COLUMN forked_at TIMESTAMPTZ;
```

字段含义：

- `share_id`
  对外分享链接使用的稳定标识。
- `share_enabled`
  是否开启分享。建议未来逐步替代 `is_public` 的业务含义。
- `shared_at`
  首次开启分享的时间，便于排序与审计。
- `forked_from_deck_id`
  当前卡组是否由他人卡组复制而来，用于来源追踪。
- `forked_from_share_id`
  保存时记录来源分享链接，避免原卡组后续删除后失去来源上下文。
- `forked_at`
  复制保存时间。

### 4.3 与现有 `is_public` 的关系

为降低迁移成本，建议分阶段处理：

- 第一阶段：保留 `is_public`，并将其视为 `share_enabled` 的兼容字段。
- 新逻辑统一读写 `share_enabled`，同时同步更新 `is_public`。
- 后续稳定后，再考虑将 `is_public` 标记为废弃字段。

这样可以避免一次性大改现有公开列表、权限判断和历史数据。

## 5. 后端接口方案

### 5.1 查询分享卡组

新增公开接口：

- `GET /api/decks/share/:shareId`

返回字段建议包含：

- 卡组基础信息：`name`、`description`
- 作者公开信息：`user_id`、`display_name`
- 卡组内容：`main_deck`、`energy_deck`
- 验证状态：`is_valid`、`validation_errors`
- 分享信息：`share_id`、`share_enabled`、`shared_at`
- 只读标识：`can_fork`

权限规则：

- `share_enabled = true` 时，任何人可访问。
- `share_enabled = false` 时，返回 404，而不是 403，避免探测卡组是否存在。

### 5.2 开启/关闭分享

新增受保护接口：

- `POST /api/decks/:id/share`
- `DELETE /api/decks/:id/share`

建议行为：

- 开启分享时：
  - 若 `share_id` 为空则生成；
  - 设置 `share_enabled = true`；
  - 同步 `is_public = true`；
  - 若 `shared_at` 为空则写入当前时间。
- 关闭分享时：
  - 设置 `share_enabled = false`；
  - 同步 `is_public = false`；
  - 默认不重置 `share_id`，方便以后重新开启后仍沿用原链接。

可选增强：

- `POST /api/decks/:id/share/regenerate`
  用于重新生成 `share_id`，使旧链接失效。

### 5.3 从分享卡组复制保存

新增受保护接口：

- `POST /api/decks/share/:shareId/fork`

请求体建议：

```json
{
  "name": "默认可选，未传时使用原卡组名 + 副本",
  "description": "默认可选，未传时复制原描述"
}
```

行为：

1. 按 `shareId` 查询分享卡组，要求 `share_enabled = true`。
2. 读取其卡组内容。
3. 创建一条新的 `decks` 记录，`user_id = 当前登录用户`。
4. 将 `forked_from_deck_id`、`forked_from_share_id`、`forked_at` 写入新记录。
5. 新卡组默认 `share_enabled = false`、`is_public = false`。
6. 返回新卡组记录。

关键约束：

- 禁止把分享卡组直接“转移所有权”给当前用户，必须创建副本。
- 即使当前用户就是原作者，也走复制逻辑或直接提示“这是你的卡组，去编辑原卡组”。

## 6. 前端交互方案

### 6.1 DeckManager 列表页

在每个云端卡组条目增加“分享”入口：

- 未开启分享：显示“分享”
- 已开启分享：显示“已分享”状态和“复制链接”
- 菜单项可包含：
  - 复制分享链接
  - 打开分享页
  - 关闭分享
  - 重新生成链接（可放二级确认）

### 6.2 卡组编辑页

在编辑页工具栏增加分享按钮：

- 若当前是自己的云端卡组，可直接开启/关闭分享。
- 若当前还未保存到云端，则先提示“请先保存卡组后再分享”。
- 若当前是从分享页 fork 得来的副本，可展示“来源于某分享卡组”的轻量提示。

### 6.3 分享预览页

建议页面信息结构：

- 卡组名称
- 作者名称
- 分享时间 / 更新时间
- 卡组说明
- 卡组统计与合法性状态
- 三个分区的卡牌清单
- CTA 区域：
  - 未登录：`登录后保存到我的卡组`
  - 已登录且非作者：`保存到我的卡组`
  - 已登录且作者本人：`编辑原卡组`

### 6.4 登录跳转与回跳

为避免登录后丢失上下文，建议在登录页保留回跳参数：

- 进入登录页时附带 `redirect=/decks/share/{shareId}`
- 登录成功后读取 `redirect` 并返回分享页

如果用户点击的是“登录后保存到我的卡组”，可增加动作参数：

- `redirect=/decks/share/{shareId}?action=fork`

分享页加载后若检测到 `action=fork` 且用户已登录，可自动弹出确认对话框或直接发起 fork。

## 7. 权限与安全

### 7.1 访问控制

- 仅卡组拥有者和管理员可开启/关闭分享。
- 分享页只允许读取 `share_enabled = true` 的卡组。
- 分享页仅返回作者公开资料，不返回邮箱等敏感信息。

### 7.2 防枚举

- 分享链接必须使用高熵随机标识，例如 UUIDv4。
- 对 `GET /api/decks/share/:shareId` 的未命中统一返回 404。
- 可对分享接口增加基础限流，防止暴力探测。

### 7.3 数据隔离

- fork 时始终新建卡组记录，不能原地修改分享者卡组。
- 新副本默认关闭分享，避免“保存后意外继续公开”。
- 副本的后续更新不影响来源卡组。

## 8. 验证与兼容性

### 8.1 卡牌可见性

分享页仍只展示当前系统内可识别、已发布的卡牌数据。若历史卡组中存在本地 registry 中缺失的卡牌，建议：

- 服务端仍返回原始卡组数据；
- 前端预览时对缺失卡牌显示占位卡片与编号，避免整页失败。

### 8.2 卡组合法性

分享页应展示分享时保存下来的 `is_valid` 与 `validation_errors`，同时前端可基于当前规则再做一次轻量校验，但以后端保存状态为主。

### 8.3 向后兼容

- 对历史 `is_public = true` 且 `share_id` 为空的卡组，首次开启分享时自动补 `share_id`。
- 旧公开卡组列表可暂时继续基于 `is_public` 工作，后续逐步迁移到 `share_enabled`。

## 9. 实施步骤

### Phase 1：后端与数据层

1. 为 `decks` 表增加 `share_id`、`share_enabled`、fork 来源字段。
2. 实现分享查询、开启/关闭分享、fork 三组接口。
3. 补充接口权限校验和单元测试。

### Phase 2：前端分享页

1. 新增分享预览路由 `/decks/share/:shareId`。
2. 实现分享页的数据获取、只读展示和缺失卡牌降级显示。
3. 接入登录回跳逻辑。

### Phase 3：前端分享入口

1. 在 DeckManager 和编辑页增加分享按钮与复制链接交互。
2. 增加“已分享”状态展示。
3. 接入 fork 流程，保存成功后跳转编辑页。

### Phase 4：增强能力

1. 支持重新生成分享链接。
2. 支持分享访问统计。
3. 支持分享页 Open Graph 元信息，优化社交平台预览。

## 10. MVP 建议

若希望先尽快上线，建议 MVP 范围控制为：

- 新增 `share_id` + `share_enabled`
- 支持开启分享 / 关闭分享
- 支持公开分享页只读预览
- 支持登录后“保存到我的卡组”

以下内容可延后：

- 重新生成分享链接
- 分享访问统计
- 社交卡片预览
- 来源追踪在 UI 中展示

## 11. 待确认问题

- 分享页是否允许显示作者昵称以外的资料，例如头像。
- fork 后的新卡组默认命名规则是“原名 - 副本”还是“原名（来自分享）”。
- 作者本人打开自己的分享链接时，是否直接展示“编辑原卡组”而不是“保存到我的卡组”。
- 分享页是否需要支持未登录时查看卡牌详情弹窗。
- 是否需要给分享链接设置失效机制，还是默认永久有效直到用户手动关闭。

## 12. 推荐结论

推荐按“独立 `share_id` + 只读分享页 + 登录后 fork 保存”的思路实施。

这个方案和当前系统最兼容，原因是：

- 可以复用现有 `decks` 数据结构和大部分展示组件；
- 只需在现有“公开卡组”能力上补上分享语义、链接和 fork 行为；
- 用户心智清晰，权限边界也容易保证；
- 后续要做链接轮换、访问统计、分享社区时不需要推倒重来。
