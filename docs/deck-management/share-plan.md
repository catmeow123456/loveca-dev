## 2. 目标

卡组分享功能用于补齐“分享链接 + 访客预览 + 登录后另存为我的卡组”的闭环，满足以下诉求：

- 用户可以对自己的卡组生成稳定分享链接，并发送给好友。
- 未登录访客可以直接打开链接预览卡组内容，但不能修改原卡组。
- 登录用户可以从分享页复制一份到自己的卡组列表。
- 复制后的卡组与来源卡组解耦，后续修改只影响自己的副本。

## 3. 核心用户流程

### 分享者

1. 在卡组管理入口开启分享。
2. 系统生成并展示分享链接。
3. 用户复制链接发送给好友，也可以关闭分享。

### 访客/好友

1. 打开分享链接。
2. 查看只读卡组预览，包括卡组名称、作者公开信息、说明、分区列表、统计与合法性状态。
3. 未登录时可被引导登录。
4. 登录后可将该卡组保存为自己的副本。

## 4. 方案取舍

### 方案 A：复用卡组主键作为分享标识

优点：

- 改动小。
- 可以复用现有卡组读取模型。

缺点：

- 资源标识和分享标识耦合，语义不清晰。
- 未来支持链接轮换、失效、访问统计或来源追踪时扩展性差。
- 容易暴露内部资源标识。

### 方案 B：使用独立分享标识与分享状态

优点：

- 分享链接与数据库主键解耦，语义更清晰。
- 更适合后续支持链接轮换、失效、访问统计、导入来源追踪。
- 前端路由和用户心智更稳定。

缺点：

- 需要额外字段和迁移成本。
- 需要明确旧公开语义与新分享语义的兼容边界。

已落地方向采用方案 B。具体字段和接口事实以 [卡组管理设计](design.md) 为准。

## 5. 权限与安全原则

- 只有卡组拥有者和具备相应权限的管理员可以开启或关闭分享。
- 分享页只允许读取已开启分享的卡组。
- 分享页只返回作者公开资料，不返回邮箱等敏感信息。
- 分享标识必须具备足够随机性，避免被枚举。
- 对未命中、未公开或已关闭分享的链接，不应泄露私有卡组是否存在。
- 复制分享卡组时必须创建新副本，不能转移或修改来源卡组。
- 新副本默认不继承公开状态，避免意外继续公开。

## 6. 兼容性原则

- 历史公开卡组与新分享语义之间应保持可迁移、可回退的兼容关系。
- 来源追踪用于审计和用户提示，不应让副本继续依赖来源卡组的存在。
- 分享页应能处理历史卡组中缺失卡牌数据的情况，优先降级展示编号或占位信息，而不是整页失败。
- 卡组合法性可根据当前规则重新计算；若未来需要冻结分享时的合法性状态，应作为独立规则设计。

## 7. 后续增强方向

仍未实现或仍需确认的增强项：

- 卡组编辑页中的直接分享入口。
- 分享链接重新生成与旧链接失效。
- 分享访问统计。
- Open Graph 等社交平台预览信息。
- 分享链接失效时间、访问限流和风控策略。
- 作者本人打开分享链接时是否直接进入原卡组编辑。
- fork 后的新卡组默认命名规则。
- 来源追踪在 UI 中的展示方式。
