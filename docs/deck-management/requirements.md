# 用户卡组管理系统 - 需求文档

> 版本: 1.0.0
> 创建日期: 2026-03-12
> 文档类型: 需求文档
> 适用范围: 卡组构筑、云端保存、DeckLog 导入、分享与游戏入口选组
> 当前状态: 已实现

## 1. 概述

### 1.1 系统目标

为 Love Live 卡牌游戏（Loveca）的玩家提供一套完整的卡组构筑与管理工具。玩家可以在线创建、编辑、保存自己的卡组，并在进入对局前选择已构筑好的卡组进行游戏。

### 1.2 设计动机

卡组构筑是卡牌游戏的核心体验之一。玩家需要在游戏规则约束下，从已发布的卡牌池中选择合适的卡牌组合，形成自己的战术方案。系统需要在保证规则合规的前提下，尽可能降低构筑门槛，提供直观的筛选、搜索和可视化反馈，让玩家专注于策略思考而非机械操作。

同时，卡组数据需要持久化到云端，使玩家可以跨设备访问自己的卡组，并为未来的卡组分享、社区功能打下基础。

### 1.3 用户角色

| 角色     | 说明                                                                                                                     |
| -------- | ------------------------------------------------------------------------------------------------------------------------ |
| 普通用户 | 可以创建、编辑、删除自己的卡组，选择卡组进入游戏                                                                         |
| 管理员   | 在普通用户的基础上，可以按已知卡组 ID 执行管理员级读取、修改、删除和分享开关操作；当前没有全量查看所有用户卡组的列表入口 |
| 访客     | 可以查看被设置为公开的卡组（只读）                                                                                       |

## 2. 卡组构筑规则

以下规则来源于 Love Live 卡牌游戏官方规则第 6.1 节，是系统进行卡组验证的依据。

### 2.1 卡组结构

一副完整的卡组由两部分组成：

- **主卡组**：包含 60 张卡牌，其中成员卡（MEMBER）恰好 48 张，Live 卡恰好 12 张。主卡组不能包含能量卡。
- **能量卡组**：包含恰好 12 张能量卡（ENERGY）。能量卡组只能包含能量卡类型的卡牌。

### 2.2 同编号限制

主卡组中，同一**基础编号**的卡牌最多放入 4 张。这一限制确保卡组具有足够的多样性。

**基础编号**的定义：将卡牌编号（`cardCode`）去掉最后一个 `-` 及之后的稀有度后缀后得到的编号，系列前缀中的 `!` 会保留。例如 `LL-bp1-001-N` 和 `LL-bp1-001-R+` 的基础编号均为 `LL-bp1-001`，`PL!-sd1-050-SEC` 的基础编号为 `PL!-sd1-050`。

不同稀有度但基础编号相同的卡牌被视为"同一张卡"，它们在主卡组中的数量**合计**不得超过 4 张。在编辑界面中，这些同类卡牌上显示的数量徽章也会同步，均显示同基础编号卡在卡组中的总数。

### 2.3 卡牌来源限制

只有处于"已发布"（PUBLISHED）状态的卡牌才能被添加到卡组中。处于草稿（DRAFT）状态的卡牌对构筑界面不可见，从根本上防止了将未发布卡牌纳入卡组的可能。

这一限制通过应用启动时的数据加载实现：`App.tsx` 在初始化时仅请求 PUBLISHED 状态的卡牌数据并写入 `gameStore.cardDataRegistry`，而卡牌编辑器（`CardEditor`）从这个 registry 读取可选卡牌，因此天然只展示已发布的卡牌。

## 3. 功能需求

### 3.1 卡组 CRUD

#### 创建卡组

用户可以从空白开始创建新卡组。创建时需要指定卡组名称，可选填写描述。创建后进入编辑模式，从卡牌库中逐张添加卡牌。

#### 编辑卡组

用户可以对已保存的卡组进行修改。编辑界面由左侧的卡牌库和右侧的当前卡组构成。**左右两侧均支持添加与删除卡牌**，具体交互如下：

- **左侧卡牌库**：点击卡图查看卡牌详情；卡图底部常驻 `− / 数量 / +` 控制条，分别用于减少和增加当前精确编号卡牌数量。若同基础编号卡已在卡组中，卡图中央显示同基础编号的总数遮罩。
- **右侧卡组预览**：以图片网格展示当前卡组条目，每张卡底部常驻 `− / 数量 / +` 控制条。点击或右键卡图查看卡牌详情。

右侧的卡组列表按成员卡、Live 卡、能量卡三个分区展示，每个分区可独立折叠。每个分区头部显示当前数量与目标数量的对比（如 "32/48"），帮助玩家直观了解构筑进度。

- **成员卡分区排序**：按费用（`cost`）从低到高排序；若费用相同，则按卡牌编号（`cardCode`）字典序排序。
- **Live 卡分区排序**：按分数（`score`）从低到高排序；若分数相同，则按卡牌编号（`cardCode`）字典序排序。
- **能量卡分区排序**：维持现有顺序，不额外重排。

#### 保存卡组

保存前系统会自动验证卡组是否符合构筑规则，并将验证结果写入 `is_valid` 与 `validation_errors`。当前实现不会阻止未完成卡组保存：新卡组通过 `POST /api/decks` 创建，编辑已有卡组通过 `PUT /api/decks/:id` 原地更新，保存完成后自动刷新卡组列表并返回列表页。

#### 删除卡组

删除前需要用户二次确认，防止误操作。当前删除接口会移除卡组记录并从前端本地列表中删除该项；卡组计数维护边界见 [当前实现限制](../current-limitations.md#数据库初始化与-schema)。

### 3.2 卡牌筛选与搜索

卡牌编辑器提供多维度的筛选能力，帮助玩家快速定位目标卡牌：

- **卡牌类型切换**：在成员卡、Live 卡、能量卡三个标签页之间切换，一次只展示一种类型。
- **文字搜索**：按卡牌名称或编号进行模糊匹配。
- **高级筛选**（可展开/收起）：
  - **稀有度筛选**：根据卡牌编号后缀解析出的稀有度进行过滤（如 N、R、R+、SEC 等）。
  - **组合筛选**：按卡牌所属的偶像组合过滤（如 μ's、Aqours、Liella! 等）。
  - **收录商品筛选**：按卡牌 `product` 字段过滤，匹配时会忽略空格差异。
  - **小组筛选**（仅成员卡）：按组合内的小组过滤。选择组合后，小组选项自动联动，只显示该组合下的小组。
  - **费用区间筛选**（仅成员卡）：按卡牌出场费用过滤，默认范围 0-22。
  - **心颜色筛选**（成员卡 / Live 卡）：成员卡按持有心颜色过滤；Live 卡按需求心颜色过滤。7 种颜色可选（粉/红/黄/绿/蓝/紫/虹）。
  - **BLADE 心效果筛选**（成员卡 / Live 卡）：按 `bladeHearts` 中的加分、抽卡或指定颜色 HEART 效果过滤。
  - **Live 分数区间筛选**（仅 Live 卡）：按 Live 卡成功时获得的分数过滤，默认范围 0-10。
- 能量卡类型下仅保留稀有度、组合、收录商品等适用筛选，不显示小组、费用、心颜色、BLADE 心效果或分数筛选。
- 所有筛选条件可一键清除重置。

筛选按卡牌类型 → 文字搜索 → 稀有度 → 组合 → 收录商品 → 小组 → 费用区间 → 心颜色 → BLADE 心效果 → 分数区间的顺序链式应用，最终结果按卡牌编号排序。

### 3.3 实时验证与反馈

在编辑过程中，系统持续检查当前卡组状态：

- 各分区的当前数量实时显示，颜色反映是否达标。
- 添加卡牌时，若对应分区已满或同编号数量已达上限，操作会被静默阻止。
- 卡组列表页通过内联状态徽章展示“完整 / 未完成”，用户无需打开卡组即可了解其是否合规。

### 3.4 YAML 导入/导出

系统支持通过 YAML 格式进行卡组的导入和导出，便于跨平台传输和备份。

- **导出**：将当前编辑中的卡组序列化为 YAML 文件并触发浏览器下载。YAML 结构与 `DeckConfig` 类型一致，包含 `player_name`、`main_deck`（分 `members` 和 `lives`）、`energy_deck` 三部分。
- **导入**：用户选择本地 `.yaml` 或 `.yml` 文件后，当前 `DeckManager` 直接解析 YAML，并对缺失的 `main_deck` / `energy_deck` 做默认值补齐后填充到编辑界面；尚未接入 `DeckConfigSchema.safeParse()` 做完整结构校验。

### 3.5 卡组图片下载（当前未提供）

当前 `DeckManager` 没有管理员下载卡组图片 ZIP 的入口或实现。若后续需要印刷、展示等运营场景，应作为新需求单独设计，包括图片来源、文件命名、权限边界、打包方式和失败重试策略。

### 3.6 DeckLog 卡组导入

用户可以输入 DeckLog 网站上的卡组 ID 或完整 URL，系统通过服务端调用 DeckLog 内部 JSON API 获取卡牌数据并自动填充到编辑界面。

**输入格式**：

- 纯卡组 ID：如 `2D6XL`
- 完整 URL：如 `https://decklog.bushiroad.com/view/2D6XL`

**爬取流程**：

1. 前端将用户输入发送到后端 `POST /api/decks/scrape-decklog` 接口。
2. 后端调用 DeckLog 内部 JSON API（`POST /system/app/api/view/{deckId}`），获取包含 `list`（主卡组）和 `sub_list`（副卡组）的 JSON 数据。
3. 卡牌编号标准化处理：全角加号 `＋` 转为半角 `+`，通过 `normalizeCardCode()` 统一格式。
4. 返回解析后的卡牌列表和卡组名称（来自 API 返回的 `title` 字段）。

**前端匹配逻辑**：

- 收到爬取结果后，逐一在本地 `cardDataRegistry` 中查找卡牌数据。
- 根据卡牌的 `cardType`（MEMBER / LIVE / ENERGY）自动归类到对应分区。
- 未在本地数据中找到的卡牌以警告形式展示，不阻断导入流程。
- 匹配成功后自动进入编辑模式，卡组名称默认使用 DeckLog 返回的标题。

**错误处理**：

- 无效输入格式：前端提示"无效的 DeckLog ID 或 URL"。
- 网络超时（15 秒）：提示"请求 DeckLog 超时"。
- 非 JSON 响应：提示"DeckLog API 返回了非 JSON 响应，可能该卡组不存在"。
- 卡组数据为空：提示"卡组数据为空"。
- 无法标准化卡牌编号：提示"未能标准化任何卡牌编号"。
- 无匹配卡牌：提示"没有匹配到任何已知卡牌"。
- 无后端服务：提示"需要连接后端服务才能使用此功能"。

**前提条件**：此功能依赖后端 API 服务，离线模式下不可用。

### 3.7 卡组公开分享

用户可以将卡组设置为"公开"，公开后所有人（包括未登录的访客）都可以通过分享链接查看该卡组内容；登录用户可将公开分享卡组复制到自己的账号下继续编辑。

### 3.8 卡组选择（游戏入口）

在进入对局前，玩家通过卡组选择器（`DeckSelector`）从已保存的卡组中选择一副用于游戏。`DeckSelector` 组件具备同时接收云端卡组和本地卡组的能力；当前游戏准备页、在线房间和调试入口主要传入云端卡组。选择器会显示每个卡组的统计信息和有效性状态，帮助玩家做出选择。

## 4. 数据安全与隔离需求

### 4.1 数据隔离

普通用户只能查看和操作自己的卡组，不能访问其他用户的私有卡组。当前实现通过 Express 路由中的 JWT 鉴权、所有者校验和管理员角色校验实现隔离；公开或开启分享的卡组只开放只读查看与复制入口。

### 4.2 管理员权限

管理员角色可以按已知卡组 ID 查看、修改和删除其他用户卡组，用于内容审核和运维支持；但 `GET /api/decks` 当前只返回当前登录用户自己的卡组，尚未提供管理员全量列表入口。管理员权限通过服务端鉴权中间件与路由中的 `req.user.role === 'admin'` 判断实现。

### 4.3 自动维护

`profiles.deck_count`、`decks.updated_at`、分享字段与初始化脚本之间的当前维护差异集中记录在 [当前实现限制](../current-limitations.md#数据库初始化与-schema)。本文档只保留业务需求，不重复维护部署限制细节。

### 4.4 外键完整性

卡组通过 `user_id` 外键关联到 `profiles` 表；`profiles.id` 再级联关联到自托管 `users` 表，确保用户账号删除时其所有卡组数据自动清理。

## 5. 非功能性需求

### 5.1 离线支持

当前 `isApiConfigured` 固定为 `true`，应用启动仍依赖 `/api/cards` 加载已上线卡牌。若 REST API 服务不可用，当前实现会进入卡牌数据加载错误页，而不是自动进入完整离线卡组编辑流程。登录页仍保留手动离线模式入口，但云端卡组、DeckLog、分享和卡牌管理等后端功能不可用；若要恢复完整离线卡组编辑，需要重新梳理启动加载和本地卡牌数据来源。

### 5.2 向后兼容

数据库中存量卡组的 `main_deck` 字段可能不包含 `card_type` 字段（旧格式）。系统在读取这些卡组时统一使用 `deckRecordUtils` 转换：优先读取 `card_type`，可用本地卡牌数据时按真实 `cardType` 识别，最后再按历史编号规则做兜底推断，确保旧数据在卡组管理、分享页、卡组选择和游戏入口中表现一致。

### 5.3 性能

- 卡牌筛选使用 `useMemo` 进行计算缓存，避免在每次渲染时重复过滤全量卡牌数据。
- 云端操作设置 15 秒超时，防止网络异常时界面长时间挂起。
