# 用户卡组管理系统 - 需求文档

> 版本: 1.0.0
> 创建日期: 2026-03-12
> 状态: 已实现

## 1. 概述

### 1.1 系统目标

为 Love Live 卡牌游戏（Loveca）的玩家提供一套完整的卡组构筑与管理工具。玩家可以在线创建、编辑、保存自己的卡组，并在进入对局前选择已构筑好的卡组进行游戏。

### 1.2 设计动机

卡组构筑是卡牌游戏的核心体验之一。玩家需要在游戏规则约束下，从已发布的卡牌池中选择合适的卡牌组合，形成自己的战术方案。系统需要在保证规则合规的前提下，尽可能降低构筑门槛，提供直观的筛选、搜索和可视化反馈，让玩家专注于策略思考而非机械操作。

同时，卡组数据需要持久化到云端，使玩家可以跨设备访问自己的卡组，并为未来的卡组分享、社区功能打下基础。

### 1.3 用户角色

| 角色 | 说明 |
|------|------|
| 普通用户 | 可以创建、编辑、删除自己的卡组，选择卡组进入游戏 |
| 管理员 | 在普通用户的基础上，可以查看和管理所有用户的卡组，可以下载卡组对应的卡牌图片 |
| 访客 | 可以查看被设置为公开的卡组（只读） |

## 2. 卡组构筑规则

以下规则来源于 Love Live 卡牌游戏官方规则第 6.1 节，是系统进行卡组验证的依据。

### 2.1 卡组结构

一副完整的卡组由两部分组成：

- **主卡组**：包含 60 张卡牌，其中成员卡（MEMBER）恰好 48 张，Live 卡恰好 12 张。主卡组不能包含能量卡。
- **能量卡组**：包含恰好 12 张能量卡（ENERGY）。能量卡组只能包含能量卡类型的卡牌。

### 2.2 同编号限制

主卡组中，同一**基础编号**的卡牌最多放入 4 张。这一限制确保卡组具有足够的多样性。

**基础编号**的定义：将卡牌编号（`cardCode`）去掉 `!` 字符和末尾的稀有度后缀后得到的编号。例如 `LL-bp1-001-N` 和 `LL-bp1-001-R+` 的基础编号均为 `LL-bp1-001`，`PL!-sd1-050-SEC` 的基础编号为 `PL-sd1-050`。

不同稀有度但基础编号相同的卡牌被视为"同一张卡"，它们在主卡组中的数量**合计**不得超过 4 张。在编辑界面中，这些同类卡牌上显示的数量徽章也会同步，均显示同基础编号卡在卡组中的总数。

### 2.3 卡牌来源限制

只有处于"已发布"（PUBLISHED）状态的卡牌才能被添加到卡组中。处于草稿（DRAFT）状态的卡牌对构筑界面不可见，从根本上防止了将未发布卡牌纳入卡组的可能。

这一限制通过应用启动时的数据加载实现：`App.tsx` 在初始化时仅请求 PUBLISHED 状态的卡牌数据并写入 `gameStore.cardDataRegistry`，而卡牌编辑器（`CardEditor`）从这个 registry 读取可选卡牌，因此天然只展示已发布的卡牌。

## 3. 功能需求

### 3.1 卡组 CRUD

#### 创建卡组

用户可以从空白开始创建新卡组。创建时需要指定卡组名称，可选填写描述。创建后进入编辑模式，从卡牌库中逐张添加卡牌。

#### 编辑卡组

用户可以对已保存的卡组进行修改。编辑界面由左侧的卡牌库和右侧的当前卡组构成。**左右两侧均支持添加与删除卡牌**，具体交互如下：

- **左侧卡牌库**：点击未在卡组中的卡牌直接添加。若卡牌已在卡组中（数量 > 0），悬停时在卡牌上方显示 `+` / `−` 按钮覆盖层，分别用于增加和减少数量。右键仍为查看卡牌详情。
- **右侧卡组预览**：每个卡牌条目右侧显示 `+` / `−` 操作按钮，分别调用添加和移除逻辑。右键查看卡牌详情。

右侧的卡组列表按成员卡、Live 卡、能量卡三个分区展示，每个分区可独立折叠。每个分区头部显示当前数量与目标数量的对比（如 "32/48"），帮助玩家直观了解构筑进度。

#### 保存卡组

保存前系统会自动验证卡组是否符合构筑规则。验证通过后将卡组数据同步到云端数据库。若编辑的是已有卡组，采用先删后建的方式更新。保存完成后自动刷新卡组列表并返回列表页。

#### 删除卡组

删除前需要用户二次确认，防止误操作。删除操作同时更新用户的卡组计数。

### 3.2 卡牌筛选与搜索

卡牌编辑器提供多维度的筛选能力，帮助玩家快速定位目标卡牌：

- **卡牌类型切换**：在成员卡、Live 卡、能量卡三个标签页之间切换，一次只展示一种类型。
- **文字搜索**：按卡牌名称或编号进行模糊匹配。
- **高级筛选**（可展开/收起）：
  - **稀有度筛选**：根据卡牌编号后缀解析出的稀有度进行过滤（如 N、R、R+、SEC 等）。
  - **组合筛选**：按卡牌所属的偶像组合过滤（如 μ's、Aqours、Liella! 等）。
  - **小组筛选**（仅成员卡）：按组合内的小组过滤。选择组合后，小组选项自动联动，只显示该组合下的小组。
  - **费用区间筛选**（仅成员卡）：按卡牌出场费用过滤，默认范围 0-22。
  - **持有心颜色筛选**（仅成员卡）：按成员卡持有的心颜色过滤。7 种颜色可选（粉/红/黄/绿/蓝/紫/虹），选中某颜色后仅显示持有该颜色心的成员卡。
  - **判心颜色筛选**（仅 Live 卡）：按 Live 卡所需判定心颜色过滤。选中某颜色后仅显示需要该颜色心的 Live 卡。
  - **Live 分数区间筛选**（仅 Live 卡）：按 Live 卡成功时获得的分数过滤，默认范围 0-10。
- 能量卡类型下不显示高级筛选按钮，因为能量卡不具备上述属性。
- 所有筛选条件可一键清除重置。

筛选按卡牌类型 → 文字搜索 → 稀有度 → 组合 → 小组 → 费用区间 → 持有心颜色 → 判心颜色 → 分数区间的顺序链式应用，最终结果按卡牌编号排序。

### 3.3 实时验证与反馈

在编辑过程中，系统持续检查当前卡组状态：

- 各分区的当前数量实时显示，颜色反映是否达标。
- 添加卡牌时，若对应分区已满或同编号数量已达上限，操作会被静默阻止。
- 卡组列表页通过 `DeckValidityBadge` 展示每个卡组的完成度状态，用户无需打开卡组即可了解其是否合规。

### 3.4 YAML 导入/导出

系统支持通过 YAML 格式进行卡组的导入和导出，便于跨平台传输和备份。

- **导出**：将当前编辑中的卡组序列化为 YAML 文件并触发浏览器下载。YAML 结构与 `DeckConfig` 类型一致，包含 `player_name`、`main_deck`（分 `members` 和 `lives`）、`energy_deck` 三部分。
- **导入**：用户选择本地 `.yaml` 或 `.yml` 文件后，系统解析内容并使用 Zod schema（`DeckConfigSchema`）进行结构校验，校验通过后填充到编辑界面。

### 3.5 卡组图片下载（管理员功能）

管理员可以将某个卡组包含的所有卡牌图片打包为 ZIP 文件下载。此功能用于印刷、展示等运营场景。

下载流程：
- 遍历卡组的所有卡牌条目，根据 `count` 生成对应数量的图片副本。
- 图片 URL 优先从 MinIO 获取高清大图（600px WebP），若不可用则回退到本地静态文件。
- 卡牌编号到文件名的转换需要处理特殊字符（移除 `!`）和稀有度后缀（如 `-N`、`-R+`、`-SECE`）。
- 所有图片通过 JSZip 打包，通过 file-saver 触发浏览器下载。

### 3.6 卡组公开分享

用户可以将卡组设置为"公开"，公开后所有人（包括未登录的访客）都可以查看该卡组的内容。这一功能为未来的卡组分享社区和卡组参考功能预留了接口。

### 3.7 卡组选择（游戏入口）

在进入对局前，玩家通过卡组选择器（`DeckSelector`）从已保存的卡组中选择一副用于游戏。选择器统一展示云端卡组和本地卡组，显示每个卡组的统计信息和有效性状态，帮助玩家做出选择。

## 4. 数据安全与隔离需求

### 4.1 数据隔离

每个用户只能查看和操作自己的卡组，不能访问其他用户的私有卡组。这一隔离通过数据库层的 Row Level Security（RLS）策略强制执行，即使前端代码被篡改也无法绕过。

### 4.2 管理员权限

管理员角色可以查看、修改和删除所有用户的卡组，用于内容审核和运维支持。管理员权限通过独立的 RLS 策略实现，检查操作者在 `profiles` 表中的 `role` 字段。

### 4.3 自动维护

- **卡组计数**：用户档案中的 `deck_count` 字段通过数据库触发器自动维护，卡组新增时加 1、删除时减 1，确保计数与实际数据一致。
- **更新时间**：卡组的 `updated_at` 字段在每次修改时通过触发器自动更新，无需前端手动设置。

### 4.4 外键完整性

卡组通过 `user_id` 外键关联到 `profiles` 表（而非直接关联 `auth.users`），并设置 `ON DELETE CASCADE`，确保用户账号删除时其所有卡组数据自动清理。

## 5. 非功能性需求

### 5.1 离线支持

当 REST API 服务不可用时（如环境变量未配置），系统应能以离线模式运行。离线模式下云端卡组功能不可用，但本地卡组的编辑和验证功能仍然正常。

### 5.2 向后兼容

数据库中存量卡组的 `main_deck` 字段可能不包含 `card_type` 字段（旧格式）。系统在加载这些卡组时需要通过卡牌编号前缀推断类型，确保旧数据仍可正常使用。

### 5.3 性能

- 卡牌筛选使用 `useMemo` 进行计算缓存，避免在每次渲染时重复过滤全量卡牌数据。
- 云端操作设置 15 秒超时，防止网络异常时界面长时间挂起。
