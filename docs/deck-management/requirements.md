# 用户卡组管理系统 - 需求文档

> 版本: 1.0.0
> 创建日期: 2026-03-12
> 文档类型: 需求文档
> 适用范围: 卡组构筑、云端保存、DeckLog 导入、分享与游戏入口选组
> 当前状态: 卡组管理业务需求；当前实现边界见 [设计文档](./design.md) 与 [当前实现限制](../current-limitations.md)

## 1. 概述

### 1.1 系统目标

为 Love Live 卡牌游戏（Loveca）的玩家提供一套完整的卡组构筑与管理工具。玩家可以在线创建、编辑、保存自己的卡组，并在进入对局前选择已构筑好的卡组进行游戏。

### 1.2 设计动机

卡组构筑是卡牌游戏的核心体验之一。玩家需要在游戏规则约束下，从已发布的卡牌池中选择合适的卡牌组合，形成自己的战术方案。系统需要在保证规则合规的前提下，尽可能降低构筑门槛，提供直观的筛选、搜索和可视化反馈，让玩家专注于策略思考而非机械操作。

同时，卡组数据需要持久化到云端，使玩家可以跨设备访问自己的卡组，并为未来的卡组分享、社区功能打下基础。

### 1.3 用户角色

| 角色     | 说明                                             |
| -------- | ------------------------------------------------ |
| 普通用户 | 可以创建、编辑、删除自己的卡组，选择卡组进入游戏 |
| 管理员   | 可以在运维和审核场景下管理卡组数据与分享状态     |
| 访客     | 可以只读查看被公开分享的卡组                     |

## 2. 卡组构筑规则

以下规则来源于 Love Live 卡牌游戏官方规则第 6.1 节，是系统进行卡组验证的依据。

### 2.1 卡组结构

一副完整的卡组由两部分组成：

- **主卡组**：包含 60 张卡牌，其中成员卡（MEMBER）恰好 48 张，Live 卡恰好 12 张。主卡组不能包含能量卡。
- **能量卡组**：包含恰好 12 张能量卡（ENERGY）。能量卡组只能包含能量卡类型的卡牌。

### 2.2 同编号限制

主卡组中，同一**基础编号**的卡牌最多放入 4 张。这一限制确保卡组具有足够的多样性。

**基础编号**的定义：将卡牌编号（`cardCode`）去掉最后一个 `-` 及之后的稀有度后缀后得到的编号，系列前缀中的 `!` 会保留。例如 `LL-bp1-001-N` 和 `LL-bp1-001-R+` 的基础编号均为 `LL-bp1-001`，`PL!-sd1-050-SEC` 的基础编号为 `PL!-sd1-050`。

不同稀有度但基础编号相同的卡牌被视为"同一张卡"，它们在主卡组中的数量**合计**不得超过 4 张。在编辑界面中，这些同类卡牌上显示的数量徽章也会同步，均显示同基础编号卡在卡组中的总数。

### 2.3 卡牌来源限制

只有处于"已发布"（PUBLISHED）状态的卡牌才能被添加到卡组中。处于草稿（DRAFT）状态的卡牌对构筑界面不可见，从根本上防止了将未发布卡牌纳入卡组的可能。

这一限制应由数据加载、服务端权限和构筑界面共同保证，不能只依赖前端隐藏未发布卡牌。

## 3. 功能需求

### 3.1 卡组 CRUD

#### 创建卡组

用户可以从空白开始创建新卡组。创建时需要指定卡组名称，可选填写描述。创建后进入编辑模式，从卡牌库中逐张添加卡牌。

#### 编辑卡组

用户可以对已保存的卡组进行修改。编辑界面应同时展示可选卡牌池和当前卡组内容，支持添加、删除、查看详情和数量反馈。

卡组内容应按成员卡、Live 卡、能量卡三个分区展示，并显示当前数量与目标数量的对比，帮助玩家直观了解构筑进度。

- **成员卡分区排序**：按费用（`cost`）从低到高排序；若费用相同，则按卡牌编号（`cardCode`）字典序排序。
- **Live 卡分区排序**：按分数（`score`）从低到高排序；若分数相同，则按卡牌编号（`cardCode`）字典序排序。
- **能量卡分区排序**：维持现有顺序，不额外重排。

#### 保存卡组

保存前系统应自动验证卡组是否符合构筑规则，并记录验证结果。未完成卡组可以保存为草稿式数据，但必须在列表和游戏入口中清楚标识其不可用于正式对局。

#### 删除卡组

删除前需要用户二次确认，防止误操作。删除后应更新卡组列表、统计信息和相关入口状态。

### 3.2 卡牌筛选与搜索

卡牌编辑器提供多维度的筛选能力，帮助玩家快速定位目标卡牌：

- **卡牌类型切换**：在成员卡、Live 卡、能量卡三个标签页之间切换，一次只展示一种类型。
- **文字搜索**：按卡牌名称或编号进行模糊匹配。
- **高级筛选**（可展开/收起）：
  - **稀有度筛选**：根据卡牌编号后缀解析出的稀有度进行过滤（如 N、R、R+、SEC 等）。
  - **组合筛选**：按卡牌所属的偶像组合过滤（如 μ's、Aqours、Liella! 等）。
  - **收录商品筛选**：按卡牌 `product` 字段过滤，匹配时会忽略空格差异。
  - **小组筛选**（仅成员卡）：按组合内的小组过滤。选择组合后，小组选项自动联动，只显示该组合下的小组。
  - **费用区间筛选**（仅成员卡）：按卡牌出场费用过滤，默认范围 0-22。
  - **心颜色筛选**（成员卡 / Live 卡）：成员卡按持有心颜色过滤；Live 卡按需求心颜色过滤。7 种颜色可选（粉/红/黄/绿/蓝/紫/虹）。
  - **BLADE 心效果筛选**（成员卡 / Live 卡）：按 `bladeHearts` 中的加分、抽卡或指定颜色 HEART 效果过滤。
  - **Live 分数区间筛选**（仅 Live 卡）：按 Live 卡成功时获得的分数过滤，默认范围 0-10。
- 能量卡类型下仅保留稀有度、组合、收录商品等适用筛选，不显示小组、费用、心颜色、BLADE 心效果或分数筛选。
- 所有筛选条件可一键清除重置。

筛选按卡牌类型 → 文字搜索 → 稀有度 → 组合 → 收录商品 → 小组 → 费用区间 → 心颜色 → BLADE 心效果 → 分数区间的顺序链式应用，最终结果按卡牌编号排序。

### 3.3 实时验证与反馈

在编辑过程中，系统持续检查当前卡组状态：

- 各分区的当前数量实时显示，颜色反映是否达标。
- 添加卡牌时，若对应分区已满或同编号数量已达上限，操作会被静默阻止。
- 卡组列表页通过内联状态徽章展示“完整 / 未完成”，用户无需打开卡组即可了解其是否合规。

### 3.4 YAML 导入/导出

系统支持通过 YAML 格式进行卡组的导入和导出，便于跨平台传输和备份。

- **导出**：将当前编辑中的卡组序列化为项目支持的 YAML 结构，便于备份和分享。
- **导入**：用户选择本地 `.yaml` 或 `.yml` 文件后，系统解析并校验结构，遇到缺失字段或未知卡牌时应给出可理解的错误或警告。

### 3.5 卡组图片下载

若后续需要印刷、展示等运营场景，应作为独立需求设计，包括图片来源、文件命名、权限边界、打包方式和失败重试策略。

### 3.6 DeckLog 卡组导入

用户可以选择日版 DeckLog 或国际版 DeckLog，并输入对应网站上的卡组 ID 或完整 URL，系统获取公开卡组数据并自动填充到编辑界面。

**输入格式**：

- 纯卡组 ID：如 `2D6XL`
- 日版完整 URL：如 `https://decklog.bushiroad.com/view/2D6XL`
- 国际版完整 URL：如 `https://decklog-en.bushiroad.com/ja/view/60G2Q`

**导入要求**：

- 用户选择的来源应决定请求的 DeckLog API；完整 URL 与选择来源不一致时应提示错误，避免同一短码跨站撞号导入错卡组。
- 系统应标准化外部卡牌编号，减少全角符号、稀有度写法等格式差异带来的导入失败。
- 系统应根据卡牌类型自动归类到对应分区。
- 未在本地卡牌池中找到的卡牌应以警告形式展示，不应导致整副卡组直接丢失。
- 匹配成功后应进入可编辑状态，并尽量保留外部卡组名称。

**错误处理**：

- 无效输入格式应提示用户检查卡组 ID 或 URL。
- 外部服务不可用、超时或返回异常时，应提示导入失败原因。
- 卡组数据为空、无法识别卡牌编号或没有匹配卡牌时，应给出明确反馈。

**前提条件**：此功能依赖在线服务，离线模式下不可用。

### 3.7 卡组公开分享

用户可以将卡组设置为"公开"，公开后所有人（包括未登录的访客）都可以通过分享链接查看该卡组内容；登录用户可将公开分享卡组复制到自己的账号下继续编辑。

### 3.8 卡组选择（游戏入口）

在进入对局前，玩家需要从已保存的卡组中选择一副用于游戏。选择入口应显示每个卡组的统计信息和有效性状态，避免玩家误选不合法卡组。

## 4. 数据安全与隔离需求

### 4.1 数据隔离

普通用户只能查看和操作自己的卡组，不能访问其他用户的私有卡组。公开或开启分享的卡组只开放只读查看与复制入口。

### 4.2 管理员权限

管理员角色可以在内容审核和运维支持场景下查看、修改或删除用户卡组。当前实现支持管理员在已知卡组 ID 的情况下进行读取、修改、删除、开启分享和关闭分享；面向管理员的全量卡组列表、搜索和审核工作台不属于当前已落地能力。管理员能力必须由服务端权限控制保证。

### 4.3 自动维护

卡组数量、更新时间、分享状态等派生信息应由系统统一维护。数据库初始化与 schema 边界见 [当前实现限制](../current-limitations.md#数据库初始化与-schema)。

### 4.4 外键完整性

卡组数据必须与用户账号建立可靠关联，确保权限判断、账号删除和数据清理具备一致性。

## 5. 非功能性需求

### 5.1 离线支持

离线模式下应明确区分可用能力和不可用能力。云端卡组、DeckLog 导入、分享和卡牌管理等依赖在线服务的能力不可用时，界面应提供清晰提示。

### 5.2 向后兼容

系统应尽量兼容历史卡组数据，保证旧格式卡组在卡组管理、分享页、卡组选择和游戏入口中表现一致。具体转换策略由设计文档维护。

### 5.3 性能

- 卡牌筛选和统计应保持响应迅速，避免卡牌池增长后明显卡顿。
- 云端操作应具备超时、失败提示和重试边界，避免网络异常时界面长时间挂起。
