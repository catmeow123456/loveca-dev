# Loveca 联机模式准备文档

> 文档类型：联机设计总览 / 收敛文档
> 适用范围：基于当前 `game_system_design.md` 与现有对局引擎实现，收敛 Loveca 从本地对局推进到真实双人联机前必须统一的设计边界、术语与改造顺序。
> 最后更新：2026-04-02

---

## 1. 文档目的

这份文档用于做三件事：

- 说明当前实现为什么还不能直接上线联机。
- 固定联机首版已经达成一致的边界。
- 作为其它联机草案文档的总索引，避免多份文档各自发散。

本文档不定义最终协议字段细节；协议细节分别放在：

- `docs/online-mode-visibility-matrix.md`
- `docs/online-mode-event-classification-draft.md`
- `docs/online-mode-public-event-minimum-schema.md`

## 2. 从当前系统设计得到的事实

结合 [game_system_design.md](/root/loveca/game_system_design.md) 与现有代码，当前引擎的真实形态是：

- `GameSession` 在单进程内维护权威状态，但仍是本地内存会话。
- 前端 `gameStore` 直接持有 `GameSession`，并不是远程客户端。
- `getStateForPlayer()` 已收口为兼容旧 UI 的脱敏兼容态，`getPlayerViewState()` 已提供按 seat 投影的安全视图。
- 当前“信任玩家 + 规则层事后修正”的模型主要服务于本地调试、对墙打和规则验证。
- 当前服务端只有 REST API，没有房间、实时同步、断线恢复、事件流持久化等联机基础设施。

这意味着当前系统虽然已经有“权威状态”、基础公共事件时间轴和“按座位可见性投影”，但还没有完整的联机协议、私密事件/审计输出和可恢复的联机基础设施。

---

## 3. 联机首版的总定位

联机首版不应被定义为“全自动数字裁判”，更准确的定位是：

- 服务端维护权威会话。
- 服务端严格维护可见性、时机、顺序、随机结果和座位权限。
- 玩家仍可在较高自由度下完成很多实体卡式操作和声明。
- 服务端记录的是“桌面上实际发生了哪些过程”，而不是强行把所有行为先翻译成高层规则语义。

因此，联机首版的主链路应优先收敛为：

`Command -> Validate -> Apply -> Emit Public/Private Events -> Project View -> Persist`

这里的重点不是先接入 WebSocket，而是先把引擎输出形状改造成：

- 可验证
- 可投影
- 可回放
- 可恢复

补充约束：

- 联机首版的世界本体仍建立在共享牌桌对象模型上，但不再采用“所有隐藏区对象都对双方持续可追踪”的激进模型。
- 首版采用“局部强跟踪，进入混淆态后断跟踪”。
- 各 seat 的差别不仅体现在看到 `BACK` 还是 `FRONT`，也体现在某对象是否仍对该 seat 保持可追踪身份。

## 4. 已锁定的首版边界

### 4.1 服务端必须硬性维护的边界

- 非法信息泄露必须拒绝，不能依赖事后追责。
- 阶段、子阶段、窗口、可操作座位必须严格校验。
- 正式阶段边界必须固定：`PERFORMANCE_PHASE` 承载单侧表演的完整流程，包含判定成功后的效果处理窗口；`LIVE_RESULT_PHASE` 从结果确认开始，不承载“先攻成功效果 / 后攻成功效果”这类正式子阶段。
- 跨公开/非公开边界的移动必须校验来源、去向和公开方式。
- 有顺序要求的公开区与私密区内部顺序必须由服务端维护。
- 洗牌、抽牌、检视顶牌、更新洗回等随机结果必须以服务端为唯一真值来源。
- 已进入公共世界的已提交事实不允许撤销；只能用新的后续动作改变状态。
- 服务端不对非法命令做静默纠错；不合法就拒绝。

### 4.2 首版允许保留“信任玩家”的范围

- 复杂费用链、替代支付、减费、置换效果等，不要求首版完整自动裁判。
- 能量支付、能量恢复、减费、替代支付等资源结算，首版允许由玩家通过显式桌面操作自行完成；服务端不在 `PLAY_MEMBER_TO_SLOT` 等命令里隐式代扣、返还或自动挑选能量牌。
- 复杂卡文解释、复杂自动能力连锁，不要求首版完整自动裁判。
- 判定、分数等声明型步骤，首版以“记录声明 + 少量客观校验”为主。
- 争议处理首版目标是“完整审计回放”，不是自动判责。

### 4.3 当前已锁定的提交流程

- 私密且未公开的步骤，优先采用“本地草稿 -> 一次确认提交”。
- 公开可见的桌面动作，优先采用“直接正式提交”。
- 客户端允许本地拖拽预览，但预览不是事实。
- 服务端事件流只记录已提交事实，不记录可任意编辑的本地草稿。
- “成功效果窗口”在联机文档中属于窗口语义，不等于新的结果阶段子阶段；若当前实现仍暴露 `RESULT_FIRST_SUCCESS_EFFECTS` / `RESULT_SECOND_SUCCESS_EFFECTS`，应视为旧状态机残留，而不是联机规范。
- 能量区与成员下方能量的卡牌位置、张数和朝向仍由服务端维护；但“为了支付费用而横置了哪些能量”“因卡牌效果恢复了哪些能量”首版作为玩家显式操作处理，不由服务端从高层命令隐式推导。
- 检视区相关流程按单独规则处理：开始检视与检视区内原子操作均为正式事实，不采用“检视草稿 -> 最终统一提交”的模型。
- 若某个公开对象从公开区进入手牌、主卡组、能量卡组等私有或隐藏混淆区，则“进入目标区”这一步的公共移动事件仍沿用原 `publicObjectId`；对象是在该事件之后，才对非持有者不再持续投影。
- 对象进入手牌后，对非持有者不再持续投影该对象；但若该牌之后重新进入公开链路，首版可继续沿用同一个 `publicObjectId`。
- 对象进入主卡组、能量卡组等隐藏混淆容器后，对非持有者默认不再持续投影该对象；若之后重新进入公开链路，首版可继续沿用同一个 `publicObjectId`；若伴随洗牌，则仍按隐藏区处理。
- 解决区中的应援牌采用“两步公开”模型：先以 `BACK` 进入解决区，再翻开；解决区内仅为阅读和排布的拖拽不是正式事实。

### 4.4 当前已锁定的系统自动动作边界

以下处理应由服务端自动执行，并作为显式系统事件进入日志与回放：

- 更新。
- 胜利处理。
- 重复成员处理。
- 不正卡牌处理。
- 不正解决领域处理。
- 演出阶段开始时的 Live 统一翻开，以及翻开后的客观后处理。
- 表演结束后进入结果阶段时，分数确认、胜者判定、结果动画、成功 Live 结算之间的客观推进。
- Live 胜败判定后的客观清理和先后攻调整。
- 成功效果发动不属于结果阶段系统自动动作；它属于 `PERFORMANCE_PHASE` 内的正式流程或窗口。
- 费用支付与能量恢复不属于系统自动动作；若要体现在联机状态中，应由玩家发出显式命令或桌面操作。

这些系统动作必须满足三个条件：

- 规则文本明确要求自动发生。
- 触发条件客观。
- 不是为了替玩家偷偷修错。

## 5. 统一术语

为避免几份草案混用抽象层级，联机文档统一采用以下术语：

- `Command`
  客户端发给服务端的玩家命令。
- `System Event`
  服务端主动触发的规则处理事件。
- `Public Event`
  可安全发送给双方的公共事件。
- `Private Event`
  仅发送给某一座位的私密事件。
- `Sealed Audit`
  仅服务端保存的密封审计记录。
- `Authoritative State`
  服务端完整权威状态。
- `Player View`
  面向某一座位投影后的安全状态。
- `publicObjectId`
  共享牌桌对象 ID。它不是权威卡牌实例 ID，而是每张牌在一局对战内的稳定对象标识；对象处于隐藏区时，不同观察者未必持续收到它的投影。
- `surface`
  某座位看到该对象的牌面：`NONE | BACK | FRONT`。

统一约束：

- 不再混用 `publicOccupantId`、`publicCardId`。
- 共享牌桌世界中一律用 `publicObjectId` 表示该牌在本局中的共享对象标识。
- `publicObjectId` 在服务端和公共链路中都可保持稳定；对象处于隐藏区时，对非持有者可以暂时不投影该对象，但对象之后重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。
- 首版采用“局部强跟踪，进入混淆态后断跟踪”：
  - 检视区、解决区、Live 放置区和各类公开区中的连续桌面过程，允许对象级跟踪。
  - 进入手牌后，对非持有者不再持续投影该对象。
  - 进入主卡组、能量卡组等隐藏混淆容器后，对非持有者默认不再持续投影该对象。
  - 洗牌、洗回、进入已混淆牌堆后，对非持有者同样不持续投影该对象。
- 当对象已经公开到可识别卡面时，再额外附带公开牌面信息。

## 6. 文档之间的职责分工

### 6.1 `online-mode-visibility-matrix.md`

回答：

- 一个容器对谁公开张数、占位和顺序。
- 一张牌对不同座位显示 `NONE / BACK / FRONT` 的规则。
- 不同区域和动作应如何投影。
- 在共享牌桌对象模型下，哪些区域中的对象对双方通常应显示为 `BACK` 而不是 `NONE`。

### 6.2 `online-mode-event-classification-draft.md`

回答：

- 服务端内部需要区分哪些事件类别。
- 哪些是公共事件，哪些是私密事件，哪些只进入密封审计。
- 哪些系统自动处理必须作为显式事件存在。

### 6.3 `online-mode-public-event-minimum-schema.md`

回答：

- 首版最少需要哪些公共事件，才能驱动公共日志、动画和流程提示。
- 这些公共事件需要哪些最小字段。

## 7. 对当前架构的具体改造要求

### 7.1 会话层

需要从本地 `GameSession` 扩展为支持联机的对局实体，至少新增：

- `MatchState`
- `SeatState`
- `ConnectionState`
- `TimerState`
- `CommandContext`

### 7.2 状态层

需要明确拆成三层：

- `AuthoritativeGameState`
- `PlayerViewState`
- `SpectatorViewState`（若未来支持观战）

当前 `getStateForPlayer()` 直接返回权威状态副本的做法不能继续用于联机。

### 7.3 规则执行入口

联机前必须收口为单一主链路，不再允许“多个大致等价入口”并存。

目标应是：

- 命令校验只有一套入口。
- 阶段推进只有一套入口。
- 自动规则处理只有一套入口。
- 事件产出和审计落库只有一套入口。

### 7.4 动作层

当前的“万能拖拽动作”不能原样进入联机正式协议。

联机需要把动作拆成带可见性和审计语义的命令，例如：

- 公开移动命令
- 公开区进入私有区专用命令
- 检视区开启命令
- 检视区内即时移动命令
- 检视结束命令
- 声明/确认命令
- 过优先权命令

补充说明：

- “语义明确的命令”指服务端能够判断玩家当前在声明执行哪一类规则动作，而不是接收一个没有规则语义的万能拖拽事实。
- 首版服务端重点校验流程、时机、对象所在区域、当前操作者和窗口边界，不要求完整裁定复杂卡文条件是否成立。
- 检视类流程不再按“私密候选集统一确认”建模。
- `INSPECTION_ZONE` 是正式区域；对象真实进入该区域，区内原子操作即时生效，对手可感知变化过程。

## 8. 持久化与联机基础设施

联机至少需要新增以下持久化对象或等价存储：

- `matches`
- `match_seats`
- `match_connections`
- `match_commands`
- `match_events`
- `match_snapshots`
- `match_results`

最小恢复能力要求：

- 命令具备幂等键。
- 事件具备严格递增序号。
- 快照可用于断线恢复和服务重启恢复。
- 最新快照之后的事件可回放重建状态。

WebSocket 只是传输层；在它之外还必须定义：

- ack 语义
- 版本号校验
- 补快照机制
- 心跳与断线恢复

## 9. 当前文档中已发现的主要问题

本轮整理前，几份草案主要有以下问题：

- 把“世界本体”“内部事件分类”“公共事件 schema”写在同一层，阅读时很难分辨哪些是原则、哪些是最终输出。
- 字段名不统一，`publicOccupantId`、`publicCardId`、`CardMoved`、`CardMovedPublic` 同时存在。
- 有些文档在讲“完整事件分类”，有些文档其实只在讲“公共事件最小集合”，但标题和正文没有清晰区分。
- 已锁定边界和待讨论问题混在一起，导致读者无法判断哪些可以直接实现。
- 个别区域规则仍有歧义，例如能量卡组顺序策略写成 `OWNER_ORDERED_HIDDEN 或 NONE`。

本次整理目标就是把这些歧义收拢掉。

## 10. 仍待确认的问题

以下问题目前不建议继续散落在正文里，应作为明确的待决事项管理：

1. 能量卡组在联机首版中是否视为有序私密容器。
2. 是否需要首版观战视图；若需要，观战视角的私密信息上限是什么。
3. 超时策略是否允许系统自动过阶段、自动不发动、自动不换牌。
4. 是否要支持好友房、邀请码房、观战和裁判介入模式。

这些问题不影响先实现联机基础骨架，但需要在产品和规则层面尽快定稿。

---

## 11. 建议的实施顺序

1. 先统一术语和事件/视图分层。
2. 再完成权威状态到玩家视图的脱敏投影。
3. 再把动作输出改造成 `public events + private events + audit`。
4. 再补持久化、快照、重放和幂等。
5. 最后再接入 WebSocket、房间、重连和产品层能力。

如果顺序反过来，先接实时层，再回头补事件和投影，返工会很大。
