# Loveca 联机模式事件分类草案

> 文档类型：联机设计草案
> 适用范围：整理联机首版中服务端需要区分的事件层级、事件类别和它们与公共/私密/审计输出的关系。
> 最后更新：2026-04-02

---

## 1. 文档目标

这份文档不直接定义最终协议字段，而是先回答两个更基础的问题：

1. 服务端内部到底要区分哪些事件类别。
2. 这些事件类别最终分别如何落到 `Public Event`、`Private Event` 和 `Sealed Audit`。

本轮整理后，本文档只讨论“事件分类”，不再和“公共事件最小 schema”混写。

---

## 2. 抽象层级

为了避免几份草案继续混层，联机事件统一按三层理解：

### 2.1 权威事实层

回答：

- 某张牌真实在哪里。
- 当前阶段、子阶段、窗口是什么。
- 哪个座位可操作。
- 随机结果是什么。

这是服务端权威状态本体，不直接等于“事件”。

### 2.2 按座位感知层

回答：

- 某座位能否看到一个单卡对象。
- 该座位看到的是 `NONE`、`BACK` 还是 `FRONT`。
- 该座位是否能持续追踪这个共享牌桌对象。

这层是投影规则，不直接等于“公共日志”。

补充约束：

- 首版采用“局部强跟踪，进入混淆态后断跟踪”。
- 检视区、解决区、Live 放置区和各类公开区中的连续桌面对象，通常可对双方持续追踪。
- 手牌、主卡组、能量卡组等隐藏混淆区，对非持有者通常只保留摘要，不持续投影该对象。
- `publicObjectId` 在首版中可保持全局稳定；对象隐藏期间对非持有者可以不投影，但对象重新进入公开链路时，首版可继续沿用同一个 `publicObjectId`。

### 2.3 事件投影层

回答：

- 这次状态变化应该对双方广播什么。
- 应该只对一侧发送什么。
- 应该只保存在服务端什么。

因此事件的地位应理解为：

- 事件不是世界本体。
- 事件是“权威事实变化 + 按座位感知变化”的差分投影。

---

## 3. 分类原则

### 3.1 优先记录桌面过程

联机首版应优先记录：

- 某张牌被移动
- 某个共享牌桌对象出现、移动、重排或离场
- 某张牌被翻开或展示
- 某位玩家检视了多少张牌
- 某位玩家提交了某个声明或确认

而不是优先记录：

- 发动了哪一条复杂效果
- 某段复杂效果是否合法成立
- 复杂费用链是否被程序完整裁定

补充说明：

- 事件优先记录“玩家声明自己做了什么桌面动作”以及“桌面上客观发生了什么”。
- 首版不要求事件层证明复杂卡文条件已经被程序完整裁定。

### 3.2 已进入公共世界的事实不可回滚

联机首版不设计“公开事件撤销”语义。

如果一个动作已经让对手感知到：

- 共享牌桌对象变化
- 公共张数变化
- 公共日志变化
- 随机结果或公开顺序变化

那么后续只能通过新的过程事件改变状态，不能抹除历史。

### 3.3 系统自动处理必须显式化

规则文本明确要求自动发生的客观处理，不应无声合并进状态，而应产出显式系统事件。

---

## 4. 事件输出模型

任一已提交动作或系统处理，最终都应映射到三层输出：

- `publicEvents`
- `privateEventsBySeat`
- `sealedAudit`

注意：

- 不是每个内部事件类别都一定会产生公共事件。
- 也不是每个公共事件都需要独立保留同名私密事件。

---

## 5. 一级分类

联机首版建议将事件先分成六大类：

1. 阶段与窗口事件
2. 共享牌桌对象移动事件
3. 卡面变化与公开展示事件
4. 检视区与检视流程事件
5. 玩家声明与提交事件
6. 系统规则处理事件

这六类已经足够覆盖当前首版目标。

---

## 6. 各类事件说明

### 6.1 阶段与窗口事件

用途：

- 锚定时间轴。
- 通知当前阶段、子阶段和窗口状态。
- 表达系统目前在等谁。

典型候选：

- `PhaseStarted`
- `SubPhaseStarted`
- `WindowStatusChanged`
- `MatchEnded`

说明：

- `WindowOpened/WindowClosed` 可以视作更细粒度实现。
- 对外公共 schema 当前优先收口为 `WindowStatusChanged`，避免事件名膨胀。
- 检视流程当前已与其它文档统一，使用独立窗口类型表达，而不是塞进普通窗口的模糊上下文里。

### 6.2 共享牌桌对象移动事件

用途：

- 记录共享牌桌世界里可观察到的对象移动、出现、离场与重排。

典型候选：

- `CardMovedPublic`

设计结论：

- 对外公共 schema 优先只保留一个主事件名。
- 在新的共享牌桌模型下，大多数单卡移动都应真实体现在公共事件中，而不是退化为纯摘要。

### 6.3 卡面变化与公开展示事件

用途：

- 对象不移动，但可见面发生变化。
- 某张牌被公开给双方。

典型候选：

- `CardRevealed`
- `CardRevealedAndMoved`
- `CardSurfaceChanged`（更偏内部分类）

设计结论：

- 内部可以保留更通用的“surface change”概念。
- 对外公共 schema 优先收口为“公开展示”和“公开展示并移动”两类事件。
- 首版默认优先拆步表达；只有当“公开”和“移动”确实不需要区分两个可观察阶段时，才投影为 `CardRevealedAndMoved`。

### 6.4 检视区与检视流程事件

用途：

- 记录某位玩家开启检视、对象进入检视区、检视区内发生的移动与排序、以及检视流程结束。

典型候选：

- `CardsInspectedSummary`（公共摘要）
- 检视区对象进入/离开事件
- 检视区对象重排事件
- 私密 seat-specific 检视补充信息事件

设计结论：

- 检视开始与区内原子操作都是正式过程，不再默认为“草稿 -> 一次确认”。
- `CardsInspectedSummary` 可以保留，但更适合作为日志友好型补充事件；它不是检视流程成立的必要条件，也不是首版必发事件。
- 检视区中的对象移动和重排，应优先作为共享牌桌过程事件存在。
- 候选全集、额外只读细节和规则上下文仍可进入私密事件和密封审计。
- 与检视相关的正式命令强调“语义明确”，即服务端知道玩家当前在声明执行哪一类规则动作；但首版不要求服务端完整裁定复杂效果条件是否成立。
- 检视区在命令层、事件层和公共位置引用里统一按 `INSPECTION_ZONE + ownerSeat` 理解；若前端视图需要 `FIRST_INSPECTION_ZONE` / `SECOND_INSPECTION_ZONE`，那属于视图展开，不属于公共事件命名。
- 成员下方能量在事件层应优先作为“成员槽位上的 overlay 位置变化”理解，而不是新的顶层区域移动。

### 6.5 玩家声明与提交事件

用途：

- 记录玩家提交了一个公开声明或流程确认。

典型候选：

- `PlayerDeclared`

适用范围：

- Mulligan 提交
- 分数确认
- 判定确认
- 阶段/子阶段完成确认

### 6.6 系统规则处理事件

用途：

- 记录服务端主动触发的、规则文本明确要求自动发生的处理。

当前建议的语义类别：

- `DeckRefreshed`
- `VictoryResolved`
- `IllegalBoardStateResolved`
- `PerformanceRevealResolved`
- `LiveResultCleanupResolved`

说明：

- 这些名字目前是语义类别，不是最终协议字段名。
- 关键要求是“显式存在”，而不是是否沿用这些命名。

## 7. 当前已锁定的提交语义

联机首版当前建议按以下方式理解提交：

- `Mulligan`
  本地草稿，确认后一次正式提交。
- 检视区流程
  开始检视与区内原子操作均为正式提交；`FinishInspection` 负责结束流程，而不是统一应用草稿。
  这些提交强调流程、时机、对象所在区域和当前操作者的客观边界；不要求服务端完整证明复杂卡文条件已经成立。
- 从手牌盖放 Live
  每放一张就是一个正式过程事件。
- 已盖放的 Live 拿回手牌
  是新的正式过程事件，不是撤销。
- 从手牌 play 成员到公开区
  直接正式提交。
- 应援翻牌
  每次翻下一张都是独立正式命令，由服务端决定真实顶牌与事件投影。
- 阶段结束 / 子阶段确认 / 分数确认
  一次正式提交，不可撤销。

## 8. 与公共事件 schema 的关系

为了减少混乱，这里明确：

- 本文档讨论的是“内部应该区分哪些事件类别”。
- [docs/online-mode-public-event-minimum-schema.md](/root/loveca/docs/online-mode-public-event-minimum-schema.md) 只讨论“最终要下发给双方的最小公共事件集合”。

也就是说：

- “事件分类”范围比“公共事件 schema”更大。
- 私密事件和系统事件在本文档里有位置，但不一定全部进入公共 schema。

---

## 9. 与当前代码的主要差距

当前引擎输出更接近：

- 输入一个动作
- 返回一个新的 `GameState`
- 附带少量内部 `triggeredEvents`

而联机需要的输出更接近：

```ts
type EngineResult = {
  authorityState: GameState;
  publicEvents: PublicEvent[];
  privateEventsBySeat: Partial<Record<Seat, unknown[]>>;
  sealedAuditRecords: unknown[];
};
```

也就是说，目前最大的差距不是“还没有 WebSocket”，而是：

- 引擎还没有稳定的事件投影层。
- 引擎还没有把公共/私密/审计输出显式分开。

---

## 10. 当前最需要避免的混乱

本轮整理后，以下混乱应视为已废弃：

- 用一个总表同时描述内部事件分类和公共事件 schema。
- 混用 `CardMoved`、`CardMovedPublic` 来指同一层输出。
- 把 `publicObjectId` 和权威卡牌实例 ID 混为一谈。
- 把“对象在隐藏区期间不再持续投影”误写成“对象重新公开时必须生成新的 `publicObjectId`”。
- 把“系统自动规则处理”写成静默状态修正。

---

## 11. 仍待确认的问题

1. 系统规则处理事件是否需要在公共日志中统一显示“由系统执行”。
2. 私密检视流程里，哪些效果需要对对手逐步公开过程，哪些只公开摘要。
3. 某些复合处理是否要拆成多条公共事件，还是允许“公开并移动”这类复合事件存在。

这些问题不影响当前分类框架成立，但会影响最终协议颗粒度。
