# Loveca 项目 UI 设计现状分析

## 1. 文档目的

本文基于当前仓库前端实现，对各模块的 UI 设计现状进行梳理，说明：

- 当前有哪些主要页面与模块
- 每个模块采用了怎样的视觉风格、布局结构与交互方式
- 当前 UI 在一致性、可维护性与产品完成度上的主要问题

本文描述的是“当前代码中的真实现状”，不等同于目标方案。

---

## 2. 前端 UI 模块总览

当前前端主要位于 `client/src`，页面入口集中在 `client/src/App.tsx`。从页面级别看，UI 可以分为以下模块：

| 模块 | 入口/核心组件 | 主要职责 |
| --- | --- | --- |
| 认证模块 | `components/auth/*` | 登录、注册、找回密码、重置密码 |
| 首页模块 | `components/pages/HomePage.tsx` | 登录后的导航首页 |
| 游戏准备模块 | `components/pages/GameSetupPage.tsx` | 选择模式、选择卡组、确认开局 |
| 卡组管理模块 | `components/deck/DeckManager.tsx` | 卡组列表、导入导出、创建编辑 |
| 卡组编辑模块 | `components/deck-editor/*` | 卡组构建、筛选、浏览、详情、侧边栏 |
| 对战模块 | `components/game/*` | 主游戏棋盘、阶段流程、拖拽交互、日志、判定 |
| 卡牌管理模块 | `components/admin/CardAdminPage.tsx` | 后台卡牌数据检索、分页、编辑 |
| 通用组件模块 | `components/common/*` | 卡组选择器、统计等可复用视图 |

从实现方式看，当前 UI 技术栈主要是：

- React 19
- Tailwind CSS v4
- Framer Motion
- Zustand
- Lucide React

整体上已经形成“页面组件 + 业务组件 + 状态仓库”的结构，但设计系统仍处于半统一状态。

---

## 3. 全局 UI 现状

### 3.1 样式组织

- 全局样式文件为 `client/src/index.css`
- 以 Tailwind 原子类为主，辅以少量全局 CSS 变量与传统 class
- 存在一部分早期全局 class，例如 `.card`、`.zone`、`.card-cost` 等
- 页面级配色与布局仍主要写在组件内，缺少统一的语义化设计 token

### 3.2 颜色与主题现状

当前实际存在三套明显不同的视觉语言：

1. 认证页：浅色暖橙/奶油色
2. 首页、游戏准备、卡组管理、卡牌管理：深棕橙色玻璃化面板
3. 对战界面：深灰蓝/石板色游戏桌 UI

结果是模块内局部统一，但跨模块切换时割裂感明显。

### 3.3 图标与视觉元素现状

- 首页、游戏准备、卡组选择器等仍大量使用 emoji 作为主图标
- 卡组管理、卡牌管理开始使用 Lucide 图标
- 游戏区域混合使用文本、emoji、Lucide 和真实卡牌图片
- 图标体系尚未统一，视觉成熟度不稳定

### 3.4 动效现状

- 首页、认证页、游戏准备页动效较多，主要是淡入、位移、hover scale
- 对战界面含更复杂的状态动效，如横幅、拖拽覆盖、判定与结果动画
- 动效使用广泛，但不同模块的节奏、风格和目的不完全一致

### 3.5 设计系统完成度

当前更接近“共享少量配色习惯的页面集合”，还不是完整设计系统。主要体现为：

- 有一些重复使用的颜色与玻璃面板写法
- 但缺少统一组件规范、状态规范、图标规范和排版规范
- 同一产品内存在多个审美阶段并存

## 4. 各模块 UI 现状

### 4.1 认证模块

核心组件：

- `components/auth/AuthLayout.tsx`
- `components/auth/LoginPage.tsx`
- `components/auth/RegisterPage.tsx`
- `components/auth/ForgotPasswordPage.tsx`
- `components/auth/ResetPasswordPage.tsx`

现状特征：

- 使用浅色暖橙主题，与站内其余深色页面明显不同
- 背景为大面积柔光渐变和装饰性光斑，情绪感较强
- 中间是单卡片式表单容器，带 blur 与半透明白底
- 表单交互完整，错误提示、重发验证、离线模式入口等信息表达清晰

优点：

- 登录流程视觉完整度较高
- 表单层级清楚，输入、提示、操作按钮关系明确
- 相比业务页，更接近完整产品页

主要问题：

- 与登录后的深色主界面风格断层明显
- 仍使用图片 logo + emoji 装饰，品牌表达偏杂糅
- 视觉更偏“欢迎页/活动页”，与游戏产品主体语言尚未打通

### 4.2 首页模块

核心组件：

- `components/pages/HomePage.tsx`

现状特征：

- 使用深棕橙渐变背景和卡片式入口
- 页面结构明确：顶部导航、欢迎区、功能入口卡片、状态条、底部 footer
- 入口卡片采用大 emoji + 标题 + 描述 + 标签胶囊的方式
- 使用 Framer Motion 做首屏进入和 hover 放大

优点：

- 功能入口非常直观，信息架构简单
- 首页作为“路由导航中心”的作用明确
- 深棕橙主色在业务页中有一定连续性

主要问题：

- 核心入口依赖 emoji，成品感不足
- 卡片信息密度偏低，视觉上更像概念首页而非成熟产品控制台
- footer、状态条、卡片、header 各自风格接近但未完全统一

### 4.3 游戏准备模块

核心组件：

- `components/pages/GameSetupPage.tsx`
- `components/common/DeckSelector.tsx`

现状特征：

- 延续首页的深棕橙风格
- 采用 step-by-step 向导式流程：模式选择、卡组选择、确认开始
- 步骤指示器、模式卡片、摘要卡片都有明确视觉反馈
- `DeckSelector` 作为独立复用组件，承担列表选择与状态反馈

优点：

- 流程型页面完成度较高，步骤逻辑和按钮状态清晰
- 调试模式与对墙打模式区分明确
- 卡组摘要信息表达有效，能支持开局前确认

主要问题：

- 仍大量依赖 emoji 作为功能识别
- 页面风格和首页接近，但组件细节没有抽象成统一规范
- `DeckSelector` 使用“可爱滚动条/cute”命名与深棕主题语言不一致，透露历史遗留风格混用

### 4.4 卡组管理模块

核心组件：

- `components/deck/DeckManager.tsx`

现状特征：

- 同样使用深棕橙系背景与玻璃 header
- 分为卡组列表视图和卡组编辑视图
- 列表页包含创建、YAML 导入、DeckLog 导入、推荐卡组、删除下载等操作
- 这部分已经比首页更偏“工具型后台”

优点：

- 信息密度比首页更合理，操作链路完整
- Lucide 图标的引入提升了专业感
- 在线/离线/本地状态、用户名等辅助信息表达较清楚

主要问题：

- 同一页面里同时存在圆角胶囊按钮、渐变按钮、卡片按钮等多种按钮语言
- 功能很多，但视觉分组仍偏“堆叠”，缺少更稳固的信息区块层级
- 与卡组编辑器之间虽能衔接，但整体上还不是一体化工作台体验

### 4.5 卡组编辑模块

核心组件：

- `components/deck-editor/CardEditor.tsx`
- `FilterPanel.tsx`
- `SearchBar.tsx`
- `CardBrowserGrid.tsx`
- `DeckSidebar.tsx`
- `CardDetailDrawer.tsx`

现状特征：

- 是当前业务复杂度最高的 UI 之一
- 采用“筛选栏 + 卡牌浏览区 + 侧边栏卡组结构”的编辑器布局
- 宽屏下左右双栏，窄屏下侧边栏悬浮抽屉化
- 已明显考虑响应式和编辑效率

优点：

- 布局思路正确，接近专业编辑器
- 移动端/窄屏的侧边栏处理说明已有实际使用考虑
- 筛选、浏览、详情、构筑四个任务区被分开了

主要问题：

- 视觉语言仍主要依赖大量 Tailwind 内联色值，维护成本高
- 编辑器内部组件虽然功能完整，但审美上仍偏“开发工具”
- 卡组编辑器与外层 DeckManager header 的整体视觉衔接一般，界面被切成上下两套语气

### 4.6 对战模块

核心组件：

- `components/game/GameBoard.tsx`
- `PlayerArea.tsx`
- `PhaseIndicator.tsx`
- `GameLog.tsx`
- `JudgmentPanel.tsx`
- `ScoreConfirmModal.tsx`
- `MulliganPanel.tsx`
- `CardDetailOverlay.tsx`
- `DebugControl.tsx`

现状特征：

- 对战页视觉语言与其他模块差异最大
- 背景使用牌桌/场景图，前景使用大量 slate 深色悬浮面板
- 主要布局为：上下玩家区域 + 中央 VS 分隔 + 四周悬浮功能面板
- 交互复杂，包含拖拽、阶段推进、日志侧栏、判定面板、弹窗、卡牌悬浮查看等

优点：

- 交互能力最强，已经形成可玩的桌面对战界面
- 玩家区布局有明确规则映射，业务表达清楚
- 游戏过程中的辅助面板比较全，便于调试和规则验证

主要问题：

- 视觉完成度与功能完成度不匹配，明显偏“内部工具/规则验证器”
- 面板样式不统一：有的偏 slate 纯色，有的偏半透明，有的仍用 emoji 文案
- 游戏桌上信息很多，但层级管理一般，界面容易显得拥挤
- 与首页/卡组管理的深棕橙风格不一致，产品身份被切断
- 一些区域仍保留虚线边框、硬编码颜色、小字号标签，原型感较强

#### 4.6.1 玩家区域

核心组件：

- `components/game/PlayerArea.tsx`

现状判断：

- 功能结构复杂，规则映射强，属于“先满足玩法表达”的设计
- 成员槽、能量区、卡组、休息室、成功区等都已具备基础可视化
- 通过真实卡图和局部堆叠实现了基本桌游感

主要问题：

- 各区块标签、边框、尺寸、留白不统一
- 小尺寸区域很多，可读性依赖用户已理解规则
- 视觉上更像调试桌面而不是正式对战桌面

#### 4.6.2 阶段与日志面板

核心组件：

- `components/game/PhaseIndicator.tsx`
- `components/game/GameLog.tsx`

现状判断：

- 两者都承担流程辅助功能，但设计语言不一致
- `PhaseIndicator` 更像右下角控制卡片
- `GameLog` 更像左侧开发日志抽屉

主要问题：

- 配色和排版不统一
- 文案中仍有 emoji
- 日志面板层次简单，信息密度高但可读性一般

#### 4.6.3 判定与结算面板

核心组件：

- `components/game/JudgmentPanel.tsx`
- `components/game/ScorePanel.tsx`
- `components/game/ScoreConfirmModal.tsx`

现状判断：

- 这类组件解决的是复杂规则操作，功能优先级高于审美
- 已经有面板、排序、统计、确认等关键交互
- 适合作为未来重构的重点对象

主要问题：

- UI 表达偏重“把所有操作放上来”，视觉组织压力大
- 不同弹层之间样式继承关系不强
- 在深色桌面上可以用，但尚未形成高级感

### 4.7 卡牌管理模块

核心组件：

- `components/admin/CardAdminPage.tsx`
- `components/admin/CardEditModal.tsx`

现状特征：

- 这是最接近“后台管理台”的模块
- 使用深棕橙头部 + 搜索过滤条 + 卡牌网格 + 分页
- 卡面缩略图承担了主要视觉内容，界面本身相对克制
- 图标语言主要依赖 Lucide，成熟度高于首页和游戏准备页

优点：

- 信息架构清晰，列表检索与批量操作完整
- 批量上线/下线、分页、筛选状态都比较明确
- 相比其他模块，更接近正式业务后台

主要问题：

- 与首页相比更“后台”，与对战页相比更“管理工具”，整体产品风格仍分裂
- 头部过滤区操作较多，压缩在一行时密度偏高
- 卡片 hover 操作较轻，批量管理的效率感还有提升空间

### 4.8 通用组件模块

重点组件：

- `components/common/DeckSelector.tsx`
- `components/common/DeckStats.tsx`

现状判断：

- 已有一定复用意识
- `DeckSelector` 的视觉质量尚可，能独立承担卡组选择任务
- 但通用组件范围仍较窄，更多是业务组件复用，而不是设计系统复用

主要问题：

- 目前复用的是“功能块”，不是“视觉规范”
- 相同的 badge、按钮、panel、header 仍反复在页面内手写

## 5. 当前 UI 的横向分析

### 5.1 一致性分析

当前一致性主要停留在局部：

- 首页、游戏准备、卡组管理、卡牌管理之间共享深棕橙色倾向
- 认证页自成一体
- 对战页自成一体

也就是说，当前产品不是“一套系统中的不同页面”，而是“几组风格接近但彼此独立的页面集合”。

### 5.2 产品完成度分析

如果按完成度分层，大致可以这样判断：

- 认证页：首页级视觉完成度较高
- 首页/游戏准备页：产品页雏形较完整，但偏展示感
- 卡组管理/卡牌管理：工具页功能成熟度较高
- 对战页：玩法功能最强，但视觉完成度最低

当前最明显的落差是：游戏核心页最重要，但也最像内部原型。

### 5.3 可维护性分析

当前 UI 的主要维护问题：

- 大量颜色、透明度、边框、阴影直接写在组件 class 中
- 缺少统一 token，后续整体换肤成本高
- 一些旧全局样式和新 Tailwind 写法并存
- emoji、Lucide、图片装饰混用，替换成本不低

### 5.4 响应式分析

已有一定响应式意识：

- 卡组编辑器针对宽窄屏有明确策略
- 首页、游戏准备、后台列表有基础栅格适配

但整体还不是系统级响应式设计：

- 很多页面仍以桌面端为主要目标
- 复杂功能区在小屏上的信息层级需要继续压缩与重组

### 5.5 品牌与题材表达分析

项目题材是 Love Live! 卡牌游戏，但当前品牌表达仍不稳定：

- 优势在于：真实卡图、Logo、心色、卡背图已经存在
- 问题在于：这些素材尚未被提炼成统一视觉语言

因此当前页面能看出“和卡牌游戏有关”，但不够稳定地传达“这是 Loveca 的正式数字产品”。

---

## 6. 当前最核心的 UI 问题

综合来看，当前 UI 最核心的问题有五类：

### 6.1 风格割裂

- 认证页、业务页、对战页分别使用不同的主视觉逻辑
- 用户从登录到首页再到对战，会经历明显的视觉切换

### 6.2 图标与组件语言不统一

- emoji 与 Lucide 并存
- 相同层级的按钮、badge、面板没有统一样式标准

### 6.3 游戏核心界面原型感偏强

- 对战页功能最复杂，但也最像规则验证工具
- 缺少足够稳定的视觉层级和氛围控制

### 6.4 设计 token 不足

- 色值、透明度、阴影大量散落在组件中
- 当前难以做统一主题化和批量升级

### 6.5 通用设计资产沉淀不足

- 已复用一些业务组件
- 但还没有形成真正可复用的 panel、toolbar、badge、button、modal 规范

---

## 7. 后续 UI 重构建议优先级

结合当前代码现状，建议后续按以下优先级推进：

### P1：先统一设计基础层

- 收敛主题颜色、文字颜色、边框、阴影、圆角为统一 token
- 统一按钮、标签、面板、输入框、弹层样式
- 停止新增 emoji 图标，逐步迁移到 Lucide 或自定义图标

### P2：统一业务页视觉语言

- 将首页、游戏准备、卡组管理、卡牌管理归入同一套业务工作台风格
- 保持 header、卡片、筛选栏、状态标签的一致性

### P3：重点重做对战界面

- 先重构玩家区域、阶段指示器、日志、判定面板
- 目标不是改功能，而是把现有功能包装成更稳定的桌面体验

### P4：最后处理认证页与整体品牌衔接

- 认证页不一定要和业务页完全相同
- 但应在字体、品牌色、图标和组件语言上和主系统建立关联

---

## 8. 结论

当前项目 UI 已经覆盖了完整业务链路，说明产品功能面已经具备较强基础：

- 有认证入口
- 有首页导航
- 有卡组管理与卡组编辑
- 有游戏准备流程
- 有可操作的对战界面
- 有后台卡牌管理

但从设计角度看，当前仍处于“功能完整、视觉系统未统一”的阶段。最大问题不是单页不好看，而是：

- 模块之间缺少统一产品身份
- 核心对战页的视觉成熟度落后于功能成熟度
- 样式资产还没有沉淀成真正可维护的设计系统

因此，现阶段最合理的方向不是零散美化，而是先完成设计基础层统一，再分模块推进业务页和对战页重构。
