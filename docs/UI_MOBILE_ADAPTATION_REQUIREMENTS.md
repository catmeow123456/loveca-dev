# Loveca UI 移动端适配需求文档

> 文档类型: 需求文档
> 适用范围: 手机/平板布局、触控、覆盖层和移动端验收要求
> 最后更新: 2026-06-12
> 当前状态: 移动端适配需求与验收标准；具体实现以代码和 UI 规范为准
> 关联设计: [Loveca UI 重设计方案](./UI_REDESIGN.md)

本文档定义移动端适配目标、页面要求和验收标准，不维护具体组件实现、样式片段、阶段任务或排期。

## 1. 背景与目标

Loveca 的桌面端 UI 已形成统一主题和组件语言，但手机和平板场景需要重新组织信息层级、覆盖层和触控交互。移动端适配的目标不是把桌面布局等比缩小，而是保证核心流程可读、可点、可完成。

本需求关注：

- 手机和平板上的统一断点与布局原则。
- 首页、准备页、卡组管理、卡组编辑器、对战页、认证页和卡牌管理页的移动端体验。
- 触控、滚动、抽屉、模态框、安全区和横竖屏要求。
- 可执行的验收标准。

## 2. 范围

覆盖页面与组件：

- 首页 / 大厅页 `HomePage`
- 游戏准备页 `GameSetupPage`
- 卡组管理页 `DeckManager`
- 卡组编辑器 `CardEditor` 及其筛选、卡牌库、详情、侧栏
- 对战页 `GameBoard` 及相关浮层
- 认证页 `AuthLayout`、登录、注册、找回密码、重置密码
- 卡牌数据管理页 `CardAdminPage`
- 全局基础能力：断点、间距、滚动、覆盖层、触控、横竖屏、安全区

不覆盖：

- 新视觉主题方向
- PWA 安装体验重构
- 独立移动端业务流程
- 手势驱动的复杂动画系统

## 3. 设备分层

| 设备层级 | 宽度范围 | 设计策略 |
| --- | --- | --- |
| 手机竖屏 | 小于 640px | 单列优先，核心任务优先，次要信息折叠 |
| 大屏手机 / 小平板竖屏 | 640px 到 767px | 单列为主，可增加局部并排 |
| 平板竖屏 / 小平板横屏 | 768px 到 1023px | 可使用双列，但主要操作必须优先可见 |
| 桌面 / 大平板横屏 | 1024px 及以上 | 保持桌面结构，补充触控与覆盖层约束 |

功能页必须支持竖屏与横屏。游戏页至少需要手机竖屏可玩方案和平板横屏接近桌面方案。

## 4. 总体原则

- 沿用 UI 重设计中的深浅双主题和语义 token，不为移动端另起配色体系。
- 手机端优先确保看得清、点得到、流程能完成。
- 不把 hover 作为唯一信息入口。
- 页面主体不得出现意外横向滚动。
- 手机端覆盖层优先使用底部抽屉或全屏层。
- 顶部、底部固定元素必须考虑安全区。
- 动画继续以反馈为目的，避免影响低性能设备上的操作流畅度。

## 5. 基础系统需求

### 5.1 布局与断点

- 断点、页面边距、顶栏高度、卡牌缩略图尺寸、浮层宽高上限应有统一约定。
- 固定宽度组件应改为响应式宽度或最大宽度约束。
- 手机端默认单列布局，平板可按信息优先级展开双列。

### 5.2 滚动

- 页面应尽量只有一个主滚动容器。
- 抽屉、模态框、筛选面板和卡牌详情必须支持内部内容滚动。
- 内部滚动区需要有明确高度来源和滚动边界，避免手势冲突。

### 5.3 触控

- 主要按钮、图标按钮、标签切换和抽屉开关应满足最小触控热区。
- 拖拽交互需要在手机端评估是否保留、降级或替换为点击确认。
- 长按、双击和单击不应承担互相冲突的核心语义。

### 5.4 覆盖层

- 手机端卡牌详情优先使用全屏或底部抽屉。
- 手机端筛选器优先使用抽屉，不在页面顶部堆叠过多控件。
- 所有覆盖层都需要明确关闭入口、背景滚动锁定、内容滚动和安全区内边距。

## 6. 页面需求

### 6.1 首页 / 大厅页

- 顶栏在手机端可以拆成双层或压缩布局，优先保留品牌、主题切换和主要账户入口。
- “卡组管理 / 开始游戏 / 卡牌管理”等入口在手机端使用单列卡片或等价结构。
- 入口 CTA 必须易于触达，装饰信息可以减量。
- 页脚在手机端轻量展示，不抢主操作注意力。

验收标准：

- 手机宽度下首屏无横向滚动。
- 顶栏按钮不互相挤压。
- 主要 CTA 在首屏或极短滚动距离内可触达。

### 6.2 游戏准备页

- 步骤信息在手机端简化为当前步骤和必要进度提示。
- 模式选择、卡组选择和确认信息在手机端垂直堆叠。
- 底部主操作按钮保持稳定位置，适合连续操作。

验收标准：

- 手机端可以单手完成选择模式、选择卡组、确认开始。
- 关键状态信息不依赖桌面双列布局才能理解。

### 6.3 卡组管理页

- 列表页顶部操作按优先级分层：导航与标题、主要操作、说明与状态。
- 卡组列表项在手机端允许统计信息和操作按钮换行。
- 编辑页的名称、描述、保存和导出操作需要适合窄屏。
- DeckLog 导入在手机端应使用底部抽屉或接近全屏的表单层。

验收标准：

- 手机宽度下卡组项无内容遮挡。
- 用户无需横向滚动即可完成卡组名称修改、描述修改和保存。

### 6.4 卡组编辑器

- 手机端以卡牌库为主视图，卡组内容通过底部抽屉或全屏层查看。
- 平板端可使用可展开侧栏，但不能压缩主操作区到不可用。
- 搜索、类型切换和筛选入口优先保留，复杂过滤条件进入二级层。
- 卡牌详情在手机端必须清晰展示卡图、效果文本和可用操作。
- 卡组统计、完整性状态、分析面板和卡组清单需要明确层级关系。

验收标准：

- 手机端编辑器主体无全局横向滚动。
- 用户可以完成搜索、查看详情、加卡、查看当前卡组和保存。
- 筛选、详情和卡组抽屉不会叠加到难以关闭或理解。

### 6.5 对战页

对战页是移动端适配风险最高的页面。手机竖屏应优先保证可玩，而不是强求桌面战场完整同屏。

需求：

- 手机竖屏采用“主战场 + 次级面板”结构。
- 对手区域、日志、判定区、阶段操作和卡牌详情可以通过摘要条、抽屉、标签页或全屏层进入。
- 平板横屏可以接近桌面结构，但仍需重新检查浮层、详情、阶段提示和结果动画的遮挡关系。
- 手机端必须提供不依赖 hover 的卡牌详情查看方式。
- 关键操作按钮必须始终可达，包括阶段推进、确认、撤销和判定入口。

验收标准：

- 手机竖屏可以完成一局基础流程。
- 平板横屏不出现覆盖层互相遮挡导致无法关闭或确认。
- 当前阶段、当前玩家、己方关键区域和主要确认操作始终可理解。

### 6.6 认证页

- 手机端缩减头部视觉占比，优先展示表单字段与主按钮。
- 输入、错误提示和辅助链接保持清晰间距。
- 软键盘弹起时页面应可自然滚动，提交按钮不能被永久遮挡。

验收标准：

- 常见手机高度下，用户在软键盘打开时仍可访问当前输入框和提交按钮。

### 6.7 卡牌数据管理页

- 筛选区在手机端拆为搜索、筛选入口和筛选抽屉。
- 分页器在移动端简化，不要求展示过多页码。
- 触屏上编辑、上线、下线等操作不能只依赖 hover。
- 卡牌编辑在手机端应使用全屏或接近全屏的编辑层。

验收标准：

- 手机端可以完成搜索、筛选、进入编辑和新建卡牌。
- 用户不依赖 hover 也能发现关键操作。

## 7. 组件级要求

| 组件类型 | 移动端要求 |
| --- | --- |
| 顶栏 | 可双行，明确返回、标题、主题切换和账户入口优先级 |
| Tabs / 筛选条 | 超出屏宽时使用横向滚动或抽屉，不强行压缩文字 |
| 列表项 / 卡片 | 标题、状态、统计和操作允许换行，不遮挡内容 |
| Drawer / Modal | 支持关闭、滚动、安全区和背景滚动锁定 |
| 卡牌组件 | 定义手机和平板缩略图尺寸，详情入口不依赖 hover |
| 游戏操作 | 关键确认和撤销入口在触屏上可达 |

## 8. 性能要求

- 移动端动画优先使用 transform 和 opacity。
- 高频界面避免叠加过多模糊、阴影和大面积半透明层。
- 图片与卡牌资源应使用适配尺寸，避免默认加载过大资源。
- 首屏不应因装饰层过多导致明显卡顿。

## 9. 验收基线

建议至少覆盖以下视口：

| 视口 | 场景 |
| --- | --- |
| 390 x 844 | 手机竖屏 |
| 430 x 932 | 大屏手机竖屏 |
| 768 x 1024 | 平板竖屏 |
| 1024 x 768 | 平板横屏 |

验收内容：

- 页面主体无意外横向滚动。
- 文本、按钮和浮层不重叠、不截断。
- 关键按钮点击热区达标。
- 模态和抽屉在手机端可完整关闭和滚动。
- 主要业务流程不依赖 hover。
- 顶部和底部固定元素不被安全区遮挡。
- 深色和浅色主题下均可读。

## 10. 后续推进建议

后续移动端适配应按风险由低到高推进：

- 先统一基础断点、滚动、安全区和覆盖层规则。
- 再处理首页、认证页、游戏准备页和卡组管理页等低复杂度页面。
- 随后处理卡组编辑器的手机和平板布局。
- 最后单独设计并验收对战页的平板和手机方案。

每个开发任务应输出对应页面的验收清单。已完成的任务记录不应长期留在本文中。

## 11. 实现边界

本文档只维护移动端适配的需求、页面范围和验收标准，不维护具体组件路径、样式片段或阶段任务记录。当前 UI 实现以代码为准；通用组件、主题和布局约束见 [前端 UI 开发规范](./coding-standard/ui-standard.md)。
