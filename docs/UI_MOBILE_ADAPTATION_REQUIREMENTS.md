# Loveca UI 移动端适配需求文档 v1

> 文档类型: 需求文档
> 适用范围: 手机/平板布局、触控、覆盖层和移动端验收要求
> 当前状态: 需求与后续实施标准；仅描述部分已落地移动适配，未实现项以“建议/应”表述
> 基于 `docs/UI_REDESIGN.md` 的分层双主题方案，补充手机与平板场景下的布局、交互和验收要求。

---

## 1. 背景与目标

当前 UI 重设计已经完成了主题统一、组件语言统一和核心页面改版，并且首页、准备页、卡组管理与卡组编辑器已经补入了部分移动端适配。但整体实现仍缺少统一的移动端系统化约束，现有页面在窄屏下仍存在以下共性问题：

1. 顶栏、工具栏、筛选区和操作区已有局部折行或压缩处理，但仍散落在各页面中，缺少统一断点与信息密度策略。
2. 若干核心区域仍依赖固定定位或桌面悬浮层，如游戏页左侧判定抽屉、右下角阶段面板、右侧 hover 详情、右上角主题切换按钮等。
3. 游戏主界面仍采用上下双玩家桌面视角，没有针对手机竖屏、小平板竖屏、平板横屏定义专门的信息层级。
4. 列表页和管理页虽然使用了响应式网格和部分移动端操作区，但尚未统一筛选抽屉、分页、常显操作与触控尺寸策略。
5. 覆盖层实现不一致：卡组编辑器的卡组/筛选/详情已使用底部抽屉，DeckLog、卡牌编辑、游戏内若干弹窗仍偏桌面居中或侧边固定。
6. 目前没有明确的触控优化要求，拖拽、点击目标尺寸、横向滚动、手势冲突仍存在风险。

本需求文档的目标是：

- 定义 Loveca 在手机和平板上的统一适配原则。
- 明确不同页面在移动端的信息优先级与交互重组方式。
- 为后续设计实现、拆分任务和验收测试提供可执行标准。

## 2. 范围

本期移动端适配覆盖以下页面与组件：

- 首页 / 大厅页 `HomePage`
- 游戏准备页 `GameSetupPage`
- 卡组管理页 `DeckManager`
- 卡组编辑器 `CardEditor` 及其筛选、卡牌库、详情、侧栏
- 对战页 `GameBoard` 及相关浮层
- 认证页 `AuthLayout`、登录、注册、找回密码、重置密码
- 卡牌数据管理页 `CardAdminPage`
- 全局基础能力：断点、间距、滚动、覆盖层、触控、横竖屏、安全区

不在本期范围内：

- 新的视觉主题方向
- PWA 安装体验重构
- 手势驱动的复杂动画系统
- 为移动端新增一套独立业务流程

## 3. 设备分层与断点策略

为了避免“只做一个缩小版桌面”，需要先明确设备分层。

### 3.1 目标设备层级

- 手机竖屏：`< 640px`
- 大屏手机 / 小平板竖屏：`640px - 767px`
- 平板竖屏 / 小平板横屏：`768px - 1023px`
- 桌面与大平板横屏：`>= 1024px`

### 3.2 布局原则

- 手机竖屏以单列布局为默认，不允许依赖双列主结构完成核心任务。
- 平板允许双列，但必须保证主要操作区先出现，次要信息区可折叠或次级展开。
- `>= 1024px` 保持当前桌面设计为主，仅做少量结构修正以统一代码路径。

### 3.3 方向策略

- 功能页必须同时支持竖屏与横屏。
- 游戏页至少要明确定义两套策略：
  - 手机竖屏：优先保证可玩和可读，不强求完整桌面式全局同屏。
  - 平板横屏：尽量接近桌面版，但控件密度、浮层与触控尺寸需重新调优。

## 4. 总体设计原则

### 4.1 保持语义令牌一致，不做移动端独立主题

- 继续沿用 `UI_REDESIGN` 中的深浅双主题与语义色令牌。
- 移动端只调整布局、层级、尺寸和交互，不另起一套配色系统。

### 4.2 以任务完成优先，减少同时展示的信息量

- 手机端优先确保“看得清、点得到、能完成流程”。
- 不追求桌面端所有信息同屏；允许通过分段、抽屉、标签页、底部面板来分层承载。

### 4.3 优先触控，不默认鼠标悬停

- 不将 `hover` 作为唯一的信息入口。
- 所有关键操作必须可通过点击直接触发。
- 所有主要点击目标最小热区应不小于 `44x44px`。

### 4.4 移动端覆盖层优先使用底部抽屉或全屏层

- 手机端优先使用 bottom sheet / full screen modal。
- 平板可继续使用居中 modal，但尺寸必须收敛并支持滚动。

### 4.5 控制横向滚动范围

- 页面主体不允许出现全局横向滚动。
- 若确有必要，只允许局部区域横向滚动，并且必须有明确视觉边界与滚动提示。

### 4.6 考虑安全区与移动浏览器视口变化

- 顶部和底部固定条需要兼容 `safe-area-inset-*`。
- 避免仅依赖 `100vh`，需要兼容移动浏览器地址栏收起/展开导致的高度抖动。

## 5. 基础系统需求

### 5.1 断点与布局 token

当前代码已有 `useMediaQuery()`、`safe-top`、`safe-bottom`、`100dvh`、卡牌尺寸和基础间距/圆角 token，但响应式断点和布局参数仍主要散落在 Tailwind 类、组件内 `matchMedia` 查询和局部 `clamp()` 中。需要补充统一的响应式 token，包括但不限于：

- 页面横向 padding
- 顶栏高度
- 面板圆角等级
- 卡牌缩略图尺寸
- 手机 / 平板下的浮层宽高上限
- 网格列数与最小列宽

要求：

- 断点策略不能只散落在页面组件里，需要有统一约定。
- 对 `fixed width` 组件进行梳理，优先改为 `max-width + clamp + responsive width`。

### 5.2 可滚动区域规范

- 页面只保留一个主滚动容器，避免嵌套滚动造成手势冲突。
- 若页面必须存在内部滚动区，需要明确其高度来源与滚动指示方式。
- 抽屉、模态框、筛选面板、卡牌详情必须支持内容溢出滚动。

### 5.3 触控规范

- 主要按钮、图标按钮、标签切换、抽屉开关需要满足最小点击面积。
- 拖拽交互需要评估在手机端是否保留、降级或替换。
- 长按、双击、单击如有复合语义，必须重新定义触屏行为，避免误触。

### 5.4 覆盖层规范

- 手机端卡牌详情优先使用全屏或底部抽屉。
- 手机端筛选器优先使用抽屉，不在主页面顶部堆叠过多控件。
- 模态框内容过长时必须可滚动，并保证关闭按钮始终可达。

## 6. 分页面需求

### 6.1 首页 / 大厅页

现状观察：

- 顶栏已经压缩为品牌、用户状态、主题切换、登出按钮的两列结构，用户名在窄屏下会截断，登出文字在 `sm` 以下隐藏。
- 主入口卡片当前默认为单列，桌面才切为多列；管理员入口也会在手机端纵向排列。
- 页脚已经支持多行换行，但仍包含版权、版本和数据来源，手机端信息密度需要继续验收。

需求：

- 手机端顶栏改为双层或压缩布局，优先保留品牌、主题切换、主要账户入口。
- “卡组管理 / 开始游戏 / 卡牌管理”入口在手机端改为单列卡片。
- 入口卡片的装饰信息可减量，但 CTA 必须始终首屏可见。
- 页脚在手机端改为多行轻量展示，不抢主操作注意力。

验收标准：

- `375px` 宽度下首屏不出现横向滚动。
- 顶栏所有按钮都可单手点击，不互相挤压。
- 主页主要 CTA 在不滚动或极短滚动距离内可触达。

### 6.2 游戏准备页

现状观察：

- 步骤条已经改为可横向滚动的紧凑数字步骤，手机宽度下不会强行压缩所有文字。
- 模式选择卡片默认单列，`md` 以上才双列；卡组选择器使用全宽容器。
- 确认页在调试模式下 `md` 以上双列，手机端垂直堆叠。
- 底部“上一步 / 下一步”操作区在手机端已使用 sticky 容器。

需求：

- 手机端步骤条需简化为“步骤数字 + 当前标题”或可横向滚动的步骤组件。
- 模式选择卡片改为单列，卡组选择器全宽展示。
- 确认页中的双方信息卡在手机端改为垂直堆叠。
- 底部主操作按钮在手机端需保持稳定位置，可考虑 sticky action bar。

验收标准：

- 手机端从选择模式到开始游戏可以单手连续完成。
- 关键状态信息不依赖横向双列才能理解。

### 6.3 卡组管理页

现状观察：

- 列表页顶部操作已重排为纵向工具区，导入 DeckLog、导入 YAML、创建新卡组在手机端全宽堆叠。
- 卡组列表项已经按窄屏重排统计与操作按钮，移动端操作按钮采用横向局部滚动兜底。
- 编辑页手机端显示卡组名称 + 保存按钮，备注默认折叠为“编辑备注”，导出放入次级小按钮。
- 保存按钮目前位于移动端编辑工具栏右侧，并非底部固定操作区。
- DeckLog 导入当前仍是居中 `max-w-md` modal，不是底部抽屉或接近全屏表单层。

需求：

- 手机端列表页顶部操作需重组为分层结构：
  - 第一层：返回、标题、主题切换
  - 第二层：主要操作按钮
  - 第三层：状态与说明
- 卡组列表项在手机端改为更明确的上下分层，统计信息和操作按钮允许换行。
- 编辑页中的名称与描述输入框改为纵向排列。
- 导出 / 保存按钮在手机端优先固定在底部操作区，或至少保证滚动后仍易于触达。
- DeckLog 导入弹窗在手机端改为底部抽屉或接近全屏的表单层。

验收标准：

- `375px` 下列表页每个卡组项无内容遮挡，操作按钮不低于两行可读布局。
- 编辑页在手机端无需横向滚动即可完成卡组名称修改、描述修改和保存。

### 6.4 卡组编辑器

现状观察：

- 当前逻辑已区分三类体验：`>=960px` 常驻侧栏，`768px-959px` 使用右侧悬浮侧栏（宽度 `min(82vw,420px)`），`<768px` 使用卡组底部抽屉。
- 手机端工具栏只常驻搜索与筛选入口，类型切换和复杂筛选已放入底部筛选抽屉。
- 手机端卡组侧栏已改为 `84dvh` 底部抽屉，并提供“查看卡组”浮动入口。
- 卡牌详情已使用 `CardDetailDrawer`：手机端底部抽屉，平板/桌面右侧抽屉。
- 详情抽屉当前是纯查看，不包含增减卡按钮；加减卡仍主要在卡牌库/卡组侧栏中完成。

需求：

- 明确三档策略：
  - 手机：卡牌库全宽，卡组信息通过底部抽屉或全屏抽屉查看。
  - 平板：卡牌库主视图 + 可展开侧栏，避免固定 `480px` 导致压迫。
  - 桌面：保留常驻侧栏。
- 手机端筛选器默认折叠，点击后以抽屉形式展开。
- 搜索栏、类型切换和筛选入口需优先保留，复杂过滤条件放入二级层。
- 卡牌详情在手机端优先全屏查看，保证卡图、效果文本、增减卡按钮都清晰可达。
- 卡组统计、完整性状态、分析面板与卡组清单之间需明确切换关系，避免同层并存。

验收标准：

- 手机端编辑器主体无全局横向滚动。
- 在 `390px` 宽度下，用户可完成搜索卡牌、查看详情、加卡、查看当前卡组、保存。
- 侧栏、详情、筛选不会同时叠加成三层以上覆盖，避免失控。

### 6.5 对战页

现状观察：

- 当前对战页是典型桌面布局：上下双玩家区域 + 中部 VS 分隔 + 左侧判定区抽屉 + 右上角主题切换 + 多个浮层。
- 交互高度依赖拖拽与悬浮详情；手机端竖屏无法直接容纳完整战场关系。
- 对墙打模式会弱化对手区域并提供左上角离开按钮，但没有形成手机竖屏专用主视角。

这是本次移动适配中风险最高、工作量最大的一部分，需要单独定义优先级。

需求：

- 先定义手机端“可玩优先版”而不是强求桌面布局完全等比缩放。
- 手机竖屏建议采用“主战场 + 可切换次级面板”结构，例如：
  - 主视图显示当前玩家关键区域
  - 对手区域折叠为摘要条或可展开面板
  - 日志、判定区、阶段操作、卡牌详情通过底部抽屉/标签切换进入
- 平板横屏可保留更完整战场，但仍需重新梳理：
  - 判定区抽屉宽度
  - 卡牌详情浮层位置
  - 主题切换与调试控制的占位
  - Live 结果动画和阶段横幅在小高度下的遮挡关系
- 必须重新评估拖拽交互：
  - 平板可保留拖拽
  - 手机需验证是否需要点击选中 + 目标区确认的降级方案
- 手机端必须提供不依赖 hover 的卡牌详情查看方式。
- 游戏中的关键操作按钮必须始终可达，包括阶段推进、确认、撤销、判定入口。

验收标准：

- 手机竖屏可以完整进行一局基础流程，不因布局导致关键区域不可操作。
- 平板横屏不出现覆盖层互相遮挡导致无法关闭或确认的情况。
- 关键游戏信息始终可见：当前阶段、当前玩家、己方关键区域、主要确认操作。

### 6.6 认证页

现状观察：

- 认证页使用 `fixed inset-0 overflow-y-auto`、`100dvh`、安全区和手机端顶部留白，整体已较接近移动友好。
- Logo 在手机端为 `64px`，标题与表单仍以居中卡片为主，软键盘场景仍需要实机/浏览器验证。

需求：

- 手机端缩减头部视觉占比，优先展示表单字段与主按钮。
- 表单输入、错误提示、辅助链接需保持清晰间距。
- 键盘弹起时页面应能自然滚动，提交按钮不能被遮挡。

验收标准：

- 常见手机高度下，用户在软键盘打开时仍可访问当前输入框和提交按钮。

### 6.7 卡牌数据管理页

现状观察：

- 顶部筛选条信息密度高，包含搜索、类型筛选、状态筛选、刷新、导出、新建。
- 当前卡牌列表虽然有响应式网格，但整体仍偏桌面工具页。
- 卡片本体点击即可进入编辑，不完全依赖 hover；但“编辑 / 上线 / 下线”辅助按钮仍通过 `group-hover` 显示，触屏发现性不足。
- `CardEditModal` 当前是居中 `max-w-2xl` modal，并未切换为手机全屏编辑层。

需求：

- 手机端将筛选区拆分为：
  - 搜索 + 筛选按钮
  - 筛选抽屉
  - 顶部主要操作区
- 分页器需要移动端简化，不要求同时展示过多页码按钮。
- 卡片 hover 操作在手机端要改为常显按钮、更多菜单，或进入详情页后操作。
- 编辑卡牌的 modal 在手机端应切换为全屏编辑层。

验收标准：

- 手机端可以完成搜索、筛选、进入编辑、新建卡牌的基础流程。
- 不依赖 hover 才能看到“编辑 / 上线 / 下线”操作。

## 7. 交互与组件级要求

### 7.1 顶栏

- 手机端允许从单行改为双行。
- 返回、标题、主题切换、账户状态必须有优先级，不要求全部始终同一层展示。

### 7.2 Tabs / 筛选条

- 超出屏宽时优先采用横向滚动 tabs 或抽屉，不要强行压缩文字。
- 超过 4 个筛选按钮时，优先合并为“筛选”入口。

### 7.3 列表项与卡片

- 统计信息、状态标签、操作按钮在手机端允许换行。
- 不允许因为标签过多导致标题区域被挤压到无法阅读。

### 7.4 Drawer / Modal / Overlay

- 手机端优先底部抽屉或全屏层。
- 所有覆盖层都要支持：
  - 明确关闭入口
  - 背景滚动锁定
  - 内容滚动
  - 安全区内边距

### 7.5 卡牌组件

- 需要定义手机 / 平板下卡牌缩略图尺寸档位。
- 需要统一卡牌详情在不同设备上的展示规格。
- 若使用拖拽，拖动热区和落点反馈要针对触屏重新验证。

## 8. 性能与实现约束

- 移动端动画应继续遵循只动画 `transform` 和 `opacity` 的原则。
- 对高频界面避免大面积 `backdrop-filter` 叠层，尤其是游戏页和多抽屉叠加场景。
- 图片与卡牌资源在移动端应优先使用适配尺寸，避免默认加载过大资源。
- 移动端首屏不得因过多装饰层导致明显卡顿。

---

## 9. 测试与验收基线

建议至少覆盖以下视口：

- `390 x 844` iPhone 竖屏
- `430 x 932` 大屏手机竖屏
- `768 x 1024` iPad 竖屏
- `1024 x 768` iPad 横屏

验收方式：

- 逐页面静态检查：无横向滚动、无内容截断、无重叠失控
- 基础交互检查：点击、输入、打开抽屉、关闭浮层、保存、提交
- 游戏流程检查：开局、阶段推进、查看详情、判定、确认
- 主题检查：深色与浅色主题下均可读

通用验收标准：

- 页面主体无意外横向滚动
- 关键按钮点击热区达标
- 模态和抽屉在手机端可完整关闭和滚动
- 主要业务流程不依赖 hover
- 顶部和底部固定元素不被安全区遮挡

## 10. 实施优先级建议

### P0

- 建立统一断点和移动端布局 token
- 修正全局高度、安全区、滚动容器问题
- 完成首页、认证页、游戏准备页、卡组管理页基础适配

### P1

- 完成卡组编辑器的手机 / 平板布局重组
- 完成管理页筛选抽屉和移动端 modal 规范
- 建立覆盖层与抽屉的统一交互组件规范

### P2

- 完成游戏页的平板版适配
- 完成游戏页的手机竖屏可玩方案
- 对拖拽交互做移动端降级或替代设计

## 11. 输出要求

3. 编辑器适配
4. 游戏页平板适配
5. 游戏页手机方案

## 12. 结论

Loveca 的移动端适配不是单纯“把桌面页面缩小”，而是要在保留 `UI_REDESIGN` 视觉体系的前提下，重新定义手机和平板上的信息层级、交互入口和覆盖层策略。尤其是卡组编辑器与对战页，必须从“桌面默认”转向“任务优先、触控优先、分层展示优先”。
