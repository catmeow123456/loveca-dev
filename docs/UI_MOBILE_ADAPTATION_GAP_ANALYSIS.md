# Loveca UI 移动端现状差距清单

> 文档类型: 现状差距清单
> 适用范围: 移动端适配需求与当前前端实现之间的差距
> 最后更新: 2026-06-13
> 当前状态: 现行事实清单；每次完成移动端适配任务后需要同步更新
> 对照需求: [Loveca UI 移动端适配需求文档](./UI_MOBILE_ADAPTATION_REQUIREMENTS.md)

本文档记录当前代码相对于移动端目标态的差距。需求、验收标准和目标状态仍以 `UI_MOBILE_ADAPTATION_REQUIREMENTS.md` 为准；本清单只描述当前实现是否满足、风险在哪里、后续应优先处理什么。

## 维护规则

- 修复移动端布局、覆盖层、触控路径或验收自动化后，应同步更新对应条目状态。
- 如果代码实现与本文不一致，以代码和最新验收结果为准，并更新本文。
- 不在本文维护详细设计稿或阶段排期；复杂方案继续放在专题设计文档中。
- 状态使用 `未满足`、`部分满足`、`基本满足`、`待验证`。

## 总体现状

当前低复杂页面已经具备一定移动端基础，卡组编辑器、卡组管理页和卡牌数据管理页也已经有专门的手机抽屉或全屏层结构，并已补充覆盖四个基线视口的移动端 E2E 横向滚动检查。主要缺口集中在对战页、软键盘/安全区/覆盖层等专项验收和部分覆盖层统一规则。本文中的“基本满足”只表示具备主要响应式结构和基础基线覆盖，不代表所有触控热区、软键盘、安全区和覆盖层滚动已经完成实测验收。

## 页面差距

### 首页 / 大厅页

状态: 基本满足

已具备:

- 手机端顶栏压缩，品牌、主题切换和账户入口仍可见。
- 入口卡片默认单列，桌面端再展开网格。
- 页脚使用安全区底部类。

待验证:

- 390 x 844 和 430 x 932 下首屏是否无横向滚动。
- 长用户名、管理员入口显示时是否挤压顶栏按钮。
- 深色和浅色主题下 CTA、状态标签是否仍可读。

### 游戏准备页

状态: 基本满足

已具备:

- 步骤条在窄屏下可横向滚动。
- 模式、卡组选择和确认步骤按窄屏单列处理。
- 手机端底部操作区使用 sticky 结构，按钮保持稳定位置。

待验证:

- 小高度手机上，步骤内容与底部操作区是否互相遮挡。
- 卡组列表较长时，主滚动和内部滚动边界是否自然。

### 卡组管理页

状态: 基本满足

已具备:

- 列表项允许标题、状态、统计和操作换行。
- 手机端操作条允许局部横向滚动，编辑和删除保持显式按钮。
- 编辑页手机头部保留卡组名称、保存、备注开关和导出入口。
- DeckLog 导入在手机端已改为底部表单层，包含关闭入口、内部滚动区、底部安全区和背景滚动锁定。

待验证:

- DeckLog 导入在 390 x 844、小高度手机和软键盘打开时是否仍可完整输入、取消和导入。
- DeckLog 导入警告列表较长时，内部滚动边界是否自然。

### 卡组编辑器

状态: 基本满足

已具备:

- 手机端以卡牌库为主视图，卡组内容通过底部抽屉查看。
- 手机端筛选条件进入底部抽屉，搜索保留在主工具条。
- 手机端详情使用底部抽屉，桌面端使用右侧抽屉。
- 加卡、减卡按钮在卡牌单元底部常驻，不依赖 hover。

主要缺口:

- 详情抽屉目前以查看为主，不提供加卡、减卡等编辑操作；如果把详情作为手机端等价操作路径，还需要补齐操作入口。
- 卡牌库单元底部加卡、减卡按钮在手机端约为 32 x 32 CSS px，低于 44 x 44 建议触控热区；卡组抽屉内的同类微按钮更小，约为 20 x 20 CSS px。需要通过详情、菜单或其他路径提供等价操作，或在专项验收中确认可用性。

待验证:

- 筛选抽屉、卡组抽屉、详情抽屉不会形成难以关闭的叠加状态。
- 390 x 844 下卡牌网格、底部查看卡组按钮和系统安全区不互相遮挡。
- 抽屉打开时背景滚动锁定是否符合需求。

### 对战页

状态: 部分满足

当前实现:

- 手机竖屏下 `GameBoard` 已有第一版“己方主战场 + 对手摘要 + 底部任务入口”结构。
- 对手区和游戏日志已可通过手机底部任务层打开；阶段操作浮窗在手机端改为底部 dock。
- 判定区在手机端已从左侧外露抽屉调整为全屏任务层样式。
- `PlayerArea` 手机端已拆出专用布局：成功 Live 区和能量区固定在左侧，主卡组/休息室/能量卡组固定在右侧，中间 Live/成员战场使用固定牌桌结构，手牌改为自适应扇形轨道。
- 桌面和平板横屏仍保留原有对手区、VS 分隔、己方区上下完整展示结构。
- `PlayerArea` 桌面端仍存在绝对定位的左右资源区、成功区和中间成员/Live 区。
- 主交互仍以 `@dnd-kit` 拖拽为核心，部分操作依赖双击或拖到目标区。
- 卡牌详情主要由 hover 触发右侧浮窗。

主要缺口:

- 手机竖屏已有基础信息结构和主战场分区；资源区目前是固定右侧紧凑堆，需要继续补充更完整的详情入口与精确操作层。
- 详情和解决区动作尚未整理为稳定的手机详情抽屉或任务层。
- 卡牌详情、判定区快捷操作、解决区动作等仍有 hover 路径。
- 拖拽和双击仍承担大量核心语义，触屏误操作风险高。
- 右下阶段面板、左日志、左判定侧栏、右侧详情浮窗在平板横屏仍需要重新检查遮挡关系。
- 换牌面板、赛前提示、分数确认、Live 结果动画、判定区和详情浮窗等多个浮层没有统一的移动端层级、关闭入口和遮挡验收清单。

建议处理:

- 单独设计移动端 `GameBoard` 信息架构，不直接缩放桌面桌。
- 手机端优先实现己方主视角、对手摘要、资源快捷栏、底部流程条和底部动作条。
- 将判定区改为手机全屏任务页或底部任务层。
- 为卡牌详情提供点击或长按入口，不能只依赖 hover。
- 为主操作提供点击式动作路径，拖拽只保留为高级或桌面增强能力。
- 在移动端重构任务中同步维护浮层清单，逐项验收打开、关闭、内部滚动、背景滚动和互相遮挡关系。

### 认证页

状态: 基本满足

已具备:

- 认证布局使用固定视口加内部纵向滚动。
- 手机端头部视觉占比较低，表单主体优先显示。
- 顶部和底部使用安全区类。

待验证:

- 软键盘弹起时，当前输入框和提交按钮是否可自然滚动到可见区域。
- 注册、找回密码、重置密码和邮箱验证页是否与登录页一致。

### 卡牌数据管理页

状态: 基本满足

已具备:

- 页面主滚动容器存在。
- 手机端保留搜索和筛选入口，类型、状态和批量状态操作已进入底部筛选抽屉。
- 点击卡牌可进入编辑层，新建卡牌入口始终可见。
- 单卡编辑提示和上线、下线操作在触屏宽度下常驻可见，不再只依赖 `group-hover`。
- 手机分页器已简化为上一页、当前页和下一页。
- 卡牌编辑在手机端已改为全屏编辑层，内容区和底部操作区分离，并补充安全区。

主要缺口:

- 卡牌编辑层内部仍有较多密集微按钮和长表单，需要专项检查触控热区、软键盘和底部操作遮挡。

待验证:

- 390 x 844 下筛选抽屉、编辑层和分页器是否无横向滚动。
- 软键盘打开时，卡牌编辑层当前输入框和保存按钮是否仍可访问。
- 大量筛选结果和长卡牌名称下，卡牌网格操作按钮是否换行自然。

### 联机与共享辅助页

状态: 部分满足

当前实现:

- `App` 中存在联机房间、联机调试、共享卡组和联机房间监控入口或路由。
- 这些页面使用了一定的 `sm`、`lg`、`xl` 响应式网格和换行结构，但此前未在移动端适配需求范围中单独登记。
- `OnlineRoomsAdminPage` 使用 `overflow-x-auto` 包裹 `min-w-[1040px]` 表格，属于受控局部横向滚动，但还不是堆叠卡片或分组列表形态。

待验证:

- `OnlineRoomPage` 在手机端是否可以顺畅完成创建/加入房间、选择并锁定卡组、查看成员准备状态和退出。
- `OnlineDebugPage` 的调试状态、服务信息和主操作在手机端是否不会挤压主要路径。
- `SharedDeckPage` 的卡组摘要、卡牌清单和复制链接操作在 390 x 844 下是否无全局横向滚动。
- `OnlineRoomsAdminPage` 的受控横向表格在窄屏下是否不会影响页面主体滚动，关键状态与刷新入口是否仍易访问。

建议处理:

- 将这些辅助页纳入低复杂页面的四个基线视口截图验收。
- 监控页如果面向手机高频访问，优先从受控横向表格进一步改为堆叠卡片或分组列表。

## 基础系统差距

### 断点与布局约定

状态: 部分满足

当前已有:

- 全局 CSS 有卡牌尺寸、间距、半径、层级、主题 token 和安全区类。
- 多数页面使用 Tailwind `sm`、`md`、`lg` 断点。
- 卡组编辑器额外使用 `960px` 区分桌面常驻侧栏。

缺口:

- `767px`、`960px`、`560px`、`480px`、`420px` 等局部断点分散在组件中，缺少统一登记。
- 卡牌缩略图尺寸、顶栏高度、浮层宽高上限尚未整理为清晰的移动端约定表。

建议处理:

- 在 UI 规范中登记通用断点和允许的页面过渡断点。
- 后续新增过渡断点时，在任务验收中说明原因。

### 覆盖层与滚动

状态: 部分满足

当前已有:

- 通用 `modal-backdrop`、`safe-top`、`safe-bottom`、`touch-scroll`。
- 卡组编辑器的手机筛选、卡组侧栏和详情已使用底部抽屉。
- DeckLog 导入和卡牌数据编辑已具备手机专用弹层形态。

缺口:

- 多数覆盖层仅渲染 backdrop 和内部滚动区，未统一通过 body scroll lock 或等价机制锁定背景滚动。
- 游戏桌内仍存在多个不同方向的浮层，层级和关闭入口不统一。

建议处理:

- 优先收口手机端弹层形态：轻任务 bottom sheet，重任务 full-screen。
- 为每类覆盖层补充关闭入口、内部滚动、安全区和背景滚动锁定验收。

### 触控与 hover

状态: 部分满足

当前已有:

- 多数普通按钮有 `min-h-10` 或 `min-h-11`，基础触控尺寸较好。
- 卡组编辑器加减卡按钮常驻，不依赖 hover。

缺口:

- 对战页仍有关键或半关键操作依赖 hover。
- 对战页存在双击切换状态、拖拽移动等触屏风险较高的核心交互。
- 个别游戏桌快捷按钮、卡组编辑器加减卡微按钮和卡牌数据编辑层内的密集微按钮尺寸小于 44 x 44 CSS px。

建议处理:

- 所有手机核心操作提供点击式入口。
- 将 hover 详情替换或补充为点击/长按详情。
- 小尺寸微按钮保留时，必须提供等价的详情页、抽屉或菜单路径。

## 验收差距

状态: 部分满足

已具备:

- 已新增 `client/playwright.config.ts` 和 `client/tests/e2e/mobile-layout.spec.ts`。
- `pnpm --dir client test:e2e:mobile` 覆盖 `390 x 844`、`430 x 932`、`768 x 1024`、`1024 x 768` 四个视口。
- 当前基础场景包含登录页、首页、游戏准备、卡组管理、DeckLog 导入、卡组编辑器、卡牌数据管理、筛选抽屉、卡牌编辑层、联机房间和共享卡组页。
- 基线会保存截图并检查页面级横向滚动。

当前缺口:

- 缺少软键盘、背景滚动锁定、安全区遮挡等移动端专项验收记录。
- 对战页移动端重构前后仍需要单独建立基础流程验收。

建议处理:

- 在现有基线基础上补软键盘、安全区、覆盖层滚动和遮挡断言。
- 为卡组编辑器补更细的抽屉打开/关闭、内部滚动和触控热区验收。
- 对战页移动端重构前后单独建立基础流程验收。

## 推荐推进顺序

1. 在现有移动端 E2E 基线上继续补专项断言：软键盘、安全区、覆盖层滚动锁定和遮挡关系。
2. 继续修复低成本缺口：重点验证并细化 CardAdmin 编辑层密集微按钮、软键盘和安全区表现。
3. 整理断点和覆盖层约定，把已有 `767px`、`960px` 等过渡断点登记清楚。
4. 单独推进对战页移动端重构，避免在桌面结构上继续局部补丁。
