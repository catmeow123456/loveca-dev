# 文档的编写规范

> 文档类型：编码标准
> 适用范围：`docs/` 与根目录长期维护文档
> 当前状态：现行规范

1. 文档应明确类型、适用范围和当前状态。常见类型包括：
   - 总览文档：说明项目是什么、如何运行、主要模块和阅读入口；
   - 需求文档：描述产品、规则或工程目标，不涉及具体代码实现；
   - 设计文档：根据需求描述数据库、代码架构、数据流和模块职责；已经完成代码编写的功能需要带上关键代码路径（只需要代码路径，无需代码行号）；
   - 编码标准：描述开发约束、边界规则、代码风格和评审标准；
   - 专题说明：描述局部问题的当前事实、边界或约束；
   - 历史/计划文档：保留尚未落地或已经被替代的方案，并清楚标注状态；
   - 临时记录：用于短期状态沟通，不作为长期事实来源。

2. 文档的保存位置：`docs/` 下存放需求文档、设计文档、编码标准、专题说明和历史参考。根目录保留 README、系统设计和项目进度等主入口文档。需求文档和设计文档可以成对出现，也可以只保留一份专题说明；但一旦标为需求文档，就不应承担实现说明职责。新增长期文档时，应同步登记到 `docs/README.md`。

3. 需求文档不得写入具体实现代码、接口调用步骤、文件级改造步骤、迁移脚本或某个组件/函数的当前处理细节。`GameState`、`GameSession`、`PlayerViewState` 等核心架构名词可以在需求或专题文档中保留，用于描述稳定边界和设计取舍。设计文档可以保留架构图、设计思想、数据流、模块职责和关键代码路径，但不应复制源码或维护大段命令/API 示例。用于解释外部契约的短示例只应出现在设计文档或数据规范中。

4. 描述当前实现时，应与代码路径和实际行为保持一致；描述未来计划时，应明确标注为计划、建议或未实现能力。

5. 对于架构图、流程图等，使用 Mermaid 而不是 ASCII 图，以确保图的结构化、清晰性。

6. 对于来自外部的文档，包括第三方 SDK 使用方式、接口文档等，都需要标注原始链接，便于后续更新维护。

7. 同一事实只保留一个权威描述；其他文档应链接到权威文档，不重复维护。跨模块限制和部署差异优先集中记录在 `docs/current-limitations.md`。

8. 文档更新应控制半径。若变更影响产品行为、规则边界、用户流程、API 契约、数据 schema、部署方式、鉴权模型、联机同步边界或已知限制，应更新对应权威文档。若只是局部重构、命名调整、组件内部实现优化、测试补充或不改变外部行为的代码整理，通常不需要同步修改长期文档。

9. 专题文档只维护本专题的权威事实。跨专题限制、部署差异和已知实现偏差应集中到 `docs/current-limitations.md`；计划文档、方案建议和历史记录不得重复维护当前实现事实。

10. 临时进度、阶段实施记录、一次性 checklist、评审意见和“本次修订说明”不应混入需求文档或设计文档，也不应在提交前准备阶段为了说明本次改动而追补到 `PROJECT_PROGRESS_TODO_*.md`、根目录临时计划、临时 TODO 或临时 checklist。仍有价值的历史信息应提炼为背景原因、设计约束、当前限制或后续计划，写入对应权威文档。

11. 新增长期文档时，应先判断是否已有权威文档可承载该事实。确需新增时，需要在 `docs/README.md` 登记分类、用途和维护边界。

12. 提交前文档审查应拦截以下内容，除非用户明确要求或当前任务的正式流程指定必须更新：
   - 新增或修改只记录“本次/本阶段做了什么”的过程日志；
   - 新增未被 `docs/README.md` 登记的临时 TODO、临时 PLAN、一次性 checklist；
   - 提交已经废弃、被替代或只剩参考价值的旧方案，但没有写明状态、背景、替代方案或保留原因；
   - 把验证命令结果、评审过程或执行流水账作为长期事实提交。

13. 如果确实需要保留旧方案或计划，应把文档类型标为“历史/计划文档”，在开头说明当前状态、保留原因、是否仍有效、已被哪个权威文档或实现替代，并在必要时登记到 `docs/README.md`。否则应从提交中移除或转写成当前权威文档的背景/限制说明。
