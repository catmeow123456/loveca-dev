# 文档的编写规范
1. 有两类文档：
* 需求文档用来描述需求，不涉及具体实现；
* 设计文档：根据需求来编写数据库、代码架构方面的设计，对于已经完成代码编写的功能还需要在设计文档中带上代码路径（只需要代码路径，无需代码行号）

2. 文档的保存位置：docs/下存放需求文档、设计文档。文档的命名，同一个功能的文档拆分无需严格区分为需求文档和设计文档，只要合理即可

3. 文档中不应当带有具体的实现代码（除了用于解释上下游的api格式而编写的示例调用代码），这是因为已经带上了代码路径，读者根据代码路径自行查看代码即可

4. 对于架构图、流程图等，使用mermaid而不是ASCII 图，以确保图的结构化、清晰性

5. 对于来自外部的文档，包括第三方的sdk使用方式、接口文档等，都需要标注原始链接，便于后续更新维护
